# D234: Filesystem-Derived Doc Registry — Single Root from mkdocs.yml + Frontmatter

## Context

BareCount documentation lived in a **dual-root** system:

1. **legacy v2 archive files** — the actual `.md` content under `docs/`
2. **docs.registry.json** — a hand-maintained JSON manifest mapping sections, modules, and paths
3. **DevHub `documents` table** — 1,440 rows tracking type, authority, domain, freshness, cross-references

The drift chain was three levels deep:

```
mkdocs.yml  →  docs.registry.json  →  DevHub DB
     ↕                ↕                     ↕
  filesystem     drifts from ①        drifts from ②
```

Any of the three could go stale independently. `docs.registry.json` was a middleman — it duplicated what `mkdocs.yml` already expressed (nav structure) and what the filesystem already contained (file existence). DevHub's DB duplicated what frontmatter already carried (uid, domain, status).

## Decision

### Single Root: `docs/` filesystem + `mkdocs.yml` + frontmatter

**One source of truth chain:**

```
mkdocs.yml + filesystem + frontmatter  (SSOT)
              ↓ scan
         DevHub DB  (derived index — never edited directly for classification)
```

### What Dies

| Artifact | Action |
|----------|--------|
| `docs.registry.json` | **Deleted.** All information derivable from mkdocs.yml + filesystem. |
| `devhub_doc_register` | **Removed.** No manual registration — scanner handles it. |
| `devhub_doc_batch` | **Removed.** Same reason. |
| Manual type/domain classification in DevHub | **Removed.** Derived from path. |

### What Survives (DevHub as enrichment layer)

| Tool | Purpose | Why it stays |
|------|---------|-------------|
| `devhub_doc_list` | Query the derived index | Faceted search (type, domain, authority, freshness) |
| `devhub_doc_get` | Single doc with cross-refs | Cross-references are DevHub-only |
| `devhub_doc_update` | Enrichment fields only | `last_validated_at`, `validated_by`, notes |
| `devhub_doc_link_ref` | Cross-references | doc↔decision, doc↔task links |
| `devhub_doc_validate` | Staleness detection | Linked decision newer than last validation |

### Scan Scope

**`docs/` only.** No scanning outside the MkDocs docs directory. This means:

- All `.md` files under `legacy-v2-docs-root\docs\`
- Navigation structure from `mkdocs.yml` `nav:` section
- No `.pages` files (none currently exist; not needed)

### Frontmatter Standard

Every `.md` file under `docs/` SHOULD have frontmatter. The scanner works without it (derives from path), but frontmatter provides precision.

#### ADRs (decisions)

```yaml
---
uid: DEC-xxxxxx
title: "Short descriptive title"
description: "One-liner for scanning"
status: proposed | decided | implemented | superseded | reversed
date: 2026-04-01
project: bc-core | bc-admin | bc-portal | bc-docs | bc-ai | global
domain: contracts | metrics | sources | database | platform | foundation | ...
authority: locked | authoritative | evolving | reference | retired
---
```

#### Dossier sections

```yaml
---
title: "Source Provider — Specification"
authority: evolving
---
```

#### All other docs

```yaml
---
title: "Document title"
authority: evolving
---
```

**`authority`** is the key frontmatter field. It replaces the DevHub-only `authority_code` column:

| Value | Meaning | Examples |
|-------|---------|---------|
| `locked` | Immutable — formal review required to modify | Foundation spec, patent |
| `authoritative` | Canonical and current — trusted for implementation | Decided ADRs, published dossiers |
| `evolving` | Work in progress — correct but incomplete (default) | New dossiers, draft guides |
| `reference` | Supporting material — not authoritative for implementation | Templates, glossary, help articles |
| `retired` | Superseded or no longer applicable | Reversed ADRs |

### Scanner Algorithm

```
scan_docs(docs_root):
  1. Walk filesystem: find all .md files under docs_root
  2. For each file:
     a. Parse frontmatter (if present)
     b. Derive type from relative path
     c. Derive domain from relative path (ADRs: use frontmatter domain)
     d. Read authority from frontmatter (default: evolving)
     e. Read title from frontmatter, or first H1, or filename
  3. UPSERT into DevHub documents table:
     - Match on file_path (relative to docs_root)
     - Set: title, type, domain, authority
     - Preserve: last_validated_at, validated_by, cross-references, notes
  4. For any DB row whose file_path no longer exists on disk:
     - Set status → retired
  5. Return: { added, updated, retired, unchanged, total }
```

### When the Scanner Runs

- **On DevHub boot** — ensures index is always fresh on start
- **On demand** via `devhub_doc_scan` MCP tool — manual refresh
- **Never** as a background watcher — unnecessary complexity

### Navigation

- `mkdocs.yml` `nav:` section is the SSOT for navigation order
- Decisions: `mkdocs.yml` points to `decisions/index.md` which renders the full ADR table
- Reference: `mkdocs.yml` points to `reference/sources/index.md` which renders the source listing
- No `.pages` files needed

## Consequences

### Positive

- **Zero drift** — filesystem IS the truth, DB is always derived
- **One place to edit** — change the `.md` file, everything follows
- **Frontmatter is portable** — survives repo moves, DevHub rebuilds, tool changes
- **Simpler MCP surface** — fewer tools, clearer semantics
- **MkDocs-native** — respects the tool already used for publishing

### Negative

- **Frontmatter discipline** — every new file should have `authority` field
- **Derivation rules maintenance** — path→type and path→domain rules need updates when structure changes
- **Slight boot delay** — scanning ~900+ files on DevHub start (should be <2s for a filesystem walk + frontmatter parse)

### Migration Steps

1. Add `authority` frontmatter to all existing `.md` files (bulk script)
2. Build scanner in DevHub (`services/doc-scanner.js`)
3. Run scanner, verify output matches current DB state
4. Delete `docs.registry.json`
5. Remove `devhub_doc_register` and `devhub_doc_batch` MCP tools
6. Add `devhub_doc_scan` MCP tool
7. Wire scanner into DevHub boot sequence
8. Update CLAUDE.md to remove docs.registry.json reference
