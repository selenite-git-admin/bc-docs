# Reference Landscape vs Source Catalog — separation of concerns

## Context

Loading all 9,810 reference objects into the Source Catalog creates false expectations (objects without readers/contracts look hollow), dilutes the governed inventory, and pollutes the catalog with unactionable entries. The Reference Landscape already serves the breadth/marketing story better as a dedicated view. Keeping them separate preserves architectural clarity and creates a meaningful promotion step aligned with the three-phase onboarding model (Phase 1 auto-match, Phase 2 gap closure, Phase 3 consulting).

## Decision

**Reference Landscape** and **Source Catalog** are distinct concepts with different purposes:

- **Reference Landscape** — read-only browseable index of all known objects in a source system (e.g., 9,810 SAP T1+T2 tables). Marketing/discovery/credibility. No governance overhead. Lives at `/registry/landscape/`.

- **Source Catalog** — governed inventory of objects with a credible path to admission. Entry requires a reader flavor (exists or planned) and the ability to create a source contract. Lives at `/registry/sources/`.

**Promotion**: an object enters the Source Catalog from the Reference Landscape when a reader flavor is built and a source contract can be meaningfully created. This is a governed act.

Reference Landscape = everything we know (breadth).
Source Catalog = everything we can act on (depth).
The gap = product roadmap.
