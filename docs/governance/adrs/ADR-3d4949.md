# Backend lives in separate bc-core repo

## Context

User preference for clean separation. Independent deployment cycles, separate CI/CD pipelines, and clearer ownership boundaries. The tradeoff is managing shared types across repos, which we'll solve with a contract-first approach — frontend types in bc-portal are the source of truth, backend must conform to them.

## Decision

The NestJS backend will live in a separate bc-core repository, not inside bc-portal/apps/api. The bc-portal repo remains frontend-only. Shared TypeScript types will be managed via a shared types package or contract-first approach.
