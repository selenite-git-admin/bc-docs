# Contract-Typed Payload Tables — known schema gets real columns, JSONB payload columns dropped

## Context

Superseded by DEC for D210 v3. v2 dropped JSONB entirely with no safety net. v3 drops JSONB from RDS but archives raw payloads to S3 per run — cheap WORM storage as the safety/recovery/audit layer.

## Problem

Tenant transactional data (SO/CO/MS/AO payloads) stored as JSONB despite schema being known at contract activation time. Three concerns: un-queryable business data, storage bloat from repeated field names, conflicting read/write workload on same table.

## Key Insight

**We know the schema exactly before the JSON is formed.** Contracts define fields and types before data flows. Known-schema data in JSONB is the wrong format.

## Decision

**Contract activation = DDL generation.** When a contract is activated for a tenant, generate a typed SQL table. **Drop the JSONB payload columns from envelope tables.**

### Typed table generation

Example: OC "BKPF" v1 activated for tenant t_demo:

```sql
CREATE TABLE t_demo.or_bkpf_v1 (
  source_object_id  uuid PRIMARY KEY
    REFERENCES t_demo.source_object(source_object_id),
  posting_date      date,
  company_code      text,
  document_type     text,
  amount            numeric,
  currency_code     text
);
```

Table naming: `{boundary_prefix}_{entity_or_contract}_v{version}`
- Observation: `or_{entity}_v{N}`
- Canonical: `co_{contract}_v{N}`
- Metric: `ms_{metric}_v{N}`

### JSONB columns are dropped, not kept

The earlier draft said "JSONB stays as frozen audit artifact." That's wrong — it's dead weight. The typed table IS the data. The envelope's structural FKs provide lineage. Keeping JSONB "just in case" is the same thinking that put data in JSONB in the first place.

Envelope tables after D210:
- **Keep:** identity (PK), structural FKs (predecessor, contract, run), status/outcome, timestamps
- **Drop:** observed_payload_json, canonical_payload_json, metric_value_json, action_payload_json

The payload lives in the typed table. The envelope is purely execution-model metadata.

### Applies at every boundary

| Boundary | Envelope (metadata only) | Typed table (payload) |
|----------|--------------------------|----------------------|
| Observation | observed_record | or_{entity}_v{N} |
| Canonical | canonical_object | co_{contract}_v{N} |
| Metric | metric_snapshot | ms_{metric}_v{N} |

### Runtime write

Envelope INSERT (metadata) + typed table INSERT (payload) in same transaction.

### Business queries

Hit the typed table. Full SQL: WHERE, ORDER BY, GROUP BY, JOIN, proper indexes, proper planner stats.

### Contract versioning

New contract version → new table version. co_journal_entry_v1 stays (immutable). co_journal_entry_v2 generated for new version. Implements DOMNS versioning recommendation from foundation-gaps.md.
