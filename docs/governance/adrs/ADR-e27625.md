# Tenant-Scoped Admission Contracts — no shared production contracts

## Context

Scenario planning (SES-6bb15f): Scenario 1 (standard-only) rejected as no business value. Scenario 2 confirmed Option B (separate contracts per tenant). Scenario 6 showed same object name with different field sets across tenants — impossible to share one contract. Scenario 7 showed drift resolution requires per-tenant contract versioning.

## Decision

Every tenant gets their own admission contracts per source object. Seed admission contracts serve as reference templates for platform team authoring — they are never bound to real tenants in production.

**Rules:**
1. Admission contracts are always tenant-specific (e.g., sfdc-account-admission-kohinoor, sfdc-booking-admission-prestige).
2. Seed contracts (e.g., sfdc-account-admission) are templates. Platform team uses them as starting point when authoring tenant contracts.
3. Admission contracts declare ONLY metric-driving fields — not all discovered fields. Field selection is a BareCount decision based on which metrics the tenant wants to activate.
4. Standard-only SFDC tenants (zero customization) are rejected — no business value. Every real onboarding involves custom objects/fields.

**Rationale:** Each tenant's source schema is unique (custom objects, custom fields, different field sets). A shared admission contract cannot accommodate this variation. Tenant-specific contracts allow independent versioning, independent governance, and clean drift resolution per tenant.
