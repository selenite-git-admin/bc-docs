# Source Contract Correction — Platform Template (Full Schema) + Observation Contract Per Flavor (Field Selection)

## Context

Foundation spec: Source Contract declares expected schema of projected fields. Contract Registry D054: Source uses Pattern A (Template Instantiation) — platform master is reference template. UinBAT Reader spec: Source Contract for a flavor declares the expected schema; Observation Schema determines WHAT is observed. The field selection (projection) is the Observation Contract's job, not the Source Contract's. The Source Contract declares what's AVAILABLE. The Observation Contract selects FROM it. One Reader has N observation contracts (one per flavor) but references existing Source Contracts.

## What D200 Got Wrong

D200 Step 5 said "Create Source Contract (business-scoped projection)" — implying a NEW source contract with only the projected fields. This is wrong. The existing 14,459 platform Source Contracts ARE correct — they declare the full available schema of each source entity as a platform template (Pattern A).

D200 also proposed a `scope_code` column (reference/projection) to distinguish "full schema" contracts from "projected subset" contracts. This is unnecessary — the Source Contract IS the full schema. Field selection happens at the Observation Contract level.

## The Correct Model

**Source Contract (platform, Pattern A template):**
- Declares the full available field schema of a source entity
- One per source_object per source_version
- The 14,459 existing contracts are these — they're correct
- Tenant instantiation (Pattern A) adds custom fields (Z-fields, custom objects)

**Observation Contract (per Reader Flavor):**
- References a Source Contract ("from these available fields...")
- Declares WHICH fields to project (the subset)
- Declares the source→business field mapping (observation_field_map)
- One per source entity per Reader flavor
- Different source systems have different observation contracts for the same Reader

**Reader Flavor:**
- Carries observation_contract_id (FK → observation_contract)
- Carries observation_schema in config_json (projected fields, standard names, field map)
- The observation_schema is derived from the observation_contract's field_map

## Example: AR Invoice Reader

```
AR Invoice Reader (reader, domain: accounts_receivable)
│
├── Flavor: sap_s4_bsid
│   ├── source_version_id → SAP S/4HANA
│   ├── observation_contract → "SAP BSID → AR Invoice"
│   │   ├── source_contract_id → Source Contract for BSID (86 fields available)
│   │   └── field_map: BUKRS→company_code, BELNR→document_number (12 selected)
│   └── observation_schema: { observedFields: [BUKRS,BELNR,...], standardFields: [company_code,...] }
│
├── Flavor: oracle_ar_invoices
│   ├── source_version_id → Oracle EBS
│   ├── observation_contract → "Oracle AR_INVOICES → AR Invoice"
│   │   ├── source_contract_id → Source Contract for AR_INVOICES_ALL (45 fields available)
│   │   └── field_map: ORG_ID→company_code, TRX_NUMBER→document_number (12 selected)
│   └── observation_schema: { observedFields: [ORG_ID,TRX_NUMBER,...], standardFields: [company_code,...] }
│
└── All flavors emit SAME standard fields: [company_code, document_number, ...]
```

## What Changes in D200

1. **REMOVE** D200 Step 5 (Create Source Contract) — the platform Source Contract already exists.
2. **REMOVE** D200 Phase 4 (TSK-4a9149: scope_code reclassification) — not needed.
3. **FIX** D200 Step 3: When selecting a source entity, FIND the existing platform Source Contract for it.
4. **FIX** D200 Step 6: The Observation Contract references the existing Source Contract and declares the field subset + mapping.
5. **FIX** the wizard flow from 8 steps to 7:

```
Step 1: Select Business Object (what we want)
Step 2: Ensure Canonical Contract (universal form)
Step 3: Select Source Entity (where to get it) — FIND existing Source Contract
Step 4: AI Field Mapping (source fields ↔ business fields)
Step 5: Create Observation Contract (field selection + mapping from Source Contract)
Step 6: Create Reader + Flavor (runtime execution)
Step 7: Create Canonical Mapping (tenant wiring)
```

## Existing 14,459 Source Contracts

**Correct as-is.** They are platform Source Contract templates (Pattern A, tenant_id = NULL). They declare the full available schema of each source entity. No reclassification, no deletion, no scope_code needed.
