# MCF Reference-Dimension Grouping — group-by / top-N over a stamped reference dimension (CB-008 Component B)

## Context

The customer-axis metric family (top-N concentration, per-customer aging) is blocked only by the absence of reference-dimension grouping in the metric grammar+evaluator — the substrate is ready (Component A stamped the reference; the computed_dimension_ref table already has a `group_by` role and a reserved grouping_inputs rowset field). Reusing the fiscal-period grouping mechanism (rather than inventing an AST grouping node) keeps one grammar for one operation, is Foundation-clean (aggregation over a stamped immutable attribute per the-governed-selection.md), and is the minimal additive change (one enum value + evaluator partition extension + two validation extensions). Distribution output reuses the existing per-grain rowset → fact.ms_* 1:N storage, so no snapshot-shape change.

## Decision

CB-008 Component B (M1-pattern MCF gate, beyond M0–M20). Implements reference-dimension grouping: a metric may partition its aggregation by a STAMPED reference canonical field and produce a distribution (per-dimension-value) or a top-N ranked aggregate. Output is ONE distribution-valued Metric Snapshot — no new object type. Authority for the Foundation permission: the-governed-selection.md (a metric is "a governed selection, then an aggregation over the resolved set"); the-object-model.md Metric Snapshot; the-contract-grammar.md (reference vocabulary). Scope doc: bc-docs-v3/docs/implementation/metric-context-framework-reference-dimension-cross-entity-scope-2026-06-30.md §6.2.

CORE DECISION: reference-dimension grouping is the SAME mechanism as the established fiscal-period grouping (the existing `SUM(amount) GROUP BY fiscal_period` computed-dimension pattern), applied to a new dimension class whose source is a stamped reference attribute. It is NOT a new paradigm.

GRAMMAR (repair location B):
- mcf.metric_computed_dimension_ref.dimension_class_code gains the value `reference_dimension` (additive enum/CHECK extension; table currently empty; applied via the governed path with operator approval per the Database Change Protocol — pre-approved SES-4dd398).
- role_in_formula_code = `group_by` (already a valid enum value — the declarative grouping surface; no change).
- source_business_concept_id = the reference-kind Business Concept whose stamped canonical field is the grouping key (e.g. 01c23173 / bill_to_customer, the corrected invoice→customer reference from Component A / DEC-7f2e73).
- resolver_config_ref_json carries an optional rank_spec { order_by: asc|desc, limit: N } for top-N. Absent rank_spec ⇒ full per-value distribution.
- The Section A rowset already reserves a `grouping_inputs` field (formula-execution.engine.ts) — the stamped reference value rides there; no AST node-kind addition required for the MVP (the computed-dimension metadata surface, like fiscal-period grouping, is the declaration). A `grouping` AST node is explicitly NOT introduced (avoid a second, redundant grammar for the same operation).

EVALUATOR (repair location D):
- Extend the governed-metric-runtime partition step (governed-metric-runtime.ts partitionByGrain / metric-evaluation-engine.service.ts evaluateWithGrain + groupInputsByGrain) to partition by (grain ∪ grouping dimension): read the stamped reference value from each resolved CO's grouping_inputs, aggregate per partition, then apply the optional rank/limit per grain across partition values.
- Aggregate ops unchanged (sum/avg/count/count_distinct/min/max + median/percentile). Top-N is a post-aggregation rank, not a new aggregate.

RESULT SHAPE: a distribution fits the EXISTING per-grain rowset model (ExecutionResult.rowset; GrainResult). Storage: progression.metric_evaluation → fact.ms_* is already 1:N per grain slice (D369) — one fact row per (grain × grouping-dimension value), or top-N rows. The grouping-dimension value is carried as an additional partition key alongside grain. No progression/snapshot schema change.

VALIDATION:
- PE-MC-7 (computed-dimension coherence, metric-publication-eligibility-evaluator.service.ts) extended: for a `reference_dimension` group_by row, assert source_business_concept_id resolves to an active stamped reference canonical field, and the dimension is NOT also a grain key (ambiguity refused).
- C-FX-6 (fixture-structural-check.service.ts) extended: the grouping dimension must be present in the fixture rowset grouping_inputs (it already validates role_in_formula_code in {grain, grouping/group_by}).

FOUNDATION GATE: repair B (grammar) + D (evaluator); no A/C/E/F compensation. Invariant I — grouping key is canonical-stamped (guaranteed by Component A / DEC-7f2e73); the predicate reads a stamped immutable attribute and performs NO metric-boundary resolution. Invariant II — inputs are CO versions (primary metric ordering). Invariant IV — the grouping rule is a versioned predicate (the computed_dimension_ref row + resolver_config, versioned with the MCV). Invariant VI — the partitioned resolved set is recorded in Lineage.

SCOPE BOUNDARY (vs Component C): Component B groups by the stamped reference VALUE on the invoice CO (top-N concentration, per-customer aging, distribution metrics) and does NOT read the referenced Customer CO. Reading the Customer CO (e.g. credit_limit) joined by the stamped reference is Component C (cross-entity binding) — a separate ADR. Component A (reference stamping, DEC-7f2e73/D468) is the prerequisite and is COMPLETE. Component D (secondary-metric DAG) shipped as DEC-0f3e57/D467. Unblocks ~8 customer-axis distribution/top-N metrics (CB-008).
