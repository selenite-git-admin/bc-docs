# Pre-production Legacy Active-Runtime Retirement — authorize archival ahead of MCF re-authoring; preserve carve-outs

## Context

DEC-6c57e2 / D417 was filed under the implicit assumption that operational runtime continuity matters — that something depends on the legacy active CC/MC/OC/Reader artifacts resolving as-is. That implicit assumption produced a §4 ("Operational continuity") that, in practice, is preserving a hypothetical. The current operational observation is that no external user consumes the legacy runtime, and we explicitly accept that internal dev metric-chain work can stop until MCF replaces it. The implicit assumption is therefore false in practice.

DEC-02f5a9 §4 already named the relevant precondition — "pre-production: demo tenant, synthetic / SDG data, no tenured customer, no customer-relied-upon historical evidence" — and noted that *"After a production tenant relies on the old chain, discard would require a separate historical-preservation ADR. ... The greenfield window is now (§4 Precondition). Greenfield is cheap while the platform is pre-production and expensive after the first real tenant onboards; the cutover should not be deferred."* This ADR is the operational realization of that precondition for the active-runtime layer specifically.

Without this ADR, the §6 end condition of DEC-6c57e2 is reachable only through the months-long MCF re-authoring path. That path is correct architecturally — it produces Foundation-correct replacement bindings — but it is also the entire MCF arc. Forcing legacy retirement to block on MCF completion inverts the dependency: MCF substrate / architecture work is what should be unblocked, and legacy retirement is what should be cheap. This ADR inverts the inversion: legacy retirement is unlocked under the precondition; MCF arc proceeds without legacy-coexistence constraints.

The sibling-ADR shape (rather than amending DEC-6c57e2 with a §7 acceleration clause) maintains abstraction-level separation. DEC-6c57e2 is about a *semantic prohibition surface* — what BCF/MCF authoring may and may not consume. This ADR is about a *runtime-state authorization* — what state the active runtime artifacts sit in. Reading rules vs. lifecycle-state rules. Different scopes; different invariants in play (DEC-6c57e2 invokes Foundation Invariants I and IV; this ADR invokes Foundation Invariant III). Different audiences (DEC-6c57e2 binds authoring panels and operators; this ADR binds a future archival gate). Mixing the two in one document would dilute both.

The ADR is permissive, not prescriptive — it authorizes a category of follow-on gate; it does not execute any archival. Each follow-on (archival, verification, UI removal, controller disablement, test cleanup, physical disposition) is its own future operator-opened gate with its own scope, deliverables, and verification. The verification gate (§Guardrails #2) is the structural prerequisite for the physical-disposition DBCP gate (#6); the dependency is enforced by gate ordering, not by this ADR's authorization.

The stopping condition is the safety mechanism. The pre-production precondition is the load-bearing input; if that precondition becomes false (a real customer onboards), the entire authorization expires immediately. Operators monitoring tenancy state have a single explicit trigger to watch. This prevents the pre-production retirement from accidentally leaking into a production-relevant action — the precondition itself is the gate, not a calendar date or a procedural check.

The reversibility constraint (no DROP / DELETE / FK break) preserves the optionality of un-archival. Archival is lifecycle-state work; lifecycle-state work is reversible under Foundation Invariant III. Physical deletion is not reversible; it is therefore deferred to the post-verification DBCP gate, where the cost / safety trade-off can be re-evaluated with archival evidence in hand.

## Context

The BCF Business Concept Registry backend MVP is complete (close-out `589a07d` + amendment `b0f3fcf`). The BCF operator console MVP is complete (UI-S1 → UI-S5; close-out `faf1914`; first UI-driven publication proven by `cycle time` `7042ca4f-…`).

ADR DEC-6c57e2 / D417 (filed 30 minutes ago as commit `394720c`) declares BF/BO/CF/Canonical Mapping **semantically quarantined**, with a §4 operational-continuity carve-out that lets existing runtime paths "continue resolving as-is" and a §6 structural end condition: *"the quarantine compatibility period ends when no active Canonical Contract, active Metric Contract, active Observation Contract, active reader, or active metric runtime path carries a reference to any quarantined surface."*

DEC-6c57e2 §4 was written under the implicit assumption that runtime continuity matters because *something* depends on the active runtime paths resolving. The current operational observation contradicts that implicit assumption:

- **There are no external users on the platform.** Per ADR DEC-02f5a9 §4 the platform remains *"pre-production: demo tenant, synthetic / SDG data, no tenured customer, no customer-relied-upon historical evidence."* That precondition has not changed; this ADR makes it operative.
- **We explicitly accept that internal dev metric-chain work stops** until MCF replaces it. Internal exercising of the legacy metric chain for testing or development purposes is no longer required.

Under those two facts, the §6 end condition becomes reachable on two distinct paths:

1. **Re-author every active CC/MC/OC/Reader binding through MCF against BCF objects.** This is DEC-6c57e2's implicit path. Months of work; blocks legacy retirement until MCF is operational.
2. **Archive the active CC/MC/OC/Reader rows directly**, because nothing consumes the values they produce. End condition is met immediately on archival completion. Physical-disposition DBCP gate opens.

This ADR authorizes path 2 for the pre-production period. It is the operational interpretation of DEC-02f5a9 §4 — the precondition exists; we may act on it.

This ADR is a *sibling* of DEC-6c57e2, not an amendment. DEC-6c57e2 is about *what BCF/MCF authoring may and may not read from quarantined surfaces*. This ADR is about *what state the legacy active runtime artifacts sit in*. Different abstraction levels; different invariants in play. DEC-6c57e2 is unchanged.

## Decision

### §1 Trigger / precondition

This authorization is operative when all three hold:

1. DEC-02f5a9 §4's pre-production precondition is unchanged: demo tenant, synthetic / SDG data, no tenured customer, no customer-relied-upon historical evidence.
2. The platform has no external users.
3. We accept that internal dev metric-chain work stops until MCF replaces it.

If any of (1)-(3) becomes false, this authorization expires immediately. See §5 (stopping condition).

### §2 Authorized future archival action

A separate follow-on operator-opened gate may archive active Canonical Contract, Metric Contract, Observation Contract, and Reader rows that carry references to DEC-6c57e2 §2 quarantined surfaces.

"Archive", in this ADR's narrow sense, means:

- Set the row's `lifecycle_state` to the appropriate inactive state (`archived` or `superseded`, per Foundation contract grammar §Lifecycle, per row's existing supersession discipline).
- Set `archived_at` to the archival timestamp where the schema carries it.
- Keep the row physically present in the table.
- Keep FK integrity intact — referenced rows are not orphaned.
- Runtime treats the row as inactive: the canonical / metric / admission boundaries do not consume archived rows at evaluation time.

The archival gate's exact scope (which active rows, which lifecycle column, which archival timestamp source) is the follow-on gate's design responsibility. This ADR authorizes the *category* of action; the follow-on gate specifies the mechanics, runs the dry-run, and executes the archival under operator review.

### §3 Actions explicitly NOT authorized

This ADR does **not** authorize:

- `DROP TABLE` on any legacy table (quarantined or otherwise).
- `DELETE` of any row from any legacy table.
- Schema migrations on legacy tables (column adds, drops, type changes).
- FK breaking (any change that severs the integrity of `target_registry_id`, `business_field_id`, `business_object_id`, `object_id`, etc.).
- Semantic migration: converting a BF row into a BCF Characteristic, a BO row into a BCF Entity, a CF row into a BCF BusinessConcept, or any other row-shape conversion across the quarantine boundary.
- Bridge table creation: any cross-reference index that resolves a quarantined identifier to a BCF / MCF identifier (the §4 shim DEC-02f5a9 already forbade, restated at the runtime layer).
- Legacy-to-BCF/MCF mapping in any structural form (lookup tables, view aliasing, ORM-side mappings, ID coercion at read time).
- Reads of quarantined surfaces for BCF/MCF authoring — DEC-6c57e2 §3 prohibitions remain in full force regardless of legacy active runtime state.
- Touching DEC-02f5a9 §5 / DEC-6c57e2 §5 carve-outs.

The archival gate authorized by §2 changes lifecycle state only. Schema, rows, FKs, and the BCF/MCF read prohibition surface are unchanged by this authorization.

### §4 Preservation — DEC-02f5a9 §5 and DEC-6c57e2 §5 carve-outs unchanged

The following surfaces remain preserved exactly as DEC-02f5a9 §5 and DEC-6c57e2 §5 define them. They are **not** in scope for the §2 archival action:

- **`bc-seed` catalog with provenance** — BCF PE1 (b) evidence source. Preserved.
- **Source Contract family** — source-reality contracts; reference source fields, not vocabulary. Preserved.
- **Admission Contract family** — same. Preserved.
- **Observation Contract family** as source-reality / observation evidence — preserved. Note: the OC's `observation_field_map.business_field_id` *column* is in DEC-6c57e2 §2 quarantine scope, and *active OC rows that carry such references* are in §2 archival scope of this ADR. The OC family as a contract artifact survives; the active runtime artifacts that bind through the quarantined column are archived.
- **Source-system observations and source-reality evidence** — `source_system`, `source_table`, `source_field`, raw observed values. Preserved.
- **KPI catalog / metric definition knowledge as operator intent** — `contract.metric_definition` + `metric_definition_knowledge` rows. The intent fields (metric name, prose definition, owner, thresholds, formula prose, references) are preserved. Per DEC-6c57e2 §3, BF/CF *binding* reads on those rows remain prohibited. The intent survives; the bindings do not. **No archival of KPI catalog intent rows.**
- **Contract body knowledge** — resolution rules, type coercion, unit conversion, temporal interpretation, reduction-over-grain. Per DEC-02f5a9 §2: may be re-authored into future governed OC / CC / MCF bodies. Not inherited as authority. The knowledge survives; the discarded CC / Canonical Mapping rows that historically carried it are subject to §2 archival.

### §5 Stopping condition

The authorization granted by §2 expires the moment any of the §1 preconditions becomes false. Specifically:

- A real external customer onboards or depends on the legacy runtime.
- A production user is established.
- A decision is made to resume internal dev metric-chain work against the legacy chain.

If any of these happens **before archival is executed**: the follow-on archival gate pauses and re-decides. The authorization is not yet acted upon; the precondition change is non-destructive.

If any of these happens **after archival has executed**: any restoration of archived runtime artifacts must be an explicit, operator-opened gate justified by the customer / dev need. Un-archival is reversible per §6, but it must be an explicit decision — not a tacit undo.

### §6 Reversibility

The §2 archival action is intended to be reversible by un-archival where the row's lifecycle / archive fields permit. Specifically:

- A row archived to `lifecycle_state = 'archived'` may be returned to its prior `active` state by clearing `archived_at` and restoring `lifecycle_state` (Foundation Invariant III preserves the row's state; no information is lost in archival).
- A row archived to `lifecycle_state = 'superseded'` follows the existing supersession-reversal discipline (which, per Foundation Invariant III, is not in-place mutation but a forward supersession to a successor that restores the prior identity).

Physical deletion (DROP / DELETE) is **not** reversible without backups. This ADR deferring physical disposition to a separate future DBCP — gated on the archival having proven safe over time — is a deliberate choice that preserves reversibility through the pre-production window.

## Consequences

- **DEC-6c57e2 §6 end condition is reachable immediately on archival completion.** No active CC/MC/OC/Reader row carries a quarantined reference once the §2 action runs. The structural condition for opening the physical-disposition DBCP gate is then met.
- **MCF substrate / architecture work is unblocked from legacy-coexistence constraints.** MCF substrate design does not have to specify how the new schema coexists with active legacy CC/MC/OC tables; the legacy active runtime is offline.
- **Internal dev metric-chain work stops until MCF replaces it.** Synthetic data flows that exercise the legacy metric chain are no longer available. Dev workflows that depended on the legacy chain must wait for MCF or adapt to a non-runtime testing pattern.
- **bc-admin Business Chain nav group may be removed in a separate follow-on UI gate.** With the legacy active runtime archived, the Business Chain UI surfaces a useless catalog. The removal is a separate gate's responsibility; this ADR does not modify any UI.
- **Legacy controllers / services that only served quarantined-table reads may be disabled in a separate follow-on backend gate.** The disablement is a separate gate's responsibility; this ADR does not modify any code.
- **The legacy test-suite that exercises the archived runtime breaks.** Test cleanup is a separate gate's responsibility; this ADR does not modify any tests.
- **The §1 precondition window has a clear semantic boundary**: the moment a real customer arrives, this authorization expires. Operators monitoring tenancy state have a single explicit trigger to watch.
- **DEC-6c57e2 is unaffected.** The quarantine rule (no reads / no inspiration / no lineage / no bridge / no migration for BCF/MCF authoring) is unchanged regardless of whether archival happens or not.

## Guardrails / follow-ons (named, not implemented in this ADR)

Each is its own future operator-opened gate. None is authorized to execute by this ADR alone.

1. **Archival execution gate** — `bc-core`. Inventory the active CC/MC/OC/Reader rows that reference DEC-6c57e2 §2 quarantined surfaces; build a dry-run report; execute the archival under operator review; produce per-row evidence. The gate's specific deliverables, scope precision, and verification protocol are defined when the gate opens.
2. **DB verification gate** — `bc-postgres` read-only. Prove post-archival: (a) DEC-02f5a9 §5 + DEC-6c57e2 §5 carve-outs are untouched (row counts, content hashes); (b) no active CC/MC/OC/Reader row carries a quarantined reference; (c) FK integrity is intact (no orphaned references); (d) DEC-6c57e2 §6 end condition is met. Issues a verification report consumable by the physical-disposition DBCP gate.
3. **bc-admin Business Chain nav removal gate** — `bc-admin`. Removes the Business Chain nav group + its routes + the BF/BO/CF UI pages. Companion to the legacy-controller-disablement gate.
4. **Legacy controller disablement gate** — `bc-core`. Disables (route 410 Gone or equivalent) the controllers that only served quarantined-table reads or quarantined-runtime-table writes. The exact controller list is the gate's scope work.
5. **Legacy test-suite cleanup gate** — `bc-core`, `bc-qa`. Removes or quarantines test cases that exercise the now-archived runtime artifacts. The cleanup is per-test-file; test fixtures referencing quarantined rows may be replaced with BCF/MCF fixtures or simply removed.
6. **Physical-disposition DBCP gate** — `bc-core` + DB review. The DROP TABLE / column drop / FK drop work. Opens **only** after the verification gate (#2) confirms safety. Per DEC-6c57e2 §6 the disposition gate has its own DDL review and DB approval; this ADR does not authorize the DBCP.

None of these is mandatory in any specific order beyond the dependencies named (verification before physical disposition). Each may be opened, deferred, or cancelled by the operator based on the unfolding state.

## Non-goals

This ADR explicitly does not:

- Execute any archival.
- Drop any table.
- Delete any row.
- Migrate any schema.
- Break any FK.
- Convert any quarantined row into a BCF or MCF artifact.
- Build any bridge / mapping / lookup / shim.
- Implement BCF Enrichment work (vocabulary authoring is the full Enrichment Plan's job).
- Implement MCF work.
- Modify any UI.
- Modify production-customer behavior in any form (there are no production customers; the authorization itself is gated on that fact remaining true).
- Modify the DEC-02f5a9 §5 / DEC-6c57e2 §5 carve-outs.
- Re-decide DEC-6c57e2 / D417 or DEC-65dc86 / D416 or DEC-02f5a9.
