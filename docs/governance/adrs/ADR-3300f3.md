# MCF authoring panel admits direct-API third-party LLMs; customer data confined to the runtime layer

## Context

The trust question for an external LLM is not vendor nationality ("Chinese bad / US good" is too crude — the platform already sends panel payloads to US sub-processors Anthropic/OpenAI). The operative questions are: what trust boundary is crossed, with what payload, and can a bad/leaky answer change governed state. For the authoring panel all three are contained: the payload is public-domain vocabulary + synthetic fixtures (no customer data), and the LLM is advisory behind an operator-confirmed gate. Meanwhile the direct APIs unlock the models that actually work for these seats (GLM-5.2 checker, DeepSeek-V4 judge — the Bedrock-hosted GLM-5/DeepSeek-V3.2 either failed the tool-calling Checker or are older) and cut panel cost materially (measured combined potential ~-73%, $0.203->~$0.054/run). Customer data-residency concerns are real but belong to the runtime/deployment layer, which is deterministic and deployment-flexible — so they do not constrain the authoring-panel roster. Operator decision 2026-07-05 (SES-31ff5e), after an independent (Codex) review; grounded in the roster A/B (memory project_mcf_panel_roster_cost_ab.md, TSK-dc87d9).

## Decision

The MCF metric-authoring panel (Maker / Checker / Judge, POST /api/mcf/panel-runs and /with-maker) MAY use third-party direct-API LLMs — including non-US, non-Bedrock vendors such as Z.ai (GLM) and DeepSeek — for any of its three seats. The roster is NOT restricted to US vendors or to AWS-Bedrock-hosted models.

This is admissible because of a hard architectural data boundary, which is the operative reason (not vendor nationality):

1. The authoring panel operates on PUBLIC-DOMAIN business vocabulary. It develops metric CONTRACTS (formula AST, bindings, grain, temporal gate) over generic KPI/business-concept definitions (e.g. "days payable outstanding", "accounts receivable") and validates them against SYNTHETIC SDG fixtures. It does NOT process customer operational or business data. How a public-domain metric is derived is not proprietary and not customer-confidential.

2. Customer operational/business data is confined to the RUNTIME evaluation layer (progression.metric_evaluation, fact.*, tenant DBs), which is DETERMINISTIC engine code with NO LLM in the path. Customer data-residency requirements are met at THIS layer via deployment options (per-tenant DB, VPC, region, self-host) — independent of which LLM authored the contract.

3. The panel is ADVISORY, not authoritative. State-changing acts (activation/publication M14, supersession M15) require the operator-confirmed two-phase gate with a >=40-char human rationale. A leaky or wrong LLM verdict therefore CANNOT unilaterally change governed state.

Guardrails (part of this decision):
- Direct-API LLMs are admissible ONLY for the authoring panel (public-domain vocabulary + synthetic fixtures). They are NEVER admissible in the runtime tenant-data evaluation path, which stays deterministic.
- Any adopted third-party LLM vendor MUST be disclosed as a sub-processor in the platform's security documentation (the trust/governance posture is part of the product USP).
- Prefer AWS-Bedrock-hosted versions (data stays in-AWS/in-region under the existing AWS DPA) WHEN they meet the quality bar; use the direct APIs where the working models are direct-only (today: GLM-5.2, DeepSeek-V4; Bedrock carries only the older/failing GLM-5, DeepSeek-V3.2).
- A production roster change still requires a clean verdict-fidelity calibration on FRESH (non-active) reservoir candidates before flipping the default (engineering gate, separate from this data-governance decision).

Scope: this governs the MCF M12 authoring panel only. The BCF registry-authoring panel roster remains calibration-locked under DEC-ffee4e (D483); this decision does not change it.
