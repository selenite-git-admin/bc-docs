# Business Object Model — independent canonical entity definitions as contract infrastructure

## Context

1. Canonical contracts previously mixed business object identity with governance — schema was buried in contract_json JSONB, not queryable or reusable.
2. "Business Object + Business Field" is natural language anyone can understand — canonical entity was technical jargon.
3. Independent layer enables: (a) define objects before contracts exist, (b) query field composition via JOINs not JSONB, (c) show "what does BareCount understand?" without contract knowledge, (d) cleaner canonical mappings with proper FKs.
4. Many-to-many junction needed because fields are shared (company_code appears in GL Journal, AP Invoice, AR Invoice, etc.) and carry per-object metadata (required/key/ordinal differs per object).
5. Patent check: does NOT create new runtime boundary. Business Objects are design-time definitions referenced by contracts. The execution model (Source → BO → CO) is unchanged.

## Decision

Introduce `contract.business_object` and `contract.business_object_field` as an independent vocabulary layer. Rename `contract.standard_field` → `contract.business_field`.

**Business Object** = reusable domain concept definition (e.g., "GL Journal Entry", "AP Invoice"). Independent of contracts, sources, tenants. Classified by existing D103 taxonomy: industry_code + function_code + subfunction_code.

**Business Field** = atomic field vocabulary (rename from standard_field). Shared across objects via many-to-many junction.

**Business Object Field** = composition junction — which fields belong to which object, with per-object metadata (is_required, is_business_key, ordinal_position, semantic_role).

**Hierarchy:** Industry Category → Industry → Business Function → Business Sub-function → Business Object → Business Field (first 4 levels already exist in master schema via D103).

**Independence principle:** Business objects have no outward FKs. Contracts reference them (FK inward):
- canonical_contract.object_id FK → business_object (governs this BO)
- observation contracts map source fields → business fields
- metric contracts declare which BOs they consume

**Patent alignment:** Business Objects are design-time contract infrastructure, NOT a new runtime boundary. The runtime execution model (Source → BO → CO via canonical contract) is unchanged. COs remain the authoritative semantic artifact. Business Objects are the reusable template that canonical contracts reference — separating "what does the CO look like" from "how do we govern it."

**Tables:**
- `contract.business_object` (object_id, object_name, display_name, definition_text, industry_code FK, function_code FK, subfunction_code, status_code, audit cols)
- `contract.business_object_field` (object_id FK, field_id FK, is_required, is_business_key, ordinal_position, semantic_role) PK: (object_id, field_id)
- `contract.business_field` (renamed from standard_field — no schema changes, only table/index rename)
