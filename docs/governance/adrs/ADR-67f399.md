# Metric Directory class doctrine — base/derived only, hybrid forbidden

## Context

Codex's auditor doctrine permits a hybrid class but its published projection schemas enum only base/derived, and the current bc-core substrate is 2-class. Forbidding hybrid keeps both engines aligned with zero enum churn, gives the cross-layer conformance audit a crisp base-vs-derived line (a metric either reads the world or composes metrics), and settles the DIR-02 relabels deterministically. The only cost — an occasional thin intermediate base metric when decomposing a would-be hybrid — is cheap and arguably better modelling (the intermediate becomes independently auditable and reusable). No finance-corpus metric requires hybrid. If a future regional/regulatory track (D502-504) produces genuinely non-decomposable mixed formulas, that would be the argument to revisit; until then hybrid is explicitly rejected rather than mislabelled base or derived.

## Decision

The Metric Directory class taxonomy is exactly two classes, defined computationally by input composition (not by formula operator complexity), per the external auditor's cross-layer doctrine (Codex bc-external-audit bcf-directory-mcf study §7.5) and operator ruling 2026-07-11:

- base = the metric's quantitative inputs are direct concept/entity quantities, with NO metric inputs.
- derived = the metric has at least one metric input and NO direct concept/entity quantitative inputs; numeric constants are allowed.
- hybrid (metric inputs AND direct concept/entity quantitative inputs in one metric) is FORBIDDEN.

A would-be hybrid MUST be decomposed: wrap the direct-concept aggregation in its own base metric, then compose it as a derived metric over that base plus the other metric inputs. This keeps every metric on one side of the base/derived line, makes the intermediate independently auditable and reusable, and matches the auditor's projection enum (directory-intent-projection-v1 / mcf-realization-projection-v1 class_code enum is base/derived only — no enum extension on either side).

Enforcement: the member and group class_code CHECK constraints remain the current two-value set ('base','derived') — hybrid is forbidden by omission AND by this explicit doctrine (no schema change; a third value is NOT added). The Group/Member class-coupling constraint (DIR-007, TSK-a679db) enforces that a member's class agrees with its group's class. Metric class in the MCF realization projection is derived computationally: presence of any metric_input binding ⇒ derived, else base.

DIR-02 remediation: the ~35 Metric Directory members currently labelled 'derived' that have NO metric inputs (computed-from-concepts) are mislabelled — under this doctrine they are base and are relabelled base. Only true metric-input compositions remain derived. The relabel is a governed metadata correction performed after this ADR is citable.
