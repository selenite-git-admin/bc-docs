# Canonical value normalization — raw source-signal inputs to CC-body derivations; meaning is produced at the canonical boundary, never by the Reader

## Context

Wave 2 (138→370) triage found a family of canonical coded dimensions blocked because their source signals are raw technical fields with no legitimate business-concept representation: JE status draft/cancelled (BKPF.BSTAT), entry_method (BKPF.AWTYP+BATCH multi-signal), and — once a source is defined — accrual_indicator. The BCF panel correctly rejected registering raw BSTAT as a business concept (homonym of the canonical status BC; SAP-only implementation detail; Invariant-I meaning drift). But the panel's verdict prose and several BCF concept definitions say source codes are "normalised at the reader boundary" — language that, read literally, assigns meaning-production to the Reader/observation boundary and contradicts Invariant I and D462 (derivation home = CC body; the OC may NOT declare canonical evaluation logic). The operator flagged this tension explicitly. This ADR resolves the language authoritatively and adds the missing governed mechanism.

## Decision

CONTEXT: Canonical coded dimensions (JE status, entry_method, accrual_indicator) exist as governed BCs with closed value sets, but their discriminating source signals are RAW technical fields (BKPF.BSTAT, AWTYP, BATCH) that are not — and per the BCF panel must never be — business concepts. Today there is no governed way to produce the canonical value from them: `classify` (D461/D462) accepts only business-concept inputs, and OC `code_lookup` is single-field vocabulary translation. Result: ~10+ metric-directory members are structurally blocked (JE status draft/cancelled ×4, entry_method ×6+2, accrual ×4 pending a source). The interim workaround language ("reader normalization") is REJECTED as a misnomer.

DECISION 1 — BOUNDARY ASSIGNMENT (authoritative language). Value normalization IS canonical evaluation. It lives in the CC body, produced once per Canonical Object version at the canonical evaluation boundary (Invariant I). The Reader and the Observation Contract OBSERVE and PRESERVE raw source fields as evidence — they never conditionally produce, translate-across-fields, or reinterpret meaning (D462: the OC may not declare canonical evaluation logic; DEC-02f5a9 superseded canonical_mapping for the same reason). The phrase "normalised at the reader boundary" appearing in BCF concept definitions (e.g. Journal Entry · status f5be51cc) and panel verdict prose is LOOSE LANGUAGE and is hereby given its authoritative reading: "normalised before consumption, at the canonical evaluation boundary — never inside the registry, the metric layer, or the Reader." New and amended concept definitions MUST phrase it as "normalised at the canonical evaluation boundary". A follow-up governed concept-version amendment corrects the status BC's definition prose; panel prompt guidance inherits the corrected phrasing at its next governed revision. The term for this capability is CANONICAL VALUE NORMALIZATION — "reader normalization" is banned.

DECISION 2 — GRAMMAR (additive, canonical-v2). Extend `derivations[].inputs[]` with a second, explicit input kind: alongside the existing {role, business_concept_id}, permit {role, source_table, source_field} — a RAW-SIGNAL input. Constraints, all fail-closed: (a) the referenced field MUST be directly observed by an ACTIVE observation-v2 leg of this CC's chain (declared in its so_schema/field_mappings) — the 1-hop rule extended to fields: a raw input is never itself derived, never fetched implicitly (Invariants II + IV: explicit declaration in the contract body, no hidden derivation); (b) the derivation output MUST be a governed business concept with a closed canonical value set, and every `then`/`default` value MUST be a member of that set (the BC's canonical_value_set is the validation vocabulary); (c) raw-signal inputs are legal ONLY inside derivations — they never appear in field_selection, resolved_schema, or metric contracts (metrics stay source-agnostic, filtering canonical values only). Function library unchanged: `classify` already expresses single- and multi-signal CASE logic; this ADR changes what its inputs may reference, not what it computes. OC `code_lookup` (single-field vocabulary translation into a bound concept's value set) is unchanged and remains sanctioned by Foundation's own OC definition ("field selection from the Source Contract structure to the business vocabulary preserved on the Source Object") — the line is: single-field vocabulary translation may bind at the OC; ANY cross-field or conditional logic is canonical-only.

DECISION 3 — MULTI-SOURCE PORTABILITY (repair-location split). The canonical concept + closed value set live at B (universal, authored once). The raw-signal cases live per source chain: each source system's OC observes its own raw fields, and the CC declares that source's cases explicitly (SAP: AWTYP populated→system_generated, BATCH populated→automated, default→manual; Oracle: GL_JE_HEADERS.JE_SOURCE map; NetSuite: transaction-origin map) — inputs name source_table explicitly, so multi-source CCs (DEC-0a187d/D508 legs) compose without ambiguity. The cross-system portability test is the acceptance test: onboarding a second source system adds an OC leg + its declared cases and touches NEITHER the canonical concept NOR any metric contract. Standard-SAP maps are platform-level and universal (never per-tenant), per the standing standard-mapping rule.

DECISION 4 — GATES. D431 O↔C observability gains a parallel FIELD-level check: every raw-signal input must resolve to a field observed by an active OC leg (fail-closed, mirroring the concept-identity check). D430 field→concept integrity extends clause (b): derivation outputs validate against the output BC's canonical_value_set. PE-MC-11 chain resolvability inherits both transparently. The BCF panel's anti-pattern ruling STANDS and is strengthened: raw source fields must NOT be proposed as business concepts; this grammar is the sanctioned alternative.

IMPLEMENTATION PATH (governed): meta-schema change request per D244 (precedent: DEC-0a187d/D508 join_semantics) amending canonical-v2.schema.json (additive, optional); CcV2CanonicalResolverService raw-input resolution path (repair-location D); D431/D430 gate extensions. Zero DDL (contract bodies are JSONB); zero impact on existing contracts, MCF, chain-status, CAS. First consumer: cc__journal_entry v3 entry_method (unblocks 6 members: manual/automated/system_generated counts + percentages; partially_system_generated stays deferred — no ECC signal), then status draft/cancelled (BSTAT cases; runtime numbers additionally await SDG parked/cancelled emission, TSK-facf65 — contract realization does not). accrual_indicator remains blocked on an operator-defined source (no BKPF field exists; accounting-principle axis) — this ADR provides the mechanism, not the missing source semantics.

FOUNDATION: Invariant I — meaning produced once, at the canonical boundary, by the CC (this ADR moves nothing to the Reader; it forbids exactly that). Invariant II — no hidden derivation: raw inputs are declared, 1-hop, directly observed. Invariant III/V — normal CC version succession; no rewrite, no replay. Invariant IV — explicit references: inputs name source_table.source_field; outputs name the BC. Invariant VI — evidence unchanged: the raw field values remain preserved on the Source Object; the derived value is emitted on the CO with the CC version that produced it. Repair-location: B (grammar) + D (resolver implementation); explicitly NOT A (source emits what it emits) and NOT F (no read-side fixes).
