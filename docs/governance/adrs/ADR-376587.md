# Section rename: The Platform → Implementation

## Context

"The Platform" reads as generic in the spine: Foundation, The Operating Model, The Platform, Governed Authoring, Platform Operations, Compliance. The whole documentation is about the BareCount platform; calling one section "The Platform" implies the others are not, which is wrong. A reader scanning the TOC reads the label and does not learn what is inside; the discoverability cost is real and recurring. The seven chapters are uniformly about the built artifact: Architecture and Services (services), Infrastructure (where it runs), Data Model and Schema (DB), API Surface (endpoints), Frontend Experience (UI), DevHub and Governance (tooling), Synthetic Data and Testing (test methodology). Each is the technical realization of a layer Foundation and the Operating Model spec. "Implementation" names this directly: the built artifact that realizes the spec. The spine reads as a clear progression: Foundation (laws) → Operating Model (operational spec) → Implementation (built artifact) → Governed Authoring (procedures) → Platform Operations (running) → Compliance (audit). Three alternatives were considered and rejected: "The Implementation" (article carries the texture of "The Operating Model" but the article asymmetry across the six sections — Foundation/Governed Authoring/Platform Operations/Compliance carry no article — pulls toward dropping the article here), "Engineering" (broader but reads as activity rather than artifact, and Governed Authoring already covers some engineering activity), "The Build" (too informal and reads as CI/CD process). "Architecture" was rejected because the chapter "Architecture and Services" is inside this section (Russian-doll naming). "Platform Implementation" was rejected because the "Platform" prefix adds nothing and echoes "Platform Operations" awkwardly. The faint risk of confusion with "implementation" in the everyday sense (writing code) is mitigated by context: the section's seven chapters describe the platform's implementation specifically, not implementation as a generic activity. Pairs with DEC-ce6e2b (the Operating Model rename); same reader-discoverability discipline applied to the third section.

## Decision

The third top-level section of bc-docs-v3 is renamed from "The Platform" to "Implementation" (no article). The folder name when chapters are drafted will be `docs/implementation/` rather than `docs/platform/`. The seven chapters (Architecture and Services, Infrastructure, Data Model and Schema, API Surface, Frontend Experience, DevHub and Governance, Synthetic Data and Testing) remain unchanged in scope and order. Cross-references in already-drafted chapters (Foundation overview, The Operating Model overview, Tenant Lifecycle and Subscription, outline.md, HANDOFF.md, README.md, aws-rewrite-checklist.md, bc-admin sync-docs.js SECTION_LABELS) update from "The Platform" to "Implementation" or "the Implementation" as the new canonical name. The "Platform" word remains admissible in compound terms (Platform Operations stays unchanged; "platform" lowercase as the product noun stays in body prose).
