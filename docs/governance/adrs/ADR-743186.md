# D286: Tenant Business Data PII Detection ‚Äî Source Catalog Field-Level Marking

## Context

D285 introduced the `operations.pii_field_registry` table and the nullification framework. Platform PII (user identity, activity logs, actor names) is static and known ‚Äî seeded with 26 entries covering T1‚ÄìT3 fields across `users`, `operations`, `infrastructure`, `contract`, `metric`, and `runtime` schemas.

However, tenant business data varies per tenant and per source system. A customer admitting SAP HR data has employee PII (names, addresses); one admitting GL journals may have none. The JSONB payloads in `boundary.observed_record.observed_payload_json` and `boundary.canonical_object.canonical_payload_json` are opaque ‚Äî their PII content depends entirely on the admitted source.

## Decision

Tenant business data PII cannot be pre-enumerated. PII detection for tenant business data (boundary JSONB payloads) will be driven by **source_field-level PII marking** in the source catalog, not by static registry entries.

### Implementation

1. Add to `source.source_field`: `is_pii` (boolean, default false), `pii_category_code` (text, nullable ‚Äî `name`, `email`, `phone`, `address`, `financial`, `health`, `government_id`)
2. During source onboarding (manual review or AI verification via bc-ai), classify each field
3. When a source is activated for a tenant, auto-populate `pii_field_registry` entries for the boundary tables that store that source's data
4. This enables dynamic PII discovery for DSAR and nullification without pre-enumerating every possible tenant data shape

### Supersedes

Static T5 entries in `pii_field_registry` for `observed_payload_json` and `canonical_payload_json` serve as placeholders until this mechanism is built.

## Rationale

The only entity that knows which fields carry PII is the source catalog's `source_field` table, where each field's semantic type can be classified during onboarding. This classification feeds dynamically into the PII registry when a source is activated for a tenant.

## Consequences

- Source onboarding workflow gains a PII classification step
- AI verification (bc-ai) can auto-suggest PII categories based on field names and descriptions
- Nullification executor must resolve dynamic PII fields at execution time, not just static registry lookups
- DSAR discovery gains per-source granularity (knows exactly which fields in which payloads are PII)
