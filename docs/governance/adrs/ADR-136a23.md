# Observation Contract stays reader-scoped, field_affinity is the sharing layer

## Context

System-wide observation contracts (per source system) would contain 1000+ fields, creating ungovernable versioning overhead. Per-BO-per-system is nearly identical to per-reader-per-system since each reader maps 1:1 to a BO. The sharing benefit of decoupling is theoretical — in practice you have one reader per BO per system. field_affinity already solves the knowledge-sharing problem without versioning overhead.

## Decision

Observation Contract remains coupled to reader (reader_id FK). No system-wide observation contracts — they would create ungovernable mapping directories with version explosion. Two-layer architecture: (1) field_affinity = soft, shared, AI-suggested system knowledge ("SAP BUKRS → company_code", confidence 0.92). Not versioned, not governed. Shared across all readers. (2) observation_contract = hard, reader-scoped, governed. Locks the specific field mapping for one reader's flavor. Versioned with SemVer. The D200 wizard uses field_affinity to pre-populate observation contracts, reducing duplicate mapping work. The observation contract is the governed artifact; field_affinity is the intelligence layer.
