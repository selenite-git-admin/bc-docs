# BareCount project taxonomy — 7 projects

## Context

Clear separation of concerns: Customer Portal consumes what Core Platform produces, Admin Portal manages what Core Platform runs, Infrastructure supports all services, Documentation consolidates fragmented knowledge, Synthetic Data Generator exercises the full pipeline. Independent phase numbering per project. Each project has its own lifecycle, team ownership potential, and deployment cadence.

## Decision

BareCount is organized into 7 distinct projects: bc-portal (Customer Portal — customer-facing React app), bc-admin (Admin Portal — internal ops React app, formerly DataJetty Admin Portal), bc-core (Core Platform — backend services, pipelines, materializer, connectors, admission), bc-infra (Infrastructure — AWS IaC via CDK, CI/CD, environments), bc-docs (Documentation — consolidated from 4 source repos: barecount-documentation, legacy platform documentation, documentation-cxofacts, barecount-intra-site), bc-synth (Synthetic Data Generator — produces raw source data that flows through full Core Platform pipeline), and barecount-devhub (DevHub — development coordination hub). The three-plane model maps as: Core Plane and Platform Plane owned by bc-core, managed via bc-admin UI; Data Plane written by bc-core, read by bc-portal. bc-synth generates realistic source data (not pre-computed snapshots) to exercise the full pipeline end-to-end. Frontend mock data in bc-portal is separate — pre-computed for UI development variety. DataJetty name is retired; all references use BareCount.
