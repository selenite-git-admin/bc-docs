# S3 Boundary Archive — WORM storage for raw payloads and detailed evidence

## Context

S3 is 10x cheaper than RDS for write-once-read-rarely data. S3 Object Lock provides stronger immutability than RDS (compliance mode cannot be bypassed even by root). Patent Section 13 explicitly recommends "distributed object store with WORM for durability" — this implements that recommendation literally. Serves both D210 (raw payload backup) and D212 (detailed evidence archive).

## Problem

Raw payload preservation is critical for audit, recovery, and compliance — but storing it as JSONB in RDS is expensive (~$0.115/GB/month) and degrades query performance (TOAST overhead, table bloat, wasted IOPS). Need a cheap, immutable, durable store for data that's written once and rarely read.

## Decision

**Every boundary run archives its raw payloads to S3. S3 is the WORM vault. RDS is the query engine.**

### S3 bucket structure

```
s3://barecount-data/
  {tenant_id}/
    archive/
      observation/
        {entity_code}/
          {run_id}.jsonl.gz
      canonical/
        {contract_code}/
          {run_id}.jsonl.gz
      metric/
        {metric_code}/
          {run_id}.jsonl.gz
```

### Archive file format

Gzipped JSONL (JSON Lines). Each line is one record:

```jsonl
{"source_object_id":"uuid-001","observed_record_id":"uuid-101","contract_id":"uuid-c1","contract_version":1,"payload":{"posting_date":"2026-03-25","company_code":"1000","amount":50000},"evaluation_outcome":"admitted","created_at":"2026-03-25T00:00:00Z"}
{"source_object_id":"uuid-002","observed_record_id":"uuid-102","contract_id":"uuid-c1","contract_version":1,"payload":{"posting_date":"2026-03-25","company_code":"2000","amount":75000},"evaluation_outcome":"admitted","created_at":"2026-03-25T00:00:00Z"}
```

Each line carries: envelope metadata + raw payload + outcome. Self-contained — readable without any database access.

### Write path

Per run, after RDS batch INSERT completes:
1. Serialize all records as JSONL
2. Gzip compress
3. S3 PutObject with Object Lock (WORM — compliance mode, retention per policy)
4. Record archive_key on the run record in RDS

Can be synchronous (write-through, strongest durability) or asynchronous (fire-and-forget with retry queue, higher throughput). Default: synchronous for observation boundary (most critical), async for downstream boundaries.

### WORM enforcement

S3 Object Lock in Compliance mode:
- Objects cannot be deleted or overwritten during retention period
- Not even the root account can bypass compliance mode
- Retention: configurable per tenant (default: 7 years for financial data)
- This is stronger than anything RDS provides — RDS rows can be DELETEd by anyone with write access

### Cost model

| Volume | RDS JSONB cost/month | S3 IA cost/month | Savings |
|--------|---------------------|-------------------|---------|
| 10 GB raw payloads | $1.15 | $0.13 | 89% |
| 100 GB | $11.50 | $1.25 | 89% |
| 1 TB | $115.00 | $12.50 | 89% |

Plus: RDS IOPS saved, TOAST overhead eliminated, backup size reduced, table scans faster.

### Recovery use cases

1. **DDL generator bug:** typed table has wrong column type → read S3 archive, re-extract with corrected mapping, re-populate typed table
2. **Contract field added retroactively:** new field wasn't in v1 contract but source data had it → S3 archive has the raw payload including unmapped fields
3. **Audit request:** regulator wants to see exactly what was observed on date X → S3 archive is the WORM-sealed original
4. **Disaster recovery:** entire tenant DB lost → rebuild from S3 archives + contract definitions

### What this serves

- D210 uses it as the raw payload backup (replaces JSONB in RDS)
- D212 uses it as the detailed evidence archive (replaces per-row evidence_object)
- ISO 27001 A.12.3.1 (backup) and A.18.1.3 (records protection) compliance
