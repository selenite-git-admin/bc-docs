# Fix fail-open activation/publication gates X5/X4 (D429 Step 4)

## Context

The contract activation gates were silently passing on a dead signal: the deprecated IntegrityService (D305/DEC-bebaec) queries cc_field_mapping/canonical_field/business_field, all permanently dropped (D417/D418), so it returns empty and reports broken=0 — the activation branch never refuses (fail-open). For metric this is masked because the live D305 SSOT chain (ChainStatus → MLS-14) is the real gate; for canonical there is no such backstop, so unverifiable CCs could activate. Separately, the compatibility/version-increment gate filters governance_state_code='released', a state the lifecycle never sets (draft/review/approved/active/superseded), so it is dead and any new version passes unchecked. The narrow honest fix is to stop trusting the dead IntegrityService signal (remove the redundant metric gate; fail-closed for canonical, real check via D430) and align the compatibility lookup to 'active'. Critically, fail-closed must key on absence of a valid SSOT signal, not on the dead service returning empty — otherwise activation would be bricked since the tables are gone for good. Gate plumbing + vocabulary only; lifecycle state machine and canonical field-identity semantics (D430) are out of scope.

## Decision

D429 Step 4: make the contract activation/publication gates honest. Scope = gate plumbing + lifecycle vocabulary only; no code/schema/DB/DDL change authorized here; implementation deferred to a later DBCP. Does NOT solve D430/D431, does NOT clean parent/version desync, does NOT redesign the lifecycle state machine, does NOT resume MCF materialization.

1. Canonical (X5-C1): fail CLOSED when canonical integrity is unverifiable — replace the silent null=>pass at contract.service.ts:526-541 with an explicit refusal. Safe now (0 active Canonical Contracts); the real canonical field->concept integrity check is delivered by D430 (DEC-a6258b). CC activation is thereby coupled to D430 supplying a verifiable signal — the intended sequence (the greenfield ARPI CC is authored under D430).
2. Metric (X5): REMOVE the vestigial/deprecated IntegrityService metric gate (contract.service.ts:462-477). The live metric gate is the D305 SSOT chain — ChainStatusService.refreshChainStatusForVersion (488-497) -> MLS-14 gate (507-522). Net runtime behavior unchanged (the removed gate passed everything); the deprecated, misleading dependency is dropped.
3. X4: replace the dead 'released' lookup with 'active' — findLatestReleasedVersion (contract-version.repository.ts:175, filter governance_state_code='released') -> findLatestActiveVersion (filter 'active'), so the compatibility + version-increment check in createVersion (contract.service.ts:349-375) actually runs against the current active version. Lifecycle is draft/review/approved/active/superseded; 'released' is never set (only a UI display label). Expect this to begin enforcing semver discipline on new versions (previously dormant).
4. IntegrityService is retained ONLY for detail/read contexts (e.g. per-MC detail views) if still needed; it is no longer an activation gate.

DEAD-SIGNAL GUARDRAIL (mandatory in the DBCP): fail-closed MUST key on 'no valid SSOT signal', NEVER on 'the deprecated IntegrityService returned empty'. The IntegrityService tables (cc_field_mapping/canonical_field/business_field) were permanently dropped (D417/D418), so a naive 'fail closed when IntegrityService is empty' would brick ALL activation. The fix is to stop trusting the dead signal, not to invert its default.

Evidence (live 2026-06-07): X5 — IntegrityService @deprecated (integrity.service.ts:2, D305/DEC-bebaec) queries dropped tables -> empty -> broken=0 -> if(integrity){} never throws (462-477 metric, 526-541 canonical); metric backstopped by ChainStatus->MLS-14, canonical exposed. X4 — types/governance.ts:5-10 enum; contract-version.repository.ts:175 'released' filter; createVersion 349-375 silently skips when null. Blast radius: contract.service.transitionState.spec.ts ordering assertions need updating (test-only); compatibility.spec 'released' fixture is dead-path-safe; MCF specs orthogonal.
