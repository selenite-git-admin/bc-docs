# Conformed dimension tables (dim_*) as platform SSOT for grain

## Context

Grain values in snapshot business_key are naked strings today. Visualization, semantic verification (D366), and tenant onboarding all need a joinable dim layer. Established Kimball convention (dim_*) applied narrowly; existing master_* reference lists untouched.

## Problem

Metric snapshots carry grain values as naked strings in `business_key`:
```
{ "company_code": "1000", "fiscal_period": "FY2025-26/P01" }
```

Nothing joins these to dimension attributes (entity name, country, timezone, chart of accounts, customer industry, etc.). BI tools and bc-admin dashboards have to re-derive context every time. The `master.date_dim` and `master.fiscal_period` tables are the only conformed dims today, and they were added ad-hoc during D364.

This means:
- Visualization is hard. Every chart/report re-looks-up entity name, currency, geography from scattered sources.
- Chain integrity has no structural target. A CF with `semantic_family=identifier` declared in D366 has nothing to point at — "identifier of what?"
- Tenant onboarding has no canonical shape for "declare your legal entities, GL accounts, customers" — each tenant solves it differently.

## Decision

**1. Every grain dimension gets a `dim_*` table.** Conformed dims are platform-level (in `master` schema). Tenant-scoped dims are per-tenant (in tenant DB's `operational` schema or a new `tenant_dim` schema).

**2. Naming convention (new rule — documented in legacy v2 archive conventions page):**

> Any table whose rows are **joinable per-instance rows** — one row per key, referenced by fact/snapshot rows (directly or via another dim) — is named `dim_*`. Tables of reference values (ISO codes, closed taxonomies) also use `dim_*` when consumed as dims. True config/schema-shape tables (one row per system-wide setting, not per instance of anything joinable) keep a domain prefix.

**Decision matrix:**

| Table kind | Prefix | Example |
|---|---|---|
| Joinable per-instance rows | `dim_*` | `dim_date`, `dim_fiscal_period`, `dim_fiscal_calendar`, `dim_legal_entity` |
| Reference lists consumed as dims (ISO codes, countries, currencies) | `dim_*` | `dim_country`, `dim_currency` |
| Closed taxonomies (role enumerations, status codes) | `master_*` | `master_function`, `master_status` |
| True config rows (not per-instance) | domain prefix | `pricing.plan_config`, `infrastructure.deployment_defaults` |

**Amendment note (post-ADR):** Earlier draft classified `master.fiscal_calendar` as a config table. This was inconsistent — each row IS a per-instance calendar type, joined by code from `dim_fiscal_period`, `organization.fiscal_calendar_config`, and `tenant_dim.dim_legal_entity`. Renamed to `master.dim_fiscal_calendar` to comply with the rule.

**3. Existing rename (done as part of this ADR):**
- `master.date_dim` → `master.dim_date`
- `master.fiscal_period` → `master.dim_fiscal_period`

Small rename; touches Drizzle schema (2 files), seed scripts (1 file), FiscalCalendarService SQL (3 sites). Done in one commit, one DDL migration.

**4. Platform-level dim inventory (conformed across all tenants):**

| Table | Key | Status | Purpose |
|---|---|---|---|
| `master.dim_date` | `date` | rename from `master.date_dim` | PBI-style date dim, 2000–2100 seeded |
| `master.dim_fiscal_period` | `(fiscal_calendar_code, date)` | rename from `master.fiscal_period` | Materialized (calendar × date) → period |
| `master.dim_fiscal_calendar` | `fiscal_calendar_code` | rename from `master.fiscal_calendar` | Per-instance calendar catalog (joinable by code) |
| `master.dim_currency` | `currency_code` | **rename from `master_currency` now** | Currency with symbol, decimals, is_active |
| `master.dim_country` | `alpha2` | **rename from `master_country` now** | Geographic dim |

Rationale for rename-now (not alias): call-site count is small; rename churn is bounded today and grows linearly with every new contract/consumer. Aliasing creates two names for the same thing during the interim (long-term confusion) for a short-term saving. Clean break ships in one commit alongside the `date_dim` / `fiscal_period` renames.

**5. Tenant-level dim inventory (per-tenant instances; same-shape across tenants):**

Lives in tenant DB (`tbc_{slug}_dev`), under a new schema `tenant_dim` (schema name confirmed; see DEC-f02230 (D368) for full tenant DB schema organization):

**MVP column shape — bare shell (key + name + is_active + audit ts)** for 5 of the 6 dims. `dim_legal_entity` is the exception because it genuinely needs country / currency / fiscal_calendar FKs to do its job (fiscal binding + reporting currency resolution).

| Table | Key | MVP columns | Purpose |
|---|---|---|---|
| `tenant_dim.dim_legal_entity` | `legal_entity_code` | code, name, is_active, country_code (FK `master.dim_country`), currency_code (FK `master.dim_currency`), fiscal_calendar_code (FK `master.fiscal_calendar`), created_at, updated_at | Tenant's company codes — the anchor for fiscal binding and reporting currency |
| `tenant_dim.dim_gl_account` | `gl_account_code` | code, name, is_active, created/updated | Chart of accounts (shell; enriched when SAP SKA1/SKB1 reader lands) |
| `tenant_dim.dim_cost_center` | `cost_center_code` | code, name, is_active, created/updated | Cost centers (shell) |
| `tenant_dim.dim_customer` | `customer_id` | id, name, is_active, created/updated | Customer master (shell; KNA1 reader populates) |
| `tenant_dim.dim_vendor` | `vendor_id` | id, name, is_active, created/updated | Vendor master (shell; LFA1 reader populates) |
| `tenant_dim.dim_product` | `product_id` | id, name, is_active, created/updated | Product/material master (shell; MARA reader populates) |

Rationale for bare shell (not speculative columns): no column ships without a source reader that actually populates it with known type/nullability/semantics. Customer industry, vendor category, product group — all get added when the reader for each dim lands, not designed from imagination.

MVP acceptance: schema exists + empty tables + onboarding can insert the minimum rows (legal_entity). Other dims populate as tenant activates readers.

**6. Link back to D366:**

D366's `canonical_field.semantic_family` extended enum gets its join targets from D367:

| semantic_family | joins to |
|---|---|
| `dim-calendar-date` | `master.dim_date.date` |
| `dim-fiscal-period` | `master.dim_fiscal_period(fiscal_calendar_code, date)` |
| `dim-currency` | `master.dim_currency.currency_code` (or `master_currency.code` via alias) |
| `dim-country` | `master.dim_country.alpha2` (or alias) |
| `dim-legal-entity` | `tenant_dim.dim_legal_entity.legal_entity_code` |
| `dim-gl-account` | `tenant_dim.dim_gl_account.gl_account_code` |
| `dim-cost-center` | `tenant_dim.dim_cost_center.cost_center_code` |
| `dim-customer` | `tenant_dim.dim_customer.customer_id` |
| `dim-vendor` | `tenant_dim.dim_vendor.vendor_id` |
| `dim-product` | `tenant_dim.dim_product.product_id` |

This makes D366's deterministic compatibility filter concrete: "CF semantic_family=dim-legal-entity" must map to a BF whose values are joinable to `tenant_dim.dim_legal_entity.legal_entity_code`. Mismatch = red without AI call.

## Consequences

Positive:
- bc-admin / bc-portal can build real dashboards; every snapshot's business_key is joinable to dim attributes
- Semantic checks have explicit structural targets (CF family → dim table)
- Tenant onboarding has a canonical shape — "populate your dims" becomes a step
- Future BI tooling (PBI, Superset, Metabase) sees a clean star schema

Negative:
- Cross-DB JOIN for tenant dims (D367's tenant_dim schema is in tenant DB; fact rows may be in tenant DB too, so JOIN works locally — but cross-tenant aggregates need federation)
- Tenant onboarding SOP gains "populate dims" step
- Tables mostly empty at first; dashboards must gracefully handle missing dim rows

## Alternatives considered

- **Keep dims ad-hoc per consumer** — rejected: what we have today. Every dashboard re-derives.
- **Put all dims in `master`** including tenant-specific ones — rejected: tenant customer list isn't conformed; shoving it into platform DB violates D232 ("all data is tenant property").
- **No rename; keep `date_dim` / `fiscal_period` as-is** — rejected: naming inconsistency is the seed of future confusion. Rename cheap while only 2 tables exist.
- **Rename all `master_*` to `dim_*`** in this ADR — rejected: over-reach. `master_country` / `master_currency` / taxonomy tables aren't the focus; dim_* for grain is.

## Scope boundaries

IN this ADR:
- Naming convention rule
- Rename `master.date_dim` → `master.dim_date`, `master.fiscal_period` → `master.dim_fiscal_period`
- New platform dims: `dim_currency`, `dim_country` (aliases/views of existing `master_*` for now)
- New tenant_dim schema + table list
- MVP DDL for `tenant_dim.dim_legal_entity` (the only one with immediate need — fiscal calendar binding needs a place to point)
- D366 semantic_family enum gets `dim-*` values corresponding to these tables

OUT this ADR:
- Populating tenant dims from source systems (per-tenant seed concern)
- Renaming `master_country` / `master_currency` / other `master_*` tables
- BI tool integration / dashboards / visualizations
- Cross-tenant dim federation
- Slowly-changing-dimension (SCD) handling — future ADR if needed

## Rollout

1. ADR decided → DDL migration file applying renames + new tables
2. Drizzle schemas updated (small, 5 files)
3. FiscalCalendarService SQL updated (search/replace `fiscal_period` → `dim_fiscal_period`)
4. Seed scripts updated (1 file)
5. Tenant onboarding playbook gains step: "scaffold tenant_dim tables"
6. D366 verifier's dim-family check becomes active once semantic_family classification includes dim-* values

Rollout is narrow. Renames are the ripple. New tables are additive (no migration for existing data).
