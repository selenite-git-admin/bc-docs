# Metric contracts reference N canonical objects, not one

## Context

1. Patent Section 7.2 explicitly defines metric subject as "Which Canonical Objects [plural] are being measured?" 2. Patent Section 7.3 states metric input is "Links to the specific Canonical Objects [plural] included in the aggregation." 3. Patent Section 7.6 confirms "It aggregates the evidence from thousands of input Canonical Objects into a single Metric Object." 4. Real KPI evidence: AP KPI catalog (26 KPIs) shows every KPI references 1-5 entities across AP, GL, CT modules. E.g. DPO = AP_4101 + AP_4102 + AP_4103 + GL_5101 + CT_5101. No real business KPI derives from a single canonical object. 5. Foundation component refs (120-metric-action-controls.md) confirm quality rules are declared within metric contracts, not separately. 6. Contract categories remain 7: extraction, source, admission, canonical (source-chain, bound to one source_version_id) + metric, intervention, ai (cross-cutting, no source_version_id binding).

## Decision

A Metric Contract declares one or more Canonical Object dependencies. A Metric Snapshot is evaluated by aggregating the governed state of N Canonical Objects at the evaluation boundary. The foundation spec statement "A Metric Fact is created from exactly one Canonical Object" (metric-fact.md) is incorrect and must be corrected to "one or more". Quality rules (freshness, coverage, completeness) are declared within Metric Contracts, not as a separate contract category. Intervention (Action) contracts reference Metric Snapshots and are also cross-cutting. Neither metric nor intervention contracts are bound to a single source_version_id.
