# Derived Canonical Fields — Compute Rules in CC Field Mapping

## Context

`contract.cc_field_mapping` (D301) translates between two vocabularies: BO-scoped Business Fields (source-side) and Canonical Fields (metric-side). Today it supports five resolution rules: `sum`, `latest`, `assert_equal`, `count_distinct`, `count_where_not_null`. Each rule takes exactly ONE source BF and produces a value for one target CF.

This model cannot express CFs that are **derived from combinations of source fields**.

### Observed failure (demo-selenite, 2026-04-15)

Of 37 rejected metric contracts in tenant evaluation:

- **8 AR duration MCs** (credit_approval_time, credit_onboarding_time, avg_days_delinquent, o2c_cycle_time, customer_request_turnaround_time, billing_to_cash_cycle_time, credit_hold_resolution_time, average_invoice_processing_time)
  - Formula shape: `O1 = SUM(I1) / SUM(I2)` where I1 = "sum of durations" and I2 = count of rows
  - Expected input: numeric days (integer per row)
  - Actual mapping: cc_field_mapping points CF `sum_of_days_past_due` → BF `receivable_hdr_due_date` (a `date`) with rule `latest`
  - SAP/BSID source emits raw due_date, document_date, created_at, modified_at — **no pre-computed days field exists**
  - Metric engine correctly rejects: `Non-numeric values for 'sum_of_days_past_due_for_overdue_invoices' in SUM()`

The rejection is accurate but the diagnosis is at the wrong layer. The CF describes a *computed* value (`days` between two dates). Today's `cc_field_mapping` has no way to declare that.

### Why this is a primitive gap, not a data gap

Writing a `days_past_due` column into BSID on import would solve these 8 MCs but:

1. It would be source-specific (SAP schema already lacks it; Oracle, SFDC, etc. also do not store it)
2. It would be evaluation-moment-dependent (days_past_due is `as_of_date - due_date` — the value depends on when you ask)
3. It violates the ownership boundary — source systems store facts, BareCount computes metrics
4. It scales badly — every new derived CF would require a schema change to every source

Computation belongs in the canonical layer, where the platform sees all source systems uniformly and owns the evaluation moment.

### Relationship to D331

D331 (proposed) introduces a **typed transform library** at the OC layer: source field → BF, at admission time, coercing text to types (`to_number`, `to_date_iso8601`, etc.).

D330 introduces the **compute extension** at the CC layer: one-or-more BFs → CF, at canonical resolution time, producing derived values (`date_diff`, `row_count_where`, etc.).

Both decisions share a function-library pattern. They are orthogonal: D331 is about *shape* (getting typed values through admission), D330 is about *derivation* (computing new values from typed inputs). D330 does not depend on D331 landing first — it operates on whatever types the chain delivers — but when D331 lands, D330's inputs are guaranteed typed, which simplifies D330's evaluator.

## Decision

### D330-R1: Add a `compute` resolution rule to `cc_field_mapping`

Extend the rule enum:

```sql
-- Current rules
resolution_rule_code IN ('sum', 'latest', 'assert_equal', 'count_distinct', 'count_where_not_null')

-- New rule
resolution_rule_code IN ('sum', 'latest', 'assert_equal', 'count_distinct', 'count_where_not_null', 'compute')
```

### D330-R2: Extend `cc_field_mapping` with typed compute expression

When `resolution_rule_code = 'compute'`, `business_field_id` is nullable and a new column `compute_json` holds the expression:

```sql
ALTER TABLE contract.cc_field_mapping
  ADD COLUMN compute_json jsonb;

-- Integrity constraint
ALTER TABLE contract.cc_field_mapping
  ADD CONSTRAINT cc_field_mapping_compute_consistency CHECK (
    (resolution_rule_code = 'compute' AND compute_json IS NOT NULL AND business_field_id IS NULL)
    OR
    (resolution_rule_code <> 'compute' AND compute_json IS NULL AND business_field_id IS NOT NULL)
  );
```

`compute_json` structure:

```json
{
  "function": "date_diff",
  "inputs": [
    { "role": "end", "business_field": "receivable_hdr_due_date" },
    { "role": "start", "business_field": "receivable_hdr_document_date_time" }
  ],
  "output_type": "integer",
  "params": { "unit": "days" }
}
```

Rules:
- `function` must be a member of the compute function library (D330-R3)
- `inputs[].business_field` must reference a BF in the CC's `field_selection`
- `output_type` MUST match the target CF's `data_type` (D331 alignment)
- `params` are function-specific and validated against the function signature

### D330-R3: Compute function library (initial set)

The library starts narrow and grows additively. Each entry has a declared signature and a deterministic implementation.

| function | inputs | params | output | purpose |
|---|---|---|---|---|
| `date_diff` | `end: date or timestamp`, `start: date or timestamp` | `unit: days or hours` (default `days`) | `integer` | Duration between two dates |
| `date_diff_as_of` | `start: date or timestamp` | `unit: days`, `as_of: evaluation_period or literal_date` | `integer` | Age from a date to evaluation moment |
| `row_count` | (none) | (none) | `integer` | Count of rows in the CC grain bucket — equivalent to `SELECT COUNT(*)` |
| `row_count_where` | (none) | `filter: filter_expression` | `integer` | Count of rows matching filter |
| `sum_where` | `value: number or integer` | `filter: filter_expression` | `number` | Sum of `value` for rows matching filter — what `filter_json` should have been on `sum` rule all along |

`filter_expression` is a JSON AST: `{"op": "eq", "field": "bf_name", "value": literal}` or `{"op": "and", "args": [...]}` etc. Supported ops: `eq`, `ne`, `lt`, `lte`, `gt`, `gte`, `in`, `not_in`, `is_null`, `is_not_null`, `and`, `or`, `not`. No arbitrary code.

### D330-R4: Evaluation happens at canonical resolution time

The `CanonicalEvaluationEngine` executes compute expressions per-row for each CO within a grain bucket, then applies the outer aggregation (the metric engine's SUM/AVG/etc.). This places the computation BEFORE the metric engine's aggregation — so the metric engine sees a numeric column and operates normally.

Execution order:
1. `CanonicalEvaluationEngine` pulls BFs from admitted records per grain bucket
2. For each row in the bucket: evaluates every `compute` cc_field_mapping → produces a numeric (or typed) value
3. Emits the CO with all CFs materialized (sum'd, latest'd, or computed)
4. Metric engine consumes CO via CF names — no engine changes required

### D330-R5: Governed write path (no direct SQL)

`cc_field_mapping.compute_json` MUST be written via the governed service. Two endpoints needed:

- `POST /api/onboarding/cc/:contractId/field-mappings` accepts `compute` mappings alongside BF-based mappings (extend existing endpoint)
- `POST /api/onboarding/cc/:contractId/field-mappings/:mappingId/replace` — new endpoint — replaces a single mapping atomically (so a bad BF-mapping can be replaced by a compute-mapping, and vice versa)

Replacement is a versioned operation: it deletes + inserts within a transaction, emits an audit event, and triggers chain_status refresh for affected MCs.

The "replace" capability is the *governed patch path* the session surfaced as missing. It is not a bulk-update endpoint and is scoped to one mapping at a time, preserving discipline.

### D330-R6: Chain integrity extension

`ChainStatusService` L5 check extends: for every CF in a CC's field_selection resolved by `compute`, verify:
- Function name is in the library
- All input BFs are in the CC's field_selection (not just BO composition)
- All input BFs are mapped by at least one active CM (L6 verifiability)
- output_type matches CF.data_type

Failures downgrade the CC's L5 from green to amber; MCs binding that CC cannot reach chain_complete until resolved.

### D330-R7: Seed the 8 AR duration mappings (scope of this ADR's implementation)

Once D330 is decided + implemented, the 8 AR duration MCs get corrected cc_field_mapping entries:

| Target CF | Compute function | Inputs |
|---|---|---|
| `sum_of_days_past_due_for_overdue_invoices` | `date_diff_as_of` | start = `receivable_hdr_due_date`, as_of = evaluation_period |
| `total_time_for_credit_approvals` | `date_diff` | end = `receivable_hdr_last_modification_date_time`, start = `receivable_hdr_document_date_time` |
| `total_elapsed_time_for_credit_onboarding` | `date_diff` | same pattern |
| `total_time_to_resolve_credit_holds` | `date_diff` | same pattern |
| `total_time_to_resolve_ar_requests` | `date_diff` | same pattern |
| `total_processing_time_for_invoices` | `date_diff` | same pattern |
| `sum_of_days_from_invoice_date_to_cash_receipt` | `date_diff` | end = `receivable_hdr_clearing_date`, start = `receivable_hdr_document_date_time` |
| `invoice_to_payment_time`, `order_to_invoice_time`, `payment_to_cash_application_time` (o2c_cycle_time sub-CFs) | `date_diff` | per-segment date pairs |

Each is an independent cc_field_mapping replace, one-at-a-time, evaluated after each change — no bulk migration.

## Options Considered

### Option A: `compute` rule on `cc_field_mapping` with typed function library (CHOSEN)

Keeps the existing primitive (`cc_field_mapping` as the BF→CF translator) and extends it additively. One new column, one new rule code, one function library.

**Pros:** Minimal schema change. Reuses D301's abstraction. Compatible with D331's transform-library pattern. Function library can grow additively.
**Cons:** `cc_field_mapping` row gains a polymorphic shape (either `business_field_id` or `compute_json`). Slight model tension but enforceable via check constraint.

### Option B: Separate `canonical_field_compute` table (REJECTED)

One row per computed CF, with its own compute expression and inputs. Lives alongside (not inside) cc_field_mapping.

**Why rejected:** Two tables describing how the same CF gets its value — consumers (metric engine, chain status, CF resolution) would have to query both and merge. Adds operational complexity for a narrow capability. Also requires a separate governed service.

### Option C: Move derivation into metric_contract formula (REJECTED)

Let MC formulas express `SUM(date_diff(due_date, document_date_time))` directly, pulling raw BFs into the formula.

**Why rejected:** Violates D301's separation (CC translates vocabulary, MC consumes the canonical vocabulary). Each MC would need to know BF names of every source, losing the source-agnostic CO abstraction. Also breaks the formula language simplicity (formulas become expressions; parser complexity explodes).

### Option D: Source-side pre-computation (REJECTED)

Write a "days_past_due" column during admission into admitted_record / CO payloads.

**Why rejected:** Evaluation-moment dependency (days_past_due value depends on when you compute it), source-system-dependent (every source needs its own compute), violates the source-system ownership boundary. Discussed in Context.

## Consequences

### Positive

- 8 AR duration MCs become computable with typed, auditable inputs — proven in-session unblock
- Pattern generalizes: count-with-filter metrics, ratio-with-filter metrics, duration metrics across domains (AP days-payable-outstanding, GL close-cycle-time, etc.)
- `filter_json` on existing `sum` / `count_where_not_null` rules gets a formal structure via D330-R3's filter expression AST — consistent mental model across rules
- Introduces the "replace one mapping" governed endpoint that the session surfaced as missing — unblocks future corrective work without weakening governance
- Keeps CCs immutable at the contract level; only the mappings change (mappings are data, not contract body)

### Negative

- New evaluator surface area: compute expression parser + executor + test suite
- `cc_field_mapping` rows become polymorphic (either BF-based or compute-based) — queries and UIs that read mappings need to branch
- Function library becomes a long-term governance concern — adding functions requires ADR or lightweight RFC (the library is LOCKED like contract keys, D233 spirit)

### Risks

| Risk | Mitigation |
|---|---|
| Function library scope creep — pressure to add arbitrary SQL-like features | ADR-gate every library addition. Start with 5 functions. Grow by ≤2/quarter by default. |
| Compute expressions reference BFs not in field_selection → silent failures | D330-R6 chain integrity gate catches this at CC validation time |
| Filter AST becomes a second query language | Keep AST small (ops above only). Reject nested subqueries, aggregations inside filters, function calls inside filters. Pure boolean algebra over BF equality. |
| Compute output type drifts from CF.data_type | CR-QG-CC validation extends to check `compute_json.output_type == CF.data_type`. Enforced at mapping write time, not runtime. |
| Per-row compute cost at canonical resolution | Benchmark during implementation. If greater than 5% overhead on typical grain-bucket sizes, evaluate memoization per-bucket. |
| Expressions referencing `evaluation_period` couple CCs to temporal context | Restrict `as_of` to `evaluation_period` literal; no other temporal references. Evaluation context is still owned by CC.temporal_gate. |

### Neutral

- Metric engine is unchanged. It consumes CFs; it does not care how they are resolved
- Chain completeness SSOT (D305) is unchanged in structure; only the L5 check's condition body extends
- No impact on tenants — CCs are platform-only (D164)

## Out of Scope

- **Cross-CC compute** (a CF in one CC derived from BFs in another CC) — deferred. Current scope is one-CC-at-a-time
- **Derived metrics that need aggregations from multiple CCs** — that is MC formula territory, not CC compute
- **Rolling-window computations** (moving averages, YoY growth) — these belong to metric engine (D315/Phase 3), not CC resolution
- **Lookup joins** (JOIN against a master/reference table during compute) — explicitly not permitted in v1. If needed, introduce via separate ADR with governance review

## Migration Plan

1. **Schema change** (one migration): add `compute_json` column + check constraint + extended rule enum
2. **Compute executor**: new module `contract/compute-evaluator.ts`. Unit-tested with the 5 functions
3. **CC onboarding service**: extend `addMappings` to accept `compute` rows; add `replace` endpoint
4. **Canonical evaluation engine**: insert compute step between row-fetch and resolution-rule-apply
5. **Chain status**: extend L5 validation
6. **Seed the 8 AR duration mappings** via the replace endpoint — one-then-many, evaluate each MC after each change
7. **Audit**: run `chain-status/refresh` after all 8 are in; confirm no regression in the 576 chain-complete count

No existing data is mutated. Existing `cc_field_mapping` rows keep their `business_field_id` unchanged. New rows (or replaces) use the compute path.

## Rollback

Each step is independently reversible:
- Schema migration has a `DOWN` (drop column, drop constraint, narrow enum)
- Compute executor module can be disabled (service becomes a 501 on `compute` rules)
- Per-mapping replaces can be un-replaced via the same endpoint (restore prior BF-mapping)
- Chain status refresh is idempotent

No tenant data is touched. No MC or CC contract bodies are modified. All changes are data (mappings) + schema (additive column) + code (new module).
