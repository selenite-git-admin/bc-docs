# Tenant DB schema organization — 6 schemas with data/identity/admin separation

## Amendment — 2026-04-22 (D369 / DEC-95687d)

D369 amends this ADR on two axes:

1. **Schema rename**: `envelope` → `progression`. Once D369 drops JSONB payload columns, the schema no longer wraps a payload; it records the ordered sequence of boundary events. Foundation §2.1 already names this "Authoritative Object Progression."
2. **Table fold**: the produced-artifact rows (`source_object`, `canonical_object`, `metric_snapshot`, `action_object`) are removed. They were paired with a corresponding boundary-event row (`admitted_record`, `evaluation`, `metric_evaluation`, `action_evaluation`). After JSONB payload moves to `fact.*`, the produced-artifact row carries nothing the fact row cannot carry itself. Fold result: 4 progression tables — `admission`, `canonical_evaluation`, `metric_evaluation`, `intervention_evaluation` — one per evaluation boundary. `observed_record` folds into `admission` (rejection is an admission status, not a pre-boundary).

All other decisions in this ADR stand. The 6-schema layout (progression, fact, evidence, tenant_dim, organization, admin) is unchanged. The naming rule is unchanged. See ADR-95687d for full rationale and implementation phases.

## Context

Platform holds metadata, tenant holds data and tenant metadata — tenant self-description (legal name, industry, fiscal context, hierarchy) is tenant's own metadata, not platform's. Operational schema was a grab-bag hiding four distinct concerns (integration, identity, audit, and two misconceived concepts). Goals are not a Foundation concept — they are intervention bindings. `intervention_definition` duplicates the already-unified `tenant.contract_binding`. Renames (boundary→envelope, record→fact) make the jsonb-SOT vs typed-projection architecture visible in schema names. Adding `organization` schema gives tenant self-description a proper home respecting the principle that platform should not know how tenant's organization is formed.

D232 (DEC-771baf) locked the core split: contracts live in platform DB, data lives in tenant DB. The tenant DB was designed with 4 schemas (boundary, record, evidence, operational). That design has held through 18 months of platform evolution, but three pressures now require revision:

1. **D367 (DEC-8672d0) introduces `tenant_dim`** — first new tenant schema since D232. Adding it in isolation without codifying the full layout means the next schema addition is another ad-hoc call.
2. **The arc locked this week** — "fix at platform level, not in DBs by handpicked changes." Tenant DB layout is itself a platform-level decision.
3. **Naming has been hiding architecture.** `boundary` and `record` both carry jsonb envelopes vs typed projections of the same data; schema names don't reveal which is SOT and which is derived. `operational` is a grab-bag of four distinct concerns plus two misconceived concepts (goal, intervention_definition).

Blast radius for restructure is lowest now — no production tenants, demo-selenite broken per Apr 21 handoff, 4 dev tenant DBs.

## Problem

### Naming hides architecture

`boundary.observed_record.observed_payload_json` (jsonb envelope, SOT, append-only) and `record.so_bkpf_v1` (typed projection, derived, queryable) carry the same data in two forms. Both names fail to reveal which is authoritative and which is derived. A new engineer cannot infer the architecture from the schema names; they must read D210 + D232 ADRs.

### `operational` is a grab-bag of unrelated concerns

Current tenant DB `operational` holds:
- `connection`, `connection_config`, `connection_check` — integration plumbing (3 tables)
- `users` — identity (1)
- `activity_log` — audit (1)
- `goal` — **misconceived** (not a Foundation concept)
- `intervention_definition` — **duplicate of `tenant.contract_binding`**

"Operational" papers over the fact that integration plumbing, user identity, and audit are structurally different. And it hides two tables that should not exist.

### `goal` is not a Foundation concept

Foundation execution model: `Source Object → Canonical Object → Metric Snapshot → Action Object`. There is no `goal` object. A "goal" in business language — "DSO should be under 45 days by Q3" — is an intervention binding with a tenant-specific target and threshold, expressed via `contract_binding.override_json` on the platform-side `tenant.contract_binding` row. The separate `goal` table is an artifact of incomplete integration between the operational runtime and the contract model.

### `intervention_definition` duplicates `contract_binding`

Platform DB already has `tenant.contract_binding` — unified binding table covering all six contract families (source, admission, canonical, observation, metric, intervention) with discriminator + soft-ref + `override_json` (see [contract-binding.ts](https://github.com/selenite-git-admin/bc-core/blob/main/src/database/schema/tenant/contract-binding.ts)). Post-D250, this is the single binding mechanism. A separate `operational.intervention_definition` in tenant DB with `intervention_contract_id` + `trigger_config_json` + `action_config_json` is vestigial — covered by `contract_binding.override_json`.

### Tenant self-description has no home

Legal name, industry, HQ country, reporting jurisdiction, reporting currency, org hierarchy, fiscal calendar choice per legal entity — these are tenant's own metadata about themselves. Per D232, platform DB should not know how the tenant's organization is formed. Currently platform has `tenant.fiscal_calendar_binding` (D364) which leaks this. Tenant DB has nowhere for the rest.

## Decision

### Tenant DB schema inventory — 6 schemas

| Schema | Status | Charter |
|---|---|---|
| `envelope` | rename from `boundary` | jsonb SOT — boundary-object envelopes (observed_record, source_object, admitted_record, evaluation, canonical_object, metric_evaluation, metric_snapshot, action_object, action_evaluation) + 4 run-trackers. Append-only, immutable, full payload as jsonb. Source of truth for all tenant execution. |
| `fact` | rename from `record` | Typed projections of envelope rows — Schema-Provisioner-created columnar tables per activated contract (`so_*`, `co_*`, `ms_*`, `ao_*`, `ev_*`). Derived, queryable, pairs with `dim_*` and `tenant_dim.*` as Kimball star schema. Can be rebuilt from envelope if lost. |
| `evidence` | existing | Provenance + lineage (evidence_object, evidence_record, lineage_object). Content-hashed, append-only. Data provenance (≠ activity audit, which lives in `admin`). |
| `tenant_dim` | new per D367 | Tenant master data — joinable per-key dim rows. MVP: `dim_legal_entity`. Added as reader flavors populate: `dim_gl_account`, `dim_cost_center`, `dim_customer`, `dim_vendor`, `dim_product`. |
| `organization` | **new** | Tenant self-description metadata — singleton identity + per-legal-entity fiscal setup + (future) org hierarchy. See Section "Organization schema contents." |
| `admin` | rename + shrink from `operational` | Tenant admin space — integration (connection×3), identity (users), audit of admin actions (activity_log). `goal` and `intervention_definition` dropped. |

### Renames

Database-level renames in one migration:
- `boundary` → `envelope`
- `record` → `fact`
- `operational` → `admin`

Table names under each schema unchanged except for the three tables dropped (see next section).

### Drops (tables removed from the schema)

1. **`operational.goal`** — removed. Not a Foundation concept. Tenant goals are expressed as intervention bindings with target values in `contract_binding.override_json` on platform-side `tenant.contract_binding`.
2. **`operational.intervention_definition`** — removed. Duplicate of `tenant.contract_binding` post-D250 unification. Any existing data migrates to `contract_binding.override_json` (demo-selenite broken per handoff; no migration load in practice).

### Organization schema contents

**`organization.org_profile`** (singleton row per tenant DB):
- legal_name, display_name, industry_code, hq_country_code, reporting_jurisdiction, reporting_currency_code, logo_url, primary_contact_name, primary_contact_email
- created_at, updated_at, created_by_name, updated_by_name

**`organization.fiscal_calendar_config`** (per tenant × legal_entity):
- Moved from platform `tenant.fiscal_calendar_binding` (D364) — honors principle that platform should not know tenant's fiscal setup.
- Columns: config_id (PK), legal_entity_code (FK `tenant_dim.dim_legal_entity.legal_entity_code`), fiscal_calendar_code (soft ref to platform `master.fiscal_calendar.fiscal_calendar_code`), is_active, effective_from, effective_to, audit columns
- Engine resolution (per D365) reads this from tenant DB during canonical evaluation to stamp `fiscal_period` onto CO payload. Tenant DB connection already required for envelope/fact writes.

**`organization.org_hierarchy`** (future, not MVP):
- Parent/child relationships between legal entities if the tenant has a complex corporate structure. Deferred until a tenant needs it. Not in D368 MVP DDL.

### Platform DB — only master renames from D367

D368 does not restructure platform DB. Platform retains 7 existing schemas (contract, master, metric, runtime, source, tenant, pricing). Changes from D367 applied:
- `master.date_dim` → `master.dim_date`
- `master.fiscal_period` → `master.dim_fiscal_period`
- `master_country` → `master.dim_country`
- `master_currency` → `master.dim_currency`
- `master.fiscal_calendar` — keep name (config table)

**Platform `tenant.fiscal_calendar_binding` is removed** (moved to tenant DB `organization.fiscal_calendar_config` per Path B).

### Fiscal calendar move — Path B (rebase PR #4)

D364 (DEC-d7e7a0) shipped `fiscal_calendar_binding` in platform `tenant.*`. That placement predated the arc's principle lockdown ("platform holds metadata, not tenant self-description"). PR #4 is still open unmerged.

**Path B: rebase PR #4 before merge.** Update D364's Drizzle schema + DDL + FiscalCalendarService to place `fiscal_calendar_config` in tenant DB `organization` schema from the start. Avoids a short-lived intermediate state where fiscal binding lives in platform DB only to move a week later.

Implementation consequence: `FiscalCalendarService.resolve()` (per D365) reads from tenant DB instead of platform DB. Engine already connects to tenant DB for envelope/fact writes; adding one more read is zero additional infrastructure.

### Naming rule (codified)

Schema design principles for both platform and tenant DBs:

1. **Schema per concern.** No grab-bags. Each schema has a one-line charter.
2. **`dim_*` prefix for joinable per-instance grain rows.** Per D367. Reference/config tables keep domain prefix (`master_*`, `pricing_*`, etc.).
3. **`fact` + `dim_*` follow Kimball** when it clarifies; use BareCount vocabulary (envelope, evidence) when Kimball doesn't fit the domain.
4. **New schema requires ADR.** No ad-hoc schema creation.
5. **No cross-DB FKs.** Platform ↔ tenant relationships use soft refs (uuid/code columns without FK constraints). Resolution via API, not JOIN. Per D232.
6. **No mixing SOT with projections in the same schema.** `envelope` is SOT, `fact` is projection. Keep separate.
7. **Platform DB = platform metadata only.** Contracts, masters, runtime, source catalog, minimal tenant registry. No tenant business data, no tenant self-description.
8. **Tenant DB = tenant data + tenant metadata.** Envelope + fact + evidence + tenant_dim (data). Organization + admin (metadata about this tenant).

### CLAUDE.md protocol update

One new bullet under "Dev Service Management" / "Stack" section documenting the 6-schema tenant DB layout as the canonical reference, so future sessions don't re-derive it from DDL.

## Consequences

Positive:
- Schema names reveal architecture (`envelope` = SOT, `fact` = projection — self-documenting)
- Tenant self-description has a proper home (`organization`)
- Grab-bag eliminated: `admin` has a tight charter (integration + identity + audit of admin actions)
- Foundation principle respected: platform does not know tenant's fiscal/entity structure
- Foundation alignment: goals correctly expressed as intervention bindings; no vestigial tables
- Pairs cleanly with D366 (semantic_family) and D367 (dim_* naming) as the three-ADR finalization

Negative / trade-offs:
- Rename churn across bc-core — Drizzle schemas, raw SQL, service references (~500-700 call sites estimated)
- D364 PR #4 rebase required before merge
- Schema Provisioner logic needs to emit `fact.*` instead of `record.*`
- `FiscalCalendarService` moves its read target from platform DB to tenant DB
- Any seed scripts referencing old schema names need update

## Alternatives considered

- **Keep `operational` as single monolithic schema, only rename it** — rejected. The grab-bag was hiding goal and intervention_definition as misconceived tables. Simply renaming would preserve the architectural mess.
- **Put organization on platform side in `tenant` schema** (original lean) — rejected per user review. Violates "platform holds metadata, not tenant self-description." Tenant's org structure is the tenant's own business; platform has no need to know legal entities, fiscal context, or hierarchy.
- **Decompose `admin` further into integration / identity / audit** — rejected. Only 5 tables; over-decomposed. Monolithic `admin` with clean charter is right-sized for current scale. Split later via ADR if any concern grows past ~6 tables.
- **Keep `operational.goal` as-is and add intervention_binding separately** — rejected. Goal is not a Foundation concept. Adding more tables to support a misconception compounds the problem. Tenant goals are intervention bindings; Foundation already covers this.
- **Leave naming alone, document the existing confusion** — rejected. The cost of rename grows linearly with every new contract, service, tenant. Today is the cheapest this will ever be.
- **Merge `organization` into `admin`** — rejected. Organization (rarely-changing identity) and admin (frequently-changing operations) have different change cadence, different access patterns, different audit requirements. Conflating them re-creates the grab-bag.

- Tenant DB 6-schema inventory with charters
- Renames: boundary→envelope, record→fact, operational→admin
- Drops: operational.goal, operational.intervention_definition
- Move: platform `tenant.fiscal_calendar_binding` → tenant `organization.fiscal_calendar_config` (Path B — rebase PR #4)
- Organization schema contents (org_profile + fiscal_calendar_config MVP; hierarchy deferred)
- Naming rule codification
- CLAUDE.md protocol update

OUT of this ADR:
- Platform DB restructure beyond D367's master renames
- bc-portal / bc-admin / bc-ai code changes (implementation session)
- Schema Provisioner changes (implementation session)
- Data migration from demo-selenite's existing state (broken per handoff; out of scope)
- Split of `admin` into sub-schemas (future ADR if any concern grows)
- `org_hierarchy` table (future when a tenant needs multi-entity parent/child)
- Subscription / workspace / pricing schema additions on platform DB (deferred)
- Tenant onboarding SOP updates (separate playbook work)
