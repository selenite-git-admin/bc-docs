# Rulebook-first execution stance — reduce ceremony, not standards

## Context

Three forces converge to make this stance worth recording now:

1. **Substrate clarity precedes runtime exposure.** PR #11 + PR #12 + PR #133 collectively lock Option A and the 5-phase A1..A5 path. The work between now and the first real M12 panel run is all substrate. During that stretch, "product readiness" metrics will look poor, and there will be pressure (internal, external, or self-generated) to take "practical" shortcuts. Recording the stance before that pressure arrives prevents re-litigation gate-by-gate.

2. **Past failures all looked practical at the time.** SES-cd4f80 (typed-fact writer silently dropped a row), SES-45e3ba (DSO inflated to 115/230 days from a fact-write key alias mismatch), SES-8f7232 (read-side double-counting from stale runs), SES-f10b35 (DSO temporal arc 38→47 not expressible) — every one of these started as a "small practical fix" against the substrate boundary. The pattern is consistent enough that the operator has codified it as a stance.

3. **Ceremony cost is uneven.** Some discipline patterns (SHA-pinned merges, BCF §13 trailers, DB invariant probes) are cheap insurance. Others (over-polished doc NITs, separate PRs for clerical work, DBCPs for parked ideas) are pure overhead. This stance draws the line explicitly so future gates do not relitigate every-time. Reductions are allowed in the second category and forbidden in the first.

The stance is recorded as a `governance` / `execution-discipline` decision because it governs HOW the team executes on the BCF/MCF substrate, not WHAT the architecture is. It complements ADR-ebf0b4 (D268, Session Discipline & Data Integrity) without superseding it: D268 codifies the ten session-discipline rules (no bulk contract generation, no cosmetic status changes, one-then-many, independent verification, self-audit at session close, etc.); this stance adds the meta-level reduction-and-preservation framework above those rules.

Future Claude (or human) sessions can reference this DEC-uid in any gate where the temptation to shortcut substrate or over-engineer paperwork arises. The expected effect is fewer re-litigations of settled boundaries and faster decisions in the safe-reduction category.

## Recording Context

This stance is recorded after PR #11 (BCF/MCF boundary DBCP, Option A chosen) merged at bc-docs-v3 main `6f8cc15`, PR #12 (D1-D11 decision record + Phase A1 substrate-design DBCP) merged at bc-docs-v3 main `70beeb7`, and bc-core PR #133 (BCF cleanup PR-1 dry-run) merged as inventory tooling at bc-core main `204c1cf`. With Option A locked, the BCF/MCF substrate-readiness work (Phase A1..A5) will take time and will make product-readiness *look* low for a stretch, because substrate clarity precedes runtime exposure. The operator anticipates pressure — internal and external — to take "practical" shortcuts during this stretch. This stance closes that door before future gates open it.

## Stance

### Direction (unchanged)

- **Continue with the rulebook path already chosen.** No re-litigation.
- **Option A remains the architecture direction.** BCF evidence gets a real `bcf.*` home; MCF evidence remains in `mcf.*`; generic `contract.*` evidence ambiguity is retired through Phases A1..A5.
- **No synthetic / mock / replay / canned data may be written to persistent substrate going forward.** This is HR1 from the BCF/MCF boundary DBCP and remains operator-locked.
- **No "practical" shortcuts that knowingly violate the substrate boundary,** even when they appear faster. Prior experience shows such shortcuts backfire through small oversights and reset trust.

### Reduce ceremony where it is safe

- Combine tightly-coupled docs/gates when they cannot reasonably fail independently (e.g. decision record + design DBCP in one PR, as PR #12 did).
- Avoid over-polishing docs after they are directionally safe — NIT hygiene patches should be the exception, not the default cadence.
- Use DevHub `parked` tasks for non-critical cleanup, not full DBCPs and PRs (e.g. TSK-46da87 for bc-core script inventory, recorded via DevHub task rather than a markdown DBCP).
- Avoid separate PRs for purely clerical work unless CI / governance requires it.

### Do NOT reduce

- **DB evidence.** Every mutating gate requires pre-state + post-state read-only DB probes. `bc-postgres` MCP `allow_write=false` throughout planning gates.
- **Rollback discipline.** Every apply gate must specify mid-apply ROLLBACK + post-apply rollback path before authorization.
- **No-synthetic enforcement.** Substrate-side via the additive HR1 CHECKs on `bcf.panel_output_record` and `bcf.certification_record`; writer-side via authoring code; pre-apply via the §12.1 probe #3 UNION ALL across both `contract.*` tables.
- **Schema-boundary clarity.** `bcf.*` for BCF authoring evidence, `mcf.*` for MCF authoring evidence, generic `contract.*` retired for the authoring cluster. No cross-schema FKs from `bcf.*` to `contract.*`. The 5 `mcf.* → contract.panel_output_record` FKs are redirected to `mcf.* → mcf.*` in Phase A5.
- **Operator authorization for mutating gates.** Every DDL apply, every data migration, every FK redirect, every writer flip, every cleanup-apply execution requires its own explicit operator instruction at the gate it governs. No gate cascades on the back of an upstream gate's authorization.
- **Tenant-results isolation.** `tbc_{slug}_dev` databases stay out of scope unless a tenant-explicit gate is authorized. Substrate-enforced via separate connections (`DATABASE_URL` vs `TENANT_DATABASE_URL`).

### On apparent product readiness

Product readiness may look low temporarily because BCF and MCF substrate readiness precedes runtime exposure. **This is acceptable.** The substrate is the foundation under the runtime — once BCF and MCF are clean, the existing UX can move quickly and product readiness should improve materially. The instinct to compress substrate work to "look more product-ready" is exactly the shortcut path this stance rejects.

### Future recommendations should optimize for

- **No shortcuts on substrate truth.** Schema boundaries, evidence integrity, no-synthetic enforcement, and operator authorization are inviolate.
- **Less fuss on paperwork where safe.** Combine docs where they cannot reasonably fail independently. Skip the PR ceremony for parked/backlog items. Skip over-polishing once content is directionally correct.

## Applicability

This stance governs all future planning gates, design DBCPs, apply DBCPs, closeout docs, and operator-authorized merges in scope of the BCF/MCF substrate work AND any adjacent work that touches the same boundaries (M-series M11..M14, tenant runtime MLS-15..MLS-25, retirement gates for legacy substrate). Recommendations that propose ceremony reductions in violation of the "Do NOT reduce" list should be rejected without re-litigation; recommendations that propose substrate shortcuts in violation of the "Direction (unchanged)" section should be rejected without re-litigation.
