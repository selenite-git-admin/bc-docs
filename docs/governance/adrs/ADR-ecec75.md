# Metric contract architecture — one contract per KPI, not per module

## Context

Implemented across bc-core (schema, taxonomy, 714 KPI definitions, generator, repository, service, controller) and bc-admin (API hooks, 3 catalog pages, contract detail KPI cards). Contract Registry page removed — three catalogs (Source, Business, Metric) replaced it.

## Decision

Each KPI/metric is its own metric-v1 contract (714 contracts), not bundled as indicators inside module-level contracts (80 contracts). A metric contract defines a computation: CO bindings (N:M per D021) + formula + temporal gate + thresholds + quality rules + output schema. The contract must support all 11 portal tabs (Overview, Settings, Contract, Schema, Output, Flow, Drivers, Actions, Quality, Logs, Snapshots). Domain mapping follows business taxonomy: domain=finance, subdomain=ar (not accounts_receivable as domain). Lineage/flow Mermaid diagrams are generated by bc-core from contract metadata, not composed by bc-portal.
