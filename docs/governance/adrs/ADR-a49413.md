# SDA Phase 1 implementation profile: CF semantic_family classification with multi-vendor advisory panel

## Authority

**DEC-a17d0f (Semantic Definitions Authority, SDA) is the governing authority for CF certification, lifecycle, gates, override matrix, AI-as-advisory posture, and evidence ledger.**

This ADR does **not** create an independent CF certification framework. It is a narrow implementation amendment within SDA Phase 1, operationalizing how the CF `semantic_family` value is sourced and the SDA G5 / G6 evidence is assembled when a CF is moved through the SDA's `proposed → reviewing → certified` transitions.

| Concern | Governed by | This ADR's role |
|---|---|---|
| Lifecycle states (proposed / reviewing / certified / deprecated / superseded / withdrawn; `is_archived`) | DEC-a17d0f §2 | Defers entirely; does not redefine |
| Deterministic gates G1–G8 (certification time) | DEC-a17d0f §4 | Defers entirely; G5 / G6 are the relevant gates for `semantic_family` |
| Reference-time gates G9 / G10 / G11 (authoring time) | DEC-a17d0f §4 + DEC-b7affa | Defers entirely |
| AI authority — advisory only, never autonomous | DEC-a17d0f §5 | Defers entirely; this ADR specifies *which* advisory inputs to collect |
| Override matrix (overridable G1 / G2b / G3 / G4; non-overridable G2a / G5 / G6 / G7 / G8 / G10 Class-A) | DEC-a17d0f §4 | Defers entirely |
| Evidence ledger `contract.certification_record` (DBCP-1c shape) | DEC-a17d0f §7 | Defers entirely |
| **Advisory panel composition for CF `semantic_family`** | **This ADR** | Specifies Gemini / OpenAI / Claude in fixed roles |
| **Advisory `cf_dedup` agent** | **This ADR** | Specifies the dedup advisory surface for G2b and downstream G10 signals |
| `LNodeSemanticService.refresh` hook after CF certification | DEC-804874 (D366) | Specifies the CF-side wiring D366 did not enumerate |

## Context

D366 (DEC-804874) introduced `canonical_field.semantic_family` to drive L2 BF↔CF compatibility verdicts via `CompatibilityFilterService`. The column was added (2026-04-21) but the workflow to populate it lives in SDA Phase 1 (DEC-a17d0f §9), which is in progress (Phase 0 complete @ bc-core `cb4b972`; Phase 1 commits `feat(sda)` / `feat(sda-4)` in flight as of 2026-05-15).

Today all 3,097 CFs are in pre-SDA `draft` state with `semantic_family IS NULL`. Until SDA Phase 1 ships the certification path (gates G1–G8 + DBCPs 1a–1f + bulk `draft → proposed` migration), no CF can be admitted to `certified`, the planned MC activation gate (Step 0) has nothing to admit, and `CompatibilityFilterService` cannot fire any rule.

The narrow question this ADR answers: **how is the `semantic_family` value sourced and the SDA G5 / G6 evidence assembled when a CF transitions `proposed → reviewing → certified`?**

## Decision

### 1. Multi-vendor advisory panel for CF semantic_family classification

When a CF enters `reviewing`, the SDA's gate evaluator invokes a mandatory advisory panel of three independent models in fixed roles:

| Participant | Role | Output |
|---|---|---|
| Gemini | Maker | Independently proposes a `semantic_family` from the master enum given the CF's name, function/subfunction, data_type, description, role |
| OpenAI / Codex | Checker | Independently evaluates whether the operator's submitted `semantic_family` is correct (accept / reject + rationale) |
| Claude | Writer / Moderator | Reads Maker proposal + Checker verdict + operator submission, writes the structured final rationale, flags any unresolved disagreement |

The panel produces advisory evidence written to `contract.certification_record.advisory_verdicts[]` per DEC-a17d0f §5. Each participant contributes one entry with `{surface, verdict, confidence, rationale, model_id, prompt_hash, timestamp, acknowledgement}`.

**The panel is advisory evidence, not a gate.** The deterministic SDA gates remain the authoritative pass/fail decision:

- **G5** (semantic_family populated and in master enum, CF only, **non-overridable**) — the structural binding decision for `semantic_family`.
- **G6** (data type / unit compatibility per `semantic_family` compatibility matrix, **non-overridable**) — the binding compatibility decision for the proposed family vs the CF's declared `data_type` / `unit_type_code`.
- **G11** (BF↔CF semantic-family compatibility, per DEC-b7affa, mostly non-overridable, fires at `cc_field_mapping` authoring time, downstream of CF certification) — covers the BF↔CF compatibility surface; CF certification under this ADR does not pre-empt or duplicate G11.

**Mandatory advisory means the panel must be attempted at every certify attempt.** AI verdicts never transition state, never block, never certify (DEC-a17d0f §5).

### 2. Panel-verdict-to-SDA-acknowledgement mapping

Per DEC-a17d0f §5's `green / amber / red / unverified` vocabulary:

| Panel outcome | Recorded as | Certifier action |
|---|---|---|
| All three return successfully, Maker proposes the submitted family, Checker accepts, Moderator accepts, no unresolved disagreement | `green` | No additional acknowledgement required for the AI evidence (deterministic gates still binding) |
| Disagreement (Maker ≠ submitted; or Checker rejects; or Moderator flags unresolved) | `amber` | Human acknowledgement with rationale ≥ 40 chars required before `reviewing → certified` (per SDA §5) |
| Strong reject (e.g., participants converge on a family incompatible with the operator's submission) | `red` | Certifier must acknowledge with rationale ≥ 40 chars to proceed; deterministic G5 / G6 still binding regardless |
| Any participant unreachable / malformed / timeout / confidence below threshold | `unverified` | Explicit `unverified` acknowledgement with rationale ≥ 40 chars per SDA §5; auto-spawned follow-up TSK tagged `sda-override` recorded in `override_followup_task_uid` |

This is the SDA's standard amber / red / unverified handling. This ADR does not introduce a new mechanic; it specifies how the panel's combined verdict maps into SDA §5's vocabulary.

### 3. `cf_dedup` advisory for G2b and G10-related signals

A complementary `cf_dedup` agent produces advisory evidence consumed by:

- **G2b** (normalized-form collision, overridable per SDA §4) — the agent surfaces semantically-near CFs that differ at the byte level but match on normalization (`lowercase + trim + replace(non-alphanumeric, '_')`). Operator may proceed via G2b's standard override path (rationale + follow-up TSK).
- **G10** (Meaning-once on `cc_field_mapping`) — operates downstream when a `cc_field_mapping` references this CF; `cf_dedup`'s output during CF certification informs early operator review of likely future G10 collisions.

`cf_dedup` output is recorded as advisory in `advisory_verdicts[]` alongside the classifier panel outputs. It is not a binding gate on its own.

### 4. Operational posture under the SDA override matrix

This ADR does **not** introduce a stricter override stance than SDA's default. SDA's matrix applies unchanged:

- Non-overridable gates (G2a, G5, G6, G7 BF-only, G8 CF-only, G10 Class-A): non-overridable; failure means the CF stays in `proposed` or returns to author per SDA's state-transition contract.
- Overridable gates (G1, G2b, G3, G4): SDA's override mechanic — rationale ≥ 40 chars + auto-spawned follow-up TSK tagged `sda-override`.
- AI advisory acknowledgements (amber / red / unverified): SDA §5 — rationale ≥ 40 chars + follow-up TSK.

If a stricter operational posture is desired for this phase (e.g., "do not use discretionary override for CF `semantic_family` classification during the 3,097-CF backfill"), it must be filed as a separate amendment to DEC-a17d0f, not specified here.

### 5. Evidence — `contract.certification_record` (per SDA §7)

All columns populated per DEC-a17d0f's contract; this ADR does not introduce a D407-specific schema:

| Column | Role |
|---|---|
| `primitive_type` | `'canonical_field'` |
| `primitive_id` | `canonical_field.canonical_field_id` |
| `action_code` | Per SDA action codes: `'submit_for_review'`, `'certify'`, `'return_to_author'`, `'deprecate'`, `'supersede'`, `'withdraw'`, `'archive'`, `'unarchive'` |
| `from_state_code` / `to_state_code` | Per SDA six-state lifecycle |
| `gate_results_json` | `GateResult[]` per `bc-core/src/registry/semantic-definitions/gates.ts` (the shipped SDA Phase 1 shape). Ordered array; one entry per gate evaluated. **Not a D407-specific shape.** |
| `advisory_verdicts_json` | Array of `{surface, verdict, confidence, rationale, model_id, prompt_hash, timestamp, acknowledgement}` per DEC-a17d0f §5. For CF semantic-family classification, three entries (Gemini, OpenAI, Claude). For dedup, one entry (`cf_dedup` or its deterministic-transitional shadow). Surface identifies which advisory contributed. |
| `override_gate_code` / `override_rationale_text` / `override_followup_task_uid` | Populated by SDA when an overridable gate is overridden or an AI amber / red / unverified acknowledgement is recorded. **D407 does not exclude these columns; SDA's rules apply.** |
| `certifier_sub` / `certifier_role_at_action` / `certifier_email` | Per SDA Q6 |
| `supersedes_primitive_id` | Per SDA Q3 |

`contract.l_node_semantic_trace` remains the MC × variable × L-node chain-trace evidence (D366). It is **not** used for CF certification evidence.

### 6. D366 refresh hook after CF certification

When a CF transitions `reviewing → certified` via the SDA's state-transition service (`POST /api/semantic-definitions/cf/:id/certify`), the certification path invokes:

```
LNodeSemanticService.refreshAffectedMetrics(ccId)  // for every CC that maps this CF
```

via the existing fan-out at `l-node-semantic.service.ts:403`. Implementation: query `contract.cc_field_mapping WHERE canonical_field_id = :cfId` for distinct `canonical_contract_id` values, fire-and-forget per CC. Same pattern as `cc-onboarding.service.ts:212-226`. This completes the CF-side hook D366 (§Decision §1) specified but did not wire into a service path.

### 7. Bulk certification path

The SDA's certify path admits bulk operations. For the one-time backfill of 3,097 pre-SDA CFs (currently `draft` / `semantic_family IS NULL` per the 2026-05-15 audit), bulk discipline follows SDA's contract:

- Per-CF gate evaluation runs independently.
- Per-CF advisory panels (Gemini / OpenAI / Claude) run concurrently across CFs (N ≤ 5 default), sequentially within a CF.
- Per-CF overrides record per-CF rationale + per-CF follow-up TSK per SDA. No batch-level override.
- Atomic per CF (one `certification_record` row per CF, per attempt).

The 3,097-CF backfill is coordinated with SDA Phase 1 DBCP-1d (bulk `draft → proposed` migration).

### 8. Allowed `semantic_family` enum

Sourced from `bc-core/src/registry/semantic/compatibility-filter.service.ts:36-59` (24 values). Per DEC-a17d0f §7, `master.semantic_family` (DBCP-1a) will normalize this closed enum into a master table with per-family compatibility metadata. This ADR does not redefine the enum; the code file remains the canonical source.

## Consequences

**Operational:** the 3,097-CF backfill is mandatory before MC activations resume. SDA Phase 1 (gates G1–G8 + DBCPs 1a–1f + bulk `draft → proposed` migration) is the unblocking path. The advisory panel + `cf_dedup` are the AI prerequisites for that phase.

**Authoritative:** the participant-trust invariant has a real authority — the SDA. `ChainTraceEngine` (future) consumes the SDA's certification-state projection (`/api/semantic-definitions/projection/certification-state` per DEC-a17d0f §6) as the participant-trust input. No new authority is created by this ADR.

**Not affected:** already-active MCs are unchanged (SDA's reference-time enforcement is shadow-mode in Phase 2 with operator-flippable block flags, never retroactive). CC / OC / AC / SC paths unchanged. `SeedContextReadinessService` shape unchanged; it gains operational meaning once SDA Phase 1 ships certified CFs.

**No contradiction with SDA:** lifecycle, gates, AI authority, override matrix, evidence schema, and authority boundary all defer to DEC-a17d0f. Where the prior `proposed` revision of this ADR diverged on these dimensions, this final version is corrected (see Change Log).

## §10. Post-certify description_text remediation (added v3, 2026-05-15)

### Context

After a CF reaches `certified`, the row carries the operator-supplied `semantic_family`, `definition_standard`, and `standard_ref` written at propose-certification time. The `description_text` column is not part of the propose-certification payload and is not touched by the certify transition. CFs that were authored before SDA Phase 1 (which is all 3,097 existing CFs as of 2026-05-15) frequently carry `description_text=NULL`.

SDA G3 (definition presence and quality) is overridable — a certify can proceed when G3 fails as long as the operator records the override with rationale ≥ 40 chars and an auto-spawned follow-up TSK. This is the SDA-designed transitional acceptance path. But it leaves a recorded data debt that, under SDA v2 today, has no resolution path: `description_text` is not editable through any governed endpoint after the CF is certified, and direct SQL is forbidden by platform policy. The follow-up TSK becomes perpetually blocked.

This section closes that resolution gap with a narrow, single-column remediation endpoint authorized by DBCP-1n.

### Decision

**1. Mutability classification of `canonical_field` columns post-certify.**

The following columns are **immutable after certify** unless the CF is moved through SDA's supersession or deprecation lifecycle:

- `field_name` — primitive identity (G1, G2a, G2b)
- `semantic_family` — SDA binding column (G5, G6)
- `definition_standard` — SDA binding column (G4)
- `standard_ref` — SDA binding column (G4)
- `data_type` — SDA binding column (G6)
- `unit_type_code` — SDA binding column (G6)
- `function_code` — primitive identity (CC binding consumer)
- `subfunction_code` — primitive identity (CC binding consumer)
- `status_code` — only SDA lifecycle transitions mutate this

The following column **is editable after certify** through the §10.3 remediation endpoint only:

- `description_text` — G3 evidence metadata; non-binding for G4/G5/G6/G7/G8; participates only in the overridable G3 gate.

The `display_name` column is also non-binding but is out of scope for this amendment. A future amendment may extend remediation to cover it; the endpoint specified in §10.3 is intentionally narrow to `description_text`.

**2. Remediation does not re-certify.**

A description_text remediation event does NOT issue a new `certify` row. The CF's certified state is preserved. The original certify `certification_record` row, including any G3 override evidence, remains the durable record of the certify act and is not invalidated. The remediation event clears the data debt represented by the override; it does not erase the historical override.

After a successful remediation, the operator MAY re-evaluate SDA gates (via the existing `GET /sda/primitives/canonical_field/:id/evaluate-gates` endpoint) to confirm G3 now passes. The remediation event itself already persists post-update G1–G8 verdicts in its `gate_results_json`, so a separate re-evaluation call is a convenience, not a requirement.

**3. Remediation endpoint.**

```
POST /api/canonical-fields/:id/remediate-description
Body: { "descriptionText": "<non-empty string>" }
Auth: @PlatformOnly() JWT
```

Service contract:

| Precondition | Result |
|---|---|
| CF not found | 404 |
| CF `status_code != 'certified'` | 422 — remediation is post-certify only; pre-cert flows use proposeCertification or createField |
| `descriptionText` missing / null / empty / whitespace-only | 400 — non-empty required |
| `descriptionText` length < 10 chars AND no terminal punctuation (G3 sentence heuristic) | 400 — preview G3 client-side before submission |
| All checks pass | Atomic write (see §10.4): UPDATE canonical_field.description_text + INSERT certification_record |

**4. Audit ledger shape.**

The remediation event is recorded in the existing `contract.certification_record` table. DBCP-1n adds `remediate_description` to the action_code enum so the audit row is honest about what it represents (not a state transition; a content-only edit with G3-clearing intent):

| Column | Value |
|---|---|
| `primitive_type` | `'canonical_field'` |
| `primitive_id` | the CF id |
| `action_code` | `'remediate_description'` (added by DBCP-1n; closed enum) |
| `from_state_code` | `'certified'` |
| `to_state_code` | `'certified'` (no state change; explicit equality records the remediation semantic) |
| `gate_results_json` | Re-run of G1–G8 against the post-remediation row. Persisting the verdicts proves G3 now passes and the other gates still pass. |
| `advisory_verdicts_json` | `[]` — no AI advisory in this slice. |
| `override_*` columns | NULL — this is not an override of any gate. |
| `certifier_sub` / `certifier_role_at_action` / `certifier_email` | Cognito identity of the remediator. |
| `supersedes_primitive_id` | NULL — no supersession. |

The state-machine in `state-transition.ts` is intentionally bypassed for this insert. The remediation endpoint writes directly via `CertificationRecordRepository` (the existing single-write repo, separate from `PrimitiveLifecycleRepository`'s dual-write path). This is deliberate: the certify dual-write is the structural backstop for state changes; remediation has no state change, only a content edit + audit row.

**Transaction shape.** The CF row UPDATE and the certification_record INSERT MUST happen in a single Postgres transaction so a failed audit write rolls back the description_text update — the platform never carries a remediated row without an audit signature. The certification_record row is inserted **once**, after the post-update G1–G8 gate results are known and confirmed to include `G3.verdict='pass'`. Sequence within the transaction:

1. UPDATE canonical_field SET description_text = $1 WHERE canonical_field_id = $2 AND status_code = 'certified' RETURNING ...;
2. If 0 rows returned (concurrent state change), abort transaction → 422.
3. Run G1–G8 evaluation against the post-UPDATE row (read-only against the same transaction).
4. If G3.verdict ≠ 'pass' or any non-overridable gate now fails, abort transaction → 500 (internal-error; pre-check disagreed with post-write evaluation).
5. INSERT INTO certification_record (...) VALUES (..., gate_results_json = $verdicts);
6. COMMIT.

**5. Non-overridable.**

If G3 still fails after the remediation (e.g., the operator supplied a single-word `descriptionText` that fails the sentence heuristic), the remediation endpoint refuses at step 4 and the transaction aborts. There is no override path on the remediation endpoint itself — its purpose is to make G3 pass, not to record another fail.

**6. Forbidden patterns.**

- Direct SQL UPDATE of `canonical_field.description_text` — bypasses the audit trail and the G3 re-verification.
- Using `proposeCertification` to remediate — that endpoint is status-gated to `draft / proposed` and writes the SDA binding columns, not `description_text`.
- Bulk remediation — not in this amendment. Each CF gets its own remediation event with its own operator-supplied text. Bulk remediation may land as a separate amendment after the single-CF flow has shipped evidence on at least one CF.

## §11. BF SDA-evidence read predicate (added v4, 2026-05-15)

### Context

`contract.business_field.status_code='certified'` is held by 6,787 / 7,062 BFs in the registry, of which **6,779 carry no `contract.certification_record` row** (legacy bulk imports certified before SDA Phase 1 shipped). The projection (`status_code`) and the ledger (`certification_record`) disagree for these 6,779 primitives. DEC-a17d0f §1 names the ledger as the canonical state for certification, so the ledger is the right source of truth for any read that asks "is this BF SDA-trusted?".

This divergence was surfaced by the SES-594568 BF participant-trust canary audit on `invoice_hdr_total_amount` (field_id `019d6db2-6b7c-7a11-b8c9-69db0580a80e`): a BF that reads `status_code='certified'`, has zero `certification_record` rows, and supplies the source path for the already-promoted `mc__revenue_collection_rate`. The same shape applies to 6,778 other BFs — i.e., effectively the entire BF catalog.

### Decision (v5 widening)

A BF is **SDA-trusted** if and only if **both** of the following hold:

1. `business_field.status_code = 'certified'`
2. `EXISTS` a row in `contract.certification_record` where:
   - `primitive_type = 'business_field'`
   - `primitive_id = <the BF's field_id>`
   - **`action_code IN ('certify', 'remediate_bf_semantics')`**

Both action codes record the same trust-bearing signal — the BF satisfied SDA G1–G8 at a specific moment with operator-supplied evidence — arrived at through different paths:

- **`'certify'`** — normal SDA certification: a BF that went through `proposed → reviewing → certified` with G1–G8 evidence recorded at the transition.
- **`'remediate_bf_semantics'`** — legacy-certified BF uplift per §12. A BF that was certified pre-SDA (no `certify` row in the ledger) reaches SDA trust by having its four binding columns (`semantic_family`, `unit_type_code`, `definition_standard`, `standard_ref`) supplied by an operator and re-evaluated against G1–G8 in the remediation transaction. The remediation row carries the **full post-update G1–G8 verdict block** in `gate_results_json` — the same evidence shape SDA `certify` produces.

The two distinct action codes are retained intentionally — they record different real lifecycle acts (greenfield certify vs legacy-cert uplift) and the audit trail must preserve that distinction. A paired-`certify` row from `certified → certified` would be semantically wrong (the `certify` action is reserved for `reviewing → certified` lifecycle transitions); the predicate widening is the foundation-correct alternative.

Read paths that gate on BF participant trust MUST use this combined predicate. New code MUST NOT consult `status_code` alone for trust decisions. The column remains a denormalized projection that the SDA state-machine writes; readers requiring trust MUST consult the ledger.

### Scope of change in Phase 1

Phase 1 implements the predicate at **two read paths** (per §10 scope discipline — narrow, foundation-correct, no schema mutation):

- **`BusinessObjectService.approve` Gate 4 (`all_bfs_certified`)** — refuses BO approval when any member BF fails the predicate. Distinguishes two refusal buckets in the error message: BFs failing on `status_code` and BFs failing on missing SDA evidence.
- **`StandardFieldService.bulkCertify`** — processes BFs that fail the predicate through CR-QG-001 gates and writes a fresh `certification_record` row. Legacy-certified-without-evidence BFs become the natural target of bulk certify rather than being silently skipped.

Other read surfaces (`IntegrityService` projection, `ChainStatusService` structural walk, runtime metric evaluation, snapshots, the SDA gate-evaluation reader, the SDA projection service, the BF immutability gate at `standard-field.service.ts:152`) are deliberately unchanged in Phase 1. Phase 2 (future, separate amendment) widens adoption.

### Phase 1 predicate is intentionally simple

Phase 1 uses `EXISTS(certify)` because the current data carries **zero** `deprecate`, `supersede`, or `withdraw` rows on BFs (verified 2026-05-15 against `contract.certification_record WHERE primitive_type='business_field'` — 8 rows total, all 8 are `certify`, no terminal-transition rows on any BF). The simple predicate is therefore behavior-identical with the stricter reading on today's data.

Phase 2 may tighten the predicate to: **"the latest relevant lifecycle event on this primitive is `certify` and not a subsequent `deprecate` / `supersede` / `withdraw`."** That tightening becomes operationally relevant only when BFs start carrying terminal lifecycle rows. No urgency in Phase 1.

### What does NOT change

- Existing certified BOs remain certified. Phase 1 does NOT retroactively flip BO status. Re-approval of an existing BO would refuse under the new gate, but the previous approval stays durable.
- Existing active MCs remain active. No recompute. No deactivation. No snapshot invalidation.
- Runtime metric evaluation, `ChainStatusService.refreshChainStatus`, `Step 0 mc_input_cf_not_trusted`, and `SeedContextReadinessService` are untouched. They never gated on BF status in the first place.
- The BF immutability gate at `standard-field.service.ts:152` (`status='certified' → reject edit`) is intentionally **not lifted** in Phase 1. Lifting it on 6,779 BFs while they're still in the catalog risks accidental edits; bulk certify is the governed uplift path. Future amendments may reconsider.

### Forbidden patterns

- **Direct SQL UPDATE on `business_field.status_code`** to "fix" legacy BFs. The ledger is the source of truth; flipping the column without writing a ledger row creates the inverse drift.
- **A new boolean flag column on `business_field`** (e.g., `legacy_certified` or `needs_sda_certification`). Duplicates the signal that `certification_record` already provides; not foundation-correct.
- **Gating runtime metric evaluation on the new predicate.** Runtime consumes active MC versions and their chain; SDA evidence at the BF layer is a write-time concern. Adding the predicate at the runtime layer would re-cast 100% of producing metrics as untrusted overnight, which is not the intended posture.
- **Consulting `status_code='certified'` alone in new code** for any trust-bearing decision. Combined predicate or nothing.

### Phase 2 deferrals (not in scope; tracked as TSKs)

- Widen predicate adoption to `IntegrityService.getKpiIntegrity` projection (enrich `StandardFieldRow` with `sdaCertified: boolean`).
- Widen predicate adoption to `ChainStatusService` (after operator decision on whether structural chain trace should harden on SDA evidence).
- Widen predicate adoption to the SDA projection (`SemanticDefinitionsProjectionService`).
- Reconsider the BF immutability gate at `standard-field.service.ts:152`.
- Tighten the predicate to "latest relevant lifecycle event = certify" once BF terminal-transition rows appear.

## §12. Governed BF semantic metadata remediation (added v5, 2026-05-15)

### Context

§11 (v4) defined the BF SDA-trust read predicate but provided no path for legacy-certified BFs (6,779 today) to acquire SDA evidence. The bulk-certify path partially addresses BFs whose metadata is already sufficient, but BFs missing any of the four SDA-binding columns — `semantic_family`, `unit_type_code` (post-DBCP-1o), `definition_standard`, `standard_ref` — cannot complete a CR-QG-001 certify because the structural-gate path doesn't itself supply those values. The operator needs a governed entry point to **write the four values and produce SDA evidence in one transaction**.

This is the BF analog of §10 (CF description_text remediation) but materially larger:

- Four columns instead of one.
- All four are SDA-binding (G4 / G5 / G6), not G3-only.
- The remediation IS the SDA evidence write — the operator-supplied values must themselves pass G1–G8 post-update for the remediation to commit.

The canary that surfaced this — `invoice_hdr_total_amount` — is structurally typical: certified pre-SDA, mapped to one already-certified CF (`total_billed_revenue`) plus 79 uncertified CFs, source-path through an SAP `TYPE_SD_S_MAP` print-program structure, no `semantic_family` / `unit_type_code` / `standard_ref` on the BF row.

### Decision

#### 1. Action code

**`'remediate_bf_semantics'`** — added by DBCP-1p (sibling to DBCP-1n which added `'remediate_description'`). Closed enum widening; no new table.

Semantics: "content-only edit of BF SDA-binding columns on an already-certified BF, with post-update G1–G8 evidence carried in `gate_results_json`."

#### 2. Mutability classification

| `contract.business_field` column | Mutable post-certify? | How |
|---|---|---|
| `field_id` | No (PK) | — |
| `name` | No | (deprecate + reseed) |
| `business_function` | No | (deprecate + reseed) |
| `object_class` | No | (deprecate + reseed) |
| `property` | No | (deprecate + reseed) |
| `representation_term` | No | (deprecate + reseed) |
| `data_type` | No | (deprecate + reseed) |
| `status_code` | Yes, via SDA lifecycle | SDA state-transition |
| `pii_classification` | Deferred (out of v5 scope) | — |
| `definition` (text) | Deferred (§13-class candidate; sibling of §10 for BF, if needed) | — |
| **`semantic_family`** | **Yes, via §12** | `POST /:id/remediate-semantics` |
| **`unit_type_code`** | **Yes, via §12** | `POST /:id/remediate-semantics` |
| **`definition_standard`** | **Yes, via §12** | `POST /:id/remediate-semantics` |
| **`standard_ref`** | **Yes, via §12** | `POST /:id/remediate-semantics` |

#### 3. Endpoint

```
POST /api/standard-fields/:id/remediate-semantics
Body: {
  semanticFamily:     string,           // required; must be in D366 SEMANTIC_FAMILY_ENUM
  unitTypeCode:       string | null,    // required when the family's G6 matrix carries compatible_unit_types != null; must FK to master.master_unit_type
  definitionStandard: string,           // required; must be in DEFINITION_STANDARDS_SET
  standardRef:        string | null,    // required when definitionStandard is external (non-BARECOUNT)
}
Auth: @PlatformOnly() JWT
```

Service contract:

| Precondition | Result |
|---|---|
| BF not found | 404 |
| BF `status_code != 'certified'` | 422 — remediation is for already-certified BFs only |
| Any required body field missing | 400 |
| `semanticFamily` not in D366 enum | 400 |
| `definitionStandard` not in DEFINITION_STANDARDS_SET | 400 |
| External `definitionStandard` with empty `standardRef` | 400 |
| Family's G6 matrix requires unit_type but `unitTypeCode` is null/empty | 400 |
| Family's G6 matrix forbids unit_type but `unitTypeCode` is provided | 400 |
| `unitTypeCode` provided but not present in `master.master_unit_type` | 400 (caught early; DB FK is structural backstop) |
| All checks pass | atomic transaction (see §12.4) |

#### 4. Transaction shape

Single Postgres transaction; one UPDATE + one INSERT; no state-machine call; the state-machine is deliberately bypassed because there is no state transition.

1. `UPDATE contract.business_field SET (semantic_family, unit_type_code, definition_standard, standard_ref, updated_at) = ($1, $2, $3, $4, NOW()) WHERE field_id = $5 AND status_code = 'certified' RETURNING ...;`
2. If 0 rows returned → abort transaction → 422 (concurrent state change — BF moved to deprecated/superseded/withdrawn between pre-check and write).
3. Re-evaluate G1–G8 against the post-UPDATE row (loaded fresh in the same transaction or projected as a pure computation over the current row + the four new values).
4. If any non-overridable gate fails (G2a / G5 / G6 / G7 / G8 / G10 Class-A) → abort transaction → 422 with the failing gate names.
5. If any overridable gate fails (G1 / G2b / G3 / G4) → **abort transaction → 422**. The §12 endpoint accepts no override. Operator fixes inputs and re-runs.
6. `INSERT INTO contract.certification_record (...) VALUES (..., action_code='remediate_bf_semantics', from_state_code='certified', to_state_code='certified', gate_results_json=$verdicts, advisory_verdicts_json='[]', override_*=NULL, certifier_*=$identity);`
7. `COMMIT.`

The remediation row is inserted **once**, after post-update G1–G8 verdicts are known and confirmed to pass.

#### 5. No override path

If any G1–G8 fails on the projected post-update state, the transaction aborts and the endpoint refuses with the gate names. The operator's remediation is to amend the four input values and re-run. There is no override mechanism on `/:id/remediate-semantics`.

This is the same posture as §10 (`/:id/remediate-description` refuses on G3 failure with no override). Rationale: the four mutable columns are SDA-binding (G4/G5/G6 are non-overridable per DEC-a17d0f §4.5). Allowing override would defeat the purpose of the remediation — the operator would be claiming SDA trust on values the gates reject.

#### 6. Audit row shape (final)

| Column | Value |
|---|---|
| `primitive_type` | `'business_field'` |
| `primitive_id` | `business_field.field_id` |
| `action_code` | `'remediate_bf_semantics'` |
| `from_state_code` | `'certified'` |
| `to_state_code` | `'certified'` |
| `gate_results_json` | post-update G1–G8 `GateResult[]` — all required gates passing |
| `advisory_verdicts_json` | `[]` (no AI advisory in this slice) |
| `override_*` columns | `NULL` |
| `certifier_sub` / `certifier_role_at_action` / `certifier_email` | Cognito identity of the remediator |
| `supersedes_primitive_id` | `NULL` |

One row per remediation. No paired `certify`. `from_state_code = to_state_code = 'certified'` records the remediation semantic (no lifecycle transition; content edit + evidence).

#### 7. Effect on §11 trust predicate

After a successful remediation:

- BF `status_code` stays `'certified'` (unchanged).
- A new `certification_record` row exists with `action_code='remediate_bf_semantics'`.
- The §11 v5 predicate (`action_code IN ('certify', 'remediate_bf_semantics')`) matches.
- The BF transitions from legacy-certified-no-evidence to SDA-trusted under §11 in one atomic transaction.

#### 8. Forbidden patterns

- Direct SQL UPDATE on any of the four SDA-binding BF columns. The ledger is the source of truth; flipping columns without an audit row creates inverse drift.
- Using `remediate_bf_semantics` to manufacture a state transition or to write a `certify`-equivalent. The action is content-only by design.
- Overriding any gate failure on the §12 endpoint. No override path exists; refuse and let the operator iterate.
- Bulk remediation. Each BF gets its own remediation event with its own operator-supplied values. Bulk remediation may land in a future amendment after the single-BF flow has shipped evidence on at least one BF (canary on `invoice_hdr_total_amount`).
- Using §12 to write `pii_classification` or `definition`. Those columns are out of v5 scope; if they need governed remediation, a future amendment defines that path.

### Prerequisite DBCPs

- **DBCP-1o** — adds `business_field.unit_type_code` (nullable text, FK → `master.master_unit_type(unit_type_code)`) + seeds three missing duration unit rows (`hours`, `minutes`, `seconds`). BF only; CF `unit_type_code` FK parity deferred (`number` divergence is a separate audit, tracked as a follow-up TSK).
- **DBCP-1p** — extends `contract.certification_record.action_code` CHECK with `'remediate_bf_semantics'`.

Both DBCPs are authored alongside this amendment; neither is applied until explicit operator approval per the Database Change Protocol.
