# MCF trust-chain resolution — concept-mediated MC→CC derivation, PE-MC-11 resolvability evidence, honest chain-green semantics

## Context

Chain-green was unreachable because the trust strip's SC/AC/OC/CC nodes were hardcoded not_declared — correct at build time (no CC/OC carried concept identity), now obsolete: D430/D431 landed field-level business_concept_id on CC field_selection and OC field_mappings, and the live ARPI v2 slices carry them. The C0 study found the architecture ~80% pre-decided: D430 forbids MC-side mappings ("not in a side mapping table, not in the Metric Contract or MCF"; "MCF does not consume runtime mappings"), DEC-29c324 fixes N:1 cardinality, DEC-02f5a9 makes concepts the only shared key. A read-side conjunction of emitted declarations is Invariant-IV/VI-clean (joins explicit references, infers nothing, evaluates nothing) and is exactly D438-D4's published wording; emitted PE-MC-11 rows give gates and the lifecycle strip evidence per Invariant VI. The live substrate supplied both proofs: billing_volume fully resolves; ARPI exposes a genuine supply-side mis-declaration (registry definitions are mutually exclusive: posted amount "distinct from net"; net amount "distinct from posted"; SAP NETWR = net value before tax) — fixing it on the supply side (Packet-1 A) preserves meaning-evaluated-once, while bending the MC (rejected B) would encode the error. Advisory-then-required (Packet-2 D) sequences gate hardness with declaration completeness and avoids both advisory-forever drift and hard-gating before the strip can explain failures. Codex independent audit of the arc recommended ADR-locking these semantics before implementation.

## Decision

Seven operator-locked decisions (2026-06-10, SES-1480d8), grounded in the C0 study + two decision packets (bc-core/scripts/audit-output/c0-mc-cc-declaration-study-2026-06-10.md and c0-decision-packets-arpi-reconciliation-pe-mc-11-2026-06-10.md):

1. MC→CC is CONCEPT-MEDIATED, never stored. The Metric Contract declares variable→business_concept (mcf.metric_variable_binding.bound_business_concept_id, exists); the Canonical Contract declares field→business_concept (D430/DEC-a6258b field_selection, exists); resolution = concept_id equality, deterministic under one active CC per grain entity (D430 C1). No CC reference is ever added to MCV body, variable bindings, or package signature.

2. The two-strip trust view's CHAIN strip is a READ-SIDE CONJUNCTION of emitted declarations (MCV bindings ⋈ CC field_selection ⋈ OC field_mappings) — the operational form of D438-D4's published wording ("OC field map covers CC selection; CC selection covers MC variable bindings"). Per-node green predicates: MC = active + M14 cert; BCF = bound concepts/entity active; CC = exactly one active CC for the grain entity AND every AST-referenced input variable concept declared by it (>1 active CC = RED, ambiguity is an integrity failure); OC = CC concepts provable-from-source via D431 O↔C concept equality; SC/AC = OC declares its source/admission linkage (grey until C2). Rollup green = all six nodes green + activation evidence (unchanged).

3. PE-MC-11 (canonical resolvability) is added to the M13 eligibility ledger: (i) determinism (exactly one active CC for grain entity), (ii) MC→CC AST-referenced concept coverage, (iii) CC→OC provability. Emitted as evidence rows like PE-MC-1..10; requires a one-line vocabulary DBCP on mper_pe_check_code_chk (verified live: CHECK currently pins PE-MC-1..10).

4. ARPI reconciliation = Packet-1 "C now + A next". C now: C1 ships with ARPI honestly AMBER (CC node detail names the gap: MCV numerator binds net amount 342669f7; active CC/OC slice declares NETWR as posted amount 1a2ac2f2 — a supply-side mis-declaration per the registry's own mutually-exclusive definitions; SAP NETWR is net value before tax). A next: author successor OC-v2/CC-v2 versions correcting NETWR→net amount 342669f7 via the governed onboarding services (SERVICES-ONLY, zero MC edits). REJECTED B: superseding the ARPI MCV to posted amount would encode the supply error into metric meaning. REJECTED D: lineage-aware concept matching — D430 pure concept identity stands.

5. PE-MC-11 policy = Packet-2 "D — advisory at C1 with pre-committed hardening trigger at C2". Advisory while C2 source-linkage is incomplete (coverage/provability gaps → OPERATOR_REVIEW); becomes REQUIRED for M14 activation when the C2 OC source-linkage requirement lands; ambiguity (>1 active CC per grain entity) is ALWAYS REJECT, including during the advisory phase. No second ADR needed for the flip — this decision pre-commits it.

6. First proofs: billing_volume is the first POSITIVE C1 proof (2/2 variable concepts covered by the active CC on the same grain entity e3963e45; CC 3/3 provable from the OC; BCF green) — proving BCF/OC/CC node derivation; ARPI is the NEGATIVE proof (honest amber from the real semantic mismatch). SC/AC remain grey for both until C2 source linkage. billing_volume's own M13/M14 progression is NOT authorized here — separate operator call.

7. ANTI-co_bindings clause (permanent): no legacy-style per-field or per-contract MC→CC authority is ever reintroduced — not in MCF substrate, not in read models as stored state, not in the bridge writer as governance input. The legacy contract.metric_contract co_bindings/metric_binding projection remains write-only output for the legacy runtime (D436: never read back as authority).

Implementation phasing (decided sequencing, NOT implemented by this ADR): C1 = read-side derivation in mcf-read service + detail builders + bc-admin detail strings (zero DDL; display-version semantics, not is_current — billing_volume's draft MCV carries is_current=false); C2 = PE-MC-11 vocabulary DBCP + evaluator extension + M14 policy flip + OC v2 authoring requires source_contract_id/reader linkage (columns exist; successor versions for existing v2 OCs, no backfill); then the Packet-1-A OC/CC successor authoring batch.
