# Operational tables moved from Platform DB to Tenant DB

## Context

Partially superseded by D168. D173 moved 8 tables from platform operational to tenant DB. The discovery tables (4) and execution run tables remain in tenant DB (correct). But connection tables (connection, connection_config, connection_check) return to platform DB runtime schema per D168. admission_run stays in tenant DB (execution data).

## Decision

8 tables formerly in platform operational schema (connection, connection_config, connection_check, admission_run, discovery_scan, discovered_object, discovered_field, discovery_diff) moved to tenant DB. The operational schema is removed from platform. Platform DB drops from 64 to 56 tables. Tenant DB grows from 13 to 24 tables. Cross-DB references (connector_id, reader_id, source_version_id) become soft references (UUID stored, no FK constraint). connection_id removed from reader_flavor (wrong level of binding — flavor is a platform template, connection binding happens at tenant level).
