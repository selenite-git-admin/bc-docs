# Section renames + scope cleanup: Onboarding, Operations; drop Demo Execution; move Clean Slate Migration to Upgrade and Migration

## Context

Spine cleanup follow-up to DEC-ce6e2b (Operating Model rename), DEC-376587 (Implementation rename), and DEC-9c58c6 (article drop). Founder reviewing the spine flagged three concerns: (a) "Governed Authoring" reads jargon-y, the "Governed" qualifier is asymmetric (every section in the documentation is governed; saying it explicitly only on one section implies the others are not), and "Authoring" connotes literal text/document writing while only some chapters fit that frame. Considered alternatives: Procedures (rejected as medical-flavored), Practices (rejected as passive), Authoring (rejected as too literal-writing-focused), Playbooks (rejected for sports-metaphor origin), Provisioning (rejected as IT-narrow), Composition / Curation / Stewardship (rejected for register noise). Onboarding chosen because the dominant verb across the 17 chapters is bringing new artifacts into the platform: source onboarding, BF/BO onboarding, contract onboarding (creation + registration), metric registration, multi-standard onboarding, tenant onboarding. (b) "Platform Operations" carries a redundant "Platform" prefix; after renaming "The Platform" → "Implementation" the "Platform" qualifier was the only place the word still appeared as a section-name modifier. Bare "Operations" reads cleanly, matches the no-article register, and the faint root-share with "Operating Model" was already addressed in the Operating Model section opener as intentional ("this section names the spec, that section names the activity"). (c) Two chapters in Governed Authoring did not fit the section's theme. Demo Execution is sales/pre-sales material (demo scripts, prospect talk tracks, walkthrough plans) — that audience is sales staff, not platform architects, developers, or auditors, and the artifact belongs in a sales toolkit repo, not platform documentation. The technical demo-tier tenant setup that platform documentation does need to cover is already part of Tenant Onboarding's tier coverage per DEC-324d9e. Clean Slate Migration is operational (platform-DB reset, source catalog export) — it was authored during the v3 cutover (DEC-3395bc) for one-time use but has enduring dev/test/disaster-recovery value, and that value belongs in Operations under Upgrade and Migration alongside platform version upgrades and contract migration. Both removals tighten Onboarding to 15 chapters all clearly onboarding-shaped. After this ADR the spine reads: Foundation, Operating Model, Implementation, Onboarding, Operations, Compliance — all uniform no-article register, each section name distinct in role and free of jargon.

## Decision

Four structural moves applied together. (1) Section "Governed Authoring" is renamed to "Onboarding" — drops the asymmetric "Governed" qualifier (every section is governed) and replaces "Authoring" with the dominant theme of the 17 chapters: registering, creating, seeding, and onboarding new artifacts into the platform. (2) Section "Platform Operations" is renamed to "Operations" — drops the redundant "Platform" prefix (every section in the documentation is about the platform). (3) Chapter "Demo Execution" is dropped from v3 entirely — it is sales/pre-sales material (talk tracks, prospect walkthroughs) not platform documentation; the demo-tier technical setup is already implicit in Tenant Onboarding's tier coverage per DEC-324d9e. (4) Chapter "Clean Slate Migration" moves from the Onboarding section to Operations and folds into Upgrade and Migration — platform-DB reset and catalog export is migration-shaped, not onboarding-shaped, and Upgrade and Migration already covers platform-version upgrades, contract migration, and tenant schema evolution. After the moves: Onboarding has 15 chapters; Operations has 8 chapters (unchanged count: Clean Slate folds into the existing Upgrade and Migration chapter rather than landing as a new 9th). Folder names will be `docs/onboarding/` and `docs/operations/` when chapters are drafted; both sections currently queued so renames are pure label work.
