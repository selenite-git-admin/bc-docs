# Schema contracts managed via DB + Admin UI, not repo seed files

## Context

Schemas have governance lifecycles (draft → review → released) that belong in the DB with approval workflows, not as static repo artifacts. The existing approval API and governance state machine already support this — the admin UI makes it accessible to the platform team.

## Decision

Schema registry contracts (admission, canonical, etc.) are runtime domain configuration. System of record is the DB (registry.schema + registry.schema_version). Repo seed JSONs are development bootstrap only. Production schemas created/versioned via bc-admin Schema Registry UI or CLI tooling.
