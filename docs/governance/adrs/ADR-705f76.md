# D353 — bc-portal Naming Convention

## Context

Navigation audit (SES-92dc4e) found **four naming generations coexisting** with no single source of truth:

| Old naming | Current naming | Still in code |
|------------|---------------|---------------|
| `DataControls` | `DataInfra` / `Data Sources` | `DataControlsConnectionsPageRefined`, `DataControlsRetentionPage` |
| `Dataspace` / `GoldenData` | `Canonical Data` | `GoldenDataOverview`, `dataspace.service.ts`, `/pages/dataspace/` |
| `CFO` / `CGO` / `CMO` / `COO` Pack | Finance / Sales / Marketing / Operations Metrics | `CFOPackPage`, `CGOPackPage`, `CMOPackPage`, `COOPackPage` |
| `FinancePage` | (current) | `FinancePage`, `FinancePageB` (two variants, unclear which is authoritative) |

Additional symptoms:
- Same component (`DataControlsConnectionsPageRefined`) used in **two routes** with different nav labels
- `"Refined"`, `"New"`, `"B"`, `"Enhanced"` suffixes scattered across filenames
- Four separate name layers diverged per page:
  ```
  URL path:         /data-sources/registered
  Component name:   DataControlsConnectionsPageRefined
  Nav label:        "Data Sources"
  API endpoint:     /api/t/connectors-catalog
  ```
- No centralized nav config — items hardcoded in `Navigation.tsx` (8 arrays) and `LeftSidebar.tsx` (page-name arrays for section detection)

Each rename attempt has updated routes + nav labels but left component names and service names behind. Developers see four different words for the same screen. This compounds every time a concept is renamed.

## Decision

### Rule 1 — The Triplet

For every user-facing page, **three names must align**:

| Layer | Source | Example |
|-------|--------|---------|
| **URL leaf** | `AppRouter.tsx` route path final segment | `registered` (in `/data-sources/registered`) |
| **Component name** | Component file name + exported symbol | `DataSourcesRegisteredPage` |
| **Nav label** | What the user sees in the menu | "Registered Sources" |

Rules:
- Component filename matches the exported symbol: `DataSourcesRegisteredPage.tsx` exports `DataSourcesRegisteredPage` (or as default)
- Component name ends in `Page` (for routable components) or the component type suffix (`Card`, `Table`, `Chart`) for non-routable
- URL leaf is the kebab-case of the component name minus `Page` suffix: `DataSourcesRegisteredPage` → `registered` (the parent segment `data-sources` is the domain)
- Nav label is the user-friendly phrasing: can differ in wording but must refer to the same concept

**Service/API endpoint names are internal** (in `src/services/` and `src/adapters/`) and may differ from the triplet. Translation happens in adapters. Document the mapping in the service file's JSDoc.

### Rule 2 — Forbidden suffixes and prefixes

The following are banned in page component names:

- `Refined`, `New`, `Enhanced`, `Improved`, `V2`, `V3`, `Old`, `B`, `A` (except when genuinely domain-meaningful, e.g., `PackA` in a game)
- `Refactored`, `Migrated`, `Updated`

Version control is git. Filename should not carry iteration history.

**Exception:** `Legacy` is permitted as a prefix for pages deliberately kept in the `/legacy` index but no longer in main nav (see Rule 4).

### Rule 3 — Exec-title codes are internal

Executive title codes (CFO, CGO, CMO, COO, CIO, CHRO) are domain codes used in database, APIs, and internal identifiers. They are **never** user-facing in:

- Component names: use `FinanceMetricsPage`, not `CFOPackPage`
- Nav labels: use "Finance Metrics", not "CFO Pack"
- Route paths: use `/kpi-store/finance`, not `/kpi-store/cfo-pack`

If an exec-title code is needed in a domain model or prop, use it explicitly: `function: 'cfo'` is fine as a typed enum value.

### Rule 4 — Legacy handling

Pages removed from main navigation but not yet retired:

- Keep the component file and route in `AppRouter.tsx`
- Add `// legacy: will-restore` comment above the route
- Surface in `/legacy` index (`LegacyPagesPage.tsx`) as "Temporarily parked — will restore at [target location]"
- Pages are **never silently deleted** — explicit retirement requires:
  1. Decision to retire (comment in DevHub task or ADR)
  2. Remove route, remove component file, remove registry entry
  3. Add redirect route for the old URL if bookmarks exist

The 72 existing legacy redirect routes in `AppRouter.tsx` (old URL → new URL) are a separate concern — they are URL-compatibility shims, not parked pages. They remain until bookmark expiration.

### Rule 5 — Nav manifest (future implementation)

A single `src/config/navigation.ts` file will be the SSOT for nav entries:

```ts
export interface NavItem {
  id: string;                               // D353 canonical, e.g. "data-sources-registered"
  label: string;                            // user-facing, e.g. "Registered Sources"
  route: string;                            // URL path, e.g. "/data-sources/registered"
  componentName: string;                    // e.g. "DataSourcesRegisteredPage"
  section: string;                          // nav section, e.g. "data-sources"
  status: 'active' | 'legacy' | 'hidden';  // legacy = will-restore per Rule 4
  icon?: LucideIcon;
  description?: string;                     // shown in mega-menu or tooltip
  registeredIn?: string;                    // ComponentRegistry ID (D352)
}

export const NAV_MANIFEST: NavItem[] = [
  // ... all nav entries here
];

export const ACTIVE_NAV = NAV_MANIFEST.filter(n => n.status === 'active');
export const LEGACY_NAV = NAV_MANIFEST.filter(n => n.status === 'legacy');
```

`Navigation.tsx` and `LeftSidebar.tsx` will consume this instead of hardcoding item arrays. `AppRouter.tsx` can validate routes against the manifest at build time (type-level check that `route` exists).

**Implementation deferred to Phase B.** The manifest is built during the rename pass — defining the manifest while simultaneously renaming is efficient.

## Options Considered

### Option A: Triplet rule + nav manifest (chosen)

One canonical name per page, enforced by pattern + manifest + SOP. Migration during rename pass.

### Option B: Accept divergence, document the mapping (rejected)

A "Rosetta stone" document mapping URL ↔ component ↔ nav ↔ API for each page. Rejected: perpetuates confusion; encourages more drift; creates N-way maintenance burden.

### Option C: Auto-generate names via codegen (rejected)

A script that generates component files from a YAML spec. Rejected: too brittle; no override for edge cases; couples filename to tooling.

### Option D: Per-rule enforcement only, no manifest (rejected)

Enforce triplet via ESLint and SOP but skip the central manifest. Rejected: `Navigation.tsx` still hardcodes labels; drift returns via nav edits.

## Consequences

### Positive

- **Single name per page** across URL, component, nav — grep-able.
- **Easier onboarding** — new developers find what they need by name.
- **Rename becomes atomic** — change the triplet once in the manifest, rest follows.
- **Legacy vs retired is explicit** — no silent orphans.

### Negative

- **Rename pass required (Phase B).** ~20 components to rename, all imports updated, all route paths normalized.
- **External bookmark breakage** on renamed routes. Mitigation: legacy redirect routes stay in `AppRouter.tsx` until bookmarks expire.
- **Manifest-first developer discipline** — adding a page requires manifest edit, not just a new file. SOP enforces this.

### Risks

- **Incremental rename drift.** Mitigation: Phase B is a dedicated migration, not opportunistic.
- **Triplet ambiguity** for nested or shared screens (e.g., a card component used in multiple pages). Mitigation: only Page components follow the triplet; non-page components follow the component-type naming convention (`Card`, `Table`, etc.).
- **Nav manifest becomes a merge-conflict hotspot.** Mitigation: group manifest entries by section; accept merge-conflict cost vs nav drift cost.
