# PII Classification at Source Field Level — Observe at Source, Propagate Through Chain

## Context

D285 (DEC-bd5492) defines a `pii_field_registry` table in the operations schema to catalog PII-bearing columns across platform and tenant databases. This registry is a **post-hoc discovery catalog** — it must be manually maintained or periodically scanned to stay current, creating drift risk.

BareCount already touches every source field during business alias assignment (Source Registration SOP Step 3). This is the natural point to classify PII — at the source, when the field is first given business meaning.

The execution model demands "observe at source, carry forward immutably." PII classification should follow the same principle: classify once at the source field, propagate through the contract chain, never duplicate.

## Decision

### Source Field Classification (two new columns)

Add to `source.source_field`:

| Column | Type | Values | Default |
|--------|------|--------|---------|
| `data_classification_code` | `text` | `public`, `internal`, `confidential`, `restricted` | `internal` |
| `pii_category_code` | `text` | `none`, `direct_pii`, `indirect_pii`, `sensitive` | `none` |

**Vocabulary alignment:** `data_classification_code` matches the screen registry and ISO 27001 information classification (A.8.2.1). `pii_category_code` aligns with D285's sensitivity tiers (T1=direct_pii, T2=indirect_pii, T3=indirect_pii, T4=sensitive, T5=structural/none).

### When Classification Happens

Classification happens **only for fields selected as business alias candidates** — the subset of source fields that receive a `business_alias` during Source Registration SOP Step 3 ("Add Fields"). These are the fields entering the contract chain; unaliased technical fields (SAP internal keys, change pointers, client codes) remain at their harmless defaults (`internal`/`none`) and never reach contracts.

When a field gets its `business_alias`, the operator also classifies it. This is zero marginal cost — you're already reviewing the field's business meaning to assign the alias. Classification is part of the same decision: "what does this field mean, and how sensitive is it?"

AI verification (bc-ai field-verify) can **suggest** classification based on field name patterns (e.g., `email`, `name`, `phone` → `direct_pii`; `created_by`, `changed_by` → `indirect_pii`). The operator confirms or overrides.

### Propagation (Derived, Not Stored)

Classification propagates through the contract chain by **derivation**, not by copying:

| Level | Classification Rule |
|-------|-------------------|
| **Source Field** | Directly classified (`data_classification_code` + `pii_category_code`) |
| **Business Object** | Highest classification of any bound source field |
| **Business Field** | Inherits from its source field binding |
| **Observation Contract** | Highest classification of any field in its field selection |
| **Canonical Object** | Highest classification of any bound observation |
| **Metric Contract** | Highest classification of any CO binding variable |

These are computed on read (SQL view or service-level derivation), never stored as separate columns on contract tables. One source of truth per value (DB Rule #4).

### Tenant Override via contract_binding

Platform sets the **conservative default** (classification at source_field level).

Tenant `contract_binding` may:
- **ELEVATE** classification (e.g., `internal` → `confidential` for a field that contains PII in their specific context)
- **NEVER DOWNGRADE** (platform floor is enforced)

This handles the case where "employee name" is PII for an HR source but "company name" isn't — the tenant context determines whether elevation is needed.

### Relationship to D285 pii_field_registry

D285's `pii_field_registry` becomes a **derived view**, not a manually maintained table:

1. Source fields with `pii_category_code != 'none'` auto-populate the registry
2. Contract chain propagation identifies which tenant DB columns (in `record.*` typed tables) contain PII
3. The registry is rebuilt/validated on demand, not maintained separately
4. Nullification operations (D285) use the registry to find all PII-bearing locations for a given data subject

This eliminates the drift risk identified in D285: "PII missed in JSONB columns → mitigated by periodic audit scan." With source-level classification, PII is known before data ever enters the system.

## Options Considered

### Option A: Post-hoc PII registry only (D285 as-is)
Separate `pii_field_registry` maintained by periodic scans. Rejected as primary approach — reactive, drift-prone, doesn't leverage the source catalog's existing field review workflow.

### Option B: Classify at source_field, derive everywhere (chosen)
Tag PII at the point of business alias assignment. Derivation through contract chain. D285 registry becomes a derived artifact. Chosen — proactive, zero marginal cost, consistent with execution model.

### Option C: Classify at business_object_field level
Tag at the BO field rather than source field. Rejected — BOs are abstractions over source fields, classification should be at the most granular level. BO-level classification is derived from source field classification.

## Consequences

### Positive
- PII known before data enters the system — proactive, not reactive
- Single point of classification — no manual registry maintenance
- AI can suggest classification during field registration (pattern matching on field names)
- D285 nullification scope is automatically correct — derived from authoritative source
- Tenant-specific PII handling via contract_binding elevation
- Auditors can trace any PII field back to its source classification decision

### Negative
- Adds 2 columns to source_field table (minimal schema impact)
- Requires backfill for existing ~7,000 source fields (AI-assisted batch classification)
- Derivation queries add modest complexity to contract-level classification lookups

### Implementation Notes
- Backfill existing fields: bc-ai can classify based on field name patterns + SAP data element semantics
- Default `internal`/`none` is safe — errs on the side of no PII, which is correct for most SAP technical fields
- Update Source Registration SOP to include classification step in "Add Fields"
- Update bc-admin AddFieldsFromSeedCatalog to include classification UI
