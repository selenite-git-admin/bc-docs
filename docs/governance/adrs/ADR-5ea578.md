# Metric Evaluation Boundary — Unification & Wiring (governed runtime promotion, run object, transactional idempotent act)

## Context

The study (verified live 2026-07-01; SQL transcript in study section 10) found the metric evaluation boundary essentially unbuilt for the governed metrics:
(a) Two engines. The governed runtime (correct: reads mcf.*, real temporal-gate selection, emits GateSelection evidence) is TEST-ONLY — evaluateGovernedMetric has no production caller. The production-wired legacy engine reads the envelope substrate contract.metric_contract_version, which is EMPTY (0 rows live). All 52 governed metric versions (51 active, 1 review) live in mcf.*.
(b) Greenfield tenant. The sole live tenant tbc_pilot1_dev has no fact.ms_*/co_*/so_* tables provisioned and progression.metric_evaluation = 0 rows. So no live metric is currently evaluated or persisted — the boundary can be built correctly from first principles rather than retrofitted onto a load-bearing engine.
(c) Non-transactional + no idempotency. fact.ms_* + index writes are awaited but progression.metric_evaluation is fire-and-forget (orphan risk); no unique constraint enforces the snapshot identity (PKs are surrogate uuids only).
(d) Scaffolding exists. progression.metric_run (full status state machine, mirrors canonical_run) and progression.metric_evaluation (with run_id, input_references_json lineage, archive_key S3 ref) already exist — unwired. A dedicated evidence schema also exists for the later Inv-VI object.
Unifying on the governed runtime + wiring the existing run object + a transactional idempotent act closes the four highest-severity gaps (wiring, governance, evidence-anchor, atomic writes) and is the foundation the other four ADRs build on. Local-first + portable seams preserves the platform's proven cost/iteration advantage while keeping AWS a deployment-time adapter swap (study section 8.9). This is shared-runtime blast-radius — one engine executes every tenant's metrics — which is precisely why it must reach BCF/MCF-grade governance.

## Decision

Adopt a single governed metric-evaluation boundary and wire it end-to-end. ADR #1 of a 5-ADR sequence from the Metric Evaluation Boundary Governance-Parity Study (bc-docs-v3/docs/implementation/metric-evaluation-boundary-governance-parity-study-2026-07-01.md).

Decisions:
1. Promote the governed runtime (evaluateGovernedMetric, src/boundary/governed-metric-runtime.ts, built under DEC-483f1e/D465) to the production evaluation act over the mcf.* substrate. Retire the legacy MetricEvaluationEngine + metric.service envelope path and its empty contract.metric_contract_version substrate.
2. Wire the existing (currently unused, 0-row) run object: every evaluation act executes inside a progression.metric_run (status_code state machine, mirrors canonical_run); progression.metric_evaluation.run_id links each evaluation to its run. The status_code set includes a fail-closed terminal state deferred_inputs_unavailable (see decision 7). No new run/evaluation tables — both already exist in the tenant progression schema. (Amendment 2026-07-01, per SS3 integration finding: status_code is NOT unconstrained text — progression.metric_run carries an inline CHECK, the shared *_run convention {pending,running,completed,failed,cancelled}. So decision 7's deferred_inputs_unavailable DID require a schema change — migration dec-5ea578-metric-run-deferred-status.sql (approved DBCP, applied to tbc_pilot1_dev 2026-07-01). Separately, SS1's success value was aligned from 'complete' to the convention 'completed' — code-only. The original "so no schema change" clause was incorrect.)
3. Idempotency = the run is the boundary (reconciles Inv V non-replayability with crash-safe retry). Snapshot identity within a fact.ms_* table = (grain/period key columns, metric_evaluation_id). Retry resumes the same run/act -> idempotent INSERT ... ON CONFLICT DO NOTHING (no rewrite; Inv III preserved). A new run -> new act -> new immutable rows that supersede prior; reads resolve the latest run.
4. Transactional evaluation act: metric_run finalize + metric_evaluation + fact.ms_* writes commit in ONE DB transaction. Remove the fire-and-forget progression.metric_evaluation write (metric.service.ts:52) that can orphan a fact row with no evaluation record.
5. Governed snapshot home = fact.ms_* (typed per-contract store, DEC-95687d/D369 generator). Deprecate the legacy snapshot home envelope.metric_snapshot (physical drop deferred; verify 0 rows first).
6. Portability seams (study section 8.9): the run executor consumes from a Queue port and is triggered by a Scheduler port — thin interfaces with local adapters now (pg-backed queue / node timer) and AWS adapters later (SQS / EventBridge). No AWS SDK in the engine core. Build local-first; deploy by adapter-swap + IaC, not rewrite.
7. Input admissibility & deferral (fail-closed; Foundation Invariants II + VI). A scheduled or triggered evaluation is a readiness check, not a compute mandate. If the required Canonical Objects for the target period are absent, or the source was not observable, the engine does NOT evaluate: producing a Metric Snapshot with no CO predecessors violates Invariant II (ordering), and fabricating a value over absence violates Invariant VI (evidence emitted, not inferred; an absent snapshot is a legitimate observable state). Instead it records a diagnostic run outcome on progression.metric_run (status_code = deferred_inputs_unavailable) with the missing-input references — this is NOT a Metric Snapshot and NOT synthesized evidence-of-evaluation. Evaluation occurs later, when inputs become admissible, as a normal FIRST act for that period — not re-evaluation and not replay (Invariant V permits distinct acts at distinct knowledge-times). Distinguish source-unreachable / zero-CO (defer) from source-observed-but-partial (evaluate as-of the knowledge-time, honestly labeled; late arrivals produce a new superseding version, never a rewrite). The check in ADR #1 is minimal (are the required COs present and was the source observed); the fuller policy — period-closure / fiscal-calendar interaction and what "required" means per metric — is ADR #2.
8. Metric snapshot output shape (operator-locked 2026-07-01, option A). A governed metric's fact.ms_<code>_v<semver> table is [grain columns..., metric_value numeric] — a UNIFORM single value column, not a per-metric named column and not per-row unit/representation tags. Rationale: (a) the store stays metric-agnostic so the shared engine, the persistence adapter, and the read-model are one code path (the shared-runtime safety argument); (b) unit_code / representation_term are properties of the METRIC not of each snapshot row — they already live on mcf.metric_variable_binding (unit_code_snapshot / representation_term_snapshot), so storing them per-row would violate D162 Rule 4 (no duplicated parent value on child rows); (c) they are surfaced by the serving layer (§14 / ADR #4) at read time — the CQRS split: the store stays uniform, the semantic layer names and unit-tags the value. This also DEFINES the never-built schema-provisioner "MC uses output fields" extraction (schema-provisioner.service.ts:127, deferred to an unimplemented "M1c"): the MC fact-table field set = the resolved grain columns + metric_value. Both the runtime MetricOutputBinding = {contractName: mc_name, factVersionCode, valueColumn: 'metric_value', snapshotKeyColumns: grain columns} AND the provisioner DDL consume this one convention. Grain columns are the gate's period column plus any grouping dimensions, resolved by the SS2 spec loader via the concept-identity join (variable_role/grain -> business_concept -> canonical field). Change A's UNIQUE(snapshotKeyColumns, metric_evaluation_id) index is over exactly these grain columns.

DB delta (Database Change Protocol — approval-gated; exact DDL delivered before any run):
- Change A: the fact.ms_* DDL generator (src/schema-provisioner/ddl-generator.service.ts) gains a snapshotKeyColumns input (the MC grain + period fields) and emits UNIQUE (snapshotKeyColumns..., metric_evaluation_id). Pure generator change; applies only to newly-provisioned tables; no existing ms_* tables to migrate (none provisioned in tbc_pilot1_dev). No new tables, no column drops.
- Change B: mark envelope.metric_snapshot for retirement; physical drop deferred to a later migration after confirming 0 rows.
Wiring (2/4) and the seam ports (6) are code-only, not DBCP.

Defers to later ADRs: temporal-shape predicates + bitemporal event-time column + input-admissibility policy (period-closure / required-input definition) (ADR #2, the metric temporal model); composite metric DAG execution (ADR #3); the evidence-schema Inv-VI evidence/lineage object + the metric_snapshot_index read-model (ADR #4); evaluation error taxonomy + version-gating (ADR #5).

Scope guard: this ADR is design-only. Change A DDL and all code land only after explicit approval; every DB change follows the Database Change Protocol.

Independent audit refinements (Codex review, 2026-07-01) — accepted:
- Fixed (P1): the spec loader now requires governance_state_code='active' (not just is_current) — a draft/review current version is never evaluated. Fixed (P1): the variable resolver surfaces constant / metric_input roles as UNRESOLVED (was silently skipped), so a formula that MIXES a CO variable with a constant/composite variable fail-closes at assembly instead of leaving that role unbound at runtime (the silent-wrong-number surface decision 2/5 guard against). Fixed (P2): run finalize verifies it matched a row (throws on silent no-op from bad routing).
- Scope clarification on decision 3 idempotency: the exactly-once guarantee holds WITHIN a run (a retry of the persist step against the same run_id is idempotent via ON CONFLICT DO NOTHING). A whole-call queue REDELIVERY opens a NEW run -> a new act that SUPERSEDES (not resume). Resume-same-run on redelivery requires the caller to pass an idempotency/run key — that belongs to SS3 / the Evaluation Orchestrator (study §13), not the ADR #1 core.
- Scope clarification on decision 8 / period_aggregate: because the period_aggregate gate is identity in ADR #1 (no temporal predicate — study §3.1), the caller MUST supply candidates already scoped to a single period for a period_aggregate metric (persistence stamps every emitted row with the requested period). This is a hard SS3 caller precondition, documented in metric-spec-assembler.ts, until ADR #2 adds the per-input period predicate. as_of metrics are unaffected (resolveAsOf selects at P).
