# bc-admin UI Development Freeze — Runtime Over Registry

## Context

Project health assessment (PLN-69129b) identified architecture-to-runtime ratio as dangerously skewed. 30+ admin screens with zero end-to-end chains. Strategic roadmap (PLN-033690) calls for immediate UI freeze. Founder agrees.

## Decision

Freeze all bc-admin development effective immediately. No new screens, no UI polish, no nav restructuring, no new pages until the bc-core runtime engine produces at least one end-to-end auditable chain (SAP FI → SO → CO → Metric Snapshot → Evidence).


## Options Considered

N/A

## Current State at Freeze

- 32 live screens, 127+ routes, 80 components, 7 contract type pages
- 17 open tasks — all deprioritized
- 3 dead pages to clean up (ActionContractsPage, ContractsPage, AIContractsPage)
- 0 TypeScript errors, 0 TODO/FIXME comments
- React Query (190+ uses), shadcn/ui, Vite — stable stack
- No ESLint configured (noted, not blocking)

## Exception Criteria

bc-admin work is permitted ONLY when:
1. A bc-core runtime feature requires a UI control surface that doesn't exist (e.g., reader execution trigger, evidence viewer for debugging)
2. A critical bug blocks runtime testing or demo scenarios
3. The freeze is formally lifted via a superseding decision

## What Happens to Open Tasks

- All 17 bc-admin tasks move to `parked` status
- Priority reset to `later` unless exception criteria met
- Tasks preserved — not deleted. Resume after freeze lifts.

## Rationale

- 32 screens built, 0 customers, 0 end-to-end runtime chains
- Every hour on UI is an hour not on the execution engine
- Admin UI should follow the runtime, not lead it
- Strategic plan PLN-033690 mandates "Runtime Over Registry"
