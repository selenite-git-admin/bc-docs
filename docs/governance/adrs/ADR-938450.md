# Universal Two-Layer Contract Scoping — Platform Master + Tenant Instance

## Context

D045 established tenant-scoped admission contracts. D044 established the canonical split (platform contract + tenant mapping binding). D047 defined Phase 1 mapping templates for zero-human-effort onboarding. D053 defined master/override for metric operational params. But these were separate decisions, each addressing one corner of a universal pattern.

D054 unifies them: the two-layer model (platform master + tenant instance) is UNIVERSAL across all contract types and mapping bindings. The classification into Pattern A (template instantiation for source chain), Pattern B (master + override for cross-cutting), and Pattern C (platform only for canonical) provides clarity on how each contract type operates within the model.

The admission contract correction (removing maps_to/semantics) follows directly from: (1) Foundation Admission Boundary: "The system must not resolve meaning at this boundary," (2) Foundation Principle "Meaning Is Resolved at Evaluation": "Admission records admissibility under a contract; it does not resolve meaning," (3) Foundation Canonical Evaluation Boundary: "the system may apply declared definitions to admitted records under a Contract." Mapping a source field to a canonical concept IS meaning resolution — it belongs at the canonical evaluation boundary via the mapping binding, not at admission.

The three-pattern classification ensures each contract type has a clear, unambiguous scoping rule. No ad-hoc decisions needed per contract type — the pattern is determined by the contract's position in the chain (source-bound → Pattern A, cross-cutting → Pattern B, canonical → Pattern C).

Cross-references: D044 (canonical split), D045 (tenant-scoped admission), D046 (discovery model), D047 (three-phase onboarding), D048 (join_context scope), D050 (canonical source-agnostic), D053 (master/override operational params), Foundation constraints/contract.md, Foundation boundaries/admission-boundary.md, Foundation principles/meaning-is-resolved-at-evaluation.md, Component Ref contract-registry.md, Arch canonical-mapping-binding.md, Arch tenant-onboarding.md.

All BareCount contracts and mapping bindings exist at two scoping layers. This formalizes the pattern that was emerging across D044, D045, D047, and D053 into a universal architectural rule.

## Two Layers

**Layer 1 — Platform Master (provider-scoped, tenant_id = NULL)**
The authoritative reference definition authored by the BareCount platform team. Serves as the governed template from which tenant instances are derived. Platform masters define the standard contract for a known source object, canonical concept, or metric. A platform master may or may not be used directly in production — its role depends on the instantiation pattern.

**Layer 2 — Tenant Instance (tenant-scoped, tenant_id ≠ NULL)**
A tenant-specific contract derived from (or independent of) a platform master. Governs actual boundary crossings for that tenant's data. May contain tenant-specific customizations: custom fields, adjusted thresholds, different field selections, or entirely custom definitions authored from scratch (Phase 2/3 onboarding).

## Three Instantiation Patterns

**Pattern A — Template Instantiation** (source chain contracts + mapping bindings)
- Platform master is a TEMPLATE — a reference design for a standard source object/extraction/admission
- Tenant instance is a FULL CONTRACT with its own identity, version, and governance lifecycle
- Instance may diverge from template over time (it is an independent artifact once created)
- Platform master is never used directly for evaluation — only tenant instances govern boundary crossings
- Applies to: **Extraction, Source, Admission** contracts and **Mapping Bindings**

**Pattern B — Master with Override** (cross-cutting contracts)
- Platform master IS the production contract — the authoritative formula, logic, rules
- Tenant override is an OVERLAY — selective customization of designated fields only (D053)
- Effective contract at evaluation time = master definition + tenant overrides, merged by evaluation engine
- Master governs boundary crossings directly; override adjusts designated parameters
- Applies to: **Metric, Intervention, AI** contracts

**Pattern C — Platform Only** (no tenant dimension)
- Contract exists only at platform scope — universal, source-agnostic
- No tenant instances, no tenant overrides — the CO form is identical for all tenants
- Source-to-canonical wiring is handled by the separate Mapping Binding artifact (D044)
- Applies to: **Canonical** contracts

## Classification by Contract Type

| Contract Type | Pattern | Platform Master Role | Tenant Artifact | What Varies Per Tenant |
|---|---|---|---|---|
| Extraction | A | Template: standard extraction config (ECB SDMX params, SAP RFC config) | Full contract: tenant-specific schedule, connection ref, lookback | Schedule, credentials, batch params |
| Source | A | Template: standard object schema (BKPF 50 fields, Opportunity schema) | Full contract: tenant schema incl. Z-fields, custom fields | Additional fields, adjusted validations |
| Admission | A | Template: standard field selection + admissibility rules (D045) | Full contract: tenant-specific field list + adjusted rules | Field selection, threshold tweaks, custom rules |
| Canonical | C | Authoritative: universal CO form (resolved_schema, business_keys, semantic_rules) | ✗ None | Nothing — CO shape is universal (D044, D050) |
| Mapping Binding | A | Template: standard source→canonical mapping (D047 Phase 1) | Full binding: tenant-specific field wiring (D044) | Field mappings, transforms, join context |
| Metric | B | Authoritative: formula, CO bindings, temporal gate, default thresholds | Override only (D053) | Thresholds, direction, alerts, schedule |
| Intervention | B | Authoritative: response rules, escalation defaults | Override only (D053) | Recipients, escalation paths, severity thresholds |
| AI | B | Authoritative: model binding, prompt governance, default confidence | Override only (D053) | Confidence gates, tenant-specific params |

## Schema Impact

**`registry.contract` table — add columns:**
```
tenant_id          TEXT          -- NULL = platform master, non-NULL = tenant instance
platform_master_id UUID          -- FK → registry.contract(contract_id), nullable
                                  -- links tenant instance to its template origin
```

Constraints:
- Platform master: tenant_id IS NULL AND platform_master_id IS NULL
- Tenant instance: tenant_id IS NOT NULL (platform_master_id may be NULL if authored independently — e.g., Phase 2/3 custom objects with no platform template)
- Unique constraint: (name, tenant_id) via COALESCE — same contract name exists once per tenant + once as platform master (NULL)

**`registry.mapping_binding` table — alter column:**
```
tenant_id  TEXT  -- change from NOT NULL to NULLABLE
                 -- NULL = platform mapping template, non-NULL = tenant mapping instance
```

Add: `platform_template_id UUID REFERENCES registry.mapping_binding(binding_id)` (nullable) — links tenant instance to template origin.

**New index:** `ix_contract__tenant` on (tenant_id) for tenant-scoped queries.
**New index:** `ix_contract__platform_master` on (platform_master_id) for "find all instances of this template."

**`registry.tenant_override` table (new, for Pattern B):**
```
tenant_id         TEXT NOT NULL
contract_id       UUID NOT NULL REFERENCES registry.contract(contract_id)
contract_version  TEXT NOT NULL
override_json     JSONB NOT NULL    -- the override fields (thresholds, schedule, etc.)
effective_from    TIMESTAMPTZ NOT NULL DEFAULT now()
created_by        TEXT NOT NULL
created_at        TIMESTAMPTZ NOT NULL DEFAULT now()
updated_at        TIMESTAMPTZ
PK: (tenant_id, contract_id, contract_version)
```

## Evaluation Engine Impact

**Pattern A resolution (source chain):**
1. Look up tenant instance contract by (contract_name, tenant_id)
2. Use tenant instance directly — platform master is NOT consulted at runtime
3. Evidence records: tenant contract ID + version

**Pattern B resolution (cross-cutting):**
1. Look up platform master contract (tenant_id IS NULL)
2. Look up tenant override for (tenant_id, contract_id, contract_version)
3. Merge: master definition + override fields → effective contract
4. Evidence records: master contract ID + version AND tenant override state (snapshot)

**Pattern C resolution (canonical):**
1. Look up platform master contract (no tenant dimension)
2. Evidence records: canonical contract ID + version

## Admission Contract Boundary Correction

Current seed admission contracts contain `maps_to` and `semantics` fields:
```json
{ "name": "TIME_PERIOD", "maps_to": "observation_date", "semantics": "temporal_key" }
{ "name": "CURRENCY", "maps_to": "quote_currency_code", "semantics": "currency_iso4217" }
```

These constitute meaning resolution — mapping a source field to a canonical concept IS resolving meaning. The Foundation Admission Boundary explicitly states: "The system must not resolve meaning at this boundary." The principle "Meaning Is Resolved at Evaluation" confirms: "Admission records admissibility under a contract; it does not resolve meaning."

**Correction:** Remove `maps_to` and `semantics` from all admission contract bodies. These fields belong in the Canonical Mapping Binding (canonical evaluation boundary, not admission boundary). Admission contracts declare ONLY:
- Field names, types, required/optional (structural constraints)
- Identity semantics (primary_key, temporal_key — for deduplication, not meaning)
- Admissibility rules (validation expressions)
- Validation policy (block/warn)

## Foundation Alignment

The Foundation specification is scope-agnostic — it defines contract types and boundary rules without addressing organizational scoping (platform vs tenant). D054 is an implementation-level architectural decision that layers scoping ON TOP OF foundation principles:

✓ Every boundary crossing still requires a contract (foundation invariant)
✓ Contracts remain immutable after release (version immutability)
✓ Evaluation evidence captures the contract version used (auditability)
✓ Admission boundary does not resolve meaning (corrected by removing maps_to/semantics)
✓ Meaning is resolved only at evaluation (canonical mapping binding, not admission)

**Foundation contradiction noted:** `constraints/contract.md` line 34 states Canonical Contract is "Source-chain bound: tied to one source version identifier." This contradicts D050 (canonical is source-agnostic) and D044 (canonical contract is platform-scoped, universal). D054 aligns with D050/D044. Foundation spec needs correction: canonical should read "Source-agnostic: defines universal canonical object form. Source wiring is handled by the Canonical Mapping Binding."

## Worked Scenarios

### Scenario 1: ECB Exchange Rate (public data, standard flow)
```
Platform masters (templates):
  ecb-sdmx-exr-extraction       [extraction template]
  ecb-sdmx-exr-source           [source schema template]
  ecb-exchange-rate-admission    [admission template]
  exchange-rate-mapping-ecb      [mapping template]

Tenant instances (demo-selenite):
  ecb-sdmx-exr-extraction-selenite       [instantiated, schedule adjusted]
  ecb-sdmx-exr-source-selenite           [instantiated, no changes]
  ecb-exchange-rate-admission-selenite    [instantiated, no changes]
  exchange-rate-mapping-selenite-ecb      [instantiated, no changes]

Platform only:
  exchange-rate-canonical                 [Pattern C, universal CO form]
  exchange-rate-volatility-metric         [Pattern B, master formula]
    └─ selenite override: alert_threshold = 5% (default 3%)
```

### Scenario 2: SAP BKPF (enterprise, standard + Z-fields)
```
Platform masters:
  sap-bkpf-source                  [50 standard fields]
  sap-bkpf-admission               [standard admissibility rules]
  sap-bseg-source                  [standard line items]
  sap-bseg-admission               [standard rules]
  ap-vendor-invoice-mapping-sap    [standard BKPF+BSEG → AP Invoice CO]

Tenant instances (megacorp):
  sap-bkpf-source-megacorp         [53 fields = 50 standard + 3 Z-fields]
  sap-bkpf-admission-megacorp      [rules adjusted for Z-fields]
  sap-bseg-source-megacorp         [standard, no changes]
  sap-bseg-admission-megacorp      [standard, no changes]
  ap-vendor-invoice-mapping-megacorp-sap  [Z-fields mapped to CO dimensions]

Platform only:
  ap-vendor-invoice-canonical      [Pattern C, universal CO form]
  ap-aging-metric                  [Pattern B, master formula]
    └─ megacorp override: aging_buckets = [45,75,120] (default [30,60,90])
```

### Scenario 3: Salesforce Custom Object (no platform template)
```
Platform masters: NONE for Booking__c (custom sObject)

Tenant instances (kohinoor) — authored from scratch in Phase 2:
  sfdc-booking-source-kohinoor       [platform_master_id = NULL]
  sfdc-booking-admission-kohinoor    [platform_master_id = NULL]
  sales-booking-mapping-kohinoor-sfdc [platform_template_id = NULL]

Platform only:
  sales-booking-canonical            [Pattern C, universal CO form]
  monthly-bookings-metric            [Pattern B, master formula]
    └─ kohinoor override: target = ₹75Cr (default ₹50Cr)
```

### Scenario 4: Multi-source convergence (SAP + SFDC → same CO)
```
megacorp has AP invoices in both SAP and SFDC:

Tenant instances:
  ap-vendor-invoice-mapping-megacorp-sap   [maps SAP BKPF+BSEG]
  ap-vendor-invoice-mapping-megacorp-sfdc  [maps SFDC AP_Invoice__c]

Both produce COs conforming to same:
  ap-vendor-invoice-canonical              [Pattern C, one universal form]

Metrics aggregate across both CO populations:
  ap-aging-metric                          [Pattern B, one master formula]
    └─ megacorp override: single override applies to ALL CO instances
```
