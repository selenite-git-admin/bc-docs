# Canonical row-arithmetic + fiscal date construction (D330/D513-additive): linear_sum and fiscal_period_end_date derivations; GL balance chain v1 at (account, fiscal year) year-end stock over GLT0

## Context

No rationale recorded.

## Decision

# Context

The Metric Directory's balance families (gl_balances, income statement lines, solvency/liquidity/return/leverage ratios, total_assets/equity/liabilities parked at review on PE-MC-11) are blocked by two facts established 2026-07-10 (SES-a87ae4 study, TSK-bb1e50):

1. **ECC stores no closing balance field.** GLT0/FAGLFLEXT persist carry-forward (HSLVT) + 16 period movement buckets (HSL01..HSL16) per (ledger, year, account). Closing balance at year end = HSLVT + Σ HSL01..16 — pure row-local arithmetic. OC-v2's transform enum [direct, date_iso8601, currency_normalise, boolean_flag, code_lookup] is strictly 1:1; mapping closing balance to any single field would be semantically false (Invariant VI violation).
2. **f86ad6c locked the temporal semantics**: a stored closing balance is a STOCK read point_in_time (as-of), NOT cumulative_to_date (Σ-of-flows would double-count a stock).

The D330 compute library grows additively by ADR only (D330-R3). D513 (DEC-3785f8) established raw-signal inputs into canonical derivations — meaning produced once at the canonical boundary, reader/OC observe and preserve only — but scoped them to `classify`.

# Decision

**D1 — `linear_sum` derivation function** (canonical-v2, D330-additive). N-ary row-local numeric sum over declared same-row inputs. Inputs may be directly-observed concepts (D462 1-hop) or raw-signal references {role, source_table, source_field} (D513 extension — previously classify-only). Optional `params.negate_roles: string[]` marks subtrahend roles for signed sums. Output concept must be numeric (amount/number representation). Raw inputs remain subject to D431 field-level observability (preserved in an active OC leg's so_schema) and never appear in field_selection, resolved_schema, or metric contracts.

**D2 — `fiscal_period_end_date` derivation function** (canonical-v2, D330-additive). Constructs a date canonical field from a fiscal year signal (+ optional fiscal period signal), resolved against the platform fiscal calendar (D363–D365, per legal entity). `params.calendar_scope: 'legal_entity'` is the explicit calendar reference (Invariant IV). Default period = last period of the fiscal year (year-end). Inputs may be concept or raw-signal (D513 extension). Contract-level declaration is complete now; runtime resolution wires FiscalCalendarService in the evaluator.

**D3 — GL balance chain v1.** Grain entity GL Account, grain keys [ledger_account_identifier, fiscal_year], over GLT0 (sc__ecc__glt0): opening_balance ← GLT0.HSLVT direct (the carry-forward IS the year-opening stock — a true 1:1 mapping); closing_balance ← linear_sum(HSLVT, HSL01..HSL16 raw signals); balance_date ← fiscal_period_end_date(RYEAR raw). One row per account per fiscal year = year-end snapshot granularity. A `fiscal year` BC on GL Account (RYEAR direct) carries the second grain key.

**D4 — Account classification is OUT of this decision and remains a named companion decision.** Classic ECC stores no asset/liability/equity class; FSV assignments (T011/FAGL_011ZC) are tenant-configured account ranges; no universal platform value-map exists. The class-filtered gl_balances members, income_statement_line_amounts, and the review-parked total_assets/equity/liabilities remain gated until the companion decision (governed platform classification reference vs D502 tenant-semantics binding) is taken. This ADR removes only the closing-balance/balance-date blockers.

**D5 — Runtime read is the D498 program.** point_in_time evaluation (latest balance_date ≤ as-of per grain key, then aggregate) is currently identity-selection at the boundary; the snapshot-read evaluator lands with the metric-runtime program. This ADR delivers contract-level realizability (chain-active), never claims runtime-live.

# Invariant test

I: closing balance meaning produced exactly once, at the CC derivation — reader/OC never compute. II: no ordering change. III: no historical rewrite; new versions via succession. IV: every input field and the fiscal calendar reference are explicit declarations. V: deterministic arithmetic and calendar lookup — no replay. VI: the derivation declaration IS the evidence; no inference of stored fields that do not exist.

# Consequences

- Balance-expression blockers (closing balance ×46, balance date ×23 blocking-concept instances) become authorable; classification blockers (account type ×34, class ×13, cash-flow category ×3) remain, honestly.
- Per-period granularity (monthly balance snapshots) deferred: requires bucket unpivot (reader-computed observation fields — grammar change at the observe boundary) or the ACDOCA line path; separate ADR when needed.
- Implementation: canonical-v2.schema.json enum + input/params blocks; canonical-concept-resolver raw-signal gates generalized from classify-only to function-aware; ccv2-canonical-resolver evaluation (linear_sum full; fiscal_period_end_date wired to FiscalCalendarService or explicit fail-closed error until the runtime program lands); specs for gates + evaluation + golden parity.
