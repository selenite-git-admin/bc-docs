# Open-Standard Publication Protocol Architecture

## Context

Symmetric architecture principle: if ingress is protocol-based and open-standard (D086, not D084 — renumbered), egress must follow the same pattern. This prevents vendor lock-in on both sides, makes BareCount a protocol-neutral boundary engine, and enables KPI as Data Product without building custom integrations per BI tool. Open standards (OData V4, CloudEvents, OpenLineage, XBRL) have broad ecosystem adoption — consumers already know how to connect.

BareCount outward integrations (publication/egress) use open-standard protocols, symmetric with D084 ingress. Adding a new consumer = configuration, not code.

## Publication Surface (by evaluation boundary)

1. **Metric Snapshots → OData V4** — KPI as Data Product. Power BI, Excel, Tableau connect natively. Metrics exposed as OData entities with standard query support ($filter, $select, $expand).

2. **Action Outcomes → CloudEvents (CNCF)** — Webhook-based event publishing in CloudEvents format. Any consumer (Slack, Jira, ServiceNow, PagerDuty, custom) subscribes via standard event envelope.

3. **Evidence & Lineage → OpenLineage** — Lineage emitted in OpenLineage format (Linux Foundation). Compatible with Marquez, Datakin, Atlan, any data catalog that supports the standard.

4. **Canonical Objects → OData V4 / Delta Sharing** — Data mesh consumers and downstream analytics access COs via OData V4 or Databricks Delta Sharing protocol.

5. **Compliance Reporting → XBRL** — Financial metrics exportable in XBRL for regulatory reporting.

## Architecture

- **Publication Executors** mirror Reader Executors — same L1 interface pattern, reversed direction (emit instead of fetch)
- **Protocol Connectors are bidirectional** — OData V4 can read SAP AND serve KPI data to Power BI
- **No custom SDKs or proprietary APIs** per consumer — protocol publishers only
- **Publication ≠ transformation** — BareCount publishes evaluation state as-is, consumers interpret

## Protocol Classification (matches D084 tab model)

| Type | Protocols | Use |
|------|-----------|-----|
| Universal | OData V4, CloudEvents, OpenLineage, Delta Sharing | Open standard, any consumer |
| Domain-Specific | XBRL, SDMX (re-publish) | Regulatory/statistical reporting |

## What this is NOT

- Not a "push connector" per destination (no SlackPublisher, JiraPublisher)
- Not an ETL/export engine — publishes boundary state, not transformed data
- Not a real-time streaming system — publishes snapshots at evaluation boundaries
