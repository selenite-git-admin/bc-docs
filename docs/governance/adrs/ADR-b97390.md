# Embedded documentation reader in bc-admin with native React implementation

## Context

# Rationale

## Why bc-admin primary

Documentation centralized away from the audience is documentation not read. legacy v2 archive has a mkdocs site that is not routinely opened by platform operators, and v2's documentation has drifted accordingly. The operator's natural surface is bc-admin, where configuration and operational decisions are made. Putting the book in bc-admin puts it where reading would actually happen during the work that prompts questions. Every other proposed solution preserved the separation between work surface and docs surface; this one closes it.

## Why native React, not embedded mkdocs

Option A (embed static mkdocs build inside bc-admin) was rejected for four reasons.

First, **two maintenance surfaces**: Option A requires maintaining both the mkdocs build pipeline and whatever integration layer embeds its output. Any future enhancement (search customization, deep-linking, design-system alignment) is implemented against mkdocs' extension model, not against bc-admin's stack. This is the pattern v2 already demonstrated — the mkdocs site exists, is not maintained, and does not align with the rest of the platform.

Second, **style mismatch is not a cosmetic problem**. Platform operators navigate between admin screens and docs screens continuously. Style changes on navigation create cognitive friction. Worse, they signal that docs are "external" to the platform work, which is exactly the perception that caused v2's mkdocs site to be under-used.

Third, **deep-linking is the real prize**. The reader's long-term value is not displaying chapters; it is clicking a `DEC-xxxxxx` in Ch 6 and landing on the live ADR in Appendix F with its supersession chain, or clicking a tenant slug in Ch 13 and landing on that tenant's admin page. None of this is possible with an iframe or static build. Native React reader makes it possible from day one.

Fourth, **Option A's ship-early benefit does not survive examination.** Option A is "a week" of work followed by indefinite technical debt to migrate to Option B. Going to B directly is weeks of upfront work followed by a stable platform that does not need migration. The integrated cost is lower.

## Why scaffold in parallel, not after

Scaffolding in parallel with chapter drafting (option ii from the pre-decision discussion) serves two concerns that deferred scaffolding does not.

First, it **proves the claim that docs-in-bc-admin works** as early as possible. If the scaffold waits until Ch 19 drafts in Session ~13, we have ten sessions of chapters drafted without a reader to display them, which reproduces the v2 pattern of docs-without-readers.

First-operator feedback also arrives earlier: the minimum-viable shell shipping in time for Ch 1 and Ch 2 lock means operators can read the thesis chapters while they are freshest, and surface reader-design issues while there is time to correct them.

Second, it **distributes the implementation load** rather than concentrating it at the end of the v3 build. Ch 19 already carries substantial content (Customer Portal, Admin Portal, Auth UX, Design System); adding a reader to its scope as a post-factum addition would make Ch 19 single-session infeasible. Distributing the scaffold work across earlier sessions keeps Ch 19 writeable in single-session density.

## Why mkdocs public site deferred

A public mkdocs site has real cost (deployment infrastructure, hosting, certificate management, DNS, drift risk) and no named consumer today. The decision is not "never" — it is "when a named consumer appears." Auditors engaging with ISO 27001 or SOC 2 work (per DEC-ae331f) will become such consumers; enterprise buyers evaluating the platform may too. The mkdocs deployment can be stood up from the same `bc-docs-v3/` source when that need is concrete. Spending infrastructure now on a site that may or may not be read is the same error v2 made.

Session 2 (SES-e1b0e4) founder decision, responding to the observation that v2 has too many documentation surfaces and none that are reliably used. Supersedes any implicit assumption that v3 would deploy a public mkdocs site at cutover.

# Decision

Platform documentation is delivered through **bc-admin as its primary surface** for platform operators. bc-admin implements a **native React documentation reader** that reads v3 Markdown content directly from `bc-docs-v3/`, renders it using bc-admin's design system, provides native search, and supports deep-linking to live platform state (tenants, ADRs, Metric Contracts, chain status, and other structured data already present in bc-admin). The reader is scaffolded in parallel with v3 chapter drafting, not deferred until Chapter 19 is drafted.

## What this decision commits to

1. **bc-admin is the primary documentation surface** for platform operators. Operators read chapters, navigate the full book, search, and follow cross-references from within bc-admin. They do not leave bc-admin to read documentation.

2. **Native React reader (not embedded mkdocs).** The intermediate option of building the mkdocs site and embedding its static output in bc-admin is rejected in favor of a native implementation. Rationale in §Rationale.

3. **Single maintained docs surface.** v3 Markdown in `bc-docs-v3/` is the single authoritative source. The reader reads it directly. No intermediate build step creates a second artifact that can drift.

4. **Public mkdocs site is deferred.** No separate public mkdocs site is deployed for v3 until a named external consumer (auditor, enterprise buyer, compliance engagement) requires one. When deferred deployment is needed, it is a derived export from the same `bc-docs-v3/` source.

5. **Customer documentation (bc-portal) stays separate.** This decision applies only to platform documentation for operators. Tenant-facing customer documentation in bc-portal is a different scope, different audience, different content. It is not consolidated with this reader.

6. **Scaffolding proceeds in parallel.** bc-admin docs reader scaffolding begins during v3 chapter drafting (not deferred to Chapter 19 drafting or to v3 cutover). This means operators can begin reading locked chapters from bc-admin as soon as the reader's minimum-viable shell is in place, rather than waiting for the full book to complete.

7. **Chapter 19 (Frontend Experience) scope expands** to include the docs reader as a first-class component. The reader gets full facet treatment (Purpose / Behavior / Inputs / Outputs / Constraints / Failure modes / Interactions).

## What this decision does NOT commit to

- Specific markdown parser or search library — to be chosen during scaffolding (candidates: remark/rehype stack, MDX, custom parser). Recorded as a followup decision when the choice is made.
- Public mkdocs site deployment timeline — deferred until a named consumer exists.
- Customer documentation in bc-portal — out of scope for this ADR.
- Offline reader support — evaluated later based on operator need.
- Publication to external docs.barecount.com — separate deployment decision.

## Reader scope (minimum viable)

The reader must, from day one of scaffolding:

- Render any v3 Markdown file as a chapter-aware page (frontmatter parsed, navigation rendered).
- Provide book-wide navigation matching outline §4 (Parts I–V + Reference Library).
- Provide full-text search across chapters and appendices.
- Cross-link ADR citations (`DEC-xxxxxx`) to Appendix F entries.
- Cross-link Errata citations (`FND-ERR-xxx`) to Appendix G entries.
- Render code fences, tables, admonitions, mermaid diagrams.

The reader may later, as book stabilizes:

- Deep-link to live platform state (e.g. clicking a tenant slug jumps to that tenant's admin view; clicking an MC UID jumps to the chain-status detail view).
- Embed contextual chapter sections inline within admin forms (e.g. "read Ch 9 §9.4 inline" from a Canonical Mapping editor).
- Support user annotations or operator notes against specific paragraphs.

## Implementation notes (non-binding)

- Reader routes at `/docs/*` in bc-admin.
- Source of truth is `bc-docs-v3/` Markdown directly; distribution mechanism (git submodule, package, fetch) to be decided in scaffolding.
- Design system: bc-admin's existing DS (no mkdocs theme borrowed).
- Parser: remark/rehype is the leading candidate; final choice in scaffolding.
- Search: client-side lunr-style index initially; server-side if operator load warrants.
- Access: platform operators only; RBAC aligned with existing bc-admin scope (platform-only, not tenant-scoped).

## Sequencing
