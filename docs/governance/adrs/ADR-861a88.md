# Providers are status-free containers — lifecycle lives on systems

## Context

SAP SE has 8 child systems at different lifecycle stages (3 active, 5 planned). Assigning a single "active" or "planned" status to the provider container is semantically wrong and misleading. The status vocabulary belongs at the system level where actual lifecycle transitions happen. Provider is a pure organizational container — like a folder grouping files of different ages.

## Decision

Provider entities in the source catalog do NOT carry their own lifecycle status. Providers are grouping containers for systems (e.g., SAP SE groups S/4HANA, ECC, Concur, Ariba). Each system has its own independent lifecycle status (planned → active → inactive → deprecated). The provider-level status badge and status filter have been removed from Source Catalog UI. The sidebar label "Registered" was renamed to "Source Catalog" since the page shows all providers regardless of system lifecycle stage.

## Consequences

This decision was **implicitly reversed** in subsequent ADRs that introduced uniform status columns across all 6 source-catalog tiers:

- **DEC-552ddd** — separated `catalog_status` from `verification_status` on every source entity (provider, system, version, module, object, field).
- **DEC-7c4c39** — locked the 4-state `catalog_status` lifecycle (`registered → approved → deprecated → archived`) on every source entity.
- **DEC-eaba02** — added the third axis `validation_status` (tenant-scoped execution proof) on every source entity.

The current DDL (`02-platform-tables.sql:184-189`) gives `source_provider` the same status triple as every other tier. The "providers are status-free containers" intent did not survive the unification of status semantics across the catalog hierarchy.

Status flipped to `reversed` (2026-04-22) to reflect this. The implicit reversal was identified during Tier 4 deep-triage (Source Catalog theme); no formal reversal ADR was authored at the time of the unification.

**Followup consideration**: D370 hygiene policy doesn't currently require a formal reversal ADR when later decisions implicitly reverse an earlier one. Worth adding rule "implicit reversals must be documented as a `reversed` status flip with a Consequences explanation" — this ADR is the first such instance.
