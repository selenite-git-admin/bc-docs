# Metric Directory Knowledge & Governance Layer (extends D506)

## Context

The mechanics (D506 schema + authoring flow) are built, but the platform's value for its three audiences is the curated knowledge produced in the process — and it is currently un-actionable (trapped in point-in-time narrative). Making it a governed, browsable, debatable, approvable artifact turns the directory from an authoring substrate into a collaborative metric knowledge base, and provides the substrate for the domain-expert advisor role (curating/approving recommended metrics per industry/standard). Capturing it now is time-critical: the dispositions, rationales, and realizability findings from the current enrichment arc (198->370 members, every scope-out reasoned) are fresh and decay quickly into hard-to-reconstruct prose. The two-laws mapping keeps the extension disciplined (no MC duplication, no cached realized state); the append-only decision log keeps governance evidence-emitted and immutable.

## Decision

Extend the Metric Directory (**DEC-b5c7ff / D506** — extends, does **not** supersede) with a governed **knowledge + governance layer** so the curated reasoning behind each metric — definition, rationale, formula-intent, provenance, dispositions, debate, approvals — becomes a first-class, queryable, audience-segmented artifact that **platform operators, domain-expert advisors, and end users** can access, debate, and approve. Today this knowledge is trapped in narrative (bc-docs manifest, change records, panel `review_reason`s); it is not per-metric queryable, governable, audience-segmented, or live.

The extension is bound by D506's two laws and the six invariants:
- the directory **OWNS intent-knowledge** (definition, rationale, formula-*intent*) as mutable enrichment — exactly as `family.decision_text`/`rationale_text` already are;
- it **REFERENCES-not-restates** realized meaning (the exact formula lives in the Metric Contract; the evidence in the BCF concept) — via explicit references, never copies;
- **governance is an APPEND-ONLY event log** (`directory_decision`) — dispositions, debate, approvals are emitted as immutable evidence (Invariant VI), never mutated (Invariant III);
- **governance state (review_state, approver) is DERIVED from the log, not stored** — mirroring D506's derive-realization law. Storing a `review_state` column would reintroduce exactly the cached-state drift D505/D506 were designed to forbid.

**Governance posture (operator decision, 2026-07-09): AUTO-APPROVE now; gate later.** With only two stakeholders today, a review/approval workflow + RBAC would be ceremony before there is anyone to govern. So Phase 1 **captures knowledge + open discussion** but applies **no approval gate** — metrics are usable/recommended by default; challenges are captured in the log; the gate + RBAC arrive when the platform actually has governors. Crucially the log is **append-only and audience-tagged from day one**, so when the workflow lands the entire discussion history is already there, immutable, ready to gate — nothing is thrown away.

Delivery is **phased**:
- **Phase 1 (now)** — `member_knowledge` + `directory_decision` (open, append-only, audience-tagged discussion/knowledge/disposition log; auto-approved, ungated) + backfill this arc's dispositions/rationales/realizability findings + a **master-detail bc-admin UI** (Families-first; family & member **detail pages** carry knowledge + discussion; lean list views; edit on detail).
- **Phase 2 (deferred, undated — "when the platform needs it")** — review/approval **state machine** + gating, `review_state` derivation, **RBAC**/approval authority, advisor review queue.
- **Phase 3 (deferred)** — end-user recommended-metrics view (approved × industry-applicable) — substrate for advisor-curated metric packs.

**No schema is applied until this ADR is `decided` (Database Change Protocol).**

> **Refinements over the one-line registry summary:**
> 1. Knowledge fields live in a **1:1 `member_knowledge` table**, not new `member` columns — `member` already has 16 columns and +6 would breach DB Rule #6 (max 20; split by concern).
> 2. `review_state` is **derived**, not stored (DB Rule #2 + Invariant III) — **and its derivation is itself deferred to Phase 2** (no governance state at all in Phase 1, since auto-approve means there is no gate to reflect). Phase 1 ships only the two additive tables + one convenience read view.

## Foundation / invariant check

Repair location: **E (storage)** + **F (read model / diagnostics)** — new tables + views for an intent/governance layer. No change to A–D (source, contracts, mapping, evaluation).

| # | Invariant | How honored |
|---|---|---|
| I | Meaning evaluated once | Knowledge is intent metadata; realized meaning still produced only at the MC boundary and referenced, never recomputed here. |
| II | Object ordering fixed | No change to SO→CO→MS→AO progression. |
| III | State immutable | `directory_decision` is append-only (UPDATE/DELETE-rejecting trigger). `member_knowledge` is editable *intent* (like `family.rationale_text`); the authoritative governance history is the immutable log. |
| IV | References explicit | `member_knowledge`→`member` FK; `directory_decision` typed FKs to family/member; provenance via `references_json` pointing at BCF concept / ADR / standard. |
| V | Evaluation non-replayable | N/A — no evaluation in this layer. |
| VI | Evidence emitted, not inferred | Governance decisions are **emitted** as log events, not inferred from state. Review state is derived *from* the emitted events. |

## What the directory owns vs references (D506 two-laws mapping)

| Knowledge | Home | Law |
|---|---|---|
| Definition, rationale, formula-*intent*, audience notes | `member_knowledge` (directory owns) | own-intent |
| Exact formula, resolved inputs, evidence chain | Metric Contract / BCF concept (referenced) | reference-not-restate |
| Dispositions, debate, approvals, rejections | `directory_decision` (append-only events) | evidence emitted, immutable |
| Review state, approver | **derived** from `directory_decision` (a view) — **Phase 2** (auto-approve now; no gate) | derive-governance (mirrors derive-realization) |
| Dependency graph | **derived** view over derivation intent | no cached graph |

## Schema (PROPOSED — not applied until this ADR is `decided`)

DDL target: new file `bc-core/docker/redesign/30-metric-directory-knowledge-layer.sql` (+ rollback). Drizzle: `src/database/schema/metric-directory/{member-knowledge,directory-decision}.ts`.

### 1. `member_knowledge` — 1:1 intent enrichment (mutable, split by concern)
```sql
CREATE TABLE metric_directory.member_knowledge (
  member_uid          text        NOT NULL,
  definition_text     text,                       -- plain-language "what it means" (end-user facing)
  rationale_text      text,                       -- "why it matters" (advisor/operator facing)
  formula_intent_text text,                       -- human "how it's computed" (INTENT; exact formula lives in the MC)
  updated_at          timestamptz NOT NULL DEFAULT now(),
  updated_by_name     text,
  CONSTRAINT member_knowledge_pkey PRIMARY KEY (member_uid),
  CONSTRAINT fk_member_knowledge_member FOREIGN KEY (member_uid)
    REFERENCES metric_directory.member(member_uid) ON DELETE CASCADE
);
```

### 2. `directory_decision` — append-only governance/annotation log (SSOT for governance)
```sql
CREATE TABLE metric_directory.directory_decision (
  decision_id     uuid        NOT NULL DEFAULT gen_random_uuid(),
  family_id       uuid,                            -- exactly one of family_id / member_uid set
  member_uid      text,
  decision_kind   text        NOT NULL,            -- rationale|disposition|deferral|debate|approval|rejection
  audience_role   text        NOT NULL,            -- operator|advisor|end_user
  author_name     text        NOT NULL,
  body_text       text        NOT NULL,
  references_json jsonb       NOT NULL DEFAULT '{}'::jsonb,  -- {bcf_concept_id?, adr_uid?, standard?, panel_run_uid?, supersedes_decision_id?}
  created_at      timestamptz NOT NULL DEFAULT now(),
  CONSTRAINT directory_decision_pkey PRIMARY KEY (decision_id),
  CONSTRAINT chk_dd_subject_exactly_one CHECK (((family_id IS NOT NULL)::int + (member_uid IS NOT NULL)::int) = 1),
  CONSTRAINT chk_dd_kind     CHECK (decision_kind IN ('rationale','disposition','deferral','debate','approval','rejection')),
  CONSTRAINT chk_dd_audience CHECK (audience_role IN ('operator','advisor','end_user')),
  CONSTRAINT fk_dd_family FOREIGN KEY (family_id)  REFERENCES metric_directory.family(family_id)   ON DELETE CASCADE,
  CONSTRAINT fk_dd_member FOREIGN KEY (member_uid) REFERENCES metric_directory.member(member_uid)  ON DELETE CASCADE
);
CREATE INDEX idx_dd_family ON metric_directory.directory_decision (family_id)  WHERE family_id  IS NOT NULL;
CREATE INDEX idx_dd_member ON metric_directory.directory_decision (member_uid) WHERE member_uid IS NOT NULL;
CREATE INDEX idx_dd_kind   ON metric_directory.directory_decision (decision_kind);

-- Append-only (Invariant III / VI)
CREATE FUNCTION metric_directory.tg_directory_decision_immutable() RETURNS trigger
  LANGUAGE plpgsql AS $$ BEGIN
    RAISE EXCEPTION 'directory_decision is append-only (Invariant III/VI): % rejected', TG_OP;
  END $$;
CREATE TRIGGER trg_directory_decision_immutable
  BEFORE UPDATE OR DELETE ON metric_directory.directory_decision
  FOR EACH ROW EXECUTE FUNCTION metric_directory.tg_directory_decision_immutable();
```

### 3. View — UI-ready member read (Phase 1; no governance state)
```sql
-- Member intent + knowledge + realized (from v_member_realized), one row for the member detail page.
-- No governance columns in Phase 1 (auto-approve = no gate). Discussion is read separately from
-- directory_decision (a timeline, not a single-row join).
CREATE VIEW metric_directory.v_member_knowledge AS
SELECT r.*, k.definition_text, k.rationale_text, k.formula_intent_text
FROM metric_directory.v_member_realized r
LEFT JOIN metric_directory.member_knowledge k USING (member_uid);
```
**Phase 2 (deferred):** `v_member_governance` — derives `review_state`/approver from the latest governance event in `directory_decision` (state machine: last proposed/under_review/approval/rejection wins; default `draft`). Not built now; the append-only log already carries the events it will read, so it drops in without a migration.

**DB-rule conformance:** `member` untouched (Rule #6, ≤20 cols); split by concern (identity/spec vs knowledge vs governance); FK-mandatory honored (Rule #3 — typed FKs + exactly-one CHECK, no polymorphic-untyped ref); no derivable stored state (Rule #2 — review_state/approver/dependencies all derived); naming ISO-11179 (`{noun}_text`, `{verb}_at`, `{noun}_json`, `idx_*`, `fk_*`, `chk_*`); append-only log needs no soft-delete/temporal columns.

## UI surface (bc-admin) — master-detail, Families-first (operator UI story, 2026-07-09)

Directory page = tabs; **Families tab is the default/first**, then Members. **Master-detail**: list views are lean (browse only); **detail pages carry knowledge + discussion + edit**.

**Families tab → Family list** — DataTable (Family · Function · Subfunction · Coverage · realized rollup). **No `Priority` column.** **No inline Edit action.** Row click → Family detail. Function/subfunction filters + search retained.

**Family detail page** — family knowledge (definition · rationale · decision · priority · industries) + **discussion timeline** (`directory_decision` for the family) + **Edit** action (here, not the list) + the family's **members** rendered by **reusing the member DataTable component**, scoped to the family (`getFamilyDetail` / members-by-family).

**Members tab → Member list** — DataTable, lean (Member · class · family · realized state). **No inline Edit action.** Row click → Member detail.

**Member detail page** — all knowledge (definition · rationale · formula-intent · provenance links · realized state) + **discussion timeline** (`directory_decision` for the member) + **Edit** action.

**Discussion (Phase 1)** — open, append-only, audience-tagged; anyone may post a `rationale`/`disposition`/`debate` note; **no approve/reject gating** (auto-approve).

Component reuse: base = the existing `MetricDirectoryPage` tabs + Families DataTable (bc-admin `ab8dcfb`); add routes for `family/:id` and `member/:uid` detail; the **member DataTable is reused inside the Family detail**.

**Deferred to Phase 2:** review-state badges, approve/reject actions, audience-filtered views, advisor review queue. **Phase 3:** end-user recommended-metrics view (approved × industry via `family_industry`).

## Alternatives considered
- **Keep it in the bc-docs manifest** — rejected: point-in-time narrative; not per-metric queryable, governable, audience-segmented, or live.
- **Add columns to `member`** — rejected: breaches DB Rule #6 (22 cols) and mixes concerns.
- **Store `review_state` on the row** — rejected: derivable from the log → violates Rule #2 + Invariant III and repeats the D505 cached-state drift the directory exists to prevent.
- **External comment tool (e.g. a doc/issue tracker)** — rejected: not integrated with the governed substrate; no provenance/FK linkage; not audience-segmented.
