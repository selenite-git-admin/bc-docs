# E2E Execution Flow — Step-by-Step Runbook

## Context

This runbook captures the exact steps, IDs, gotchas, and verification queries needed to reproduce the E2E proof. Without it, future sessions would spend hours rediscovering the same information.

This documents the proven E2E execution flow for running data through all 4 evaluation boundaries. Verified with 500 GL Journal Entry records on 2026-03-25.

## Step 1: Reader Execution (Observation + Admission)

**Endpoint:** `POST /api/readers/:readerId/execute`
**Scope:** Platform (admin client)
**Headers:** `Authorization: Bearer $TOKEN`, `x-tenant-id: demo-selenite`
**Body:**
```json
{
  "connectionId": "<connection_uuid>",
  "tenantId": "demo-selenite",
  "environment": "dev"
}
```

**What happens:**
- Reader fetches data from SDG via connection
- Creates `observed_record` + `source_object` for each row (Observation Boundary)
- Validates against observation contract field_map
- Creates `admitted_record` for valid records (Admission Boundary)
- Creates `admission_run` in operational schema with counts

**Key IDs needed downstream:**
- `run_id` — from the admission run (check `operational.admission_run` or `boundary.admitted_record.run_id`)

**Verification:**
```sql
SELECT count(*) FROM boundary.observed_record;  -- should match SDG fetch count
SELECT count(*) FROM boundary.admitted_record;   -- should match (minus rejections)
```

## Step 2: Canonical Resolution (Evaluation Boundary)

**Endpoint:** `POST /api/t/admission-runs/:runId/resolve`
**Scope:** Tenant (portal client)
**Headers:** `Authorization: Bearer $TOKEN`, `x-tenant-id: demo-selenite`
**Body:**
```json
{ "bindingId": "<canonical_mapping_id>" }
```

**Finding the right IDs:**
```sql
-- Run ID (from admitted records)
SELECT DISTINCT run_id FROM boundary.admitted_record;

-- Canonical mapping ID (from platform DB)
SELECT canonical_mapping_id, mapping_name FROM contract.canonical_mapping
WHERE mapping_name LIKE '%gl-journal%';
```

**What happens:**
- Loads mapping binding + version from `contract.canonical_mapping` + `contract.canonical_mapping_version`
- Loads all admitted records in the run, enriched with observed payloads
- Groups by business key (or 1:1 if no join_context)
- Runs `CanonicalEvaluationEngine` on each group
- Creates `evaluation` + `canonical_object` for accepted records

**Known gotchas:**
- Uses `contract.canonical_mapping` table (NOT `master.mapping_binding`)
- `mapping_json` field (NOT `binding_json`) in the version table
- If re-running: must `DELETE FROM boundary.evaluation` first (UNIQUE on admitted_record_id)
- `normalizeMappingBody()` handles both `source_dependencies` and `source_bindings` formats

**Verification:**
```sql
SELECT count(*) FROM boundary.canonical_object;  -- should match admitted count
SELECT count(*) FROM boundary.evaluation WHERE evaluation_status = 'rejected';  -- should be 0
```

## Step 3: Metric Evaluation (Metric Boundary)

**Endpoint:** `POST /api/t/metric-evaluation`
**Scope:** Tenant (portal client)
**Headers:** `Authorization: Bearer $TOKEN`, `x-tenant-id: demo-selenite`
**Body:**
```json
{
  "metricContractId": "<metric_contract_uuid>",
  "metricVersion": "v1",
  "canonicalObjectIds": ["<co_uuid_1>", "<co_uuid_2>", ...],
  "tenantId": "demo-selenite"
}
```

**Finding metric contracts:**
```sql
-- Metric contracts bound to our canonical contract
SELECT mc.metric_contract_id, mc.metric_contract_name
FROM contract.metric_contract mc
JOIN metric.metric_binding mb ON mb.metric_contract_id = mc.metric_contract_id
WHERE mb.canonical_contract_id = '<canonical_contract_id>';

-- All CO IDs
SELECT json_agg(canonical_object_id) FROM boundary.canonical_object;
```

**Metric contract JSON format (for MetricEvaluationEngine):**
```json
{
  "body": {
    "inputBindings": [{"alias": "gl_je", "canonicalContractId": "<uuid>", "role": "primary"}],
    "computation": {
      "operations": [
        {"op": "count", "as": "total_entries"},
        {"op": "sum", "as": "reversal_count", "field": "IsReversal"},
        {"op": "avg", "as": "avg_period", "field": "FiscalPeriod"}
      ]
    },
    "qualityRules": [{"type": "nonNegative", "field": "total_entries"}],
    "thresholds": [{"metric": "total_entries", "operator": "gte", "value": 100}]
  }
}
```

**Supported operations:** `count`, `sum`, `avg`, `min`, `max`
**Quality rule types:** `nonNegative`, `minCount`, `maxCount`, `fieldRange`
**Threshold operators:** `gt`, `lt`, `gte`, `lte`, `eq`

**Verification:**
```sql
SELECT count(*) FROM boundary.metric_evaluation;   -- 1 per metric contract
SELECT count(*) FROM boundary.metric_snapshot;      -- 1 per accepted evaluation
SELECT metric_value_json FROM boundary.metric_snapshot;  -- check computed values
```

## Step 4: Accuracy Verification

Cross-validate metric values against raw SQL:
```sql
SELECT count(*) FROM boundary.canonical_object;  -- should match total_entries
SELECT count(*) FROM boundary.canonical_object 
  WHERE (canonical_payload_json->>'IsReversal')::boolean = true;  -- reversal_count
SELECT avg((canonical_payload_json->>'FiscalPeriod')::numeric) 
  FROM boundary.canonical_object;  -- avg_fiscal_period
```

## Known Gaps (Follow-up Required)

1. **Evidence/Lineage:** 0 records created — evidence service has a silent failure (likely schema mismatch)
2. **Idempotency:** Resolve endpoint is not idempotent — re-running requires manual DELETE
3. **Field renaming:** CamelCase → snake_case should happen at observation contract level, not resolution
4. **Version codes:** Metric contracts use `v1`, canonical contracts use `1.0.0` — inconsistent
