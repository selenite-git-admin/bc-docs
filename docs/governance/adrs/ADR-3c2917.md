# Connection belongs on reader_flavor, not observation_contract

## Context

Contracts define schema and rules. Connections define endpoints and credentials. Mixing them couples deployment topology to contract governance. A single OC can be served by multiple connections (dev SDG, staging SAP sandbox, production SAP). The flavor is the right join point — it already carries connector_id (driver type) and now connection_id (live instance).

## Decision

connection_id column added to runtime.reader_flavor. Connection is operational (which SAP endpoint to hit), not contractual. The observation_contract defines WHAT to observe; the connection defines WHERE to connect. Redesign mistakenly dropped connection_id from flavor — restored.
