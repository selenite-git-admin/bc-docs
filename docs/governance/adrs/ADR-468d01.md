# Registry root layout — dead-code removal + six-folder consolidation of the loose root files (D484/D485 companion)

## Context

155 loose files at the registry root made the folder unreadable and hid genuinely dead surfaces (two complete unmounted HTTP controllers sat undetected). The importer graph, not name-vibes, drew every classification boundary: two agent verdicts were corrected during verification (dto/ is LIVE with 39 barrel importers; contract-analytics.repository is LIVE via contract.service) — precisely the trap class this study existed to catch. Foldering follows proven module seams (each folder = the files one NestJS module family already registers), so moves are mechanically safe and reviewable. Executed post-PR#401, so every wave verifies against the strictest bar bc-core has had: tsc 0 errors, green vitest, fully-blocking CI.

## Decision

Completes the src/registry layout arc that DEC-9428e0 (D484, bcf/) and DEC-3aa336 (D485, mcf/) explicitly deferred: "the 100+ remaining loose files at the registry root are a separate, unfiled cleanup decision." Grounded study (SES-520bd2, importer graph over all 155 root files + module-mounting map + git last-touch): barecount-devhub/.claude/registry-root-refactor-ground-study-2026-07-04.md.

**Wave 0 — dead/orphaned code deleted (unreachable today, no sunset dependency):** metric-knowledge.repository.ts (comment-only refs), formula-deterministic-evaluator.ts+spec (replaced by compute-evaluator), canonical-chain.service.ts (moved to scripts in D225 era), oagis-d292.ts (comment-only refs), mapping-binding.controller.ts (complete HTTP surface never registered in any module — tenant-views fronts the service directly), and the catalog-verification family (controller/service/repository/prompts/agent — a pre-BCF/MCF Claude-API verification surface whose controller was never mounted; superseded by the BCF registry-authoring panel and MCF M12 panel).

**Waves 1–3 — six-folder consolidation (pure git-mv relocation, D484 pattern: no file renames, no route/behavior change):** contracts/ (contract.{module,controller,service,repository}, contract-version/-validation/-governance/-listing/-metrics/-analytics, contract-families, contract-query-builder, contract-envelope.builder, contract-generator, compatibility.ts, mapping-binding.{service,repository}); resolvers/ (canonical-concept-resolver, observation-concept-resolver, concept-observability); source-catalog/ (source-catalog.*, source-reference.*, discovery.*, seed-catalog.*, seed-metrics.*, oagis-seed.*); connections/ (connector.*, connection.*, credential-resolver); readers/ (reader.*, reader-authoring.*, reader-chain-resolution{,.service}, reader-observation-binding.repository) — verified safe against bc-qa p10.pipeline globs and check-forbidden-vocab.sh (both target top-level src/readers only); execution/ (execution.*, execution-writeback, package.*, core-map/ folds in). The 3 root meta-schema-*.ts files join meta-schemas/. seed/ taxonomy files (domain-taxonomy, metric-taxonomy — the only 2 runtime-imported files of 159) extract to their consumers' side.

**Deliberate NON-moves (negative space, per the D485 pattern):** registry.provider.ts stays at root (REGISTRY_DB token hub, 58 importers); intake-queue.* stays (shared MCF+BCF substrate per DEC-3aa336); panel-output-record.* stays (BCF substrate hub, 17 importers, kept out of bcf/ by DEC-9428e0); business-catalog trio stays (cross-cutting aggregation); the ~18-file legacy metric-* sunset unit stays in place untouched (relocating dying code is churn); abc/ keeps its own future homing decision.

**Correction to DEC-3aa336's framing, recorded here:** the "~20 legacy metric-* root files will be deleted wholesale" claim is only half true. 13 files in that neighborhood are load-bearing for LIVE paths: readiness-ledger.service (D316/D337 dispatch spine — not legacy at all), metric-funnel.service + mls-14-predicate + metric-catalog-reader.repository + metric-binding.repository + contract-metrics.repository (live readiness/tenant-metrics/business-catalog), metric-reference.{service,repository} (live tenant-views), kpi-spec trio (serves the ACTIVE POST /metric-definitions surface), integrity.service (contract-activation gate until Runtime Spine R5), formula-audit.service (readiness-ledger dispatch gate), compute-evaluator + compatibility + boundary-rollup.repository (live boundary/contract paths). The legacy sunset therefore requires five repoint jobs (readiness, tenant-metrics, tenant-views, business-catalog, activation-gate reads off contract.metric_contract/metric.* onto MCF substrate) BEFORE the wholesale delete; that work belongs to the legacy-bridge sunset / Runtime Spine arc, not this layout decision.

**Policy flag:** contract-generator.service.ts (bulk SC+AC chain generation) conflicts with D268 rule 1 (no bulk contract chain generation); it gains @deprecated documentation in Wave 1. Its removal is a separate future decision, not this one.
