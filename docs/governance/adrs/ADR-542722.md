# Catalog-first metric authoring — draft MCVs are chain-independent; PE-MC-11 is the sole chain coupling

## Context

The chain-gated approach saturated at corpus 108 — the ceiling of what the two live slices' CC chains declare. That cap is structural: authoring more against the current chain yields dups and vacuous metrics (proven by the Tier-1 triage: 91 near-set survivors, only ~3 authorable-now). But the cap is artificial — the blocked metrics are valid semantic definitions; they are blocked by chain/data, not by being un-authorable. The probe proved a draft MCV needs only BCF concepts + a grain entity, not a chain. Catalog-first removes the structural cap and lets the metric count run toward the program's thousands, with the expensive/lumpy chain work built demand-pulled (per customer entitlement, D475) instead of speculatively. This is the natural extension of two already-decided moves: D475 (contracts derived from entitled metrics) and D498 (chain-ready decoupled from data-ready). It adds one tier below chain-ready. Doctrine-compatible: creation vs runtime were already separate units; this separates catalog-definition from chain-materialization, both still upstream of runtime.

## Decision

Metric authoring runs CATALOG-FIRST. An MC materializes to DRAFT bound only to BCF business-concepts + a grain entity, with NO dependency on a canonical/observation/source contract chain. The DRAFT MCV is the catalog tier. Promotion draft→active is gated solely by PE-MC-11 (chain-declaration resolvability), deferred until the SC/AC/OC/CC declaration spine for the metric's concepts is authored ON DEMAND. Contracts are demand-derivable from metrics (D475 onboard-metric reverse-walk); metrics are not derivable from contracts — so the catalog leads and the chain follows.

**Empirical proof (SES-f77411, 2026-07-07):** line_item_extension_amount (MC 5637ba5b) materialized to draft on the Customer Invoice Line Item grain (entity 734d2c51), which has ZERO active canonical contracts. PE-MC returned all-PASS except PE-MC-8 (deferred-PASS) and PE-MC-11 = REJECT (declared-source-resolvability). Materialization (M12.5), C-FX, and PE-MC-1..10/12..16 are all chain-independent; PE-MC-11 is the single gate that couples an MC to the chain, and it fires only at activation.

**Three-tier lifecycle (extends DEC-a6cdae / D498):**
- **catalog-defined** — draft MCV, BCF-bound, panel-approved, deduped, fixture-verified; chain-independent. The catalog.
- **chain-ready** — active MCV; PE-MC-11 passed (declaration spine exists); bindings/gate-fields projected. (D498 definition of done, now the SECOND tier.)
- **runtime-live** — data-backed + payload-verified via the data program (D498 / TSK-72a0a7). Unchanged.

**Guardrails (binding):**
1. BCF vocabulary is the true creation-time prerequisite (Invariant IV — explicit references). Where a measure/filter/grain concept does not exist in BCF, that BCF authoring precedes the MC. BCF is the demand-independent vocabulary layer.
2. STATUS HONESTY: never claim chain-ready or runtime-live for a catalog-defined draft. A derived read must distinguish the three tiers; catalog counts are reported as 'defined', never as capability.
3. Each catalog draft still costs a panel run (dedup + C-FX + M10 verifier apply at materialize). Dedup-first (§0.5) still gates — catalogue is cheaper than chain, not free.
4. Foundation-clean: the six invariants govern EVALUATION, not catalog definition. A draft produces no meaning; it is inert until evaluated through the boundary at runtime. Defining an MC ahead of its chain violates no invariant.
5. Do NOT author a catalog draft whose measure can never be source-observed by ANY system (authorability triage's 'chain-ABLE ever?' test replaces the old 'chain-ready now?' gate) — this prevents permanent zombie definitions.
