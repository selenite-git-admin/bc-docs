# Runtime Drift Detection — payload vs catalog probe, classification dispatch, S3 quarantine route, ticket lodging

## Context

Today the runtime detects schema delta only via per-record AJV validation failure, with no comparison against the source catalog and no path from "reader observed a new field" to "operator notices, runs catalog scan, authors new SC version." The admission contract envelope already declares validation_policy ('block' / 'quarantine' / 'warn') and outcome_thresholds, but admission.service.ts ignores both — every rejection takes the same path (422 + rejection evidence + progression status='rejected'). S3 archive only writes admitted records; quarantine route is unimplemented. No ticket service exists. The founder's documented design (lodge ticket on minor delta, quarantine to S3 + ticket on major delta, block batch on critical) was correctly captured in the admission-v1 meta-schema and DEC-3ee0f6 (per-tenant S3 archive bucket) but never wired in code. This ADR locks the bridge: runtime probe + classification dispatch + ticket service + S3 quarantine prefix, all consuming the existing source-change-classification policy and writing into the MLS substrate.

## Context

Sibling to DEC-e7b7b3 (D392, MLS State Substrate) and the v3 chapter `operating-model/source-change-classification.md` (Source Change Classification and Version-Bump Policy). This ADR locks the runtime side of the same arc.

What's documented today (verified during SES-5f806e):

- **Admission contract envelope** (`admission-v1.json` meta-schema) declares `validation_policy: 'block' | 'quarantine' | 'warn'` plus per-rule `severity` and `action` and batch `outcome_thresholds`
- **S3 archive** (DEC-3ee0f6 / D379) provisions per-tenant Object-Lock buckets `bc-archive-{slug}-{env}` for admitted records, written via `S3ArchiverService`
- **Five admission outcomes** documented in operating-model chapter `admission-and-observation.md`: `admitted | quarantine | warn | log | block`
- **v2 cc-creation-sop** plus v2 ADR-0002/-0003/-09b8e6/-05140c lock the version-bump classification (additive/destructive); promoted to v3 chapter `source-change-classification.md`

What's actually implemented (verified by reading admission.service.ts + contract-validation.service.ts + s3-archiver.service.ts):

- AJV with `strict: false` runs structural + identity + temporal + custom rules against the AC envelope
- Failed validation → `recordAdmissionRejection` + 422 + progression `status='rejected'`. **`validation_policy` is ignored.** Every reject takes the same path
- S3 archiver writes accepted records only — no quarantine route
- No ticket service exists in code. Closest analog is rejection evidence + 422
- All seeded admission contracts use `validation_policy: 'block'`. The `'quarantine'` and `'warn'` paths have likely never been exercised

The gap: documented design + meta-schema + tenant S3 buckets are all in place, but the runtime never differentiates by `validation_policy` and never lodges tickets. Drift exits the runtime exclusively via per-record rejection — the catalog feedback loop is manual.

This ADR closes that gap.

## Decisions

### D-1. Runtime drift probe at the admission boundary

A new probe runs as part of the admission sequence (after structural validation, before evidence emission):

**Step 5 in the admission sequence (per `operating-model/admission-and-observation.md`)** is augmented:

```
5a. Drift probe — diff observed payload field-set against active catalog
5b. Batch threshold evaluation (existing — per AC outcome_thresholds)
5c. Source Object emission (existing)
```

The drift probe operates per-record but aggregates per-batch:

- For each observed record, compare its field set to the source_field rows registered for the source_table the OC is bound to
- Detect three delta kinds:
  - **added_fields** — fields present in the payload, absent from `source.source_field` for this table
  - **missing_required_fields** — fields declared in the OC's `field_mappings[]` as `required: true`, absent from the payload
  - **type_mismatches** — fields where the payload's runtime type differs from `source.source_field.data_type` (best-effort; type inference at runtime is approximate)
- Aggregate per-batch: `{added_fields: [...], missing_required_fields: [...], type_mismatches: [...], affected_record_count: N}`

Type inference is best-effort because runtime payloads are JSON (typed strings vs numbers vs nulls); this ADR does not require lossless type inference. Where type comparison is ambiguous, the probe records `unknown` and does not flag a mismatch.

### D-2. Classification — uses the source-change-classification chapter

Each detected delta is classified per the chapter's `(change action) → (kind)` table:

| Delta kind | Source change classification | Severity |
|---|---|---|
| added_field (per record) | additive (new field appears) | minor |
| missing_required_field (per record) | destructive (field declared required, absent) | major |
| type_mismatch (per record) | destructive (incompatible type) | major |

The chapter defines the policy; this ADR consumes it. If the classification table evolves (e.g., new "compatible-widening" type cases), the runtime dispatch follows the chapter without ADR-level change.

### D-3. Dispatch — honor `validation_policy` per AC

The admission contract's `validation_policy` (already in the envelope, currently ignored by the runtime) is now enforced:

| `validation_policy` | Detected delta severity | Runtime action |
|---|---|---|
| `warn` | minor (additive) | Admit the record. Lodge a `warn`-severity ticket. Emit `MLS-02.field_observed_not_in_catalog` event |
| `warn` | major (destructive) | Admit the record (warn-policy is permissive). Lodge a `warn`-severity ticket with major-classification annotation |
| `quarantine` | minor (additive) | Admit with annotation. Lodge a `warn`-severity ticket. Same as warn-policy minor |
| `quarantine` | major (destructive) | **Reject the record. Route to S3 quarantine prefix (D-4). Lodge a major-severity ticket. Emit `MLS-02.destructive_drift_detected` event** |
| `block` | minor (additive) | Admit the record (block-policy applies to violations, not catalog-extension drift). Lodge a `warn`-severity ticket |
| `block` | major (destructive) | **Reject the entire batch (existing rejection path) + lodge a critical-severity ticket. Emit `MLS-02.destructive_drift_detected` event** |

The dispatch matrix:

- **minor (additive) drift** never blocks runtime regardless of policy. Worst case: ticket lodged for catalog onboarding
- **major (destructive) drift** under `warn` admits-with-warning (operator must monitor); under `quarantine` quarantines the record; under `block` rejects the batch
- The path the platform recommends per AC is `quarantine` for most contracts (preserves data, surfaces the issue, doesn't break runtime). `block` is appropriate for safety-critical contracts where bad data is worse than no data. `warn` is appropriate for low-stakes contracts where operator latency is acceptable

### D-4. S3 quarantine route

The S3 archiver gains a quarantine path. New layout under the existing per-tenant bucket (DEC-3ee0f6):

```
bc-archive-{tenant_slug}-{env}/
  ├── canonical/{contractName}/{versionCode}/{yyyy-mm-dd}/{runId}.jsonl       (existing — admitted)
  ├── metric/{contractName}/{versionCode}/{yyyy-mm-dd}/{runId}.jsonl          (existing)
  └── quarantine/{contractFamily}/{contractName}/{versionCode}/{yyyy-mm-dd}/{runId}.jsonl   (NEW)
```

The quarantine prefix carries the same JSONL line format as admitted records, with two added envelope fields:

```json
{
  "tenantSlug": "...",
  "runId": "...",
  "recordId": "...",
  "archivedAt": "...",
  "payload": { ... },
  "drift_classification": "destructive",      // additive | destructive | semantic
  "drift_evidence": {                         // matches MLS-02 evidence_schema
    "added_fields": [...],
    "missing_required_fields": [...],
    "type_mismatches": [...]
  }
}
```

Object Lock COMPLIANCE mode and 7-year retention apply to the quarantine prefix unchanged. Quarantined records are immutably preserved for the same retention period as admitted records.

`S3ArchiverService` gains a `quarantineRecord(args)` method paralleling `archiveRecord(args)`. Same fail-closed semantics — if S3 quarantine write fails, the runtime logs `OPS.s3_quarantine_write_failed` and the record is rejected per the contract's policy (no quarantine ≠ no rejection; the rejection still happens, just without the durable quarantine artifact).

### D-5. Ticket service — `OperationsTicketService`

A new service emits structured tickets. Tickets are durable records that surface in the bc-admin Tickets section (existing nav slot at `/platform/tickets`).

New platform table `operations.incident_ticket`:

| Column | Type | Notes |
|---|---|---|
| `ticket_id` | uuid PK | |
| `kind` | text | `runtime_drift_minor` / `runtime_drift_major` / `runtime_drift_critical` (initial set; extensible) |
| `severity` | text | `warn` / `major` / `critical` (CHECK constraint) |
| `tenant_slug` | text nullable | when applicable |
| `contract_family` | text | `source` / `admission` / `observation` / `canonical` / `metric` |
| `contract_id` | uuid nullable | the affected contract |
| `contract_version_code` | text nullable | the affected version |
| `source_table` | text nullable | for source-side tickets |
| `recommended_bump` | text nullable | `minor` / `major` per the source-change-classification table |
| `summary_text` | text | human-readable summary |
| `evidence_json` | jsonb | structured per kind (e.g., for drift: added_fields, missing_required_fields, type_mismatches) |
| `state_code` | text default `'open'` | `open` / `acknowledged` / `resolved` / `wont_fix` |
| `created_at` / `updated_at` | timestamptz | |
| `resolved_at` | timestamptz nullable | |

Tickets carry the same shape as the failure vocabulary's emission — `code` references a `metric.mls_failure_code` row when applicable, `evidence_json` matches that code's `evidence_schema`. This keeps the ticket surface consistent with the broader observability vocabulary (DEC-9d7a5c).

`OperationsTicketService.lodge(args)` writes a row. The service is synchronous — ticket lodging is part of the admission act when the policy demands it. Failure to lodge a ticket logs `OPS.ticket_lodge_failed` but does not block runtime (best-effort, like the recorder).

### D-6. MLS event emission via the substrate

Each detected drift event is also written to the MLS substrate (DEC-e7b7b3) via `MlsStateRecorderService.record(event)` with `source_event_kind='drift_detected'`. The trigger binding for `drift_detected` writes to MLS-02 with the appropriate code (`MLS-02.field_observed_not_in_catalog` for additive, `MLS-02.destructive_drift_detected` for destructive).

This makes drift events visible in the MLS state ledger, the Inspector, and the Operations dashboard — same surface as every other MLS state change.

### D-7. The classification escape hatch — semantic drift

Semantic drift is by definition undetectable by the runtime probe (no structural diff). This ADR does not introduce a runtime semantic-drift detector. Semantic drift surfaces through:

- AI-assisted catalog review (out of scope for this ADR; future surface)
- Operator-reported anomalies in downstream metric snapshots
- Periodic statistical review of source field value distributions (future)

When semantic drift is later identified, the operator manually files a ticket via the Tickets surface, and the standard major-version-bump cascade follows.

### D-8. Backward compatibility — existing rejection path preserved

Until the new dispatch ships, existing AC `validation_policy='block'` behavior is preserved exactly: 422 + rejection evidence + progression `status='rejected'`. The new probe + dispatch is additive and gated:

- Phase 1: ship the probe + the ticket service. Initial behavior: probe emits events, lodges tickets, but does NOT change admission outcome (observe-only mode)
- Phase 2: enable `validation_policy` dispatch per contract. Operators opt contracts into the new behavior one at a time, watching ticket volume and adjusting policies before broader rollout
- Phase 3: default new contracts to `validation_policy='quarantine'` and migrate existing `'block'` contracts where appropriate

This honors D268 session discipline (one-then-many, prove on ONE before scaling). The full rollout is multi-session; the Phase 1 ship is independently valuable for observability even before any contract changes behavior.

## Consequences

**Positive:**
- Documented quarantine + ticket design finally implemented end-to-end
- Catalog feedback loop becomes automatic — operators learn about drift from tickets in bc-admin, not from rejection-rate spikes in logs
- `validation_policy` enum becomes load-bearing instead of dead code
- S3 quarantine preserves rejected records for forensic + reprocessing
- MLS-02 events flow into the substrate, making drift visible alongside all other state
- Phased rollout (D-8) protects existing behavior

**Negative:**
- New table (`operations.incident_ticket`) — DDL change, requires founder approval
- Type inference at runtime is best-effort; some legitimate mismatches will be missed (false-negative)
- Aggressive `quarantine` policy on a noisy source could fill S3 fast; retention is 7yr per Object Lock so cost matters
- Per-record drift probing adds cost to admission — measure overhead in Phase 1 before broader rollout

**Neutral:**
- The probe doesn't change which records admit; it only adds a parallel detection + ticket emission path. Phase 1 is purely additive
- Semantic drift remains undetected by automation; humans + AI-assisted review remain the only path

## Out of scope (separate decisions)

- Semantic drift detection (AI-assisted statistical review of source field value distributions) — future ADR when bc-ai maturity warrants
- Auto-authoring of major version bumps on detected destructive drift — out of scope; operator-driven for now
- Ticket assignment / owner / SLA — `operations.incident_ticket` schema permits adding `assigned_to_user_id` later; not in initial DDL
- Cross-tenant aggregation of ticket counts — comes via the MLS substrate's cross-cutting query surface, not via direct ticket-table query
- Reprocessing of S3-quarantined records after a major bump fixes the contract — out of scope; manual replay via existing fact-reader patterns
