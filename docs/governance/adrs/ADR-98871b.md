# D268 Structured Self-Audit — session close enforces structured discipline report + backend stats

## Context

**Why v1.1 now.** D332 landed the registry. Left unused, side_effect metadata rots. D268 Rule 7 is the natural first consumer — it already mandates a discipline audit but has no programmatic surface. v1.1 closes that gap.

**Why "informational not blocking".** Blocking session close on discipline heuristics before we see real false-positive rates is premature. Rule 4 "bulk write" threshold (10+ of the same activity_type) could trip on legitimate batch operations. Ship informational first, observe real sessions for 2-3 weeks, then escalate to blocks in v1.2 with evidence.

**Why no new DB schema.** The ISO 27001 change_records table already has report_json for session close. Nesting discipline_audit inside is semantically correct (self-audit IS part of the change record) and avoids Database Change Protocol overhead.

**Why reframe "destructive gate" to "structured audit".** The original D332 framing assumed destructive tools would exist. They don't (0 in v1). The real value of side_effect is as an audit lens, and the real value of this work is making Rule 7 structured. Honest reframe beats pretending the destructive-gate story works.

**Alignment**
- D268: directly implements Rule 7 enforcement
- D332: consumes mcp_tool_registry in the activity_type → side_effect mapping
- D221 (ADR-first): this ADR auto-generates the canonical file
- D305 (SSOT): discipline reports are computed on-demand from activity (authoritative source), not persisted ad-hoc

**Non-goals.**
- Not replacing Claude's prose self-audit — augmenting it
- Not a hard gate on destructive actions — v1.2
- Not a dashboard — API-only in v1.1

## Decision

v1.1 of the MCP Tool Registry governance work. D268 Rule 7 mandates a self-audit at every session close but currently trusts Claude's prose. v1.1 introduces structured enforcement.

**What changes**

1. New library `src/lib/session-audit.js`:
   - `computeDisciplineReport(session_id)` returns {
       activity_counts: by activity_type,
       bulk_write_flags: activity_types with count > 10 (Rule 4 trip),
       checkpoint_gaps: windows of write activity with no checkpoint in between (Rule 6 trip),
       side_effect_lens: opportunistic mapping activity_type → mcp_tool_registry.side_effect, summarised
     }
   - Pure read; no mutations.

2. Modified session close (PATCH /api/projects/:slug/sessions/:uid):
   - Compute and attach disciplineReport to the response.
   - Does NOT block close if report flags concerns. Flagging is informational in v1.1 (blocking logic deferred to v1.2 after we see false-positive rates).

3. Modified devhub_session_close MCP tool:
   - Adds required self_audit_json parameter with schema: { rules_relevant: string[], rules_tested: string[], near_misses: string[], rules_obeyed: boolean }
   - Automatically injects self_audit_json into the session's change_record.report_json.discipline_audit subkey (no new DB column needed — uses existing ISO 27001 change record infrastructure).
   - Returns combined text: Claude's narrative audit + backend's computed stats.

4. No DB schema change:
   - self_audit_json lives inside existing change_record.report_json
   - No new tables
   - No new columns on sessions

**What doesn't change**
- mcp_tool_registry schema (D332 v1 is stable)
- Activity logging (reuses existing writes)
- Any other session endpoints

**Why Design C.1 over C.2**
Design C.2 (add MCP tool-call logging in every handler) would give cleaner side_effect queries but requires touching 53 tool handlers and doubles write volume to activity table. C.1 uses the activity log as-is. The mcp_tool_registry side_effect column is still useful as a standalone lens (`devhub_mcp_list --side-effect=destructive` to audit the surface) but isn't the primary gate hook in v1.1.

**Deferred to v1.2**
- Active blocking of session close when discipline concerns exceed a threshold
- Per-MCP-call logging (activity_type='mcp_call', message=tool_name) for true tool-history traces
- UI surface for discipline reports (per-session audit page)
- Cross-session trend analysis (rule-violation frequency over time)
