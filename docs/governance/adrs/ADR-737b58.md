# Developer Guides — per-repo codebase guides and code reviews in dev-guides/

## Context

See decision text below.

## Context

legacy v2 archive (D229/D230) covers architecture, data model, boundaries, contracts, and domain documentation. But no stream documents the **actual codebase** — how code is structured, what technology choices were made and why, or whether the code follows good engineering practices.

D230 reserved `dev-guides/` with prefix `GDE` but defined no internal structure.

The existing `architecture/backend-overview.md` is high-level and stale (references 42 tables/5 schemas — now 82 tables/12 schemas).

## Decision

`dev-guides/` contains two document types:

### 1. Codebase Guides (one per repo)

How the code is organized. A developer's map.

```
dev-guides/
  bc-core/
    index.md              — overview, tech stack, setup
    folder-structure.md   — annotated tree, what lives where
    patterns.md           — service pattern, controller pattern, guard/interceptor chain
    database-access.md    — Drizzle patterns, schema registration, migrations
    testing.md            — test strategy, vitest config, what to mock
    auth-flow.md          — JWT validation, tenant extraction, RBAC
  bc-admin/
    index.md
    folder-structure.md
    patterns.md           — Express + EJS + SQLite patterns
    api-integration.md    — how admin calls bc-core APIs
  bc-portal/
    index.md
    folder-structure.md
    patterns.md           — React/Vite patterns, state management, routing
    auth-flow.md          — Cognito login, token handling
    api-integration.md    — how portal calls bc-core APIs
  bc-ai/
    index.md
    folder-structure.md
    patterns.md           — FastAPI, Bedrock, RAG pipeline
  devhub/
    index.md
    folder-structure.md
    patterns.md           — Express + EJS + SQLite + MCP server
```

### 2. Code Reviews

Systematic quality findings per repo. Output of actual deep code reviews — anti-patterns, security gaps, consistency issues, actionable recommendations.

```
dev-guides/
  reviews/
    bc-core-review.md
    bc-admin-review.md
    bc-portal-review.md
    bc-ai-review.md
    devhub-review.md
```

### Technology Decisions

Technology choices (why NestJS, why Drizzle, why PostgreSQL) are recorded as ADRs in `decisions/` — not duplicated in dev-guides. Dev-guides reference the relevant ADRs.

### Frontmatter

```yaml
---
id: GDE-xxxxxx
slug: bc-core-folder-structure
repo: bc-core
type: codebase-guide | code-review
status: draft | current | stale
last_validated: 2026-03-30
---
```

### Scope boundaries

- NOT domain documentation (stays in dossiers)
- NOT API reference (stays in architecture/ or dossier section 04)
- NOT operational runbooks (stays in playbooks/)
- NOT help articles (stays in help/)

## Options Considered

1. **New stream (I01-I05)** — too heavy; dev-guides are supplementary, not a parallel documentation axis
2. **Embed in dossiers as section 10** — wrong granularity; dossiers are per-component, code structure is per-repo
3. **dev-guides/ as planned by D230** — right level, already reserved, just needs internal structure defined

## Consequences

- Every repo gets a navigable codebase guide before any new developer touches it
- Code reviews produce documented findings that persist and can be tracked
- Technology decisions stay as ADRs (no duplication)
- `architecture/backend-overview.md` becomes stale and should be retired once bc-core guide is complete
