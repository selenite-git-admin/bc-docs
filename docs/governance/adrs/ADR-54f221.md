# Controlled Semantic Refactor — three-layer model (Interpretation Surfaces / Implementation Names / Compatibility Names) — supersedes DEC-7a1c98

## Context

The Step 2 inventory filed 2026-06-22 (mms-step2-operator-facing-docs-ui-labels-inventory) revealed the original six-step model's Step 2 was too narrow — operator-facing UI + runbook prose only. The interpretation surface that actually carries the legacy codes for human and AI consumption is broader: inline source comments (originally an allowed alias context per DEC-7a1c98 Decision 3), test descriptions, error messages, generated operator reports, developer docs, and bc-ai prompt files. The axis that distinguishes 'human reading for understanding' from 'compiler reading for behavior' from 'external/persisted consumption' is the natural reversibility tier: Interpretation Surfaces are fully reversible (text-only); Implementation Names are compiler/test-suite-bound; Compatibility Names cannot be rewritten (additive only, Invariant III). bc-ai prompt files are interpretation in form but model-input in fact — renaming legacy codes in prompt bodies can shift the model's reasoning distribution; the new model gates prompt renames behind regression discipline. Telemetry / log / dashboard consumers are the sleeper risk DEC-7a1c98 buried in Step 5; the new model lifts it into Compatibility Names with a pre-inventory requirement before that layer can open. The 'inline comments as permanent alias context' rule in DEC-7a1c98 Decision 3 contradicts the goal of code-review clarity; the new model treats comments as interpretation (renamed) with transition-window annotation only. This is a material model change, not a sequencing reshuffle — therefore a successor ADR rather than an amendment.

## Three-layer refactor model

The Controlled Semantic Refactor program for the opaque workflow-code family is organised into three layers, each with a distinct reversibility tier and execution discipline.

### Layer 1 — Interpretation Surfaces

Surfaces whose purpose is to explain, interpret, review, hand off, or reason about the system. Includes: doctrine docs (`bc-docs-v3/docs/foundation/`, `/docs/operating-model/`, `/docs/implementation/`, `/docs/onboarding/`, `/docs/adrs/`); current planning, audit, and DBCP packets; UI labels, subtitles, paragraph copy, aria-labels, toast messages; AI prompt files (`bc-ai/...`); inline source comments; test descriptions; error messages and exception text; generated operator reports; developer docs.

Semantic vocabulary is primary. Legacy codes may appear as parenthetical aliases on first mention per page/file/context only — they are NOT a permanent alias context (correction from DEC-7a1c98 Decision 3). Reversibility: full (text-only changes). bc-ai prompt files are a SEPARATELY GATED cluster within this layer because prompt text is model input — see the bc-ai prompt cluster discipline below.

### Layer 2 — Implementation Names

Code identifiers and structure. Includes: class names, service names, method names, function names, filenames, internal route aliases, non-persisted internal symbols, exception class names.

Eventual semantic naming under a compiler/test-suite-backed refactor with re-export aliases during the transition window. Per-metric controllers (`mcf-arpi-materialization.controller.ts`, `billing-volume-retry-unlock/`) are quarantine or fold-in candidates inside this layer (was Step 6 in DEC-7a1c98). Reversibility: high but bound by test-suite discipline; IDE refactor + import-path sweep required. Layer 2 cannot open until Layer 1 is materially complete on the surfaces a Layer 2 rename touches.

### Layer 3 — Compatibility Names

Persisted or externally-depended names. Includes: DB enum values (`governance_state_code`, `verdict_code`, `pe_check_code`, `action_code`, etc.); telemetry keys, log-pipeline dimensions, dashboard filter values; event names, API payload values; old route paths preserved as backward-compatible aliases; immutable evidence (database rows already written, log lines already emitted).

Additive only — new semantic values join existing legacy values; writer side may emit either; reader side accepts both; **historical evidence is never rewritten** (Foundation Invariant III). Reversibility: none for historical rows; additive-only for the writer side. Layer 3 cannot open until a telemetry / log / dashboard pre-inventory is filed identifying downstream consumers and their dependencies.

## Preserved from DEC-7a1c98

- **Scope of legacy codes covered** (Decision 1 of DEC-7a1c98): M-track process codes (`M12`, `M12.5`, `M13`, `M14`, `M15`, broader `M0`–`M20` used as process names); publication/coverage/precondition codes (`PE-MC-*`, `L-V1*`, `C-FX-*`); BCF-style process/action shorthand (`B6`, `C5`, `F3`, `C1`/`C2`/`F1`/`F2` where used as process names).
- **Decision identifiers** (`DEC-…`, `D-…`, ADR UIDs) are NOT renamed by this ADR or its successors.
- **Migration appendix** mapping legacy → semantic (Decision 6 of DEC-7a1c98): preserved verbatim, including the PE-MC-N exhaustive mapping (PE-MC-1 → G9, PE-MC-2/3/4/6/7 → G2, PE-MC-5 absorbed into G5, PE-MC-8 retired to Runtime Evaluation, PE-MC-9 → G8, PE-MC-10 → G5, PE-MC-11 → G3, PE-MC-12 → G4), the M-track stage mapping (M12 → Metric Draft Review, M12.5 → Metric Contract Materialization, M13 → Publication Eligibility Evaluation, M14 → Metric Activation, M15 → Metric Supersession), the BCF B6/C5/F3 mapping, and the deferred per-code mapping for `L-V1*` / `C-FX-*` / `C1`/`C2`/`F1`/`F2`.
- **MMS hierarchy** (Decision 5 of DEC-7a1c98): MMS umbrella; MCF as grammar/substrate layer NOT renamed (`mcf.*` schema, `DEC-c3e57f / D422`, `bc-core/src/registry/mcf/` retain their established names); Metric Engine as runtime executor; Creation/Recovery/Runtime/Evolution Tracks as doctrinal track names.
- **No-execution-on-filing rule**: filing this ADR does NOT start any rename or refactor work. Each cluster within each layer requires its own operator-authorized session at the point of execution.
- **Foundation Invariant III**: historical evidence is never rewritten — database rows under legacy values stay under those values forever; historical artifact filenames are preserved; persisted log/audit rows are immutable.

## Changed from DEC-7a1c98

- **Six-step sequence** (DEC-7a1c98 Decision 4) is **replaced** by the three-layer model. The sequence: Layer 1 → Layer 2 → Layer 3, with each layer's clusters opening on independent operator authorization.
- **Inline source comments** (DEC-7a1c98 Decision 3) are NO LONGER a permanent allowed alias context. They are renamed at Layer 1 (Interpretation Surfaces). `// legacy: M12` style annotations may survive as a transition-window convenience during the Layer 1 sweep — not as a permanent alias.
- **HTTP route aliases** move from DEC-7a1c98 Step 4 (their own step) into Layer 2 (Implementation Names) where they sit alongside class/service/method renames. External-facing route deprecation (the legacy route path as a backward-compatible alias visible to external clients) moves to Layer 3 (Compatibility Names).
- **Per-metric controller quarantine** (DEC-7a1c98 Step 6) is folded into Layer 2 as one of its execution clusters.
- **Telemetry / log / dashboard** concerns (buried in DEC-7a1c98 Step 5 as "additive only") are lifted to Layer 3 with an explicit pre-inventory requirement.

## bc-ai prompt cluster discipline (Layer 1, gated sub-cluster)

Prompt files (e.g. `bc-ai/.../m12-panel-{maker,checker,moderator}.v1.md` or equivalent paths) contain legacy codes inside the prompt body that the model reads as input. Renaming legacy codes in prompt bodies may shift the model's reasoning distribution on edge cases.

Treatment:

- Identified as a SEPARATE gated cluster within Layer 1.
- Cannot rename without a before/after regression fixture run against representative prior panel-run transcripts.
- Regression delta thresholds (e.g. `verdict_code` distribution change tolerance, `candidate_proposal` shape change tolerance) must be defined and met before the rename commits.
- A prompt rename that shifts model output beyond the threshold halts the cluster and requires operator decision — either accept the shift with documentation, or hold the prompt at the legacy wording.

## Telemetry / log / dashboard pre-inventory (Layer 3 precondition)

Layer 3 cannot open until a separate inventory is filed identifying:

- All log emission keys / structured-log fields mentioning legacy codes.
- All telemetry / metrics dimensions / dashboard filter values that depend on legacy codes.
- All downstream consumers (CloudWatch dashboards, alerting rules, observability tools, defect-tracking integrations, log-pipeline subscribers) that read those keys/values.
- The migration impact per consumer (whether they need parallel-emission, parallel-read, or coordinated migration).

This inventory is new work that DEC-7a1c98's six-step model did not require. It is the sleeper risk Layer 3 surfaces; opening Layer 3 without it risks breaking downstream telemetry consumers.

## Filing posture

Filing this ADR completes the doctrine layer of Layer 1's preparation. It does NOT authorize Layer 1 execution, Layer 2 execution, or Layer 3 execution. Each execution session opens separately under operator authorization. The Recovery Track doctrine, the MMS chapter, the Audit, and the pre-doctrine note remain in force — only the rename-sequencing model changes from six-step to three-layer.
