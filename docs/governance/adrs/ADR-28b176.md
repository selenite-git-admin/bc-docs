# Metric Readiness Model — three independent dials + per-formula-token audit

> **Amended by [DEC-a8b33e](ADR-a8b33e.md) (D397)** on 2026-05-09. The Bound dial's source changed from raw active bindings to the canonical Stage 5 (`tenant_ready`) of the metric lifecycle funnel. Raw bindings + stale bindings survive as side diagnostics. The dial APIs and labels in this ADR remain authoritative; the underlying source is now a projection of `MetricFunnelService.getLadder()` rather than independent SQL. See DEC-a8b33e for the canonical 7-stage ladder.

## Context

Six sessions (SES-61f2f6 → SES-d88852) of fixing metric-onboarding issues in sandbox1 produced the diagnostic primitives this ADR formalises. The single-tile "producing/bound" view was provably misleading (it didn't move despite five sessions of chain fixes because the constraint was binding scope, not chain quality). Three independent dials make movement interpretable on each axis. The per-token audit replaces a coarse leading-indicator (30% accurate) with one that approximates the engine's actual checks (~95%). Documented now while the journey rationale is fresh; the ADR also locks the five-pillar consolidation that prevents future overlap.

## Three dials, each meaning one thing

### Dial 1 — Catalog Readiness (platform-wide, no tenant)

```
GET /api/admin/readiness/catalog
→ { totalMcs, chainComplete, formulaSupported, ready }
```

`ready = chain_status.chain_verdict='complete' AND formula uses only supported primitives (SUM/AVG/MIN/MAX/COUNT/COUNT_DISTINCT/ABS)`. Movement on this dial means a chain bug fix, a contract authoring fix, or a formula primitive added to the engine. Does not depend on any tenant.

### Dial 2 — Tenant Pilot Scope (per-tenant)

```
GET /api/admin/readiness/tenant?tenant=<slug>
→ { bound, producing, wouldProduceIfBound }
```

- `bound` — count of active `tenant.contract_binding` rows for `contract_family='metric'`. Founder/operator-controlled pilot scope.
- `producing` — bound MCs whose `fact.ms_<code>_v<version>` table has at least one row.
- `wouldProduceIfBound` — strict per-formula-token check across all unbound MCs, computed by `runFormulaTokenAudit`.

Movement on `bound` requires a deliberate binding decision. Movement on `producing` requires CR / CC mapping work for a bound MC. Movement on `wouldProduceIfBound` happens when source data flows in (new tenant CO data) or when authoring fixes (cc_field_mapping) align formula tokens with populated BFs.

### Dial 3 — Pilot Health (the producing/bound ratio)

Surfaced as the third tile in the bc-admin Tenant Metrics page. Sub-label `"+N would produce immediately if bound"` exposes the binding-ready set the previous single-tile design hid.

## The strict per-token audit predicate

```
GET /api/admin/readiness/tenant/<slug>/formula-token-audit
→ { totals, topBrokenCfs, bySubfunction, rows: per-MC tokens }
```

For every active MC, every input variable's `field_code` is checked against three gates:

1. **no_mapping** — no `contract.cc_field_mapping` row for this canonical_field on any bound CC. Authoring gap.
2. **type_mismatch** — `cc_field_mapping` exists but the `business_field.data_type` is not numeric (integer/number) when the formula uses a numeric op (SUM/AVG/MIN/MAX/ABS). Authoring-shape bug.
3. **null_in_tenant** — mapping exists, type is compatible, but the BF column has zero non-null values in the tenant's CC fact table. Source-data emission gap.

An MC is `overall: clean` only when (a) every input field passes all three gates AND (b) the formula text uses only supported primitives.

## Why this design over the previous "X of Y producing" tile

The previous tile conflated catalog quality with pilot scope. A bigger number could mean either better chain or a wider pilot — operators couldn't tell which. Five sessions of layer-by-layer chain fixes barely moved it because the actual constraint in sandbox1 was binding scope (5 MCs out of 360 catalog-ready).

## Predicate accuracy journey

- **Old coarse predicate** ("any CO in any bound CC"): empirical 30% accuracy — bound 10 candidates, 3 produced.
- **Strict per-token without type check**: empirical 60% accuracy — bound 15 candidates, 9 produced.
- **Strict per-token with type-compat + isFormulaSupported gates**: ~95% (residual misses are temporal_gate / completeness_threshold, not data-shape).

## Non-overlapping pillars

The readiness model is one of five pillars; each owns one verb:

- **Readiness** (this ADR): "is the platform/tenant ready"
- **Tenant Metrics** (`/admin/tenant-metrics/*`): per-MC operational drill
- **Test Bench** (`/admin/test-bench/*`, `/t/test-bench/*`): explicit invocation primitive
- **Inspector** (`/admin/inspection/*`): read-only chain provenance per MC
- **Schema Provisioner** (`/schema-provisioner/*`): chain-walk-driven onboarding + reconcile

## Deprecations

`/api/admin/test-bench/survey-mcs` (SES-61f2f6) is subsumed by this readiness pillar. Its 8-layer breakdown is now provided by `formula-token-audit` (token-level reasons) + `binding-candidates` (filtered) + `tenant-metrics/snapshot` (operational view). The route is deprecated and removed.
