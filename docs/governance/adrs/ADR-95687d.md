# Typed-First Tenant Runtime — Schema Provisioner as declarative reconciler

## Context

BareCount accumulated two apparently compatible but actually conflicting decisions:

- **DEC-a25931 (Mar 25 2026, decided):** "JSONB-first for boundary payloads; typed tables immediately after via D210." Pragmatic call to let E2E stand up fast while the DDL-generation machinery caught up.
- **DEC-2658ff (D210v3, Mar 25 2026, proposed):** "Drop JSONB from RDS progression; typed tables in RDS + raw JSONL in S3 with Object Lock (WORM)." The end-state.

In the months since (D300-D368), every new feature — the grain engine (ADR-c0290f), the readiness scheduler (D316), the fiscal stack (D363/D364/D365), the semantic verdict gate (D366), the dim conformance (D367), the 6-schema tenant DB layout (D368) — has been built on the JSONB-backed path. D210v3 has stayed `proposed`. Schema Provisioner remains unbuilt.

That is the local optimum of each session, not the global optimum of the platform. As of 2026-04-21:

- **There are two potential sources of truth** for the same business record (progression JSONB + fact typed projection), with the fact layer empty and the progression carrying the data. This is exactly the redundancy D162 Rule 1 forbids.
- **Sandbox1's `fact.*` schema exists and is empty**, while `envelope.metric_snapshot.business_key` (current schema) already carries `fiscal_period` as a JSONB extract — the exact "querying JSONB" anti-pattern.
- **Every code path that writes a canonical object or a metric snapshot today writes JSONB**, deepening the debt each time a new CC or MC activates.
- **No tenant chain had successfully INSERTed into the new tenant DB layout until D365's D.2 session**, when a Drizzle/DDL column-name drift was exposed and fixed. The typed-layer has been conceptual only.

DEC-a25931's premise — "E2E isn't working yet; ship JSONB first" — is no longer true. E2E is working. The conditions that justified the tactical detour have been met. The detour must now end.

## Problem

The JSONB-backed progression and the as-yet-unbuilt typed `fact.*` layer cannot both be the source of truth. Leaving the architecture mid-flight violates D162 Rule 1 ("No JSONB for queryable data"), contradicts the locked three-layer model of D210v3, and leaves D368's `fact` schema empty while `progression` grows unbounded.

Additionally, the Schema Provisioner spec as written (evolving authority, P09-platform-services/schema-provisioner) triggers table creation on **contract activation**. That is one of three legitimate trigger moments. It ignores:

1. **Late tenants** — a contract activated long ago; a tenant newly onboarded must still get the table.
2. **Late contracts** — a tenant onboarded long ago; a newly-activated contract reachable from its connectors must produce the table retroactively.
3. **Drift** — any event missed, any DDL applied outside the provisioner, must eventually be corrected.

A trigger-based model papers over these. A **reconciler** model resolves them.

## Decision

### 1. Typed-first. JSONB payloads are removed from RDS.

`progression.*` tables in the tenant DB are reduced to **immutable metadata only**: identity (PK), structural FKs (predecessor row, contract, run), status/outcome codes, timestamps, `archive_key` pointing at the S3 archive for this run.

**Schema naming + table consolidation.** The current name `envelope` (D368) is a metaphor of a sealed letter around the payload. After D369 drops the payload, the table holds only the ordered instance record of boundary events — the *progression*. Foundation §2.1 already uses this word (*"Authoritative Object Progression"*).

D368 is amended on two axes:

1. **Rename**: `envelope` → `progression`.
2. **Fold**: `source_object`, `canonical_object`, `metric_snapshot`, `action_object` are **removed** from the schema. They were the "produced artifact" rows paired with a "boundary event" row (`admitted_record`, `evaluation`, `metric_evaluation`, `action_evaluation`). Once JSONB payload moves to `fact.*`, the produced-artifact row carries nothing the fact row cannot carry itself — PK, FK to its boundary event, timestamps. Keeping both is redundancy.

### 1a. Progression schema after D369 — 4 tables

One table per evaluation boundary. Each row is a **boundary event**; its result (if accepted) lives in the corresponding `fact.*` typed table.

| Progression table | Cardinality to fact | Role |
|---|---|---|
| `progression.admission` (was `admitted_record` + `source_object` + `observed_record`) | 1:0..1 with `fact.so_<sc>_v<ver>` | Admission event: SC + AC FK + version, `status` ∈ {admitted, rejected}, source_key (composite biz key), reader_id, tenant_id, env, timestamps, `archive_key` |
| `progression.canonical_evaluation` (was `evaluation` + `canonical_object`) | 1:0..1 with `fact.co_<cc>_v<ver>` | Canonical evaluation event: CC FK + version, `status` ∈ {accepted, rejected}, 1:1 FK → `admission`, timestamps |
| `progression.metric_evaluation` (was `metric_evaluation` + `metric_snapshot`) | 1:N with `fact.ms_<mc>_v<ver>` (one row per grain) | Metric evaluation event: MC FK + version, `status`, `input_references_json` (lineage; may move to `lineage.*`), timestamps |
| `progression.intervention_evaluation` (was `action_object` + `action_evaluation`) | 1:0..1 with `fact.ao_<ic>_v<ver>` | Intervention event: IC FK + version, `status` ∈ {triggered, not_triggered}, baseline metric-snapshot refs, timestamps |

**Observed vs admission**: Foundation §3.5 says *"Rejected observations emit Evidence but do not emit Source Objects."* Rejection happens at admission, not at a separate pre-boundary. One row per observation with `status` covering both outcomes is enough. No separate `observed_record`.

### 1b. The rule (load-bearing)

> *"Progression records **boundary events**. Fact records **produced artifacts**. S3 records **immutable raw payloads**. Each row in a progression table has a status; if accepted, a corresponding fact row exists and carries the business data."*

Each `fact.*` row carries its own PK (it IS the SO / CO / MS / AO) plus a FK to the progression row that produced it. Instance-level lineage (Patent §10) is preserved through the progression FK chain:

```
progression.admission
   ← progression.canonical_evaluation (FK admission_id)
     ← progression.metric_evaluation  (FK: input_references_json lists accepted CO fact rows)
       ← progression.intervention_evaluation (FK metric_evaluation_id + baseline refs)
```

Fact rows hang off their progression rows:

```
progression.admission.admission_id            → fact.so_<sc>_v<ver>.admission_id (FK)
progression.canonical_evaluation.id           → fact.co_<cc>_v<ver>.canonical_evaluation_id (FK)
progression.metric_evaluation.id              → fact.ms_<mc>_v<ver>.metric_evaluation_id (FK, 1:N per grain)
progression.intervention_evaluation.id        → fact.ao_<ic>_v<ver>.intervention_evaluation_id (FK)
```

### 1c. Why keep progression at all (not collapse it into fact)

1. **Rejected events have no fact row.** `progression.canonical_evaluation(status='rejected')` records the rejection; no `fact.co_*` row exists. Essential for audit.
2. **Operational reporting is shape-stable.** Counts like "how many admissions → canonical → metric this week?" read `progression.*.status`. These queries never depend on dynamic fact-table shapes.
3. **Execution chain FKs are the spine.** The four progression tables form the immutable instance-lineage chain (Patent §10). Fact tables vary per contract per tenant; progression does not.
4. **Fixed vs dynamic.** Progression is 4 fixed tables, D368-locked. Fact is hundreds, dynamic. Separating them keeps ops/audit queries stable while fact evolves freely.

### 1d. JSONB columns dropped

All payload / snapshot / validation JSONB that previously lived in `envelope.*` tables is removed. None of it survives on the progression side:

- `validation_result_json`, `contract_snapshot_json` → `evidence.*`
- `observed_payload_json`, `canonical_payload_json`, `metric_value_json`, `action_payload_json`, `business_key`, `threshold_state_json`, `quality_signals_json`, `evaluation_result_json` → corresponding `fact.*` typed columns
- `provenance_json` → `evidence.*`

All payload data lives **only** in typed `fact.*` tables. Opaque / compliance / recovery copies live **only** in S3 under Object Lock, per DEC-14592e.

**Rule (locks D162 Rule 1 for this domain):** *"No business data is queryable from `progression.*`. If you need to WHERE, ORDER BY, GROUP BY or JOIN on a business attribute, read `fact.*` or `dim_*`. If you need the immutable raw payload, read S3. Progression answers 'did this run happen and what was its outcome', nothing more."*

### 2. Schema Provisioner is a declarative reconciler, not a trigger handler.

Target state is a pure function of registry state. For each tenant T:

```
desired(T) = {
   fact.so_<sc>_v<ver>,
   fact.co_<cc>_v<ver>,
   fact.ms_<mc>_v<ver>,
   fact.ao_<ic>_v<ver>
   | contract ∈ active
     ∧ tenant_binding(T, contract)  [for source contracts]
     ∧ contract_binding(T, contract) [for admission/obs/canonical/metric/intervention]
}
```

The reconciler reads `desired(T)`, diffs against `information_schema` in the tenant DB, and emits `CREATE TABLE` to close the gap. It is idempotent, re-runnable, and always converges on the same steady state regardless of the order events arrived.

### 3. Connector onboarding is the **primary** trigger.

Tenants do not reason in individual contracts. They onboard connectors ("connect my SAP"). A connector has a well-defined chain walk:

```
connector (runtime.connector, platform)
  → readers (runtime.reader)
    → reader_flavor (runtime.reader_flavor)
      → reader_binding → source_contract
        ⋯ chain walks forward through admission → observation → canonical → metric → intervention
```

**Onboarding a connector in tenant T:**
1. Tenant inserts `admin.connection` + `admin.connection_config` rows in tenant DB with credentials.
2. Platform-side onboarding flow walks the chain forward from the connector, finds every contract in the reachable subgraph, and populates `tenant.tenant_binding` (for source contracts) and `tenant.contract_binding` (for the other 5 families).
3. Reconciler detects new rows and provisions `fact.*` tables for the entire subgraph in tenant T's DB.

Result: all tables tenant T will ever write to are present the moment onboarding completes. No later "first-write lazy create," no empty tables for contracts tenant T cannot reach.

### 4. Platform-side contract activation is the **secondary** trigger.

When a new contract is activated platform-wide (new MC downstream of an existing CC, new CC derived from existing SCs, etc.), the reconciler reverse-walks: finds every tenant whose onboarded connectors transitively feed this new contract, and provisions the new table in those tenant DBs.

### 5. Nightly reconciler is the **safety net**.

Once per 24h per tenant, full `desired(T)` is recomputed from `tenant_binding` + `contract_binding` + contract activation state and diffed against `information_schema`. Any drift — missing table, extra table, renamed column — is reported; creations are re-applied idempotently; drops and renames are reported-only (never destructive).

### 6. One-time vs dynamic is a layer distinction, not a timing hack.

- **One-time at tenant provisioning** — the six schemas of D368 (`progression`, `fact`, `evidence`, `tenant_dim`, `organization`, `admin`) and the **fixed** tables within them: 4 progression tables (admission, canonical_evaluation, metric_evaluation, intervention_evaluation), ~3 evidence tables, 2 organization tables, ~7 admin tables, plus the `dim_*` tables in `tenant_dim`. These come from a DDL template; they exist from day one for every tenant.
- **Dynamic via Schema Provisioner** — every table in `fact.*`. Per-tenant, per-contract, per-version. Created on onboarding (primary), on platform activation (secondary), reconciled nightly (safety net).

### 7. Z-fields reshape per-tenant typed tables deterministically.

A Z-field can enter the platform in two ways (per contract-requirements-v1):

- **Platform-promoted Z-field**: a new SC version is issued with the field marked `z_extension: true`. All bound tenants receive a new `fact.so_<sc>_v<N+1>` table with the new column. Standard version bump.
- **Tenant-specific Z-field**: the field lives in `tenant.contract_binding.extensions_json.z_field_mappings[]`, never promoted to the platform SC. Only this tenant's typed table has the column.

The Schema Provisioner's DDL for tenant T, contract C, version V is computed as:

```
base_fields     = SC_json.fields[]  where the active SC version is V
tenant_z_fields = contract_binding(T,C,V).extensions_json.z_field_mappings[]
full_fields     = base_fields ∪ tenant_z_fields
```

So `fact.so_bkpf_v1` at tenant A can legitimately have a different column set than `fact.so_bkpf_v1` at tenant B. This is expected and clean: typed tables live in tenant DBs, the shape is determined by (platform contract) ∪ (this tenant's extensions).

Recomputing is deterministic: same contract version + same extensions_json = same DDL. Changes to extensions_json trigger reconciliation (same as a contract version change).

### 8. Z-tables are entirely new SCs.

Per contract-requirements-v1 §2: a Z-table (e.g., SAP ZZPURCHASE_REQUESTS) is not an extension but a **new platform SC**. The tenant requests it; platform authors it; a reader flavor is published; the tenant binds via its connector onboarding (or a separate binding). From the reconciler's perspective, this is identical to any other SC activation — no special case needed.

### 9. Versioning: new version → new table, always.

- **Major bump** (breaking shape): new `fact.*_v<N+1>` table. Old `fact.*_v<N>` stays, is no longer written to. Historical queries hit v<N>; new writes hit v<N+1>. Downstream readers explicitly pick a version.
- **Minor bump** (additive, non-breaking): same treatment. **No ALTER TABLE ever.** This matches the execution-model immutability principle (Foundation §5.2 — "version coexistence is structural, not transitional").
- **Patch bump** (metadata only — description, display_name): no DDL impact; reconciler no-op.

This is stricter than the semver-based "ALTER is fine for minors" proposal; the stricter rule wins because it preserves the immutability guarantee and eliminates the "same table, different row shapes at different times" hazard.

### 10. Migration: existing JSONB progression data moves to typed tables + S3.

Non-trivial for any tenant with data. The phased plan:

**Phase M1 — Build the reconciler.** Ship the Schema Provisioner service (tenant-side per D232), wire primary/secondary/reconciler triggers, generate DDL deterministically from `contract_json.body.resolved_schema`. No writes yet — it only creates `fact.*` tables.

**Phase M2 — Dual-write transition.** Write BOTH the legacy JSONB progression and the new typed table + S3 archive. Compare outputs in test. When a tenant's fact.* tables match the JSONB progression row-for-row, that tenant is ready to cut over.

**Phase M3 — Cut-over per tenant.** Tenant-by-tenant, flip reads to fact.* (dashboards, Monitor UI, metric evaluation inputs). Keep JSONB writes for rollback safety for one release cycle.

**Phase M4 — JSONB-drop.** Drop payload JSONB columns from `progression.*`. Emit a data-lineage record referencing S3 archive_key for anyone who needs the raw payload.

**Sandbox1 path (now, first):** Sandbox1 has 1368 admitted records in `envelope.admitted_record` (current name) with no `fact.so_*` tables. It can be wiped and re-seeded after M1 ships with the new 4-table progression schema + fact DDL. No migration needed; it's a test tenant with synthetic data.

**Production path (later):** No production tenants exist today. This ADR lands before any do. Migration is a non-issue for production.

### 11. Sunset plan for JSONB-writing code.

- `canonical-resolution.service.ts` — currently writes `envelope.canonical_object.canonical_payload_json` + `envelope.evaluation`. After M4 writes `progression.canonical_evaluation` (metadata) + `fact.co_<cc>_v<ver>` (typed columns). No JSONB, no `canonical_object` row.
- `metric-evaluation-engine.service.ts` — currently writes `envelope.metric_evaluation` + `envelope.metric_snapshot.metric_value_json`. After M4 writes `progression.metric_evaluation` (metadata) + N `fact.ms_<mc>_v<ver>` rows (per-grain typed columns). No JSONB, no `metric_snapshot` row.
- `reader-runtime.service.ts` — currently writes `observed_record` + `source_object` + `admitted_record`. After M4 writes a single `progression.admission` row (the admission event) + `fact.so_<sc>_v<ver>` (if admitted) + S3 archive. No JSONB.
- `intervention-*.service.ts` (when built) — writes `progression.intervention_evaluation` + `fact.ao_<ic>_v<ver>` when triggered. Pre-emptive pattern.

All three service files are in scope for the implementation session set.

## Foundation review — compliant

- **§2.1 Fixed object progression** — unchanged. SO → CO → MetricSnapshot → AO. Typed tables are the representation; the progression is the logic.
- **§4.5 Non-Retroactivity** — stronger. Versioned tables coexist forever (v1, v2, v3...); no ALTER means no accidental retroactive reshape.
- **§8.4 Evidence Capture Boundaries** — stronger. S3 archive is WORM; "Evidence is emitted not reconstructed" is enforced literally by Object Lock.
- **§5.2 Version coexistence** — directly implemented by `fact.*_v1`, `fact.*_v2` coexistence.
- **§1.2 Non-scope: storage layout** — the foundation explicitly places storage layout outside its scope. D369 is free to make this choice; the foundation does not constrain it.

## Patent review — compliant, strengthened

- **Section 13** — "Distributed object store with WORM for durability; NVMe for state transitions" — S3 Object Lock is literal WORM; RDS typed tables are the state transitions. The patent recommends this exact architecture.
- **Section 14** — "Storage/I/O Optimization via DOMNS" — separating progression (metadata) from fact (payload) from S3 (archive) is exactly this separation.
- **Section 4/6** — Typed tables carry extracted/evaluated values; progression carries metadata. Same information, optimal storage.

## D232 review — compliant

Schema Provisioner is a **tenant-side service** (per the existing P09 spec §Constraints). It runs in tenant context, reads contracts from platform API, writes DDL to the tenant DB. Platform never opens a connection to tenant DB. The one-way dependency holds.

## Consequences

### Wins

- **Single source of truth** for business data: `fact.*`.
- **S3 Object Lock compliance** (ISO 27001 A.12.3.1, A.18.1.3) by design, not configuration.
- **Query-friendly** data layer from day one — Monitor UI reads typed columns, not JSONB extracts.
- **Zero cost to add a tenant**: onboarding a connector triggers the reconciler; tables appear; data flows.
- **Zero cost to add a contract**: platform activates; reverse-walk triggers reconciler in relevant tenants.

### Costs

- **Build the Schema Provisioner.** Non-trivial; multiple sessions.
- **Sunset JSONB-writing code** in 3 services. Touching canonical-resolution.service, metric-evaluation-engine, reader-runtime.
- **Dual-write phase** (M2) temporarily doubles write volume. Acceptable for a bounded window.
- **Monitor UI work**: once `fact.ms_*` exists, the UI reads typed columns from it (e.g., `fact.ms_<mc>_v1.fiscal_period`), not JSONB extracts. Small code change, big semantic upgrade.

### Risks

- **DDL generator correctness** — per DEC-2658ff, S3 archive is the recovery net. Bugs in the generator are recoverable by re-reading S3 and regenerating; do not need source-system re-reads.
- **Per-tenant typed table drift** — if Z-field extensions diverge significantly, the "same contract v1 at tenant A and B" tables differ. This is expected; tooling (query routers, cross-tenant analytics) must respect per-tenant DDL.
- **Migration labour** for any production tenant that lands before M4. Mitigated by no production tenants existing today.

## Open issues locked by this ADR

- JSONB-first premise of DEC-a25931 is superseded in effect: JSONB was a staging state, not a parallel runtime. This ADR declares the staging phase over.
- D210v3 flips from `proposed` to being the implementation path of this ADR.
- T01 / T02 dossiers use the pre-D368 schema names (`boundary`, `record`); Schema Provisioner spec same. Those documents must be updated to use `progression` and `fact`. Listed as a follow-up task in D369 implementation.

## Explicitly not in this ADR

- The reader binding / OC entity-naming question (TSK-632f35). Orthogonal — that's a data-flow bug, not a storage decision.
- The remaining progression Drizzle/DDL drifts (TSK-a84ad2). Covered by M4 anyway.
- Anything about Action Objects — same architecture applies by analogy; the three ADRs it depends on (DEC-2658ff, DEC-14592e, D368) already cover AO identically.

## Closing

The platform has three locked decisions (D210v3 proposed, D368 decided, D14592e proposed) that together describe one coherent architecture. It has one tactical decision (DEC-a25931, decided) that was justified at the time and is no longer. This ADR removes the tactical detour, flips the two proposeds to decided, and lays the implementation on a single path: **typed in RDS, opaque in S3, progression is metadata**.
