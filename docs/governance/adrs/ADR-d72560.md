# Canonical Field as 3rd Contract Primitive — Two-Vocabulary Model with CC as Translator

## Context

The contract chain has two naming vocabularies serving different purposes:

1. **Source-side (BO-scoped BFs):** Named after source structure. D292-compliant: `receivable_hdr_amount`, `invoice_hdr_total_amount`, `journal_entry_hdr_debit_amount`. Used by OC field_selection and CC input.

2. **Metric-side (formula variables):** Named after business meaning. `accounts_receivable_balance`, `total_revenue`, `journal_entry_count`. Used by MC formula variables and metric definitions.

These are fundamentally different vocabularies. Previous attempts forced them to be the same — BFs tried to serve both as source-structure mirrors AND metric-formula inputs. This created the "binding problem": metric formula variables don't match BO-scoped BF names because they express different concepts.

DEC-2e4cb3 established that MC binds to CC outputs, not BFs directly. But it left the CC output vocabulary undefined — CC output fields were still assumed to be BFs. This ADR formalizes what was always implicit: CC outputs are a different vocabulary (canonical fields), and CC is the translator between the two.

### Evidence: Three Proof-of-One Experiments

Validated on experiment/canonical-field branch with live data in bc_platform_dev:

**1. Days Sales Outstanding (DSO)** — accounts_receivable subfunction
- MC formula: `(accounts_receivable_balance / total_revenue) * evaluation_period_days`
- 2 CCs involved: `cc__receivable_hdr` (provides `accounts_receivable_balance`), `cc__invoice_hdr` (provides `total_revenue`)
- Resolution: `receivable_hdr_amount` -(sum)-> `accounts_receivable_balance`, `invoice_hdr_total_amount` -(sum)-> `total_revenue`
- Exposed early mismatch: MC I1 expected `accounts_receivable_balance`, initial mapping used `total_accounts_receivable_balance` — caught and fixed before any runtime wiring

**2. AR Turnover** — accounts_receivable subfunction
- MC formula: `total_revenue / average_accounts_receivable`
- Reuses `total_revenue` CF from DSO (same cc__invoice_hdr mapping)
- `average_accounts_receivable` is a **derived temporal CF** — computed from `accounts_receivable_balance` over evaluation window, not a direct CC output
- Proves: CFs can be derived from other CFs (temporal aggregation), not just from BF resolution

**3. Journal Entry Error Rate** — general_ledger subfunction
- MC formula: `(erroneous_journal_entries / total_journal_entries) * 100`
- New CC: `cc__journal_entry_hdr` with BFs from a different BO domain (GL, not AR)
- New resolution rules: `count_where_not_null` (for erroneous entries), `count_distinct` (for total entries)
- Proves: model works across business subfunctions and BO domains; resolution rules extend naturally

All 3 MCs had 100% variable binding to registered canonical fields via cc_field_mapping.

## Decision

### 1. Canonical Field is a first-class entity (3rd primitive)

Alongside Business Field (BF) and Business Object (BO), **Canonical Field (CF)** is a contract-schema primitive:

```sql
CREATE TABLE contract.canonical_field (
    canonical_field_id   UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    field_name           TEXT NOT NULL UNIQUE,      -- metric-vocabulary name
    display_name         TEXT NOT NULL,
    function_code        TEXT NOT NULL,              -- FK to master_function
    subfunction_code     TEXT,                       -- FK to master_subfunction
    unit_type_code       TEXT,                       -- FK to master_unit_type
    data_type            TEXT NOT NULL DEFAULT 'number',
    role                 TEXT NOT NULL DEFAULT 'input',  -- input | output | derived
    status_code          TEXT NOT NULL DEFAULT 'draft',
    description_text     TEXT,
    created_at           TIMESTAMPTZ DEFAULT now(),
    updated_at           TIMESTAMPTZ DEFAULT now()
);
```

### 2. CC Field Mapping is the translator

Each CC declares how its BF inputs resolve to CF outputs:

```sql
CREATE TABLE contract.cc_field_mapping (
    cc_field_mapping_id   UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    canonical_contract_id UUID NOT NULL REFERENCES contract.canonical_contract(canonical_contract_id),
    business_field_id     UUID NOT NULL REFERENCES contract.business_field(field_id),
    canonical_field_id    UUID NOT NULL REFERENCES contract.canonical_field(canonical_field_id),
    resolution_rule_code  TEXT NOT NULL DEFAULT 'latest',
    created_at            TIMESTAMPTZ DEFAULT now(),
    UNIQUE(canonical_contract_id, business_field_id),
    UNIQUE(canonical_contract_id, canonical_field_id)
);
```

### 3. Two vocabularies with clear boundaries

| Vocabulary | Entity | Named after | Used by | Example |
|-----------|--------|-------------|---------|---------|
| Source-side | Business Field (BF) | Source structure (D292 BO-scoped) | OC field_selection, CC input | `receivable_hdr_amount` |
| Metric-side | Canonical Field (CF) | Business meaning | CC output, MC formula variables | `accounts_receivable_balance` |

**CC is the translator.** It takes BO-scoped BFs as input, applies a resolution rule, and produces canonical fields as output. The `cc_field_mapping` table is the Rosetta Stone.

### 4. MC binds to canonical fields, not BFs

The R1 invariant changes: MC `field_code` references = canonical field names, not BF names. MC creation becomes trivial — formula variable names ARE canonical field names.

### 5. Resolution rules

Each cc_field_mapping row declares how the BF value resolves to the CF value:

| Rule | Meaning | Example |
|------|---------|---------|
| `sum` | Aggregate BF values | `receivable_hdr_amount` -(sum)-> `accounts_receivable_balance` |
| `latest` | Most recent BF value | `exchange_rate` -(latest)-> `spot_exchange_rate` |
| `assert_equal` | All values must match (dimension) | `currency_code` -(assert_equal)-> `reporting_currency` |
| `count_distinct` | Count unique BF values | `journal_entry_hdr_id` -(count_distinct)-> `total_journal_entries` |
| `count_where_not_null` | Count non-null BF values | `journal_entry_hdr_error_flag` -(count_where_not_null)-> `erroneous_journal_entries` |

New rules can be added as metric patterns demand them.

### 6. Derived canonical fields

Some CFs are temporal derivations of other CFs, not direct CC outputs:

- `average_accounts_receivable` = temporal average of `accounts_receivable_balance` over evaluation window
- Role: `derived` (not `input`)

These are computed at MC evaluation time, not at CC resolution time.

### 7. Canonical field seeding (top-down)

The 1,241 metric definitions contain 4,226 formula variables. The unique set of input/output field names across all formulas seeds the canonical_field table. This is the demand signal — metrics declare what CFs they need, then CCs are built to supply them.

### 8. Contract chain relay (updated)

```
Source Catalog → SC+AC → BF+BO → OC → CC (with cc_field_mapping) → MC (binds to CF)
                                         ^                           ^
                                    BF vocabulary              CF vocabulary
                                    (source-side)              (metric-side)
```

The "baton" splits at the CC boundary. Before CC: source vocabulary. After CC: metric vocabulary.

## Options Considered

### Option A: Single vocabulary — BFs end-to-end (rejected)

Same BF name from OC through CC to MC. Forces metric formula variables to use BO-scoped names (`receivable_hdr_amount` in formulas). Doesn't match how metrics are actually written. Creates the binding problem that blocked 10 months of attempts.

### Option B: Two vocabularies with CC as translator (chosen)

Canonical fields as separate entity. CC maps BF to CF. MC consumes CFs. Natural vocabulary boundary at CC. Proven by 3 experiments. This was actually the original conceptual model that got lost during implementation.

### Option C: Virtual mapping — no CF table, just aliases in CC body (rejected)

Avoid a new table by storing CF names as aliases in CC contract_json body. Loses queryability, can't enforce uniqueness, can't FK from MC. Violates D089 rule 1 (no JSONB for queryable data).

## Consequences

### Positive

- MC creation becomes trivial — formula variable names = canonical field names
- Binding mismatches caught at CC mapping time, not MC evaluation time (shift-left)
- Same CF can be supplied by multiple CCs (cross-BO reuse: `total_revenue` from invoice_hdr or journal_entry_hdr)
- Clear separation of concerns: BF knows source structure, CF knows business meaning
- Resolution rules are explicit and auditable per mapping
- Natural extension point for new resolution rules as metric patterns emerge

### Negative

- New table and mapping layer adds schema complexity
- CC onboarding must now include field mapping step (not just field_selection)
- Existing CC onboarding service needs modification to create cc_field_mapping rows
- MC onboarding service needs modification to validate against canonical_field instead of business_field

### Risks

- **Migration:** Experiment branch has DDL applied directly via psql. Production path requires Drizzle schema definitions, migration files, and DDL updates to `bc-core/docker/redesign/`. Mitigation: create proper migration before merging.
- **CF explosion:** 4,226 formula variables may produce thousands of CFs. Mitigation: deduplicate by unique field_name; role-based filtering (input vs output vs derived).
- **Stale CF names:** If metric definitions change formulas, CF names may drift. Mitigation: CF status lifecycle (draft → certified → deprecated); MC validation gate checks CF status.
