# sandbox1 demo scenario — 3 fiscal years of real-estate AR data via bc-sdg + canonical AR slice

## Context

sandbox1 is the redesign cluster's only registered tenant (per `tenant.tenants`). It's been a structurally-complete-but-empty fixture since the D369 M4.4 cutover. Bc-portal redesign needs realistic data to design against, and the team has a real-estate prospect (Kohinoor) whose past MIS work shows the shape of real-estate AR: milestone-based collections, multi-component receivables, project-unit hierarchy, RERA targets. The canonical 5 chain-complete AR MCs (DSO, AR Aging, AR Turnover, Total AR Balance, Bad Debt Ratio) compute correctly on standard SAP ECC AR data, so we can prove the chain end-to-end with bc-sdg's `real-estate-co` profile (only ECC-landscape profile of the six bc-sdg ships). A separate Phase 2 ADR/session will author real-estate-specific MCs (Pending Agreements Aging, Milestone Collection %, RERA Variance, Unsold Inventory by Project) — the differentiated BareCount value-add for industry-specific demos. Phase 1 proves the chain; Phase 2 makes it credible to a real-estate buyer.

# sandbox1 — 3 Fiscal Years of Real-Estate AR Demo Data

## Phase 1 (this decision) — chain-prove on canonical AR

### Scenario shape

| Property | Value |
|---|---|
| Tenant | `sandbox1` (status `active`, schema `t_sandbox1`) |
| Data generator | `bc-sdg` profile `real-estate-co` (UrbanBuild Realty Pvt Ltd, ECC landscape) |
| Backfill window | `2024-04-01` → `2027-03-31` (3 fiscal years FY24, FY25, FY26 Apr–Mar) |
| Fiscal calendar | `IN-APR-MAR-MONTHLY` (already in `master.dim_fiscal_calendar`) |
| Source system | bc-sdg ECC OData V2 at `http://localhost:4200/ecc/sap/opu/odata/sap/` |
| Legal entities | First two company codes from the real-estate-co profile, both INR |
| Currencies | INR base; foreign-currency mix per profile defaults |
| Active reader flavors used | `ecc-bc-ar-operations`, `ecc-credit`, `ecc-credit-status`, `ecc-gaap-receivables`, `ecc-journal-entry-hdr`, `ecc-journal-entry-line` |
| Activated MCs | DSO, AR Aging Distribution, AR Turnover, Total AR Balance, Bad Debt Ratio (5 chain-complete MCs) |
| Volume target | ~100 invoices/entity/period × 36 periods × 2 entities ≈ 7,200 source rows |

### Why `real-estate-co` over a new profile

Of bc-sdg's six profiles (mfg-co, pharma-co, auto-parts, chem-co, fmcg, real-estate-co) only `real-estate-co` is `landscape: ecc`. The five `s4_cloud`/`s4_onprem` profiles serve a different OData service path (`/sap/opu/odata4/sapi`) the SAP ECC reader flavors aren't bound to. Authoring a new ECC profile is feasible (~30–45 min) but adds zero value over `real-estate-co` for proving the chain — they generate the same SAP shape, only customer/material names differ.

### Order of operations

1. `npm run db:migrate` against the `bc_sdg` database (port 5435, on the redesign cluster).
2. `sdg init real-estate-co` — seeds master data (company codes, customers, materials, COA).
3. `sdg generate real-estate-co --from 2024-04-01 --to 2027-03-31` — generates 3 fiscal years of transactions.
4. Author a `runtime.connection` row for sandbox1 — endpoint `http://localhost:4200/ecc/sap/opu/odata/sap/`, source_system_name `sap_ecc`, environment `development`. Requires explicit user approval per CLAUDE.md DB protocol.
5. Pair the 6 active SAP ECC reader flavors with the new connection (via `runtime.reader_flavor.connection_id` update or API).
6. Insert `tenant.contract_binding` rows for the 5 chosen MCs (`tenant_id=sandbox1`, `contract_family='metric'`, `is_active=true`). Requires explicit user approval per CLAUDE.md DB protocol.
7. Run the schema provisioner against sandbox1 (`scripts/d369-m1c-sandbox1-probe.mjs`) → `fact.ms_*` tables for the 5 MCs land in `tbc_sandbox1_dev`.
8. Seed sandbox1's `tenant.fiscal_calendar_config` row pointing at `IN-APR-MAR-MONTHLY` (per legal entity) — verify or insert via the FiscalCalendarService API.
9. Run readers — proving slice first: 1 entity × 1 period × 1 MC. Verify the snapshot in `progression.metric_evaluation` + `fact.ms_<mc>_v1` + bc-portal `/metrics/:id/monitor`.
10. Scale to full 2 × 36 × 5.

### Phase 1 success criteria

- `/metrics/:id/monitor` for at least one of the 5 MCs shows ≥1 real metric snapshot value (not null, not placeholder) for sandbox1 in bc-portal.
- `progression.metric_evaluation` rows count > 0 for sandbox1.
- `fact.ms_<mc>_v1` tables exist in `tbc_sandbox1_dev` and are non-empty.
- No funnel-padding (per `feedback_funnel_padding.md`) — if an MC's L4 mapping points at a SAP field bc-sdg doesn't populate, we drop that MC and pick another chain-complete one. We do NOT hand-curate source data to fit a broken chain.

## Phase 2 (separate ADR/session) — real-estate-specific MCs

To make the demo credible to a real-estate prospect, author 6–8 industry-specific MCs that mirror the KPI shape from real-world real-estate developer MIS dashboards:

| MC | Why it matters in real-estate |
|---|---|
| Total Agreement Outstanding (multi-component) | Receivable = principal + maintenance × 3 + GST + TDS. Aged separately |
| Milestone Collection Adherence % | Sales are milestone-staged; collection vs schedule is the primary AR health indicator |
| Pending Agreements Aging | Booked but not yet registered/executed — a separate aging surface |
| RERA Variance % (Residential / Commercial) | Regulatory commitment vs actual delivery; project credibility |
| Construction Variance % | Planned construction release vs actual; affects revenue recognition timing |
| Unsold Inventory by Project (Carpet sqft) | Inventory KPI in real-estate is unsold area, not stock value |
| Bad Debt Risk by Aging Bucket | Granular buckets (1-15, 16-30, 31-45, 46-60, 61-90, 91-180, 180+) per real-estate convention |
| Cancellation Rate / Bookings | Booking cancellations are a separate funnel that affects projected revenue |

These are NEW `metric.metric_definition` + `contract.metric_contract` rows. Phase 2 work order:

1. Define the new metric_definitions (industry_code = `real_estate`).
2. Author canonical contracts that target real-estate fields bc-sdg either generates or can be extended to generate.
3. Possibly extend bc-sdg `real-estate-co` profile to emit unit/agreement/milestone tables (Z-program-equivalent shapes) the new CCs can read from.
4. Promote → activate → run.

Phase 2 is **out of scope for this ADR**; tracked as a follow-up DevHub task once Phase 1 is green.

## Reproducibility

The recipe is intentionally text-locked above so anyone (or any session) can rebuild sandbox1's data from scratch with the exact same shape. All inputs are deterministic:
- bc-sdg uses seeded random — same backfill window produces same output (verify the profile uses a fixed seed).
- The 5 MCs are listed by name (chain-complete subset is platform-determined).
- Connection endpoint, fiscal calendar code, and fact-table provisioner are explicit.

## Consequences

- sandbox1 becomes a working data fixture for bc-portal redesign.
- The DSO / AR Aging / AR Turnover / Total AR Balance / Bad Debt Ratio metrics will show realistic values, but the demo narrative will say "this is canonical AR; real-estate-specific KPIs land in Phase 2."
- bc-sdg's `bc_sdg` database becomes a dependency for sandbox1's data flow. Stopping bc-sdg breaks reader runs (acceptable for a dev fixture).
- The legacy `runtime.connection` row at `localhost:6100` becomes archive-only — not deleted (audit trail), but superseded by the new sandbox1-bound connection.
