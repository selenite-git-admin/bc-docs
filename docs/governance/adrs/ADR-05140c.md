# Unified Source Catalog — drop Landscape, status-driven approval, scope attribute

## Context

The Landscape/Catalog split created two parallel hierarchies with identical shape, requiring a promotion bridge to copy data between them. This is unnecessary complexity — the only difference is verification status. A single catalog with status-driven gating is simpler, eliminates data duplication, and makes the UI straightforward: one place to see everything, filtered by status/scope. Discovery (tenant) produces the same shape and feeds directly into the same catalog. Approved status is the single gate for contract creation.

## Decision

**Drop the Landscape/Catalog duality.** There is one Source Catalog. The difference between "known" and "verified" is status, not location.


## Options Considered

N/A

## Status lifecycle (all 6 tiers)

```
registered → approved → deregistered
```

- **registered** — known to exist (seeded from reference data, or discovered by tenant). Visible in catalog but gated from contract creation.
- **approved** — SME-validated. Only approved members can create contracts, bindings, and drive the 6-tier chain.
- **deregistered** — retired. Preserved for audit, no new contracts.

## Tier-level seeding rules

| Tier | Seeded as | Rationale |
|------|-----------|-----------|
| Provider | `approved` | Verifiably exist (SAP SE, ECB). Provenance link proves existence. |
| System | `approved` | Known products (S/4HANA, ECC, Salesforce). |
| Module | `registered` | Known from reference but not validated for BareCount use. |
| Object (table/view) | `registered` | Exists in reference catalogs, needs SME approval. |
| Field | `registered` | Same — needs approval before contract binding. |

## Scope attribute (every member)

- `platform` — universal, applies across all tenants.
- `tenant` — tenant-specific (custom Z-tables, tenant-discovered fields, etc.).

## Discovery integration

Tenant discovery produces objects/fields in the **same catalog** with `scope_code = tenant` and `catalog_status = registered`. On approval they stay in place with `catalog_status = approved`. No "move" or "copy" between tables.

## Contract gating

Only `approved` objects (Tier 5) can trigger contract auto-creation (source + admission contracts per meta-schema). New fields added to an approved object enter as `registered`, go through approval, and trigger a MINOR version bump on the contract.

## What this supersedes

- D090 (Landscape vs Catalog with promotion bridge) — superseded by this unified model.
- `sap_table_reference`, `sap_cds_reference` tables — to be absorbed into `source_object` with `catalog_status = registered`.
- `/sources/landscape` route group — merged into `/sources/catalog`.
- Separate Landscape pages in bc-admin — removed, Catalog IS the full view.
