# Tenant GL account classification binding (D514 companion, D502-aligned): classify_by_binding derivation, platform defaults with onboarding/FSV override, onboarding-mandatory enforcement

## Context

No rationale recorded.

## Decision

# Context

ADR DEC-36913d (D514) delivered balance expression (closing balance, balance date) but explicitly excluded account classification (its D4): classic ECC stores no asset/liability/equity class — SKA1 gives only balance-sheet-vs-P&L (XBILK/GVTYP); the real classification lives in the tenant-configured Financial Statement Version (T011/FAGL_011ZC account ranges). No universal platform value-map exists, so a platform CC carrying one would invent meaning. ~50 directory members (gl_balances, income_statement_line_amounts, gl_account_by_cash_flow, and the ratio families over them) plus the review-parked total_assets/equity/liabilities (PE-MC-11 all legs passing since D514) are gated on this.

Operator direction (2026-07-11): option 2 architecture (tenant-scoped binding, D502-aligned) seeded with option 1 defaults (governed platform classification reference), overridable at onboarding and via the tenant's own FSV; structurally covered and ENFORCED as a mandatory tenant-onboarding step; bc-portal workspace UI as follow-up.

Precedent: the fiscal calendar (D363–D365) resolves tenant configuration at evaluation behind an explicit declared reference — D514's fiscal_period_end_date already uses it. Classification follows the identical pattern.

# Decision

**D1 — `classify_by_binding` derivation function** (canonical-v2, D330-additive). Output = a governed closed-set concept (D430 value-set gate applies); `params.binding_kind` (v1: 'gl_account_classification') is the explicit binding reference (Invariant IV); `params.output_facet` selects which classification facet the field carries ('account_class' | 'account_type' | 'cash_flow_category'). Inputs: the grain identity signal(s) the binding keys on (concept or raw-signal). The CC declares THAT the field is binding-classified and INTO which closed vocabulary; it never carries tenant mapping content — platform contracts stay universal (Invariant I: meaning declared once at the canonical boundary; the binding supplies tenant configuration, exactly like the fiscal calendar).

**D2 — `tenant.gl_account_classification_binding`** (platform DB, tenant schema, beside tenant.contract_binding — repair location C). Rows: (tenant_id FK, chart_code, account_from, account_to, account_class_code, account_type_code, cash_flow_category_code nullable, provenance_code, source_ref_text, effective_from/effective_to, created audit). Range semantics (account BETWEEN from AND to; equal for single accounts) live in the BINDING SERVICE, not the contract grammar — no range operators enter the CC grammar.

**D3 — governed platform defaults**: `contract.gl_account_classification_default` keyed by chart_template_code (INT first). Every row carries source_ref_text documenting its grounding. Defaults are explicitly DEFAULTS — provenance-stamped 'platform_default' when seeded into a tenant binding, never silently authoritative over a tenant's own FSV.

**D4 — onboarding-mandatory enforcement (operator-directed)**, three layers: (a) STRUCTURAL — tenant provisioning seeds the binding from the defaults catalog for the tenant's chart template; (b) FAIL-CLOSED — the binding service REFUSES classification resolution for a tenant with zero binding rows (loud error, never a silent platform fallback at read time), so no classified metric can evaluate for an un-onboarded tenant; (c) PROCEDURAL — the Tenant Onboarding chapter (bc-docs onboarding/tenant-onboarding.md) gains a mandatory classification-review step: confirm seeded defaults, or ingest FSV, or record operator overrides. Overrides stamp provenance ('fsv_derived' with the FSV item reference; 'operator_override' with rationale).

**D5 — FSV ingestion path**: FAGL_011ZC/T011 are registerable source tables; an onboarding-time translation converts the tenant's FSV item assignments into binding rows. FSV item labels are tenant-defined, so item→class mapping is confirmed at onboarding (one-time human confirmation), then rows land 'fsv_derived'. v1 ships the binding surface + defaults seeding; FSV ingestion is a named follow-on build on this substrate.

**D6 — reuse**: the binding mechanism generalizes to the other parked tenant-semantic classifications (ANBWA→asset movement kinds, treasury SANLF→debt types — TSK-517116) via future binding_kind values; each new kind is ADR-or-task-governed, never ad hoc.

**D7 — UI**: bc-portal workspace gains a classification-binding surface (view seeded rows, provenance, override entry, FSV upload) — tracked as a dedicated bc-portal task; not part of this bc-core build.

# Invariant test

I: classification meaning declared once at the CC; binding supplies tenant config through the declared reference. II: no ordering change. III: binding rows are temporal (effective_from/to), corrections append. IV: binding_kind + output_facet are explicit declarations; every seeded row carries provenance + source_ref. V: deterministic range lookup. VI: fail-closed for unseeded tenants — a missing classification is an error, never an inferred value.

# Consequences

- total_assets/equity/liabilities activate on cc__gl_account v2; gl_balances, income-statement lines, cash-flow counts, and dependent ratio families become drivable.
- Two new tables via DBCP (operator approved 2026-07-11): DDL 35-gl-account-classification-binding.sql + Drizzle schemas.
- Runtime classification evaluation follows the D498 posture: contract-level realizability now; the ccv2 resolver evaluates classify_by_binding against the binding service where wired, else fails loud.
- bc-docs tenant-onboarding chapter amended (mandatory step). bc-portal workspace UI task filed.

## Amendment 2026-07-11 — onboarding-stage clarification (operator)

D4c's procedural step is a **SOURCE-onboarding** step, not a tenant-onboarding step. The operator's staging model: tenant onboarding carries tenant-level context (fiscal year / fiscal calendar per D363–D365); **source onboarding** — the act of connecting a specific source system (e.g. SAP) for a tenant — carries source-semantic bindings, which is where the FSV lives and therefore where the GL account classification review belongs. The mandatory classification-review step, the FSV ingestion path (D5), and the readiness gate attach to the source-onboarding flow for GL-bearing sources; the enforcement layers (D4a structural seed, D4b fail-closed resolution) are unchanged. The seeding trigger is source connection, not tenant creation. Consequences line "bc-docs tenant-onboarding chapter amended" is corrected to the source-registration/source-onboarding chapter; follow-on tasks re-pointed (TSK-526a25, TSK-ba0625, TSK-363ad4, TSK-931f0c).
