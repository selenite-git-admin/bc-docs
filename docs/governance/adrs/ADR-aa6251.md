# D255: Contract Primitives — BO and BF as First-Class Governed Artifacts

## Context

The contract chain (SC -> AC -> OC -> CC -> MC -> IC) references two definitions that are not themselves contracts:

- **Business Object (BO)** — referenced by CC (`body.business_object_code`), OC (`body.business_object_code`)
- **Business Field (BF)** — referenced by CC (`body.field_selection`, `body.resolved_schema`), MC (`body.variables[].field_code`), OC (`body.field_mappings[].business_field_code`)

Without BOs and BFs, no CC can declare what it produces, no MC can declare what fields it computes, and no OC can map source fields to business fields. They are load-bearing. Yet they have no formal governance status — no defined shape, no governance tags, no foundation spec.

Currently: 46 BOs and 1,604 BFs exist in the DB (`contract.business_object`, `contract.business_field`), all generated by the D225 generator for finance. They have table columns but no declared governance model.

## Decision

**Business Objects and Business Fields are designated as Contract Primitives.**

Contract Primitives are the irreducible definitions that contracts reference. They receive the same D253 governance treatment as contract families:

- Every column is documented with governance classification (fixed, derived)
- All non-nullable columns are always required — no structural optionality
- Foundation docs follow the same self-contained spec format as contract family docs

### What Contract Primitives Are NOT

- **Not contracts.** No `$contract` envelope, no header/body split, no version table, no approval workflow.
- **Not JSON documents.** They are relational table rows. Governance applies to DDL columns, not JSON keys.
- **Not meta-schema validated.** No entry in `contract.contract_meta_schema`. Validation is DDL-enforced (NOT NULL, CHECK, FK).

### The Governed Artifact Set

| Type | Artifacts | Count |
|------|-----------|-------|
| **Contract Primitives** | Business Object (BO), Business Field (BF) | 2 |
| **Contract Families** | SC, AC, OC, CC, MC, IC | 6 |
| **Total** | | **8 governed artifacts** |

### Governance Model

Primitives follow the same three-level model as contracts (D233):

| Level | Contracts | Primitives |
|-------|-----------|------------|
| **Master Shape** | JSON Schema in `contract_meta_schema` | DDL + foundation doc |
| **Platform Instance** | `contract_json` in version table | Row in `business_object` / `business_field` |
| **Tenant Override** | `contract_binding.override_json` | Not applicable — primitives are platform-only |

All primitive columns are **fixed** or **derived**. Tenants do not override BO or BF definitions. If a tenant needs a different BO structure, the platform creates a new BO — same as CC.

### BO-BF Relationship

BFs are not FK-linked to BOs in the DDL (a BF may be used by multiple BOs). The link is established through:

1. **CC `body.field_selection`** — which BFs this BO's CC includes
2. **`contract.business_object_field`** — junction table mapping BO to BFs (if implemented)
3. **Naming convention** — BFs carry `business_function` matching the BO's `function_code`
