> **Approach clarification + governed-selection gating (2026-06-29 cont., SES-8f76ae).** A deep operator-led problem-framing pass clarified the model and surfaced a Foundation gate that revises the SCOPE + SEQUENCING above.
>
> **Model.** A balance/bucket is a P-parameterized **governed selection over immutable dated facts** â€” not a stored balance, status, or bucket. The receivable document carries its immutable dates (posting/recognition, clearing, due); "outstanding at P" is *derived* as `clearing-date > P (or null)`, **never** read from a stored status flag â€” a stored status is as-of-*now* (knowledge time), not as-of-P (effective time), so using it silently misclassifies any historical P (Invariant V). Aging buckets are **date arithmetic** (`P âˆ’ anchor` in days); fiscal as-of is **period-ordinal** comparison (the `cumulative_to_date` work above); both are governed selections over the same dates, differing only in predicate.
>
> **Base case reuses `arpi` â€” no new surfaces.** For full clearing (one clearing date; binary open/cleared) `cc__customer_invoice_arpi_slice` already carries posting date, clearing date, and amount. The base-case balance/aging is a governed selection over `arpi`'s existing dated Canonical Objects. The line-entity surfaces authored earlier this session (`oc__receivable_posting_bsad`, `oc__receivable_clearing_bsad`, `cc__receivable_posting`, plus the line-entity `posting date`/`clearing date` BCs) were a **C1 dodge** (a second CC cannot grain on Customer Invoice while `arpi` holds that grain entity) and are **superseded for the base case** â€” cleanup pending; the line `posting date` BC is semantically a header concept and a withdraw candidate.

> **Netting = the partials generalization, deferred.** The postingsâˆ’clearings netting is not a different model â€” it is this same selection when clearing is *many* dated+amounted events (partial settlement) instead of one. Deferred until partial settlement is in scope.
