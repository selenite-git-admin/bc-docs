# DQC Integration — Global QC Rulebook as Foundation F04, Three-Tier Rule Model for AC

## Context

AC was carrying inline quality rules without a shared rule library — every AC duplicated the same global rules (null PK, type mismatch). The DQC rule packs spec from legacy platform documentation already defined the taxonomy (8 categories, staging, scoring). Integrating DQC into the contract model gives: (1) reusable rule templates instead of per-AC duplication, (2) consistent severity/action/scoring across all ACs, (3) clear three-tier governance aligned with D233, (4) DQC stages naturally map to BareCount's three evaluation boundaries (Admission/Canonical/Metric).

## What

The Admission Contract (AC) becomes the Bronze-stage Data Quality Control layer. Quality rules are organized in a three-tier model aligned with D233:

### Three-Tier Rule Model

| Tier | Scope | Storage | Rule ID Prefix |
|------|-------|---------|----------------|
| Global | Platform-wide, all ACs | QC Rulebook (foundation F04) | GL-* |
| Platform | Table-specific | AC contract_json | PL-* |
| Tenant | Tenant-customized | contract_binding.extensions_json | TN-* |

### DQC Boundary Mapping

DQC stage-based validation maps to BareCount boundaries:

| DQC Stage | BareCount Boundary | Contract | Rule Focus |
|-----------|-------------------|----------|------------|
| Bronze | Admission | AC | Structural: schema, completeness, type conformance, uniqueness, freshness |
| Silver | Canonical | CC (semantic_rules) | Semantic: referential integrity, range, cross-field |
| Gold | Metric | MC (quality_rules) | Business: reasonability, statistical, business logic |

### 8 Rule Categories (from DQC spec, adapted to BareCount vocabulary)

completeness, uniqueness, range, referential_integrity, freshness, reasonability, duplicate_detection, business_logic

Each rule in an AC body carries: rule_id (with scope prefix), category, field_code, type, severity (critical/warning), action (quarantine/warn/log).

### Global QC Rulebook

Foundation-locked artifact defining:
- 8 rule categories with templates
- Global rules (GL-*) that apply to every AC: primary_key completeness, type_mismatch, structural_drift, duplicate_detection
- Default penalty weights for DQ scoring
- Execution precedence order

Global rules are NOT duplicated in AC bodies — loaded from rulebook at runtime and merged with platform (PL-*) and tenant (TN-*) rules.

### AC Body Changes

- Added `rulebook_version` — references the global QC rulebook version
- Added `structural_validation` — schema-level checks (field count, unknown/missing field policy)
- Added `category` per rule — maps to 8 DQC categories
- Added `severity` + `action` per rule — from DQC spec
- Rule IDs carry scope prefix (GL/PL/TN)
- `admission_bands` removed as separate object — band assignment derived from rule outcomes

### Foundation F04: Data Quality

New foundation component at `legacy-v2/docs/system/foundation/data-quality/`:
- `index.md` — DQC overview, boundary mapping
- `qc-rulebook-v1.md` — global rule library
- `qc-scoring.md` — penalty weights, composite scoring
- `qc-categories.md` — detailed category definitions + templates
