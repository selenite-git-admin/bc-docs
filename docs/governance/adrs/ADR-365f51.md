# D261: DevHub Housekeeping Agents — Intelligent Autonomous Platform Maintenance

## Problem

DevHub accumulates governance debt that degrades the value of its data over time:

| Symptom | Impact | Current mitigation |
|---------|--------|-------------------|
| Docs go stale after decisions change | Claude sessions build on outdated specs | Manual `devhub_doc_validate` (rarely run) |
| Sessions crash without closing | Orphan pollution in boot output, lost context | Next session manually recovers |
| Tasks sit in WIP/planned for weeks | Invisible backlog rot, false progress signals | Human eyeball at boot (unreliable) |
| NCs open without resolution | Compliance drift, audit risk | Manual `devhub_qa_nc_list` |
| API/doc registries drift from code | Stale blast radius, wrong dependency info | Manual `devhub_api_scan` |
| Active plans with no tasks | "Invisible on agenda" — decided work never starts | Boot warning, no action |
| Risk review dates pass | Unassessed risks accumulate | Nothing |

**Root cause:** All maintenance is reactive and human-triggered. No agent monitors platform health between sessions.

## Decision

Build a **housekeeping module** in bc-ai (`app/housekeeping/`) with 8 intelligent agents that run on a daily schedule, reason about findings via LLM, and take corrective actions — all with evidence audit trails.

## Architecture: Assess-Recommend-Act

Each housekeeping agent follows a 3-phase pattern:

```
Phase 1: ASSESS    — Call DevHub REST API, gather raw state
Phase 2: RECOMMEND — LLM reasons about findings, proposes actions
Phase 3: ACT       — Execute approved actions, log evidence
```

This is **not** the maker-checker-gate pipeline. That pattern is designed for entity-level suggestions with human review queues. Housekeeping needs a different pattern:

- **Assessor** (no LLM) — pure API calls, gathers data, computes metrics
- **Reasoner** (LLM) — analyzes assessment, classifies findings by severity, recommends actions
- **Executor** (no LLM) — carries out approved actions via DevHub API

The Reasoner is the only LLM-calling component. It uses a single Haiku 4.5 call per agent run (cheap, fast, sufficient for classification and reasoning).

### Why not reuse maker-checker-gate?

| Concern | Maker-Checker-Gate | Housekeeping |
|---------|-------------------|--------------|
| Entity scope | One entity at a time | Batch of findings across whole platform |
| Confidence routing | GREEN/AMBER/RED per suggestion | Severity classification per finding |
| Human review | Individual review queue items | Consolidated daily report |
| Evidence | Per-entity evidence record | Per-run report with finding list |
| Retry logic | Feedback loop with QA gate | No retry — report what you find |
| Cost model | Variable (3 LLM calls per entity) | Fixed (1 LLM call per agent, ~$0.01/day total) |

Building a separate, simpler pattern avoids over-engineering. The maker-checker-gate pipeline remains for its designed purpose (entity AI suggestions).

## The 6 Agents

### Agent 1: Doc Staleness Monitor (`doc-staleness`)

**Assess:** Call `GET /api/documents/validate` — returns docs where linked decisions are newer than `last_validated_at`.

**Reason:** For each stale doc, the LLM classifies:
- **Critical:** Doc references a LOCKED decision that changed — spec may be wrong
- **Moderate:** Doc references an evolving decision — content may need update
- **Cosmetic:** Decision change was metadata-only (status, description) — doc content likely fine

**Act:**
- Auto-action: Update doc status to `possibly_stale` or `stale` based on severity
- Auto-action: Batch-refresh `last_validated_at` for cosmetic-only staleness
- Report-only: Critical/moderate findings listed in daily report for human review

### Agent 2: Session Hygiene (`session-hygiene`)

**Assess:** Call `GET /api/projects/:slug/sessions` for all projects. Find sessions where `ended_at IS NULL` and `started_at < NOW - 24h` (orphaned).

**Reason:** For each orphan, the LLM reads the session plan + checkpoints and:
- Synthesizes what was accomplished vs. planned
- Identifies any WIP tasks that should revert to `planned`
- Drafts a close summary

**Act:**
- Auto-action: Close orphaned sessions with LLM-generated summary, state = "crashed — auto-closed by housekeeper", next = extracted from plan
- Auto-action: Revert WIP tasks to `planned` if the session that started them is orphaned
- Report-only: Sessions closed listed in daily report

### Agent 3: Task Triage (`task-triage`) — The Investigator

The most sophisticated agent. Operates in two modes:

**Mode 1: Backlog Sweep** (first run + weekly deep clean)
Scans ALL open tasks across all projects. For each task, actively investigates real status:

1. Read task title + details
2. Search session summaries (all projects) for matching work — keyword match on task title, UID references, entity names
3. Search activity log for related entries (task UID mentions, decision references matching task tags)
4. Check if files/endpoints mentioned in the task exist in the API registry
5. Cross-reference against closed session deliverables — "session says it built X, task says build X"

Classification per task:
- **Likely completed:** Session evidence shows the work was done but task never marked. Details contain completion language ("executed", "46/46", "all fixed", "merged", "deployed")
- **Stale WIP:** In `wip` > 7 days with no session activity
- **Blocked:** Parent plan has unresolved dependencies or prerequisite tasks still open
- **Priority drift:** `now` tasks sitting idle while `next` tasks have active sessions
- **Orphan plan:** Active plan with 0 linked tasks
- **Abandoned parked:** `parked` > 30 days with no discussion

Token budget: batches tasks in groups of 10-15 per LLM call. First sweep of ~135 tasks ≈ 10 calls ≈ $0.10.

**Mode 2: Daily Delta** (routine housekeeping)
Only examines tasks touched since last run:
- New sessions closed → check if any open task matches session deliverables
- New decisions recorded → check if any task was waiting on that decision
- Tasks sitting in `wip` since yesterday → still active or forgotten?

Token budget: 1-2 LLM calls ≈ $0.01.

**State tracking:** Stores `last_sweep_at` and `last_delta_at` in `housekeeping_run` metadata. Full sweep runs weekly (or on first run). Delta runs daily.

**Assess:**
- Call `GET /api/tasks?status=wip,planned,parked` across all projects
- Call `GET /api/projects/:slug/sessions/latest` for each project (recent session summaries)
- Call `GET /api/activity?limit=100` (recent activity for cross-reference)
- Fetch active plans for orphan-plan detection

**Reason:** LLM receives task batch + session evidence + activity log. For each task, outputs:
- `status_verdict`: `likely_completed` | `stale_wip` | `blocked` | `priority_drift` | `orphan_plan` | `abandoned` | `still_valid`
- `evidence`: why it reached that conclusion (session UID, activity entry, completion language in details)
- `recommended_action`: what to do

**Act:**
- Auto-action: Add `stale` tag to WIP tasks > 7 days
- Recommend (high confidence): Mark `likely_completed` tasks as completed — includes evidence citation
- Recommend: Priority/status changes, plan archival, blocked tags
- Report: Full triage table in daily report

### Agent 4: NC Aging Monitor (`nc-aging`)

**Assess:** Call `GET /api/qa/nc?status=open,investigating`. Compute age in days for each.

**Reason:** The LLM classifies each aging NC:
- **SLA breach:** Open > 14 days with severity `block` — escalate
- **Unassigned:** No `assigned_to` after 3 days — suggest assignment based on repo ownership
- **Likely false positive:** Pattern match against known false-positive categories (e.g., `function-length` on generated code)
- **Cluster:** Multiple NCs from same check + same file — likely systemic, suggest single fix

**Act:**
- Auto-action: Add `overdue` tag to NCs past SLA
- Report-only: Escalation recommendations, assignment suggestions, false-positive candidates

### Agent 5: Registry Refresh (`registry-refresh`)

**Assess:** Check last API scan timestamp and last doc scan timestamp from activity log.

**Reason:** Simple age check — no LLM needed for this agent (pure logic):
- API registry > 24h stale → trigger rescan
- Doc registry > 24h stale → trigger rescan

**Act:**
- Auto-action: Call `POST /api/api-registry/scan` if stale
- Auto-action: Call `POST /api/documents/scan` if stale
- Report: Scan results (endpoints found, docs found, changes)

**Note:** This agent does NOT use LLM — it's pure automation. Included in the housekeeping module for scheduling consistency.

### Agent 6: Risk Review Monitor (`risk-review`)

**Assess:** Call `GET /api/risks?status=identified,assessed,mitigated`. Check `review_date` against today.

**Reason:** For each overdue risk, the LLM:
- Reviews recent project activity (sessions, decisions, completed tasks) for context
- Assesses whether the risk's likelihood/impact may have changed
- Suggests updated scores with reasoning

**Act:**
- Report-only: All risk reassessment recommendations in daily report (risk score changes require human approval)

### Agent 7: QA Patrol (`qa-patrol`)

Proactive quality enforcement — runs audits, catches regressions, verifies fixes. Distinct from NC Aging (Agent 4) which only monitors *existing* NCs. QA Patrol actively creates *new* findings.

**Three capabilities:**

**7a. Scheduled audit sweep** — runs `devhub_qa_audit` against all repos (bc-core, bc-admin, bc-portal, bc-ai, barecount-devhub) on schedule. Compares results against previous run. If new violations appear → auto-raises NCs. If violations disappeared → auto-resolves NCs.

**7b. Post-session regression check** — monitors activity log for `session_end` events. When a session closes, identifies which repos were touched (from session summary / change record `files_changed`). Runs targeted audit on those repos. Catches regressions introduced during a session before they accumulate. Uses LLM to correlate new violations with session changes ("this new `function-length` NC in integrity.service.ts was introduced by the rewrite in SES-faec38").

**7c. Resolution verification** — when an NC status changes to `resolved` (activity log: `qa_nc_resolved`), re-runs the specific check against the specific file. If the violation still exists → reopens the NC with note "fix not effective." If gone → confirms resolution.

**Assess:**
- Call `GET /api/activity?limit=50` — look for `session_end` and `qa_nc_resolved` events since last run
- Call `GET /api/qa/nc?status=resolved` — recently resolved NCs for verification
- Run `devhub_qa_audit` on repos due for sweep or touched by recent sessions

**Reason:** LLM correlates:
- New violations vs. session changes → "likely introduced by SES-xxx"
- Disappeared violations vs. resolved NCs → "fix confirmed" or "coincidental"
- Patterns across repos → "same violation in 3 repos suggests shared config issue"

**Act:**
- Auto-action: Raise NCs for new violations found in audit sweep (same as existing `devhub_qa_audit` auto-raise)
- Auto-action: Auto-resolve NCs whose violations disappeared in latest audit
- Auto-action: Reopen NCs whose resolution verification failed
- Report-only: Regression correlation (which session likely introduced which violation)
- Report-only: Cross-repo pattern alerts

**Schedule:**
- Full audit sweep: weekly (expensive — runs ESLint on all repos)
- Post-session check: triggered by `session_end` activity (event-driven, not cron)
- Resolution verification: triggered by `qa_nc_resolved` activity (event-driven)

**Note:** 7a uses no LLM (pure audit execution). 7b and 7c use LLM for correlation only.

### Agent 8: MkDocs Maintainer (`mkdocs-maintainer`)

Keeps the legacy v2 archive MkDocs site healthy. Sessions frequently create or move files (ADRs, dossier sections, playbooks) without updating `mkdocs.yml` nav or fixing cross-references. Over time this causes orphan pages, broken links, and build warnings.

**Three capabilities:**

**8a. Nav sync** — scans filesystem against `mkdocs.yml` nav section. Finds:
- **Orphan pages:** `.md` files in `docs/` that exist on disk but are not in nav (and not in `not_in_nav` exclusion). These need nav entries.
- **Dead nav entries:** nav references to files that don't exist on disk (renamed/deleted but nav not updated).
- **Misplaced files:** files in wrong directory for their type (e.g., a dossier component file outside the module directory).

**8b. Build health** — runs `mkdocs build --strict` (dry-run) and parses stdout/stderr for:
- **WARNING:** broken cross-references, missing images, invalid YAML frontmatter
- **INFO:** pages not in nav (overlaps with 8a but catches edge cases)
- **ERROR:** template failures, plugin errors, syntax problems

**8c. Frontmatter audit** — scans all `.md` files for required frontmatter fields. Docs should have `uid`, `title`, `description`, `authority` per D234. Missing fields degrade DevHub's doc scanner.

**Assess:**
- Scan `docs/` filesystem for all `.md` files
- Parse `mkdocs.yml` nav tree to extract all referenced file paths
- Run `mkdocs build --strict 2>&1` and capture output
- Scan `.md` files for frontmatter completeness

**Reason:** LLM classifies each finding:
- Which nav section should an orphan page go into (based on file path and content)?
- Is a dead nav entry a rename (suggest fix) or a deletion (suggest remove)?
- Which build warnings are content issues vs. config issues?

**Act:**
- Auto-action: Remove dead nav entries (file doesn't exist)
- Auto-action: Fix frontmatter gaps where values can be inferred from path/filename
- Recommend: Nav insertion point for orphan pages (LLM suggests where in the 879-line nav)
- Recommend: Broken cross-reference fixes
- Report: Full build health summary

**Note:** This agent needs filesystem access to legacy v2 archive (`legacy-v2-docs-root`), not just DevHub API. The `DevHubClient` is extended with a `MkDocsClient` that reads the filesystem directly.

**Schedule:**
- After every session that touches legacy v2 archive (event-driven via activity log)
- Weekly full build health check

## Base Class

```python
class HousekeepingAgent(ABC):
    """Base for all housekeeping agents."""

    agent_id: str           # e.g. "doc-staleness"
    description: str        # Human-readable purpose
    uses_llm: bool = True   # False for registry-refresh

    @abstractmethod
    async def assess(self, client: DevHubClient) -> Assessment:
        """Phase 1: Gather state from DevHub API."""
        ...

    @abstractmethod
    async def recommend(self, assessment: Assessment) -> list[Finding]:
        """Phase 2: Analyze and classify findings (LLM call if uses_llm)."""
        ...

    @abstractmethod
    async def act(self, findings: list[Finding], client: DevHubClient) -> list[Action]:
        """Phase 3: Execute auto-actions, collect report-only items."""
        ...

    async def run(self, client: DevHubClient) -> AgentReport:
        """Full cycle: assess → recommend → act → report."""
        assessment = await self.assess(client)
        findings = await self.recommend(assessment)
        actions = await self.act(findings, client)
        return AgentReport(
            agent_id=self.agent_id,
            findings=findings,
            actions=actions,
            assessment_summary=assessment.summary,
        )
```

## Data Models

```python
class Severity(str, Enum):
    CRITICAL = "critical"
    MODERATE = "moderate"
    LOW = "low"
    INFO = "info"

class ActionType(str, Enum):
    AUTO = "auto"           # Executed immediately
    RECOMMEND = "recommend" # Included in report for human decision

class Finding:
    agent_id: str
    severity: Severity
    entity_type: str        # "document", "session", "task", "nc", "risk"
    entity_uid: str         # e.g. "DOC-a7f3b2", "SES-abc123"
    title: str              # One-line finding
    reasoning: str          # LLM explanation
    recommended_action: str # What should be done

class Action:
    finding_uid: str        # Links back to finding
    action_type: ActionType
    description: str        # What was done / what is recommended
    executed: bool          # True if auto-action was performed
    api_call: str | None    # The DevHub API call made (for audit)
    result: str | None      # API response summary

class AgentReport:
    agent_id: str
    run_at: datetime
    findings: list[Finding]
    actions: list[Action]
    assessment_summary: str
    llm_cost_usd: float
    duration_ms: int

class HousekeepingReport:
    """Consolidated daily report across all agents."""
    run_id: str             # UUID
    run_at: datetime
    agents: list[AgentReport]
    total_findings: int
    total_auto_actions: int
    total_recommendations: int
    total_cost_usd: float
```

## Scheduling

**APScheduler** (lightweight Python scheduling library):

```python
from apscheduler.schedulers.asyncio import AsyncIOScheduler
from apscheduler.triggers.cron import CronTrigger

scheduler = AsyncIOScheduler()

# Daily at 06:00 local time (before work starts)
scheduler.add_job(
    run_all_agents,
    CronTrigger(hour=6, minute=0),
    id="housekeeping-daily",
    name="DevHub Daily Housekeeping",
)

# Registry refresh every 6 hours (cheap, no LLM)
scheduler.add_job(
    run_registry_refresh,
    CronTrigger(hour="*/6", minute=15),
    id="registry-refresh",
    name="Registry Refresh",
)
```

Scheduler starts in FastAPI lifespan (alongside db init and model registry init). Doesn't block request handling.

## Action Classification: Auto vs. Human

| Action | Type | Rationale |
|--------|------|-----------|
| Update doc status to `stale` | **Auto** | Read-only classification, reversible |
| Refresh doc `last_validated_at` | **Auto** | Cosmetic timestamp update |
| Close orphaned session | **Auto** | Session is already dead, just recording the fact |
| Revert orphan's WIP tasks to `planned` | **Auto** | Prevents false WIP signal |
| Add `stale` tag to old WIP task | **Auto** | Additive metadata, reversible |
| Change task priority | **Recommend** | Human judgment on priority |
| Change task status (abort, complete) | **Recommend** | Destructive state change |
| Trigger API/doc registry scan | **Auto** | Read-only scan, no mutations |
| Assign NC to person | **Recommend** | Affects workload |
| Resolve NC as false positive | **Recommend** | Compliance decision |
| Update risk scores | **Recommend** | Risk assessment needs human sign-off |
| Archive plan | **Recommend** | May discard intended work |
| Raise NC from audit sweep | **Auto** | Same as existing `devhub_qa_audit` behavior |
| Auto-resolve NC (violation gone) | **Auto** | Audit evidence confirms fix |
| Reopen NC (resolution failed) | **Auto** | Audit evidence shows violation persists |
| Regression correlation | **Report** | Attribution of violations to sessions |

**Principle:** Auto-actions are limited to additive metadata (tags, timestamps, status downgrades) and cleanup of clearly dead state (orphaned sessions). Anything that affects priority, assignment, or compliance decisions is recommend-only.

## Implementation Phases

### Phase 1: Foundation (1 session)
- `app/housekeeping/base.py` — base class, models, report structure
- `app/housekeeping/client.py` — DevHub REST client
- `app/housekeeping/agents/registry_refresh.py` — simplest agent (no LLM)
- SQLite schema additions
- `POST /api/ai/housekeeping/run` endpoint
- Test with manual trigger

### Phase 2: Intelligent Agents (2 sessions)
- Doc Staleness + Session Hygiene agents (1 session)
- Task Triage + NC Aging agents (1 session)
- Reasoner prompts for each

### Phase 3: Risk Review + QA Patrol + Scheduling (1-2 sessions)
- Risk Review agent

## Dependencies

- `httpx` — async HTTP client for DevHub API (already in bc-ai deps)
- `apscheduler` — job scheduling (new dependency, ~50KB)

## Rejected Alternatives

| Alternative | Why rejected |
|------------|-------------|
| **Claude Code scheduled tasks** | Requires Claude Code running, no evidence trail, session-scoped, can't run unattended on EC2 |
| **Cron scripts (bash)** | No intelligence — can't reason about severity, can't draft summaries |
| **DevHub-native housekeeping** | Node.js/EJS — no LLM integration. bc-ai has the infrastructure |
| **Full maker-checker-gate per finding** | 18 LLM calls/day for low-stakes metadata corrections is over-engineering |

## Security & Guardrails

- DevHub API is localhost-only (port 4000, no auth) — bc-ai runs on same machine
- Housekeeping has **no write access to bc-core DB** — only DevHub metadata
- Auto-actions limited to additive/corrective metadata (see classification table)
- All actions logged with full evidence trail
- Budget counted against daily limit (negligible at ~$0.03/day)
- Dry-run mode available for testing without side effects
