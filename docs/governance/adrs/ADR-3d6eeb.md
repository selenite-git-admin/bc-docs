# External metric-correctness audit as a permanent pre-release gate — two engines, one store, signature-bound certification

## Context

No rationale recorded.

## Decision

The external Codex audit engine (metric-correctness-audit-v1, first run 2026-07-11 over 254 active MCVs) becomes a PERMANENT pre-release certification gate under a separation-of-powers architecture. Principle: the auditor must not be the audited — the platform (and the agent sessions that author metrics) never reimplements, absorbs, or forks the external rubric. Division: (1) AUDITOR-SIDE (Codex): owns and versions the audit method (structural checks + research-driven contextual scoring); read-only access via a dedicated bc_auditor Postgres role (SELECT on mcf/contract/metric_directory/concept_registry/source; no tenant DBs in v1; no MCP/write tools by design); publishes signature-bound verdicts as artifacts (artifacts/metric-audit/&lt;date&gt;-&lt;population&gt;/audit-results.json + per-metric evidence). (2) PLATFORM-SIDE: owns the evidence read surface (stable, change-announced), the verdict store — the EXISTING mcf.chain_audit_evidence becomes a two-engine store (cas-v0 internal regression + codex-metric-correctness-audit-v1 external certification; consumers distinguish by audit_engine_version), the governed import (each result's package_signature_hash validated against the CURRENT MCV signature; stale rows refused and recorded; import is an attributed platform act — the auditor never writes to the database), and the RELEASE GATE: an MCV may be bound to a tenant or admitted to SDG evaluation ONLY with a current external-engine combined_sdg_decision=PASS for its exact package_signature_hash. Any governed remediation (succession, retire/re-mint) changes the signature and automatically voids the certificate, forcing re-audit — the correction loop is closed by construction and matches the auditor's own invalidation rule. CAS remains the fast internal nightly layer (its C2 false-positive class, TSK-c41a81, must be fixed before its verdicts share interpretive weight). CONTEXT_NOT_AUDITED is not releasable — the contextual backlog is an honest release blocker. Open items co-owned with the auditor (proposal §7, artifacts/metric-audit/PROPOSAL-external-audit-gate-2026-07-11.md): by-design OPERATOR_REVIEW allowlist, rubric-version/gate coupling, contextual PASS floor, CAS-01 interplay, directory class_code doctrine line, read-surface stability contract. Program consequence: new metric authoring is PAUSED in favor of depth (Class C remediation + evidence hardening) until the gate is live and the first re-certification loop has closed.

## Amendments (post-Codex review, 2026-07-11)

Codex reviewed the proposal (artifacts/metric-audit/RESPONSE-external-audit-gate-2026-07-11.md) and accepted the architecture in principle with blocking amendments. All are ACCEPTED; they refine the trust boundary and MUST land before enforcement. The core decision above (external gate, two independent engines, one signed store, signature-bound release) stands unchanged.

1. **Bind to the full dependency closure, not just the package signature (P0).** `package_signature_hash` covers only the MCV package; audit conclusions also depend on mutable state outside it — directory member/taxonomy/class/knowledge, OC/CC versions, business-concept/characteristic versions, active source-field mappings + transform params, upstream metric versions and their closures, chain-status/publication/certification/fixture evidence, and the audit methodology + research references. A directory relabel or mapping correction can change a verdict without changing the package signature, leaving a stale certificate current. Every result MUST carry BOTH `package_signature_hash` AND `audit_dependency_closure_hash` (computed from a canonical manifest of every versioned row/evidence object the result used); the gate matches both. The full-run population signature attests corpus completeness only, never per-metric currency.

2. **Authenticate the external artifact (P0).** Platform signing after import proves only that the platform imported a file — not who produced it or that it was unmodified. The auditor MUST publish a canonical manifest (hashes of the JSON report + every referenced evidence file) signed with an auditor-controlled key or approved workload identity; the importer records signer identity, signing method, signature, manifest digest, and verification result, and supports explicit certificate revocation + auditor-key rotation. The durable trust root is an independently governed audit repository + execution identity — NOT an ephemeral Codex session ("Codex" is the method owner, not an operational trust identity).

3. **Preserve both axes; platform recomputes the gate (P1).** The interchange MUST carry the three contextual dimension scores (definition, formula, source_field) with rationale, reference URIs + access dates + doc/version ids + content hashes where obtainable, methodology digest, and the dependency manifest — not a pre-derived combined verdict. The platform RECOMPUTES `combined_sdg_decision` from the two axes (the artifact may include it only as a cross-check); referenced paths are traversal-validated and manifest-hash-verified. A NEW versioned schema `external-metric-audit-interchange-v2` with an explicit adapter is authored auditor-side — bc-core never silently reinterprets the v1 report shape. Correction to the proposal: contextual scores are `1..5` (NOT_AUDITED ⇒ null); there is no score 0.

4. **No indefinite grandfathering (P1).** Enforcing only at the next admission event can leave an existing binding active forever. Every current binding MUST be inventoried and classified certified / blocked / time-bounded-legacy with a migration deadline; any emergency continuation is a separate, attributed, scoped, EXPIRING release exception that NEVER rewrites an audit verdict to PASS.

**Resolved open items (proposal §7):** (7.1) by-design OPERATOR_REVIEW is NEVER folded into PASS — a check is `NOT_APPLICABLE` only under an objective versioned applicability rule; an unresolved condition stays OPERATOR_REVIEW; a release exception is separate/attributed/expiring and leaves the verdict unchanged (immutable history ≠ immutable current status: add a superseding evaluation). (7.2) Gate policy keys on accepted engine versions + minimum-version-by-profile + effective/deprecation dates + emergency revocation; non-semantic patch releases preserve certificates, a major correctness change forces population recert within a migration window. (7.3) Calculator-grade PASS = structural PASS AND definition ≥4 AND formula ≥4 AND source_field ≥4 AND no unresolved critical finding (≡ contextual HIGH_CONFIDENCE/VERIFIED, since overall confidence = min dimension); CONDITIONAL is not releasable and no waiver converts it to PASS. (7.4) CAS-01 = consume only declared-healthy CAS evidence as SUPPORTING, and independently re-derive release-critical invariants — an internal engine is never the sole basis of the external certificate. (7.5) Directory doctrine is computational: `base` = direct concept/entity quantitative inputs, no metric inputs; `derived` = ≥1 metric input, no direct quantitative inputs (constants allowed); `hybrid` = both, only if deliberately supported (else reject explicitly, never mislabel). Operator complexity does NOT determine class. (7.6) Read surface = a platform-generated canonical SIGNED evidence export as the stable primary, with the read-only role retained for drill-down; role hardening adds session/read limits, restricted callable functions, and `ALTER DEFAULT PRIVILEGES FOR ROLE <object_owner>` so future owner-created tables are covered.

**NC/CR compliance loop.** Two governed objects sit between verdict and recertification: an auditor-issued **NC** (Non-Conformance; severity CRITICAL/MAJOR/MINOR, scope METRIC/FAMILY/SYSTEMIC, with the exact affected-metric list carrying both hashes — one systemic NC per shared root cause, not hundreds of tickets) and a platform-submitted **CR** (Compliance Report: root cause, containment, corrective/preventive action, changed governed objects, superseded + new signatures, fixture/chain evidence, impact). The platform signs the CR; the auditor responds INSUFFICIENT_EVIDENCE / ACCEPTED_FOR_REAUDIT / VERIFIED_CLOSED / PARTIALLY_CLOSED. An accepted plan does NOT close an NC — closure requires observed evidence + a passing independent re-audit on the new package + closure signatures. The release gate consumes the current certificate, never the NC/CR status; a CR cannot manufacture PASS.

**Code + deployment boundary.** The external engine does NOT live in bc-core. A separately governed `barecount-external-audit` repo owns the rubric, checks, contextual scoring, dependency-closure verification, interchange/NC/CR schemas, and certificate/manifest signing. bc-core owns evidence-export (canonical signed read bundle + change feed), audit-import (schema/signature/hash validation + persistence), release-gate (recomputes admission from accepted evidence), and compliance-reporting (CR generation/signing). `barecount-devhub/artifacts/` is development EXCHANGE only — the current DevHub scripts are an audit-engine PROTOTYPE and migrate (with history + tests) into the governed repo before enforcement; DevHub is never the production code authority or trust root.

**Concessions + errata (platform ↔ auditor).** (a) My SRC-01 claim (source_references rendered as empty `{}`) is WITHDRAWN — not reproducible; the published audit-results.json carries complete reference objects (claim_id/claim_text/source_uri/tool_code/tool_call_id); the `{}` I saw were legitimate empty objects (discriminator_json / resolver config). (b) The `Average Days Late` report title is NOT a renderer defect — it faithfully rendered the MCV's stored display name; the mismatch against mc_name `average_days_to_collect` is a REAL contract/directory semantic defect for the remediation backlog (a semantic-consistency check across mc_name / display name / directory name / definition / formula is now an audit requirement). (c) Codex owns a contextual-v1 errata before any v2 certificate: replace name-regex population inference with definition/filter-aware classification (average_days_to_collect's definition limits to status='paid', so BSAD cleared-item sourcing was wrongly flagged), remove the generic IFRS-18 citation fallback (unresolved grounding must LOWER confidence, not fabricate a citation), and re-run + sample-review each rule family. No existing PASS is affected — the 2026-07-11 run produced zero PASS.

**Enforcement sequencing (agreed):** (1) this ADR amendment; (2) stand up barecount-external-audit + migrate the prototype; (3) define NC/CR/response/certificate schemas + ownership; (4) publish contextual-v1 errata + v2 interchange schema; (5) define + test the canonical dependency-closure manifest; (6) signed export/import with rejection + revocation tests; (7) wire gate + NC/CR in SHADOW mode and reconcile; (8) re-certify the re-minted GL balance family end-to-end (NC → CR → verified closure) as the proof; (9) enforce only after the proof run + legacy-binding inventory pass.
