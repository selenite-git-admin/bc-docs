# Metric Catalog rebuild — governed attributes, panel-time enrichment, knowledge import, and the two-strip trust view

## Context

The June 9 dev-greenfield reset retired the definition-keyed metric corpus per the D437 golden path, leaving the Metric Catalog page reading an intentionally-empty store and raising the operator's concern that the enriched metric view (knowledge, mappings) was lost. The read-only snapshot audit proved the enrichment is fully preserved and quantified the delta beyond seed documents. Rebuilding the catalog on the governed substrate keeps one metric authority (mcf.*), makes enrichment reliable by routing it through the adversarial M12 panel instead of an ungoverned side job, preserves the legacy curation as panel input rather than resurrected truth, honestly separates seed vocabulary from master-backed classification, and answers "can I trust this metric?" in one place by rendering only emitted evidence — consistent with operator directive #6 (legacy UI keeps its intent; gaps are bridged deliberately) and the Foundation invariants.

## Decision

Operator-locked design from the 2026-06-10 discussion (SES-d58dc6), filed as proposed for operator review before any build. Five decisions:

**D1 — Catalog population.** The bc-admin Metric Catalog page lists GOVERNED metric contracts (mcf.metric_contract + current mcf.metric_contract_version with governance state draft/review/approved/active), NOT the seed reservoir. Register = the 12,501-seed reservoir; Catalog = the platform's contracted metrics. The legacy read over retired metric.metric_definition is removed.

**D2 — Enrichment is governed M12 panel output.** Metric enrichment (knowledge profile: definition summary, context, stakeholders, drivers, formula explanation, threshold source/notes, related metrics) is produced inside the M12 Maker/Checker/Judge panel: Maker drafts the profile, Checker verifies, output lands with the governed contract with panel evidence — replacing the legacy side-job enrichment service (metric.metric_enrichment_job, 1,141 rows pre-reset). The legacy curation is NOT lost: snapshot audit (read-only, 2026-06-10, golden-snapshot-pre-reset-2026-06-09.sql) confirmed metric.metric_knowledge = 1,241 rows (1:1 with definitions; cols: definition_summary, definition_context, stakeholders, formula_explanation, threshold_source, threshold_notes, drivers, related_metrics, search_tags), metric_formula 1,216, metric_binding 1,133 — all preserved. A ONE-TIME governed import carries this knowledge into the reservoir keyed by metric_name as PANEL INPUT (not truth); the import is a separate operator-gated DBCP (reservoir write). Legacy metric.* tables are not resurrected.

**D3 — Locked attribute list with sources.** Three source classes: seed (reference), panel-enriched (governed), runtime evidence (computed). Identity: name/display/description — seed → panel-confirmed. Classification: metric type (Performance/Efficiency/Quality...), BSC perspective, tier, direction — panel-enriched; FUNCTION + SUBFUNCTION are panel-NORMALIZED against the platform Function/Subfunction masters at promotion (Checker-verified) and stored on the MCV's function_code/subfunction_code — the seed keeps its free-text "Function Tag"/"Subfunction Tag" as honest provenance; the 12,501-row free-text→master mapping resolves one metric at a time at promotion. Knowledge: summary/context/stakeholders/drivers — panel-enriched (seeded from the D2 import). Formula: canonical AST + explanation — M12-governed. Thresholds: values + source + notes — panel-enriched. Relationships: related metrics — panel-enriched. Governance: state + certs — runtime evidence. Storage home for the knowledge profile fields on the MCF substrate is an explicit follow-up DBCP (DB Change Protocol applies; not decided here).

**D4 — Two-strip trust view (retires KPI-map + integrity screens for mcf-governed metrics).** Per-metric trust panel with two strips. (a) LIFECYCLE strip — audited state transitions: Seed → Contracted → Panel-verified → Activated → Bound → Evaluated → Producing; every edge renders ONLY from already-emitted evidence (M11 intake row, M12 panel run + verdict, M12.5 materialization cert, M13 PE rows, M14 activation cert, tenant.contract_binding row, progression.metric_evaluation row, metric_snapshot_index row); reads never recompute (Foundation: reads do not trigger evaluation; Invariant VI evidence is emitted not inferred); Bound/Evaluated/Producing are tenant-scoped and render grey "n/a" at platform scope. (b) CHAIN strip — structural integrity, platform/design-scoped (no source fields needed): SC → AC → OC → (BCF) → CC → MC; each node green when that contract exists, is active, and its declared link to the next holds (OC field map covers CC selection; CC selection covers MC variable bindings; BCF concepts resolve where declared); the (BCF) node shows grey "not declared" when concept linkage is absent — no placeholder greens. "All green" = both strips green, rolled up to a single dot per metric in the Catalog list. Legacy diagnostics (chain-integrity page, orphan endpoint) remain for legacy rows only.

**D5 — Metric detail page.** Re-keyed to the governed metric: current MCV + governance state, both trust strips, the knowledge profile, thresholds, formula + explanation, and a "source seed" lineage section carrying the full reference profile (5D/BSC/tier from the seed document).

**Sequencing (bridge #5):** (1) this ADR review → decided; (2) knowledge-import DBCP (operator-gated reservoir write); (3) bc-core reads: governed catalog list/detail + trust-strip evidence endpoint; (4) bc-admin Catalog + detail rebuild; held PRs per house discipline. Builds on the bridge #3/#4 ladder re-point (PR #257, holding).

## Amendment 1 — 2026-06-10 (operator-locked, SES-d58dc6)

## Amendment 2 — 2026-06-10 (operator-locked "Option B", SES-1480d8)

**Degraded-source knowledge acceptance lives in the knowledge family — `mcf.certification_record` stays pure.**

**Context.** The remediation plan's E item assumed a `metric_knowledge_accept` action in `mcf.certification_record` behind a one-line vocabulary DBCP. The read-only substrate audit (SES-1480d8) falsified that assumption: the cert table is pinned on four axes — `mcf_cert_action_code_chk` (only metric_create / metric_transition / metric_supersede), `mcf_cert_primitive_type_chk` (`primitive_type = 'metric_contract_version'` only), `mcf_cert_subject_kind_chk` (three MCV subject kinds), and `mcf_cert_state_transition_chk` with `to_state_code NOT NULL` (every action maps to a fixed governance-state transition). A knowledge-acceptance cert violates all four, and the violation is semantic, not incidental: accepting a knowledge profile is **not** a governance-state transition, which is precisely what this table is purpose-built to certify. Option A (amend the four CHECKs) was presented and rejected — it would weaken the schema guarantees protecting the MCV state machine to admit a record that does not belong there.

**Decision (operator: "B").** Acceptance evidence is recorded where the knowledge lives: one nullable column `mcf.metric_knowledge_profile.acceptance_rationale_text` with `CHECK (NULL OR char_length >= 40)` (`mkp_acceptance_rationale_len_chk`). Non-NULL **iff** the operator accepted a profile whose source panel run verdict was NOT `APPROVE_FOR_DRAFT` (the DEC-804874/D366 §4 rationale pattern). The acceptor is `created_by_name` and the acceptance time is `created_at` — the operator invoking the persist IS the acceptor by construction, so no duplicate `accepted_by`/`accepted_at` columns (D162 rule 4). Stamped at INSERT only; the table stays append-only (Invariant III); the profile row plus this column is the emitted acceptance evidence (Invariant VI).

**Enforcement.** Writer gate **KW-G5** (`McfKnowledgeProfileWriterService.persistFromPanelRun`): the enrichment path reads the run's consensus `verdict_code`; a non-APPROVE source without a ≥40-char rationale is refused (HTTP 409). APPROVE runs ignore a supplied rationale so the column's non-NULL state means exactly one thing. The M12.5 auto-persist path always stamps NULL — it only runs on APPROVE consensus. The endpoint body gains optional `rationale_text`; the detail read and the bc-admin Knowledge card surface the acceptance.

**No backfill.** The one historical degraded-source row (`7f0b60f3`, ARPI enrichment via OPERATOR_REVIEW run `601401f7`, persisted before this column existed) is deliberately NOT updated — that would violate append-only. Its acceptance is documented in change record CHG-582702 (SES-d58dc6). KW-G5 was live-proven against that run post-implementation: persist without rationale → 409; with rationale → idempotent `already_existed`, row untouched.

**Implementation:** bc-core PR #261 (substrate column applied live under the approved envelope + KW-G5 + specs) with bc-admin PR #21 (acceptance note in the Knowledge card); companion H2 auto-persist in bc-core PR #260.
