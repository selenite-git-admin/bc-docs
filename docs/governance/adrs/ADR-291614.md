# Canonical-value filters are permissible at the Metric Contract layer

## Context

Three independent evidence sources converge:

1. PE-MC-12 evaluator (machine gate, live) — strategic_filter with non-null canonical_value_set: literal membership test → PASS when literal ∈ set. This is the designed happy path, not an edge case.

2. D442 §D6.2 runtime matrix — three filter-eligible roles (strategic_filter, status, dimension) are the ONLY roles that may appear in MC filter clauses. strategic_filter is defined as "primary filter target" — it exists for exactly this use case.

3. The CC-succession + D441 + D492 arc — CC v9.5.1 code_lookup resolves source codes at the canonical boundary (layer A/C). BCV f10afed1 declares the closed vocabulary at layer B. PE-MC-12 validates membership at the publication gate. D492 Layer 1 (G4 formula leg, evaluator v11) extends this to formula ASTs. The entire multi-session engineering arc was built to enable canonical-value filters as the source-agnostic replacement for source literals.

The ruling was prompted by TSK-f8ff16 (boundary case from audit SES-464df9) where a parked review-state MCV carried document_type_code='customer_invoice' and the question was whether this canonical value — as distinct from the SAP literal 'DR' it replaces — is permitted at the MC layer.

## Decision

Canonical-value filters ARE permissible at the Metric Contract (MC) layer. An MC filter clause may reference values from a BCF concept's canonical_value_set when the bound business_concept_version carries semantic_role=strategic_filter (or status or dimension per D442 §D6.2).

This is the designed outcome of the D441→D442→D443→D492 arc: source-system literals (SAP BLART codes DR/DZ/KR/KZ) are forbidden at the MC boundary, but canonical vocabulary declared on a strategic_filter BCV is the REPLACEMENT, not a violation.

Concrete example (the ruling's immediate application): document_type_code='customer_invoice' in an MC filter clause, bound to concept f10afed1 (semantic_role=strategic_filter, canonical_value_set=[customer_invoice, customer_payment, credit_memo]), is a PASS under PE-MC-12.

Machine enforcement: PE-MC-12 (G4, evaluator mcf-m13-v11) already validates this — filter leg (pe-mc-12-evaluation.ts:126-158) and formula leg (pe-mc-12-formula-literal-evaluation.ts:140-154) both test literal ∈ canonical_value_set for strategic_filter/status roles. The gate is implemented, not aspirational.

Invariant compliance: (I) Meaning is evaluated once — the canonical value set is declared on the BCV; the MC filter references it, no meaning drift. (II–VI) Not implicated — the filter is a semantic declaration, not an evaluation, mutation, or replay.

What is NOT permitted (unchanged): source-system literals (DR, DZ, BLART values), diagnostic-role BCV references in filters, filters bound to identity/temporal/amount/reference roles. These remain REJECT under PE-MC-12.

Unblocks: TSK-853a3b (CB-013 wave-1+2 re-mints). The re-mint template: replace source literals with canonical-value equivalents bound to the strategic_filter BCV.
