# Reader Path Robustness + Chain Integrity Gates — fail-fast contracts, locked SOPs, end-to-end smoke gate

## Context

D381 Phase 1 (sandbox1 demo) surfaced multi-layer silent fragility in the reader path. We hit each layer in turn — admission "succeeded" with garbage source_keys; canonical resolution "succeeded" with groups: 0; chain_status SSOT shows 576 MCs as 'complete' but the underlying readers/OCs include draft-status entries and field-shape mismatches that prevent end-to-end snapshots. The platform leaks "looks green" through every layer until the very last (no metric snapshots in fact.*). For a real-time observation platform whose entire value proposition is trustworthy data, this is exactly the kind of failure that erodes credibility. Five contracts proposed below, sized to be implementable in 1-3 sessions each. None require deep architectural rework — they are mostly fail-fast checks, governance gates, and one alias-translation decision that's been implicit.

# Reader Path Robustness + Chain Integrity Gates

## Why now

D381 Phase 1 attempted to prove sandbox1's chain end-to-end. We progressed through admission (1,368 records admitted), found canonical_mappings, but stopped at canonical resolution with `groups: 0`. Drilling in surfaced five distinct platform-layer fragilities, each silent:

1. Reader flavor `config_json` was NULL → `keyFields=[]` → admission emitted `{_entity:'X'}` for every record. No warning.
2. Executor doesn't fall back to OC's `identity_semantics.identity_fields` when flavor `keyFields` is empty.
3. Reader emits `sourceEntityName` = OData entity set name (`CustomerOpenItemSet`); canonical resolution filters by `sourceEntityName === primaryEntity` where `primaryEntity` = SAP table name (`BSID`). Mismatch is silent — `bindingsConsidered: 1, resolved: [{ summary: { groups: 0 } }]`, status `ok`.
4. OC governance status `draft` participates in chains the platform reports as `chain_complete`. ecc-invoice-hdr (draft) underlies DSO and Bad Debt Ratio chains.
5. No end-to-end smoke gate. The 576 chain-complete MCs include chains that no one has actually exercised on a real tenant.

Each of these can be addressed independently. None require deep refactoring.

## Decisions

### D-1: `sourceEntityName` contract

`sourceEntityName` on every admission row is **the canonical SAP table name**, not the OData entity set name through which the data was fetched.

- ECC OData V2 readers translate entity set name → SAP table name before emitting observations. (E.g. `CustomerOpenItemSet` → `BSID`.) The `ENTITY_SET_MAP` already exists in `bc-sdg/src/simulators/sap-ecc/odata-server.ts`; the equivalent map needs to live in the bc-core executor or be sourced from the OC's `source_references[].source_table`.
- S/4HANA readers (V2/V4) where the entity set name already matches the SAP/CDS view name see no change.
- Other modules (admission, evidence, lineage, canonical resolution) continue to compare `sourceEntityName` to OC `source_table` directly — no ENTITY_SET_MAP-aware comparison code paths needed downstream.

Rationale for this direction (over alias maps in the resolution layer): bc-core's contract surface is SAP-canonical. The reader is the only layer that knows it fetched via an OData entity set; that detail should not leak past admission. (Option A from the four enumerated in TSK-d8d7c6.)

### D-2: Reader executor fails fast on missing identity fields

`sap-odata-v2.executor.ts` (and equivalent V4 / SDG / generic OData executors) MUST throw `UnprocessableEntityException` at execution start if the chain `flavor.keyFields ∪ OC.identity_semantics.identity_fields` resolves to empty or to a single field that the OC's `so_schema.required` reveals to be non-unique. Source keys MUST NOT silently fall back to `{_entity: <name>}`.

Order of resolution per record:
1. `flavor.config_json.keyFields` if present and non-empty.
2. else `OC.identity_semantics.identity_fields` if present and non-empty.
3. else throw at execution start (not per-record — one error per run).

### D-3: Reader Flavor Activation SOP — gated promotion

Promotion `draft → active` of a `runtime.reader_flavor` row requires:

| Gate | Check |
|---|---|
| Schema | `config_json` declares `entitySet` AND `keyFields` (non-empty array) |
| OC linkage | `observation_contract_id` references an OC with `status_code='active'` |
| OC identity | OC's `identity_semantics.identity_fields` is non-empty AND each field appears in OC's `so_schema.properties` |
| Field-shape | `keyFields` ⊆ OC's `so_schema.properties` keys (case-insensitive) |
| Preflight | A dry-run reader execution against the bound connection succeeds with at least one record returned and a non-empty `source_key` per record |

The promotion path is API-driven (e.g. `POST /api/registry/reader-flavors/:id/activate`) and runs the gate; bypassing the API path with a direct DB UPDATE on `status_code` is a process violation worth detecting in chain_status audits.

### D-4: chain_status SSOT respects governance state

`contract.chain_status.chain_verdict='complete'` must not be set when:

- Any reader flavor in the chain has `status_code != 'active'`
- Any OC in the chain has `status_code != 'active'`
- Any participating canonical_mapping is missing the field_resolutions / source_dependencies needed by the canonical resolution service's normalize pass

A new verdict value `governance_blocked` (or extension of the existing `partial`/`broken`/`unlinked` set) captures the case where structural completeness exists but governance state isn't ready. The MC catalog UI shows it distinct from chain-broken.

### D-5: Canary tenant smoke gate

Every MC whose chain reaches `chain_verdict='complete'` automatically gets exercised against a designated `qa-bench` (or dedicated `canary`) tenant: source data is generated, readers run, canonical evaluation runs, metric evaluation runs. If at least one snapshot does not land in `fact.ms_*` for the canary tenant, the chain_status is downgraded back to `partial` with an evidence reference to the failed run.

This is the integration test the platform doesn't currently have. It can be deferred until D-4 is in place — it is the runtime confirmation of D-4's structural check.

## Implementation tracking

Each decision becomes a separate DevHub task linked to this ADR. Sequencing:

1. **D-1** unblocks D381 Phase 1 directly. Highest priority. Implementation in bc-core/src/boundary/reader-runtime/executors/sap-odata-v2.executor.ts (and a config decision: where the entity-set-to-table map lives — flavor config, OC body, or platform-wide registry).
2. **D-2** is a small additional change in the same executor file plus the V4 / SDG / OER variants. Small scope.
3. **D-3** is a service + API + governance update. Medium scope.
4. **D-4** is a chain_status compute change + UI surface. Medium scope.
5. **D-5** is the largest — needs canary tenant infrastructure + scheduler hook + chain_status downgrade machinery.

## Consequences

- The 576 chain_complete MCs will likely shrink to a smaller number once D-4 lands — many will downgrade because their underlying flavors are draft or their canonical_mappings have unresolved gaps. This is honest, not a regression.
- Reader flavor seeding scripts (`bc-core/scripts/seed-fi-readers-week2.mjs` etc) need a sweep to ensure all active ECC flavors carry valid `keyFields` and `entitySet`. Not blocking D-1; downstream work.
- D-5's canary tenant changes the platform's "cost of activating a metric" — every promotion now triggers a smoke run. This is a feature, not a bug; the cost paid here is what makes downstream `chain_complete` claims trustworthy.

## Open questions

- Where does the entity-set → SAP table map live? Options: (a) flavor config, (b) OC body, (c) platform-wide registry table, (d) hard-coded in executors. (b) seems cleanest — the OC declares the SAP table; the reader executor reads OC's source_references and translates. Decide before D-1 implementation.
- Does the canary tenant share infrastructure with sandbox1, or is it a distinct tenant? Cleanest to make it distinct with hardcoded data shape so chain_status integrity is independent of any user tenant's state.

## v3 Documentation Reconciliation

After authoring the original five decisions above, we audited them against bc-docs-v3 chapters. **Two of the five decisions reduce to enforcing existing v3 contracts the platform isn't honoring; one is augmented; one new decision (D-6) was surfaced.** The reframed scope:

### D-1 — Net new (stands as proposed)

`sourceEntityName` contract is not documented anywhere in v3. Locking it via this ADR adds the contract; the chapter to update is `operating-model/connectors-and-readers.md` (or `operating-model/admission-and-observation.md` — the better fit). Implementation in the reader executor is net-new code.

### D-2 — Augmented

`onboarding/observation-contract-creation.md` already specifies that `identity_semantics.identity_fields[]` is "typically the source table PK", semi-auto pre-populated from SC key markers (line 178, 183), and gate #8 of CR-QG-OC requires every identity field to exist in a referenced source table.

The reader executor's fail-fast logic (D-2 as proposed) is still net-new. The fail-fast logic assumes OCs in the registry declare *sufficient* identity_fields per the SOP. Where they don't — surfaced today on `oc__s4hana__bsid__receivable_hdr` declaring only `["BELNR"]` instead of the BSID compound PK `[BUKRS,BELNR,GJAHR,BUZEI]` — those are data-quality drift, fixed by re-authoring the OC and (separately) tightening gate #8's wording from "identity_fields exist as source fields" to "identity_fields are sufficient to uniquely identify a record". That data + wording fix is tactical and tracked outside this ADR (TSK-050356, a D381 follow-up).

### D-3 — Reduces to enforcement of existing v3 SOP

`onboarding/reader-creation.md` already locks **CR-QG-RDR-001 / CR-QG-RDR-002 / CR-QG-RDR-003** with explicit gates: BO linked, OC bound + active, Connector bound + active, config_json carries the field map, status active, plus seven chain-integrity checks. The "Activation requires CR-QG-RDR-003 (the seven chain integrity checks below) to pass. A draft Flavor or a Flavor with missing OC binding cannot be activated" line is already authoritative.

**The implementation has drifted from the SOP.** All 9 currently-active ECC reader flavors have `config_json=NULL` despite gate CR-QG-RDR-002.3 requiring "config present: config_json carries the field map". The fix is not a new SOP — it is wiring the existing SOP's gates into the activation API path so direct DB UPDATE bypass is impossible. Reframe: D-3 = "Implement CR-QG-RDR-002 / CR-QG-RDR-003 enforcement in code; backfill drift on existing active flavors."

### D-4 — Reduces to enforcement of existing chapter

`operating-model/chain-completeness-and-verdict.md` already lists C2 ("contract's governance state is `active`") as one of the five per-contract checks; failure → `broken` verdict. L6 explicitly: "The Observation Contract names a Reader that is governance-active and bound."

The verdict semantic set (`complete`, `partial`, `broken`, `unlinked`) already covers the case. **The implementation isn't honoring the chapter** — the 576 "complete" MCs include chains where C2 should fail (draft OC, draft flavor) but the compute reports `complete` anyway. No new verdict needed; the introduction of `governance_blocked` proposed earlier was redundant.

Reframe: D-4 = "Fix ChainStatusService compute to honor C1-C5 governance state per the chapter; expect the 576 'complete' count to drop to the honest figure when chains with draft links downgrade to `broken`."

### D-5 — Net new (stands as proposed)

Canary tenant smoke gate is not documented in v3. `implementation/synthetic-data-and-testing.md` covers the bc-sdg synthetic data layer but doesn't lock a "every chain_complete MC must produce a snapshot via canary" gate. D-5 adds the contract.

### Reconciled summary

| Decision | Net new? | Lands in |
|---|---|---|
| D-1 sourceEntityName contract | yes | bc-core executor + new chapter language |
| D-2 reader executor fail-fast | yes (executor logic) | bc-core executors |
| D-3 reader flavor activation enforcement | enforcement of existing CR-QG-RDR-002/003 | bc-core API + activation service |
| D-4 chain_status governance compute | enforcement of existing chapter | bc-core ChainStatusService |
| D-5 canary tenant smoke gate | yes | platform infrastructure + new chapter |

D-1, D-5 add new platform contracts. D-2 adds new code at a layer the SOP already reasoned about. D-3, D-4 are pure enforcement of existing chapters. The BSID OC re-authoring + gate #8 tightening that surfaced during reconciliation is a tactical D381 follow-up tracked as TSK-050356, outside the ADR scope.

### What this commit DID drive

Two adjacent gaps surfaced while exercising the chain — neither belongs to D382 D-1..D-5 directly, but both fit D382's broader "fail-fast contracts" theme:

- **Tenant scope contract.** Reader runtime previously hardcoded `tenantId ?? 'demo-selenite'` when the caller omitted it. Removed; `tenantId` is now required, resolved from CLS via `@CurrentTenant` (x-tenant-id header is the source of truth). Eliminates silent cross-tenant writes — exactly the kind of "looks green" failure D382 was authored to attack.
- **M4.2b identity rule enforcement.** Canonical resolver was generating a fresh `canonicalObjectId` UUID, breaking the M4.2b identity contract (`canonical_object_id === canonical_evaluation_id === evaluation_id`) that FactReader / dispatcher / evaluation.repository all rely on. Result was every CO unreachable from metric eval. Fixed by setting `canonicalObjectId = evaluationId`.

These belong in D382's "next-revision" scope if the ADR is amended, or in a focused follow-up ADR if a cleaner separation is preferred.
