# CB-008 Component C — cross-entity governed selection (credit-limit utilization)

## Context

No rationale recorded.

## Decision

> **AMENDMENT (2026-06-30, Foundation correction — supersedes the Part 2 mechanism below).**
> An independent Foundation re-check (operator-raised: "are we modelling around contracts / crossing boundaries?") found the original Part 2 mechanism WRONG. The cross-entity metric itself is Foundation-legitimate, but the way to express it is **`metric_binding` + grain alignment at the Metric Contract level**, NOT a `reference_resolved_as_of` variable-binding selection rule.
>
> **What Foundation says.** The metric boundary reads **N Canonical Object versions across N Canonical Contracts** via `metric_binding` + **grain alignment** (the-evaluation-boundaries §metric boundary L113; the-contract-grammar §Metric Contract L198/202; DEC-29c324 / FND-ERR-003). My §"FOUNDATION GROUNDING" paragraph below misread the-governed-selection: the governed selection is the **temporal/as-of selection WITHIN the metric's referenced inputs**, not a mechanism to reach across into a foreign entity's COs. The "secondary single-version governed selection keyed by the reference" framing conflated the temporal selection with the cross-CC reference (which is `metric_binding`) and the join (which is grain alignment).
>
> **MCF grounding (live, 2026-06-30).** No explicit `metric_binding` table — referenced CCs are **implicit** via `metric_variable_binding.bound_business_concept_id` (BC → owner entity → CC). `mvb_role_target_chk` **already permits** `input` + `bound_entity_id`, so a plain `input` binding to the Customer `credit_limit` BC already expresses the cross-CC reference. PE-MC's binding-integrity / source-chain gate already validates a BC against its owning entity + OC/CC resolvability — it does NOT assume single-entity (so a `credit_limit` binding resolves now that `cc__customer_master` is active). `governed-metric-runtime` is entity-agnostic (operates on a provided CO set + grain grouping) — *which* COs and *how aligned* is the deferred SDG-phase I/O.
>
> **Corrected Part 2 (no new substrate).** `credit_limit` = a plain cross-entity `input` binding (`bound_business_concept_id`=credit_limit, `bound_entity_id`=Customer). Grain alignment = the `bill_to→Customer` reference, which is **already the Component B grouping key** (the metric is grouped by `bill_to_customer`; `credit_limit` aligns on the same customer identity). The as-of is the temporal gate. The cross-entity CO read + identity-keyed alignment is **deferred SDG-phase runtime**, like all metric execution (stop at chain integrity). So Part 2 reduces to: author the metric with a cross-entity `credit_limit` input + Component B customer grouping; verify PE-MC resolves it (`cc__customer_master` active) and it activates; **no `reference_resolved_as_of` rule, no DB change**.
>
> **Reverted.** The `reference_resolved_as_of` substrate + grammar (commit bc-core `3409bc09`) + its DB constraint (`mvb_reference_resolved_chk` dropped; `mvb_selection_rule_chk` restored to the 3-value enum) are **withdrawn** (revert `41212c4d` + DB revert, verified). Part 1 (the Customer master chain) stays correct and is unaffected.
>
> Repair location for corrected Part 2: B (the metric declares a cross-entity input + grain alignment — existing grammar) + runtime alignment deferred (D, SDG-phase). The §"FOUNDATION GATE (1) Part 2 = B + D" rationale below stands; only the *mechanism* (a new selection rule) was wrong.

CONTEXT. CB-008 Component A (DEC-7f2e73/D468) stamped the invoice.bill_to → Customer reference through the Customer Invoice contract chain and explicitly DEFERRED the Invariant-IV reference (invoice CO → Customer CO) to "governed selection at the METRIC boundary (Component C)". Component B (DEC-2c2849/D469) added reference-dimension grouping (partition a metric by a stamped reference). Component C is the deferred capability: a metric that uses a value from a DIFFERENT entity's CO, resolved via the stamped reference. Canonical proving metric (operator-locked): credit-limit utilization per customer = AR balance (Customer Invoice COs, open at P, grouped by bill_to_customer — Component B) ÷ credit limit (Customer master CO, as-of P), per customer.

FOUNDATION GROUNDING (the-governed-selection.md, authoritative, DEC-c4c742). This is NOT a new Foundation artifact. The chapter §"The reserved artifact, generalized" states that resolving a single reference to a single version IS a governed selection — the degenerate single-member case ("the single-version case is recovered by a predicate that resolves to one version"). And §"a metric evaluation is one path: a governed selection, then an aggregation over the resolved set." So the credit-limit-utilization metric is simply TWO governed selections + a ratio: (1) PRIMARY — invoice COs open at P, grouped by bill_to_customer → AR balance per customer; (2) SECONDARY (single-version) — the Customer CO that is the customer's master record as-of P, keyed by the bill_to→Customer reference → credit_limit. Both resolved sets recorded in Lineage. Clean under Invariants I (meaning once), IV (explicit reference via the stamped edge), V (non-replayable, knowledge-time per P), VI (evidence emitted).

GROUNDING FINDINGS (live substrate, 2026-06-30):
- The Customer entity (705f47e1) ALREADY carries credit_limit (BC bb7d8b50, amount/decimal, active) + a customer-account-identifier identity-bearing key (BC 68267928) + country/currency/industry/payment-terms/status codes. No BCF enrichment needed.
- BUT the Customer MASTER entity has NO contract chain — the only Customer-related CC is cc__customer_invoice_arpi_slice (the Invoice TRANSACTION chain). So NO Customer COs exist; the credit-limit DENOMINATOR is not yet a canonical object.
- Substrate precedent EXISTS: mcf.metric_variable_binding already has bound_entity_id + snapshot_selection_rule_code + snapshot_selection_params_json; 40 live 'metric_input' bindings resolve a FOREIGN object (another metric's snapshot) via a governed selection rule (as_of_period_end / period_matched / prior_period_end). Component C extends this exact pattern to resolve a foreign ENTITY's CO via a reference.

DECISION — Component C = TWO PARTS, sequenced Part 1 → Part 2.

PART 1 (prerequisite): Onboard the FULL Customer MASTER entity as a master/state canonical entity (operator-locked: full master, not a thin slice). SC → AC → OC → CC admitting customer master records, producing Customer COs carrying all 7 Customer BCs (credit_limit + key + country/currency/industry/payment-terms/status). State/as_of entity (master data, not transactional). Standard entity onboarding per the bc-docs-v3 Onboarding SOPs. The full master is chosen because it is reusable across a class of customer-attribute metrics (segmentation, per-customer master joins), not just credit-limit utilization. Repair location A/B/C.

PART 2 (the capability): MCF cross-entity variable binding + runtime.
- Grammar (repair location B): a new binding shape — an 'input' variable with bound_entity_id = Customer (≠ the metric's grain entity) + a new snapshot_selection_rule_code (proposed: 'reference_resolved_as_of') whose snapshot_selection_params_json declares {join_via_reference: bill_to_customer, target_entity_id: 705f47e1, target_key: customer_account_identifier, temporal: as_of}. The variable resolves to the credit_limit value of the Customer CO whose key matches the primary group's bill_to_customer, as-of P.
- Runtime (repair location D): governed-metric-runtime, per primary group (customer), performs the SECONDARY governed selection — resolve the matching Customer CO version as-of P (single-version predicate) → credit_limit — then the formula AST computes AR balance ÷ credit_limit per group. Reuses the existing governed-selection primitive against a SECOND candidate set (Customer COs). Both resolved sets emitted to Lineage.
- Gates: PE-MC-2/PE-MC-7 (reference DAG) MUST validate the cross-entity binding rides a STAMPED, governed, active reference edge (the bill_to→Customer reference in the chain). Fail-closed if the reference is not governed/active.
- Open implementation detail (not a separate decision): whether 'reference_resolved_as_of' is a new snapshot_selection_rule_code value or folds into the existing selection-rule vocabulary; and whether the AST engine already supports a per-group two-selection ratio or needs a small extension.

FOUNDATION GATE. (1) Why this location? Part 2 = B (the metric variable-binding grammar cannot today express a binding to a CF on a different entity resolved via a reference) + D (the runtime must perform the second governed selection over Customer COs). Part 1 = A/B/C (no Customer master chain exists). (2) Why not upper layers? The reference edge (A/B) is already correctly stamped (Component A, bill_to→Customer); the gap is specifically the metric binding grammar (B), not compensation below it. (3) Why not lower layers? Not bypassing a working impl: Customer COs do not exist (Part 1 builds them) and the cross-entity binding genuinely needs new grammar + runtime (B+D), never a read-side hack (F). No fact-shape-tied formula (the MC declares a reference-resolved semantic input, not a column).

VERIFICATION (when built): Part 1 — Customer master chain create→activate, chain_status integrity, Customer COs produced with credit_limit (stop at chain integrity; tenant snapshots = SDG phase). Part 2 — unit (cross-entity resolution + per-group ratio) + PE-MC reference-DAG fail-closed test + E2E author credit-limit-utilization through the off-pool pipeline → materialize → activate, with Lineage carrying both resolved sets.

SCOPE / NON-GOALS. Does NOT change Components A or B, the identity hash (D470), or the governed-selection Foundation definition. Does NOT introduce a new Foundation artifact (it applies the existing governed selection to a second candidate set). Implementation is a separate build effort after this ADR; this records the architecture + the two operator-locked decisions (credit-limit utilization as the first metric; full Customer master for Part 1).
