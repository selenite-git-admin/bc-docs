# Persisted L-node verification with semantic-family classification

## Context

User pivot from universal-fix-with-ripples to persist-and-enforce-at-session-close. Uses existing bc-ai agents + existing chain-status persistence pattern (D305 precedent). Adds one narrow semantic_family column for cheap deterministic filtering. Session-close gate makes sanity non-skippable for contract-touching sessions without blocking unrelated work. Joint ADR per SES-691761.

## Problem

Contract authoring has produced systematic semantic rot: funnel-padded cc_field_mapping (one BF mapped to 20+ unrelated CFs), date fields mapped to period CFs, missing semantic coherence between BF data_type and CF purpose. Every gate we have checks structural completeness (link exists, field referenced). None check meaning.

Symptoms:
- AR `cc_field_mapping` maps `receivable_hdr_document_date_time` (BLDAT, a date) to CF `fiscal_period` with rule `latest`.
- Same CC bulk-maps one currency BF to measures of different families (ratios, counts, balances) with mixed rules.
- Memory note `feedback_funnel_padding.md` explicitly warned against this anti-pattern; the warning lived as prose, never as a gate.

The causes are simple and locally addressable. Past sessions have tried universal fixes and created ripples. This ADR picks the opposite frame: **use what exists, wire it so sessions can't deviate**.

## What already exists

Structural (bc-core, already persisted, already refresh-on-write):
- `contract.chain_status` — per-MC verdict rollup
- `contract.chain_trace` — per (MC × variable × CC) L1-L8 flags
- Refreshed via `ChainStatusService.refresh()` on any input change

Semantic (bc-ai, exists but not persisted, not wired as blocking gate):
- `cc_field_audit` agent — per-CF semantic mapping verdict (map_to_bf / leave_unmapped / needs_new_bf / formula_definitional_error). Literally designed to prevent funnel-padding.
- `chain_auditor`, `source_verifier`, `table_verifier`, `field_mapper`, `metric_verifier`, `bf_dedup`, `bf_pii_classify`, `bo_composer`, `bo_dedup`, `eval_advisor`, `metric_tracer` — ~15 agents, each Maker+Checker+Gate cross-family
- Agents return verdicts on invocation; no persistence

Gap: agents aren't wired into authoring flows as blocking gates. Seed code inserts cc_field_mapping rows without running cc_field_audit. DB doesn't know, chain_status computation doesn't know, SOP doesn't enforce.

## Decision

Four parts. Smallest possible footprint. Uses what exists.

### 1. Persisted semantic verdicts (new tables, platform DB)

Parallel to chain_status/chain_trace (separate tables — no race between structural and semantic refresh):

```
contract.l_node_semantic_verdict
  metric_contract_id           uuid  NOT NULL
  metric_version_code          text  NOT NULL
  l1_semantic_verdict          text  -- 'green' | 'yellow' | 'red' | 'not_applicable'
  l2_semantic_verdict          text
  ...
  l8_semantic_verdict          text
  overall_semantic_verdict     text
  break_summary_json           jsonb
  input_hash                   text
  computed_at                  timestamptz NOT NULL
  PRIMARY KEY (metric_contract_id, metric_version_code)

contract.l_node_semantic_trace
  metric_contract_id, metric_version_code,
  canonical_contract_id, variable_field_code, node_level,
  agent_flow_id, verdict, verdict_detail_json,
  input_hash, computed_at
  PRIMARY KEY (metric_contract_id, metric_version_code, variable_field_code, node_level, agent_flow_id)
```

Refresh-on-write via a new `LNodeSemanticService.refresh(mcId, mcVersion)` hook called at the same write-path sites as `ChainStatusService.refresh()`:
- cc_field_mapping INSERT/UPDATE/DELETE
- business_field INSERT/UPDATE
- source_field INSERT/UPDATE
- admission_contract version activation
- reader binding changes
- MC version activation

### 2. New column `master.canonical_field.semantic_family` (nullable)

Refinement of existing `role` column. Closed enum:

| role (existing) | semantic_family values |
|---|---|
| dimension  | identifier, code |
| measure    | measure-currency, measure-count, measure-ratio, measure-percent, measure-score |
| identifier | identifier |
| descriptor | name, code, text |
| temporal   | date, period, datetime, duration |

Additional values for CFs that map to a conformed dim table (D367 companion):
- dim-calendar-date   → joins `master.date_dim.date`
- dim-fiscal-period   → joins `master.fiscal_period.period_label`
- dim-currency        → joins `master.master_currency.code`
- dim-country         → joins `master.master_country.alpha2`
- dim-legal-entity    → joins `master.legal_entity_dim` (new per D367)
- dim-gl-account      → joins `tenant.gl_account_dim` or `master.gl_account_template` (D367)
- dim-party           → joins tenant-scope party dim (D367)

Semantics:
- Nullable. Absence = "not classified yet". Does not block authoring.
- Required at new CF creation (cc-onboarding validator enforces).
- Existing 1,950 CFs classified gradually via admin workflow (one-by-one per prior decision — no heuristic bulk pass).
- Verifier consults semantic_family when available; absence gracefully falls through to AI agent verdict.

### 3. Deterministic compatibility filter (runs before AI agents)

For each L-node where a BF↔CF mapping is involved (L2, L4), the verifier first runs a deterministic check:

```
isCompatible(bf.data_type, cf.semantic_family, resolution_rule) → boolean + reason
```

AI agents only run when:
- semantic_family is null (classification pending), OR
- Deterministic check is inconclusive, OR
- Policy requires both (high-stakes node, e.g. L1 CF registration)

Rationale: deterministic catches the obvious-impossible combos at O(1), explainable, no LLM cost.

**Initial rule matrix — ~15 rules grouped by family.** Ships with D366 implementation. Structured by family so new rules slot cleanly. Matrix growth stays in explicit PR (not organic drift). Each rule is unit-testable.

**Temporal family** (4 rules)
- CF family=`period`, BF type=`date` → red "period-family CF requires integer or period-formatted BF"
- CF family=`date`, BF type=`period` → red "date-family CF requires a date value, not a period label"
- CF family=`datetime`, BF type=`date` → yellow "datetime CF with date-only BF loses time component"
- CF family=`duration`, BF type=`date|datetime|period` → red "duration CF requires a numeric interval, not a point-in-time"

**Measure family + rule** (5 rules)
- CF family=`measure-currency`, resolution_rule=`latest` → red "currency measures aggregate via sum/avg, not latest"
- CF family=`measure-count`, BF type=`text` → red "count-family CF requires numeric BF"
- CF family=`measure-ratio`, resolution_rule=`sum` → red "ratio CFs cannot be summed; recompute from numerator/denominator"
- CF family=`measure-percent`, resolution_rule=`sum` → red "percent CFs cannot be summed; recompute from base"
- CF family=`measure-score`, BF type=`text` → red "score-family CF requires numeric BF"

**Identifier family + aggregation** (3 rules)
- CF family=`identifier`, resolution_rule=`sum|avg|min|max|latest` → red "identifier CFs require assert_equal or assert_unique"
- CF family=`identifier`, BF type=`date|datetime|period` → yellow "identifier backed by temporal BF is unusual — verify intent"
- CF family=`code`, resolution_rule=`sum|avg` → red "code CFs are not aggregable"

**Dim-family** (3 rules, active post-D367 classification)
- CF family=`dim-legal-entity`, BF type=numeric → red "dim-legal-entity expects a joinable code, not a number"
- CF family=`dim-calendar-date`, BF type=`text` → yellow "dim-calendar-date with text BF requires date_parse transform"
- CF family=`dim-*`, resolution_rule=`sum|avg` → red "dim reference CFs use assert_equal, not aggregation"

### 4. MCP tools + session-close gate (no scanning, reads from persisted tables)

New MCP module on devhub (or bc-core if devhub can't reach bc-ai directly):

```
devhub_l_node_verify(scope, ref)
  SELECT from l_node_semantic_verdict + chain_status
  Returns: { structural, semantic, overall, computed_at, stale }
  ~ms response time, no agent calls on read path

devhub_l_node_refresh(scope, ref)
  Triggers re-audit (agent calls happen here). Used by:
    - Authoring services (automatic on write)
    - Tests / admin / batch backfill

devhub_l_node_audit(scope='all')
  Bulk read from persisted tables. Summarizes reds/yellows by L-level.
  No recomputation.
```

Session lifecycle integration:
- `devhub_session_open`: boot output includes L-node rollup for the project scope ("bc-core: X green / Y yellow / Z red")
- `devhub_session_save_plan`: if plan mentions contract authoring keywords, require structured field `plan_json.l_node_impact`
- `devhub_session_close`: if session touched contracts (see **"Touches contract" detection** below), run `devhub_l_node_diff(session_uid)`. Close blocks on regression; operator override path defined below.

**"Touches contract" detection — union of three signals.** Any one fires the gate.
1. **Plan field** — Claude declares `plan_json.l_node_impact` in `devhub_session_save_plan`. Self-reported intent; cheap; may be missed.
2. **Commit diff** — files matching `bc-core/src/registry/**`, `bc-core/src/schemas/contract/**`, `bc-core/src/services/*contract*`, `bc-core/db/migrations/**contract**`, seed/DDL files touching `cc_field_mapping` / `business_field` / `canonical_field` / `observation_field_map` / `admission_contract` / `metric_contract`. Objective file-level ground truth.
3. **DB writes during session** — audit hook records writes to `contract.*`, `master.canonical_field`, `master.business_field`, `cc_field_mapping` during the session, keyed to session UID. Catches runtime writes regardless of where the code lives (seeds, scripts, ad-hoc inserts).

Signals (1) and (2) ship day-1 with D366 implementation. Signal (3) ships when audit hook infrastructure exists; tracked as followup task, does not block D366.

**Session-close override shape — text rationale + auto-spawned followup task. No time-box.**
- Override requires text rationale in `self_audit_json.l_node_override` (required field, free-text, min 40 chars). Goes into the change record verbatim.
- `devhub_session_close` automatically spawns `devhub_task_add` with tag `l-node-regression`, linked to the session UID and the regressing MC list. Task sits in the project agenda as a forcing function until resolved.
- No time-box / auto-expiry. Rationale: the persisted regression in `l_node_semantic_verdict` stays visible as long as the red is red; the auto-task is the ownership mechanism; adding an expiry renewal loop is complexity disproportionate to the risk. If the task queue bloats with stale regressions, that is a management signal, not something to automate into a hard block.

CLAUDE.md gets one new bullet under mandatory session protocol.

## Consequences

Positive:
- Semantic rot becomes visible at the point of authoring, not months later
- Session close produces an objective artifact, not prose self-audit
- Funnel-padding caught in O(1) at the deterministic filter for classified CFs
- No boil-the-ocean migration; existing contracts stay in place, cleaned up one-at-a-time as sessions naturally touch them
- Existing 15 bc-ai agents get the missing wiring layer

Negative / trade-offs:
- Two new tables + refresh hooks = more write-path cost on contract changes
- Classification workflow creates operator task load (~1,950 CFs over time)
- Session close adds ~1 DB query for contract-touching sessions; acceptable
- Override path exists; must be audited to prevent silent bypass (flag reds that overrode)

## Alternatives considered

- **Keep scanning docs / re-running agents on every check** — rejected per D305 precedent ("replaces stateless traversal with write-through computation")
- **Merge semantic verdicts into chain_status** — rejected: races between structural and semantic refresh; separate tables align with D305's existing pattern
- **Classify 1,950 CFs up-front via heuristic** — rejected per user decision ("otherwise we are just pushing the unavoidable issues to later")
- **New platform-wide taxonomy ADR (D367 as originally scoped)** — rejected as over-engineering; semantic_family is kept narrow as one column, gate behavior is additive to existing agents
- **Run agents on read** — rejected: slow, expensive, contradicts persistence principle

## Scope boundaries

IN this ADR:
- Two tables (l_node_semantic_verdict, l_node_semantic_trace)
- Refresh hooks on existing write-paths
- semantic_family column on canonical_field (nullable)
- Deterministic compatibility filter
- MCP tools + session-close gate
- CLAUDE.md protocol update

OUT of this ADR (separate concerns):
- Conformed dim table inventory / creation → ADR-D367 sibling
- bc-admin UI for violation triage → deferred
- Migration of existing violations → none; emerges naturally from session work
- AI agent business logic changes → none; agents as-is
- Authoring SOP rewrites → one small playbook for CF classification only
