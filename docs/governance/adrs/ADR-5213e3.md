# Open-source connector strategy — npm libs as transport, ReaderExecutor as containment boundary

## Context

Connectors are commodity — they don't differentiate BareCount. The platform (boundary model, contracts, evidence) is the IP. Building connector transport from scratch means rediscovering edge cases that npm libraries and the Airbyte community have already solved in production. Risk of open-source dependency is equal or lower than custom code because: (1) battle-tested by thousands of users vs untested custom code, (2) ReaderExecutor interface contains the blast radius of any library change, (3) only 2 real dependencies needed for initial 4 connectors (SAP SDK + jsforce), rest are plain fetch.

## Decision

Connectors use open-source npm libraries as the transport layer (e.g., @sap-cloud-sdk/http-client for SAP, jsforce for Salesforce, plain fetch for simple REST APIs). Each library is wrapped in a thin BareCount ReaderExecutor implementation. The ReaderExecutor interface is the containment boundary — libraries are internal implementation details and can be swapped without affecting the rest of the system. Build executors inside bc-core first. When 5+ executors are stable, extract to @barecount/connectors (MIT license, separate npm package). Platform code (ReaderRuntimeService, CredentialResolverService, admission contracts, evidence chain) stays proprietary. For connectors #5 through #50, explore Airbyte low-code CDK YAML manifests as auto-generation source (TSK-e4243e, parked).
