# Source System Simulators -- Per-System-Type Synthetic Data Architecture

## Context

bc-sdg was built early to generate synthetic SAP data for Reader development. It bundles three SAP flavors (ECC, S/4 Cloud, S/4 On-Prem) into a single Node.js application exposing OData endpoints.

During test bench design (D303), drift analysis revealed critical mismatches between bc-sdg and the contract chain:

1. **Missing entities**: The DSO metric chain references BSID (SAP ECC AR open items table). bc-sdg does not generate BSID at all. It was built around S/4HANA CDS Views (I_JournalEntry, I_BillingDocument), not ECC legacy tables.

2. **Incomplete field coverage**: bc-sdg generates ~40 fields per entity. The source catalog registers 600+ fields for BSEG alone. AR-specific fields (clearing dates, dunning levels, special GL indicators) are entirely missing.

3. **Protocol mismatch**: ECC speaks OData V2 with SAP-specific conventions (ATOM format, field names like BUKRS/BELNR). S/4 Cloud speaks OData V4 with CDS naming (CompanyCode, AccountingDocument). bc-sdg blurs this distinction.

4. **Structural drift**: bc-sdg's entity schemas are hand-maintained. The source catalog in bc-core evolves independently. There is no mechanism to detect or prevent drift.

5. **No Salesforce simulation**: bc-sdg has optional SFDC support but it is not wired to the contract chain's Salesforce source catalog entries.

These issues make bc-sdg unusable for the D303 test bench without significant rework.

## Decision

### Architecture: Per-System-Type Simulators

Decompose bc-sdg into **dedicated simulators**, one per source system type. Each simulator:

- Speaks the **exact protocol** the real system uses
- Exposes the **exact entities and fields** registered in bc-core's source catalog
- Generates **contextually realistic data** with configurable volume
- Is **indistinguishable from the real system** to the Reader
- Has its **schema generated from the source catalog** (not hand-maintained)

### Simulator Inventory

**Phase 1 (build now):**

| Simulator | System Type | Protocol | Port | Entities |
|-----------|-------------|----------|------|----------|
| sdg-sap-ecc | SAP ECC EHP8 | OData V2 | 6100 | BSID, BSIK, BKPF, BSEG, VBRK, VBRP, EKKO, EKPO, MARA, LFA1, KNA1, T001 |
| sdg-salesforce | Salesforce Sales Cloud | REST API | 6110 | Account, Opportunity, Contact, Lead, Quote, Invoice, Contract |

**Future phases:**

| Simulator | System Type | Protocol | Port | When |
|-----------|-------------|----------|------|------|
| sdg-sap-s4-cloud | S/4HANA Public Cloud | OData V4 | 6120 | When S/4 CDS coverage is sufficient |
| sdg-sap-s4-onprem | S/4HANA On-Prem | OData V4 | 6130 | After cloud variant |
| sdg-oracle-ebs | Oracle EBS | REST | 6140 | When Oracle metrics come online |

### Port Reservation (extends D046)

Range **6100-6199** reserved for source system simulators. 10-port gap per system type for sub-services.

| Port | Assignment |
|------|-----------|
| 6100 | sdg-sap-ecc |
| 6110 | sdg-salesforce |
| 6120 | sdg-sap-s4-cloud (future) |
| 6130 | sdg-sap-s4-onprem (future) |
| 6140 | sdg-oracle-ebs (future) |
| 4200 | bc-sdg coordinator (deprecated, kept for backward compat) |

### Repository Structure

bc-sdg becomes the coordination repo:

```
bc-sdg/
  core/                          # Shared library
    odata-v2-server.ts           # Reusable OData V2 protocol handler
    odata-v4-server.ts           # Reusable OData V4 protocol handler
    rest-server.ts               # Reusable REST API handler
    data-factory.ts              # Generates realistic values by field type + semantic
    profile-engine.ts            # Volume scaling, fiscal patterns, edge cases
  simulators/
    sap-ecc/                     # SAP ECC EHP8
      entities/                  # BSID.ts, BSIK.ts, BKPF.ts, BSEG.ts, ...
      behaviors/                 # Clearing logic, dunning state machine, posting keys
      server.ts                  # OData V2 server on :6100
      README.md
    salesforce/                  # Salesforce Sales Cloud
      entities/                  # Account.ts, Opportunity.ts, ...
      behaviors/                 # Pipeline stages, forecast categories
      server.ts                  # REST API server on :6110
      README.md
  schema-gen/                    # Schema generator (reads bc-core source catalog)
    generate.ts                  # npx tsx schema-gen/generate.ts sap-ecc
    templates/                   # Entity skeleton templates per protocol
  profiles/                      # Industry personas (mfg-co, pharma-co, etc.)
    mfg-co.yaml
    pharma-co.yaml
```

### Schema Generation from Source Catalog

The critical architectural invariant: **simulator entity schemas are derived, not authored.**

```
bc-core source catalog (DB)          Schema Generator           Simulator
┌──────────────────────┐            ┌──────────────┐          ┌──────────────┐
│ source_system:       │            │ Reads fields  │          │ BSID entity  │
│   sap_ecc_ehp8       │──────────▶│ Generates:    │─────────▶│ 600+ fields  │
│                      │            │  - EDMX       │          │ OData V2     │
│ source_table: BSID   │            │  - TypeScript │          │ :6100        │
│   field: BUKRS char4 │            │  - Factory    │          │              │
│   field: BELNR char10│            │  - Seed rules │          │ Data factory │
│   ... 600 fields     │            └──────────────┘          │ knows types  │
└──────────────────────┘                                      └──────────────┘
```

When a field is added to the source catalog, running `npx tsx schema-gen/generate.ts sap-ecc` regenerates the entity definition. Drift is structurally eliminated.

### Data Factory Semantics

The data factory generates realistic values based on field metadata:

| Field Attribute | Generation Strategy |
|----------------|---------------------|
| type: char4, name contains "BUKRS" | Pick from company code master (1000, 2000, 3000) |
| type: date, name contains "AUGDT" | Generate clearing date 30-90 days after posting date |
| type: decimal, name contains "DMBTR" | Generate amount in range based on document type |
| type: char1, name: "SHKZG" | Debit/credit indicator based on posting key |
| semantic_role: "business_key" | Generate unique within entity scope |
| semantic_role: "foreign_key" | Reference existing master data records |

Behaviors layer adds cross-field consistency:
- Clearing documents reference each other (AUGDT + AUGBL)
- Dunning levels increase over time for overdue items
- Payment terms drive net due date calculation
- Three-way match links PO → GR → Invoice

### Connection to Test Bench (D303)

Test bench scenario config references a simulator:

```json
{
  "scenarioName": "Finance AR DSO 1K",
  "simulator": "sap-ecc",
  "simulatorPort": 6100,
  "sourceSystem": "sap_ecc_ehp8",
  "sourceTables": ["BSID", "VBRK"],
  "volumeTarget": 1000
}
```

The Reader connects to `http://localhost:6100/sap/opu/odata/sap/` -- identical to connecting to a real ECC system. The full chain runs unchanged: Reader -> SO -> Admission -> Canonical -> Metric.

### Existing bc-sdg Disposition

- `bc-sdg` repo is preserved as the coordination repo (monorepo for all simulators)
- Current bc-sdg code (port 4200) remains running for backward compatibility
- Gradually migrate current S/4 Cloud entities into `simulators/sap-s4-cloud/`
- Once all simulators are live, deprecate the 4200 coordinator

## Options Considered

### Option A: Patch bc-sdg to add missing fields (rejected)

Adding BSID + 600 fields to the existing monolithic bc-sdg. Rejected because:
- Does not fix the structural drift problem (still hand-maintained schemas)
- ECC OData V2 protocol differs from S/4 V4 -- shoehorning both into one server creates protocol confusion
- No separation of concerns -- ECC behaviors (clearing, dunning) mixed with S/4 CDS logic

### Option B: Per-system-type simulators with schema generation (chosen)

Chosen because:
- Drift is structurally eliminated (schemas derived from source catalog)
- Each simulator is protocol-accurate (V2 for ECC, V4 for S/4, REST for SFDC)
- Reader cannot distinguish simulator from real system
- Independent lifecycle per simulator (add SFDC without touching ECC)
- Behaviors layer captures system-specific business logic (SAP clearing != Salesforce pipeline)

### Option C: Skip simulation, inject test data directly (rejected as primary)

Generating SO records directly in the tenant DB (bypassing Reader). Rejected as primary strategy because:
- Does not test the Reader layer
- Different code path than production
- Acceptable as a temporary fallback for early test bench iterations, not as architecture

## Consequences

### Positive

- Drift between synthetic data and source catalog is structurally impossible
- Reader integration is tested end-to-end (simulator is indistinguishable from real system)
- Each system type gets dedicated protocol accuracy (V2/V4/REST)
- Behaviors layer captures domain-specific realism (SAP clearing, SFDC pipeline)
- Test bench scenarios become truly representative of production

### Negative

- More repos/services to maintain (mitigated: shared core library)
- Initial build cost for ECC simulator is significant (BSID alone is 600+ fields)
- Schema generator needs access to bc-core source catalog (coupling)

### Risks

- **Source catalog completeness**: Schema generator is only as good as the field registry. If BSID has 100 fields registered instead of 600, the simulator will be incomplete. Mitigation: prioritize field registration for AR/AP tables before building the simulator.
- **Behavior realism**: Data factory produces syntactically correct but potentially semantically unrealistic combinations (e.g., clearing date before posting date). Mitigation: behaviors layer with explicit SAP business rules.
- **Maintenance overhead**: N simulators instead of 1. Mitigation: shared core library handles 80% of the work; entity-specific code is generated.
