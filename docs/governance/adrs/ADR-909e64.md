# Tenant DB: single schema per tenant, no sub-schemas

## Context

PostgreSQL schemas are flat (no nesting). Splitting into 6 schemas per tenant (t_acme_connection, t_acme_discovery, etc.) would: (1) multiply schema count 6× (600 schemas at 100 tenants), (2) require cross-schema FK grants within a single tenant, (3) complicate search_path from 1 entry to 6, (4) bloat pg_catalog, (5) make future migration to RLS harder. 24 tables is small — single schema is clean. At real scale the move is toward shared tables with RLS, not more schemas.

## Decision

Each tenant gets one PostgreSQL schema (e.g. t_acme) containing all 24 tables. The 6 logical groups (Boundary, Connection, Discovery, Execution, Evidence, Users) are expressed as comments in SQL and sections in documentation — not as separate PostgreSQL schemas. No table name prefixes.
