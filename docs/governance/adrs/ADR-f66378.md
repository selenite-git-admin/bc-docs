# BO-Scoped BF Composition — No Shared Observation Fields Across BOs

## Context

OAGIS 10.12 defines Core Component Types (CCTs) — shared field definitions like `TotalAmount`, `Identifier`, `DocumentDateTime` — that appear across multiple Nouns (Invoice, PurchaseOrder, SalesOrder, etc.). This sharing works for OAGIS because it is a **message schema**: each XML document is self-contained, and there is no cross-entity mapping layer.

BareCount's architecture is fundamentally different. The **contract chain** maps source system fields to Business Fields (BFs) via Canonical Contracts (CCs), and Metric Contracts (MCs) compute from BFs. This creates a cross-entity dependency:

```
Source Field (SAP NETWR) → CC mapping → total_amount (field_id X) → MC formula
Source Field (SAP WRBTR) → CC mapping → total_amount (field_id X) → MC formula
                                                                      ↑ COLLISION
```

When `total_amount` has one `field_id` shared across 18 BOs, CCs from different source systems all map to the same field_id, and MCs cannot distinguish which BO's amount they are computing.

### Evidence

Discovered during BF/BO seeding (Apr 6-7, 2026):

- `actual_amount`: exists in 1 OAGIS Noun (maintenance-order), was force-injected into **53 BOs** as a "measure filler" to pass CR-QG-002
- `effective_date`: exists in 1 OAGIS Noun (remittance-advice), was injected into **43+ BOs**
- `total_amount`: shared across 18 OAGIS Nouns — in OAGIS this is fine, but in BareCount each BO needs its own variant
- `identifier`: shared across 131 OAGIS Nouns — same problem

## Decision

### The Rule

**Every Business Field composed into a Business Object must be semantically scoped to that BO.** The BF name must unambiguously identify which BO it belongs to.

### Naming Convention

For BFs that appear in multiple OAGIS Nouns, prefix with the BO name:

| OAGIS CCT (shared) | BareCount BO-scoped BF |
|---|---|
| `total_amount` | `po_total_amount`, `invoice_total_amount`, `so_total_amount` |
| `extended_amount` | `po_line_extended_amount`, `invoice_line_extended_amount` |
| `identifier` | `po_identifier`, `invoice_identifier`, `maintenance_order_identifier` |
| `document_date_time` | `po_document_date`, `invoice_document_date` |
| `amount` | `payable_amount`, `receivable_amount` |

### Exceptions — Shared Reference Dimensions

The following BFs may be shared across BOs because they are **reference lookups**, not observation values. Their semantic meaning is identical everywhere and they will never be the target of a CC source-field mapping or MC computation:

- `currency_code` — ISO 4217 currency
- `language_code` — ISO 639 language
- `country_code` — ISO 3166 country
- `unit_of_measure` — UN/CEFACT UoM

**Test:** A BF may be shared only if it would never appear as a column in a typed table (D210) or as a variable in a metric formula. If it could be measured, summed, filtered, or mapped from a source field — it must be BO-scoped.

### CR-QG-002 Update

Add Gate 7: **No shared observation BFs**. For each BF in the BO composition, check: if the BF appears in more than one approved BO AND is not in the shared-exceptions list, gate fails.

### CR-QG-001 Update

BF names for BO-scoped fields should follow the pattern `{bo_prefix}_{oagis_field}`:
- Object class = BO name (e.g., `purchase_order`)
- Property = OAGIS field name (e.g., `total_amount`)
- Result: `po_total_amount` with object_class=`purchase_order`, property=`total_amount`

## Consequences

1. **~800 new BO-scoped BFs** need to be created (104 BOs x avg 8 shared fields)
2. **All 104 BOs** need composition rebuilt with BO-scoped BFs
3. **Finance domain** (21 BOs) is also affected — was manually curated but uses shared CCTs
4. Old shared BFs remain in the registry (they document OAGIS CCT vocabulary) but are removed from BO compositions
5. **Onboarding service** (TSK-f02759) must implement this rule: for each OAGIS Noun field, check if it is a reference dimension — share; otherwise — prefix with BO name
6. **CC/MC creation** is blocked until all BOs have BO-scoped compositions

## Alternatives Considered

1. **Keep OAGIS sharing** — rejected because BareCount's CC/MC chain creates cross-entity dependencies that OAGIS's message model does not have
2. **Scope at CC level instead of BF level** — rejected because it pushes the disambiguation problem downstream; BFs should be unambiguous at the vocabulary level
3. **Use BO+BF compound key in CC mappings** — rejected because it violates the principle that a BF field_id should have one semantic meaning
