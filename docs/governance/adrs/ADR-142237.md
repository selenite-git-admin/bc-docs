# D352 — bc-portal Design System Governance

## Context

Design system archaeology (audit SES-92dc4e) found **four coexisting DS traces** in bc-portal:

| Trace | Location | Status |
|-------|----------|--------|
| shadcn/ui primitives | `src/components/ui/` (47 files, unmodified) | Strong canonical candidate |
| CSS variable tokens | `src/styles/globals.css` (58 variables + `@theme inline`) | Canonical token source |
| Hardcoded hex colors | 135 occurrences in 66 files — Tailwind arbitrary values (`from-[#A25BFF]`, etc.) | Escape hatch — bypasses tokens |
| ComponentRegistry | `src/components/design-system/ComponentRegistry.ts` (68 entries vs ~140 actual components) | Informational, not enforced |

Additional context:
- The `/design-system` route hosts `DesignSystemBlueprint.tsx` — 15 modular sections demoing components, typography, spacing, icons, charts, forms
- **Previous state:** 40+ separate DS library pages existed, consolidated into the single Blueprint page
- No dark mode implementation yet; hardcoded hex values would break any future dark mode

Without governance, each new component would drift further from the token system.

## Decision

### 1. Canonical DS sources (three pillars)

- **Primitives:** `src/components/ui/` (shadcn/ui, unmodified). Do not fork, do not patch inline. Upgrade by replacing wholesale.
- **Token source:** `src/styles/globals.css` — 58 CSS variables in `:root` and `.dark`, wired through `@theme inline` block. Add new tokens here. No other location is authoritative.
- **Living reference:** `src/pages/design-system/DesignSystemBlueprint.tsx` (route `/design-system`). All tokens, components, and patterns are demoed here. A developer asking "what colors do we have?" visits this page.

### 2. Registry enforcement

`src/components/design-system/ComponentRegistry.ts` is the catalog of all reusable components **above the `ui/` tier**. Fields per entry: `id`, `name`, `path`, `category`, `version`, `isDeprecated`, `isProtected` (IP-sensitive), `useCases[]`.

**New components in `common/` or any feature directory (above `ui/`) must be registered.** Enforced via the DS Contribution SOP (D352-SOP) and review gate.

Components in `ui/` are shadcn primitives and are implicitly registered by their presence.

### 3. Rule 1 — No arbitrary hex in Tailwind classes

ESLint rule `no-restricted-syntax` (deployed in `eslint.config.js`, TSK-42e6ee) bans:

```tsx
// ❌ Banned
<Button className="bg-[#7263FF]">         // arbitrary Tailwind value
const variants = { default: "bg-[#7263FF]" }  // cva variant

// ✅ Allowed
<Button className="bg-primary">            // uses CSS variable
```

- **Severity:** `warn` (135 grandfathered violations)
- **Exemption:** `src/pages/design-system/**` and `src/components/design-system/**` (they demo the violations intentionally)
- **Escape hatch:** `// eslint-disable-next-line no-restricted-syntax` with a justification comment. Review gate checks justifications.

### 4. Rule 2 — No inline `style={{}}` for static values

Dynamic values (computed sizes, positions) may use inline style. Static colors, spacing, typography must use Tailwind classes referencing tokens.

Current state: 0 inline-style violations for static values.

### 5. Rule 3 — Dark mode readiness

`globals.css` has `.dark` selector with OKLCH values for all color tokens. Any component using CSS variable tokens automatically responds to dark mode.

Migrating the 135 hardcoded hex values to tokens is the path to full dark mode support.

### 6. Deprecation and migration path

| What | Policy |
|------|--------|
| Inline styles (static) | 0 current — remain banned |
| Hardcoded hex in className | 135 grandfathered, ESLint warns; migrate opportunistically during rename pass (Phase B) or feature work |
| Duplicate components (e.g., 3× KPICard) | Consolidate during Phase B rename pass; delete the deprecated copy |
| Unregistered components | Registry catch-up in Phase C (TSK-4b7cb8) |

## Options Considered

### Option A: Token-first + ESLint + Registry enforcement (chosen)

All three pillars. Migration is opportunistic (no big-bang rewrite).

### Option B: Accept hardcoded hex as a "semantic layer" (rejected)

Argued that business logic (red = bad, green = good) warrants inline colors. Rejected because: (a) CSS variables can express semantic intent (`--status-negative`), (b) dark mode becomes impossible, (c) any single-color change becomes a 135-file edit.

### Option C: TypeScript color constants in `src/theme/colors.ts` (rejected)

A typed constants module would centralize colors but duplicates the CSS variable system. Dark mode still fails (constants are static). Pattern exists in Option A with `var(--primary)` references in JS-evaluated styles when genuinely needed.

### Option D: Status quo + documentation (rejected)

Documentation without a lint rule or review gate doesn't prevent drift. Status quo is why this ADR exists.

## Consequences

### Positive

- **Single canonical source** per pillar (primitives / tokens / reference).
- **Dark mode is achievable** once hex migration completes.
- **New components follow the pattern** automatically (ESLint enforces).
- **Onboarding:** "go to `/design-system`" answers "what's available."

### Negative

- **135 grandfathered warnings** until migrated — IDE/lint noise during development.
- **Registry maintenance** is manual — drift risk if SOP not followed.
- **DesignSystemBlueprint is manually curated** — new components don't auto-appear.

### Risks

- **Liberal `eslint-disable` use.** Mitigation: review gate requires justification; bc-qa audit could scan for undocumented disables.
- **`ColorsSection.tsx` itself uses hex swatches in its gradient demo** — that file is in the exempted `src/pages/design-system/**` path, as expected.
- **Token system can't express every gradient nuance.** Mitigation: add new gradient tokens to globals.css when a third component needs one.
