# D232: Split RegistryModule into 6 Domain Modules

## Context

`apps/api/src/registry/registry.module.ts` is a single NestJS module containing 49 providers (24 repositories + 22 services + 3 utilities), 34 controllers, and 30 exports covering 13+ distinct business domains. It was identified as the sole critical finding (F001) in the bc-core code review (SES-e7f6c7, 2026-03-31).

The original decision D020 (DEC-a5df75) chose to keep the monolith because "the code is already cleanly modular at the TypeScript level, making future extraction straightforward." The trigger conditions listed in D020 have not been met (no separate teams, no divergent SLAs), but a new condition has emerged: the bc-docs v2 documentation framework (D229) organizes platform documentation into 10 domain modules (P01-P10) that map naturally to the registry's internal domains. The code structure should match the documentation structure.

### Problems with current monolith

1. **DI bloat** — BoundaryModule imports RegistryModule and receives all 49 providers when it uses 11.
2. **No isolation** — A bug in MetricWizardService affects the entire RegistryModule, including unrelated ContractController.
3. **Test complexity** — Unit testing a ContractService requires standing up the full 49-provider module.
4. **Cognitive load** — 91 files in one flat directory with no sub-grouping.
5. **ISO 27001 A.14.2.2** — Modular architecture enables independent testability per domain.

## Decision

Split RegistryModule into 6 domain modules + 1 shared utility module. Module-only refactoring — no file moves, no DB changes, no API surface changes.

### Target Modules

| Module | Providers | Controllers | D229 Mapping |
|--------|-----------|-------------|--------------|
| SharedRegistryModule | IdService, CoreMapService, EnrichmentProcessor | CoreMapController | Cross-cutting |
| ContractModule | 3 (repo, service, validation) | 1 | P02: Contract Registry |
| ConnectorModule | 5 (connector + connection repo/svc, credential resolver) | 2 | P05: Runtime Definitions |
| SourceCatalogModule | 6 (catalog + reference + discovery repo/svc) | 12 | P01: Source Catalog |
| ReaderModule | 3 (repo, service, wizard) | 4 | P05: Runtime Definitions |
| MetricModule | 9 (binding + reference + definition + kpiSpec repo/svc, wizard) | 5 | P04: Metric Catalog |
| BusinessObjectModule | 14 (BO + verification + enrichment + mapping + standard field + canonical + catalog verification) | 5 | P03: Business Definitions |

### Remaining in slim RegistryModule (rename to ExecutionModule later)

PackageRepo/Svc/Controller, ExecutionRepo/Controller, ExecutionWritebackService, BoundaryRollupRepo, FieldMappingService/Controller, IntegrityService/Controller.

## Cross-Domain Dependency Graph

```
SharedRegistryModule (IdService, CoreMapService)
    ↑ imported by all domain modules

ContractModule ←─── (leaf, no domain imports)
ConnectorModule ←── (leaf, no domain imports)
SourceCatalogModule ← (leaf, no domain imports)

ReaderModule ───→ ConnectorModule (CredentialResolverService)
MetricModule ───→ ContractModule (ContractRepository, read-only)
BusinessObjectModule ─→ ContractModule + MetricModule + SourceCatalogModule
```

No circular dependencies. All cross-domain dependencies are read-only metadata queries.

## Options Considered

### A. Keep monolith (status quo — D020)
- Pro: Zero effort, zero risk.
- Con: Violates NestJS modularity, prevents domain isolation, misaligns with D229 documentation structure, increases test complexity.

### B. Split into 6 domain modules (chosen)
- Pro: Aligns with D229, enables domain isolation, reduces DI bloat, improves testability.
- Con: ~4-5 sessions of refactoring work, risk of missing exports during extraction.
- Mitigation: Phase 0 dependency graph, commit per extraction, full test suite verification.

### C. Split into 2 modules (registry + boundary-support)
- Pro: Simpler, lower risk.
- Con: Doesn't solve the core problem — registry would still be 40+ providers.

### D. Extract to separate microservice
- Pro: Maximum isolation.
- Con: Premature — D020's trigger conditions (separate teams, divergent SLAs) not met. Adds operational complexity.

## Consequences

### Positive
- Each consumer imports only the domain modules it needs
- Independent unit testing per domain module
- Code structure matches D229 documentation modules (P01-P05)
- Future microservice extraction trivial (each module already self-contained)
- ISO 27001 A.14.2.2 compliance improved

### Negative
- ~4-5 sessions of refactoring work
- Temporary increase in module import statements in app.module.ts
- Existing wrapper modules (BusinessCatalogModule, MetricCatalogModule) must be replaced

### Neutral
- No API changes — consumers (bc-admin, bc-portal) unaffected
- No database changes — shared REGISTRY_DB token continues
- bc-docs v2 platform stream structure unaffected (D229 already models correct boundaries)

## Risks & Mitigation

| Risk | Severity | Likelihood | Mitigation |
|------|----------|------------|------------|
| Circular dependency between new modules | High | Low | Dependency graph proven acyclic. SharedRegistryModule for cross-cutting. `forwardRef()` as last resort. |
| Missing export breaks consumer at runtime | High | Medium | Grep @Inject() for every provider before moving. Full test suite after each step. BoundaryModule endpoints verified. |
| BoundaryModule regression | High | Medium | BoundaryModule is heaviest consumer (11 exports). Dedicated endpoint verification after each extraction. |
| NestJS module resolution order | Medium | Low | Test startup after each extraction. Watch for "Nest can't resolve dependencies" errors. |
| BusinessCatalog/MetricCatalog wrapper breakage | Medium | Medium | Replace wrappers in Phase 3 AFTER dependencies stable. |
| File move import path breakage | Low | N/A | Constraint: no file moves in Phase 1-3. File reorganization is separate follow-up. |
