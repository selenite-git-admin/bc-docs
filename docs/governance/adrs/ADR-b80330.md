# Documentation Implementation — Greenfield with Split Dossiers, Diagram Layer, Foundation Stream

## Context

D229 defined the 3-stream, module-component documentation architecture (Platform P01-P10, Tenant T01-T06, Common C01-C02). This decision defines HOW to implement it.

Two approaches were considered:
1. **Greenfield (chosen):** Create the framework, templates, and tracking schema first. Then author fresh dossiers per component by reading existing docs, ADRs, and code. Finally retire old docs with clear status markers.
2. **Brownfield (rejected):** Correct 1,440 existing docs in-place to match the framework. Impractical — vocabulary drift, stale content, inconsistent structure make per-file correction more expensive than fresh authoring.

Three refinements emerged during design:
- **Split dossiers:** A single 9-section mega-file per component is too large and hard to maintain. Split into index + 9 child files, each independently updatable with its own frontmatter.
- **Diagram layer:** Diagrams are a distinct content type that crosses component boundaries. They need a parallel directory structure, not a 10th dossier section.
- **Foundation stream:** Foundation (execution model, naming, forbidden vocabulary) and Patent are not modules — they're static reference pillars that every module refers to. They sit outside the P/T/C streams as locked references (F01-F02).

## Decision

### 1. bc-docs Directory Structure

```
bc-docs/
  README.md                  ← human entry point: structure, reading order, navigation
  docs.registry.json         ← machine-readable manifest of ALL docs (for AI/sessions)
  system/
    foundation/              ← F01-F02, locked reference pillars
    platform/                ← P01-P10 modules + component dossiers
    tenant/                  ← T01-T06 modules + component dossiers
    common/                  ← C01-C02 modules + component dossiers
    ai/                      ← A01-A05 modules + component dossiers
  decisions/                 ← ADRs (decision records, not system docs)
  help/
    admin/                   ← bc-admin help articles (platform operators)
    portal/                  ← bc-portal help articles (tenant users/CXOs)
    shared/                  ← concepts that apply to both apps
  dev-guides/                ← how to develop/extend the system
  playbooks/                 ← how to perform operational tasks
  under-review/              ← ALL existing docs land here on migration
  archived/                  ← explicitly retired docs with pointers
```

**`README.md`** — explains the 5-stream structure, how to find docs, reading order for new sessions. Replaces the CLAUDE.md "Documentation Reading Order" section (which will point here instead).

**`help/`** — user-facing product documentation. bc-ai's KPI Assistant uses these as a RAG source (DEC-e82f0a). Each help article gets frontmatter:

```yaml

app: admin | portal | shared
screen: SCR-xxxxxx
role: platform-admin | tenant-user
difficulty: beginner | intermediate | advanced
tags: [contracts, source, wizard]
status: draft | published
---
```

Help articles are editorially maintained — bc-ai indexes them for RAG retrieval but does not generate them.

### 1b. Universal Document Identity

Every document in bc-docs gets two identifiers in its frontmatter:

| Field | Purpose | Format | Example |
|-------|---------|--------|---------|
| `id` | Machine ID — stable, globally unique, never changes | `{PREFIX}-{6 hex}` | `DOS-e4b221` |
| `slug` | Human ID — readable, URL-friendly, derived from title/component | kebab-case | `source-table-data-structure` |

**ID prefixes by document type:**

| Prefix | Document type | Used in |
|--------|--------------|---------|
| `DOS` | Dossier (index + section files) | `system/` |
| `DEC` | Decision (ADR) | `decisions/` |
| `HLP` | Help article | `help/` |
| `DGM` | Diagram | `*/diagrams/` |
| `GDE` | Dev guide | `dev-guides/` |
| `PLB` | Playbook | `playbooks/` |
| `FND` | Foundation reference | `system/foundation/` |

**Rules:**
1. `id` is assigned once at creation, never changes even if title/content changes
2. `slug` may change if title changes — but old slugs should redirect (help articles especially)
3. `docs.registry.json` indexes both `id` and `slug` for lookup
4. Cross-references between documents always use `id`, never `slug` (slugs can change)
5. URLs for help articles use `slug` (human-friendly): `/help/admin/creating-a-source-contract`
6. DevHub document registry syncs on `id` as the primary key

**Dev guide frontmatter:**

```yaml
---
id: GDE-c3a892
slug: adding-a-contract-family
title: "How to Add a New Contract Family"
tags: [contracts, development, bc-core]
status: draft | published
---
```

**Playbook frontmatter:**

```yaml
---
id: PLB-d1e4f7
slug: kpi-enrichment-playbook
title: "KPI Enrichment Playbook"
tags: [metrics, enrichment, bc-ai]
status: draft | published
---
```

**Migration rule:** All 1,440 existing docs (essentials/, domains/, foundation/, architecture/, component-references/, etc.) move to `under-review/` preserving their subdirectory structure. During Phase 3 (populate), each doc either gets absorbed into a `system/` dossier and then moves to `archived/`, or is promoted back if still valid standalone. Nothing is deleted.

**ADRs** stay in `decisions/` (currently `architecture/decisions/`). They are decision records, not system documentation. Dossier Section 02 (architecture) references ADRs by UID — it does not absorb them.

**D226 superseded:** The essentials/domains/tier model was a stepping stone. This structure replaces it. The CLAUDE.md "session reading order" will point directly into `system/foundation/` instead of `essentials/`.

### 2. Stream Architecture (revised from D229)

| Stream | Modules | Nature |
|--------|---------|--------|
| **Foundation (F01-F02)** | Foundation, Patent | Static reference, authority: locked |
| **Platform (P01-P10)** | Source Catalog → Master Data | Active, platform-scoped |
| **Tenant (T01-T06)** | Boundary Data → Tenant Views | Active, tenant-scoped |
| **Common (C01-C02)** | Auth, Infrastructure | Active, cross-stream |
| **AI (A01-A05)** | Agent Platform, Construction Agents, KPI Assistant, Enrichment, Guardrails | Active, cross-app |

**F01: Foundation (LOCKED)**
- Execution Model specification
- Forbidden Vocabulary
- Naming Conventions (ISO 11179)
- Evaluation Boundaries
- Object Lifecycle
- Binding Chain

**F02: Patent (LOCKED)**
- Claims mapping
- Prior art references
- Patent-protected patterns

Every dossier's Section 01 (Specification) references F01/F02 — it does not duplicate their content.

**A01: Agent Platform** (5 components) — Model Registry, Evidence Store, Budget Controls, Cache & Pin, Prompt Management
- Core infrastructure that all AI capabilities depend on
- Model health, fallbacks, cost tracking
- Immutable evidence trail with QA gate
- Versioned prompt templates with flow configs and locks
- Refs: D222 (DEC-e9294b)

**A02: Construction Agents** (7 components) — BO Composer, Field Mapper, Metric Tracer, Evaluation Advisor, Chain Auditor, Checker Agent, QA Gate
- Domain-intelligent agents that generate inputs for bc-admin wizard constructors
- Maker-checker-gate pipeline: every suggestion through two independent models + quality gate
- Serves bc-admin wizard "AI Suggest" buttons
- Refs: D222 (DEC-e9294b), DEC-d214ed (superseded — original BO enrichment pattern)

**A03: KPI Assistant** (6 components) — Intent Resolver, Catalog Retriever, Snapshot Retriever, Grounded Composer, Output Validator, Access Control
- Conversational metric intelligence for CXOs and business users
- 5-step pipeline: intent → catalog → snapshots → compose → validate
- 4-fence access control: tenant contract, function, subfunction, access tier
- Serves bc-portal (customer) and bc-admin (platform)
- Refs: D223 (DEC-c566f3), DEC-e82f0a (bc-portal AI drawer)

**A04: Enrichment Pipeline** (4 components) — Formula Enrichment, Knowledge Enrichment, Semantic Dedup, Maturity Promotion
- Batch AI processing for metric catalog population
- Dual-model formula decomposition (Nova Pro maker + Nova Lite checker)
- Knowledge enrichment (Sonnet — context, stakeholders, thresholds, search tags)
- Semantic dedup (formula similarity scan + AI triage)
- Refs: PLN-86b3fc (Metric Catalog enrichment plan)

**A05: Guardrails & Trust** (4 components) — Bedrock Guardrails, Tenant Fence, Help Article RAG, Exposure Protection
- Trust and safety layer across all AI capabilities
- Bedrock Guardrails: topic blocks, PII filtering, injection defense
- Tenant fence: data isolation enforced before data reaches any model
- Help article RAG: indexing bc-docs help articles for bc-portal assistant
- Exposure protection: sanitization gateway — what customers never see (model names, costs, QA reasoning)
- Refs: D223 Addendum B (4-fence model), DEC-e82f0a (dual retrieval)

**Why AI is a separate stream (not inside Platform or Common):**
1. Different language (Python) and runtime (FastAPI) — separate repo `bc-ai`
2. Serves ALL consumers: bc-admin, bc-portal, Claude Code sessions, DevHub
3. Has its own evidence store, model registry, budget controls — a platform unto itself
4. Different deployment lifecycle — AI models change weekly, independent of bc-core releases

### 2. Split Dossier Structure

Each component produces one directory with an index file and 9 numbered child files:

```
{module}/
  {component}/
    index.md              ← overview, summary, completeness score, cross-reference map
    01-specification.md   ← foundation refs, invariants, patent refs (→ F01, F02)
    02-architecture.md    ← ADR refs, relationships, position in master flow
    03-data-structure.md  ← DDL tables (authoritative), Drizzle schemas, columns, FKs, indexes
    04-ui-api.md          ← endpoints, DTOs, screens, permissions
    05-seeding.md         ← seed scripts, golden snapshot coverage, sample data
    06-enrichment.md      ← AI verification, batch processing, enrichment scripts
    07-integrity.md       ← validation rules, cross-component checks, chain completeness
    08-status.md          ← what works, what's scaffolding, DDL↔Drizzle alignment
    09-open-issues.md     ← contradictions, parked tasks, tech debt, known gaps
    diagrams/             ← component-scoped diagrams
```

**Index file frontmatter:**

```yaml
---
id: DOS-e4b221                # system-generated UUID, globally unique, never changes
slug: source-table            # human-readable, URL-friendly, derived from component name
component: source-table
module: P01-source-catalog
stream: platform
status: draft | in-progress | authored | verified
completeness: 3/9             # how many sections are authored
last_validated: 2026-03-30
xref:
  ddl_tables: [source.source_object]
  drizzle_schemas: [source-object.ts]
  api_routes: [/api/source-objects]
  adrs: [DEC-09b8e6, DEC-f0eb14]
  screens: [SCR-xxxxxx]
  existing_docs: [component-references/source-catalog.md]
---
```

**Child file frontmatter:**

```yaml
---
```

### 3. Diagram Layer

Diagrams live as a parallel directory at both module and component levels:

```
{module}/
  diagrams/                    ← module-level (cross-component)
    erd-source-schema.mermaid
    flow-catalog-onboarding.mermaid
  {component}/
    diagrams/                  ← component-level
      state-source-table-lifecycle.mermaid
```

**Diagram frontmatter:**

```yaml
---
id: DGM-b82c41
slug: erd-source-schema
component: source-table          # omit for module-level diagrams
module: P01-source-catalog
diagram_type: erd | flow | sequence | state | architecture | deployment | xref
scope: component | module | cross-module
format: mermaid | excalidraw | png
related_sections: [02-architecture, 03-data-structure]
status: draft | verified
---
```

**Diagram types:**

| Type | What it shows | Example |
|------|--------------|---------|
| erd | Table relationships, FKs, schemas | Source schema 7-table ERD |
| flow | Data/object progression through boundaries | SO admission flow |
| sequence | Runtime interaction between components | Reader → Connector → Source |
| state | Object lifecycle states | Contract: draft → active → superseded |
| architecture | System-level component layout | Platform stream module map |
| deployment | Infrastructure, services, ports | Docker + pm2 service topology |
| xref | How modules connect | P01 → P02 → P05 linkage |

**Format preference:** Mermaid first (version-controlled, diffable, renderable). Excalidraw for complex visuals. PNG as last resort (not diffable).

### 4. Implementation Phases

**Phase 1 — Schema & Templates**
- DevHub tables: `doc_module`, `doc_component`, `doc_dossier_section`, `component_xref`
- Register all modules (F01-F02, P01-P10, T01-T06, C01-C02) with component lists
- Create dossier section templates (what goes in each of the 9 sections, with examples)
- Cross-reference map linking components to DDL tables, APIs, ADRs, screens, existing docs

**Phase 2 — DevHub UI**
- Module browser → component list → dossier viewer
- Section status dashboard (how many sections authored/verified per component)
- Cross-reference panel
- Diagram viewer (mermaid rendering)

**Phase 3 — Populate (module by module)**
- For each component: read existing docs, ADRs, code, DDL → author the dossier
- AI-assisted but human-verified — not a bulk migration
- Start with P02 Contract Registry (highest confusion area) to prove the pattern
- Incremental, trackable per-component

**Phase 4 — Retire**
- Mark old docs as `retired` with pointer to new dossier location
- Don't delete — clear status so no session reads stale theory as truth
- Existing ADRs remain as-is (they're decision records, not component docs)

## Options Considered

1. **Greenfield with split dossiers (chosen)** — fresh authoring, clean structure, incremental
2. **Brownfield correction** — fix existing 1,440 docs in-place. Rejected: more forensic reading than writing
3. **Single mega-file dossiers** — one 9-section file per component. Rejected: too large, hard to maintain, poor diffability

## Consequences

- Every component in the system will have a predictable, findable documentation location
- Sessions can read exactly the section they need (e.g., just `03-data-structure.md`) without loading the whole dossier
- Cross-reference map eliminates the "where is the DDL for this component?" question
- Diagrams are first-class citizens, not afterthoughts buried in prose
- Foundation and Patent are universally referenced but never duplicated
- Old docs don't disappear — they get clear retirement status
- Phase 3 (populate) is the largest effort but is bounded per-component and trackable in DevHub

## Addendum A: Refinements After P02 Contract Registry (Phase 3a)

Recorded after authoring the first real module (P02, 10 components, 101 files). These override the original spec where they conflict.

### A1. ID Scheme — Readable, Not Random Hex

Original spec: `{PREFIX}-{6 random hex}` (e.g., `DOS-e4b221`).

**Revised:** Readable hierarchical IDs:

| Level | Pattern | Example |
|-------|---------|---------|
| Module | `MOD-{module}` | `MOD-P02` |
| Component index | `DOS-{abbrev}{seq}` | `DOS-sc0001` |
| Section | `DOS-{abbrev}{seq}-{section}` | `DOS-sc0001-03` |

Self-documenting. No lookup needed to understand what `DOS-sc0001-03` is.

### A2. Frontmatter — 4 Required Fields, Rest Optional

**Required on every file:**

```yaml
id: DOS-sc0001-03
slug: source-contract-data-structure
section: "03"
parent: DOS-sc0001
```

**Optional (add when relevant):**

```yaml
component: source-contract
module: P02-contract-registry
status: authored
last_validated: 2026-03-30
authored_by: claude
refs:
  - type: ddl
    path: bc-core/docker/redesign/02-platform-tables.sql
```

### A3. Parent Field Replaces Component + Module

Original spec used `component` + `module` on every section file. **Revised:** Use `parent` to reference the component index `id`. The parent carries the component/module context — no need to repeat it.

Cross-reference chain: section `parent` → component index `id` → module index.

### A4. Diagrams — Inline Mermaid in Sections

Original spec: separate `diagrams/` directory per component with standalone `.mermaid` files.

**Revised:** Inline mermaid blocks directly in the section where they're relevant. ERDs in Section 03, flow diagrams in Section 02, state machines in Section 01.

Standalone diagram files (`diagrams/`) only for:
- Module-level diagrams that span multiple components
- Complex diagrams referenced by multiple sections

This eliminates empty `diagrams/` directories and keeps context with content.

### A5. Navigation — awesome-pages Plugin

Original spec: explicit nav in `mkdocs.yml` (every file listed).

**Revised:** Switch to `awesome-pages` plugin with `.pages` files per directory. The explicit nav grew to 80+ lines for one module alone — 10 modules would be 800+ lines in mkdocs.yml, unmaintainable.

Each directory gets a `.pages` file controlling order:
```
# P02-contract-registry/.pages
title: P02 Contract Registry
nav:
  - index.md
  - source-contract
  - admission-contract
  - ...
```

Top-level nav in mkdocs.yml stays minimal (streams only). awesome-pages handles the rest from folder structure.
