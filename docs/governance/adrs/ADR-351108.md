# Contract-Typed Payload Tables + Reader Batch Write Model

## Context

Superseded by three separate decisions: DEC-9e0cd0 (D210 — typed tables only), DEC-4472ca (D211 — batch writes only), DEC-8c489b (D212 — evidence as structural lineage). The original combined decision mixed storage, workload, and evidence concerns into one record.

## Problem

Three compounding issues with the current tenant boundary table design:

### 1. Query: JSONB payloads are un-queryable for business use

The core tenant transactional data — observed_payload_json (OR), canonical_payload_json (CO), metric_value_json (MS) — is stored as JSONB blobs. This violates D162 Rule #1 ("No JSONB for queryable data") because business users WILL query, filter, sort, and aggregate on payload fields (e.g. "invoices over $10K", "revenue by customer", "POs by vendor").

JSONB cannot use regular indexes for range/sort queries, has no planner statistics, forces per-row extraction and casting, and makes cross-object JOINs on business fields effectively impossible at scale.

### 2. Storage: JSONB repeats field names per row

A canonical object with 30 fields repeats all 30 field name strings in every row. At 1M rows/month: ~50-100MB of pure key name repetition per entity. No columnar compression. TOAST adds read amplification on analytical scans.

### 3. Workload: Readers fetch bulk, write row-by-row

Current reader executor fetches efficiently (OData pagination at 1000 rows/page, accumulates full result set). But ObservationService.recordObservation() writes ONE ROW AT A TIME with 6 DB round-trips per record (reader validation, binding resolution, INSERT observed_record, INSERT source_object, INSERT evidence, INSERT lineage). Batch method just loops the single-row method, capped at 100.

10K journal entries = 60,000 individual DB operations. This doesn't scale.

## Key Insight

The schema is NOT dynamic at write time. Contracts define the exact fields, types, and structure BEFORE data flows. When a canonical contract is activated with fields [posting_date:date, company_code:text, amount:numeric], we KNOW the schema. There is no reason to serialize known-schema data into a JSON blob.

## Decision

### Part A: Contract-Typed Payload Tables

**Contract activation generates a typed table in the tenant schema.**

When canonical contract "Journal Entry" v1.0 is activated for tenant t_demo with fields {posting_date:date, company_code:text, document_type:text, amount:numeric, currency_code:text}:

```sql
CREATE TABLE t_demo.co_journal_entry_v1 (
  canonical_object_id  uuid PRIMARY KEY
    REFERENCES t_demo.canonical_object(canonical_object_id),
  posting_date         date,
  company_code         text,
  document_type        text,
  amount               numeric,
  currency_code        text
);

CREATE INDEX idx_co_journal_entry_v1__posting_date
  ON t_demo.co_journal_entry_v1 (posting_date);
CREATE INDEX idx_co_journal_entry_v1__company_code
  ON t_demo.co_journal_entry_v1 (company_code);
```

**Applies at every boundary:**

| Boundary | Envelope table (keeps) | Typed payload table (new) |
|----------|----------------------|--------------------------|
| Observation | observed_record (envelope + lineage) | or_{entity}_v{N} (observed fields, typed) |
| Canonical | canonical_object (envelope + lineage) | co_{contract}_v{N} (resolved fields, typed) |
| Metric | metric_snapshot (envelope + lineage) | ms_{metric}_v{N} (metric values, typed) |

**The envelope tables remain unchanged** — they are the execution model backbone (immutable chain, evidence, lineage, contract binding). The JSONB payload column stays as a frozen audit artifact / backup, but is NOT the queryable representation.

**Runtime write = envelope INSERT + typed table INSERT in same transaction.**

**Business queries hit the typed table.** Full SQL: WHERE, ORDER BY, GROUP BY, JOIN, proper indexes, proper planner stats.

**DDL generation is deterministic:** contract fields → CREATE TABLE DDL. Contract version change → new table version. Old version stays (immutable data).

### Part B: Reader Batch Write Model

**Replace row-by-row writes with batch operations per run.**

The reader execution model changes from:
```
fetch 10K rows → loop 10K × recordObservation() → 60K DB ops
```

To:
```
fetch 10K rows → 1 batch write call → ~6 DB ops total
```

**Specifically:**

1. **Reader validation + binding resolution: ONCE per run** (not per row). The reader/binding don't change mid-run.

2. **Envelope writes: batch INSERT.** Single multi-row INSERT for observed_record, single multi-row INSERT for source_object. PostgreSQL handles this natively.

3. **Typed payload writes: batch INSERT.** Single multi-row INSERT into the contract-typed table (e.g. or_bkpf_v1).

4. **Evidence + lineage: batch INSERT.** Emit per-run evidence (not per-row). A run that observes 10K records produces 1 evidence record ("run observed 10K records from BKPF") not 10K evidence records.

5. **For very large batches (>10K): chunk at the batch level** (e.g. 5K per INSERT), not at the row level. Each chunk is a single transaction.

**Per-row evidence is NOT needed.** The observed_record → source_object 1:1 FK IS the lineage. The run_id groups them. Adding 10K evidence rows that each say "this row was observed" adds no information — the row's existence IS the evidence.

### Part C: Interaction — Typed Tables Make Batch Writes Natural

These two changes reinforce each other:

- Typed tables mean the INSERT is a simple multi-row VALUES clause with proper types — no JSONB serialization overhead.
- Batch writes mean fewer transactions, fewer WAL entries, fewer index updates.
- Combined: 10K journal entries becomes ~4 multi-row INSERTs (observed_record, source_object, or_bkpf_v1, evidence) in 1 transaction — not 60K individual operations.

## What Does NOT Change

- The fixed object progression (OR → SO → AR → Eval → CO → ME → MS → AO → AE)
- Immutability of all boundary objects
- Contract-binding at every boundary
- Evidence and lineage recording (but at run granularity, not row granularity)
- Envelope table schemas (all existing columns stay)
- The JSONB column on envelope tables (kept as frozen audit backup)
