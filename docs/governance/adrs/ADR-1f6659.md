# Safe Delete Guard — Referential Integrity Check Before Any Hard Delete

## Context

1. Raw DB FK constraint errors (23503) are uninformative — users get cryptic error messages. 2. Service-layer checks let us return human-readable blocker lists. 3. Contract soft-delete (archived) prevents data loss while keeping the registry clean. 4. Only connection.service currently has this pattern — all others are unguarded.

## Decision

All registry entity DELETE operations must check for linked/dependent entities at the service layer BEFORE executing the delete. If dependencies exist, return a 409 Conflict with a structured response listing the blocking entities.

Entities covered: contract, reader, reader_flavor, connector, connection, kpi_spec.

Additionally, contracts get a `governance_state` column supporting soft-delete via `archived` state. An archived contract cannot be deleted but is hidden from active listings. Contracts bound to active readers cannot be archived.

Pattern: Each service's delete method calls a dependency check function that queries all known FK relationships and returns a structured blockers list. This replaces relying on raw PostgreSQL FK constraint errors.

Future: Other entities may get governance_state/archived support when needed, but for now the dependency guard is the universal safety mechanism.
