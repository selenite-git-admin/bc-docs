# Chain Enrichment Engine (CEE) v0 — planner-only over harness v1.1 governed-apply

## Context

See decision text below.

## Context

The metric promotion path today expects every binding element (entity, characteristic, business concept, canonical contract field, observation contract mapping, MC, MCV, variable bindings, filter clauses) to be present and active before M12 panel admission. When something is missing or stale:

- C-FX checks fail (C-FX-1..11)
- L-V1 a-i materialization gates block
- PE-MC checks reject
- `contract.chain_status` reports `partial` or `broken`
- L-node semantic verdict reports `red`

Each detection surface stops. The operator then manually:

1. Identifies which substrate element is missing
2. Invokes the right governed write endpoint (B6 / S1 / C5 / F3 / publication / CC field_selection / MCV rebind)
3. Re-runs the failing detection
4. Repeats until all green
5. Then re-runs M12 panel → M12.5 materialize → M13 evaluate-pe-mc → M14 activate

This is N round trips, where N grows with chain depth and substrate freshness gaps. Iteration cost dominates.

CEE is the sibling of the Chain Audit Service (CAS, DEC-1fa08f / D445, shipped). **The audit defines what "correct enough to proceed" means; CEE is built to satisfy the auditor.** CAS shipped first deliberately so the enrichment engine has a concrete target to plan against.

## Decision

Introduce a **Chain Enrichment Engine** (CEE) as a planner over the existing governed-write surface. CEE:

1. **Detects gaps** by reading `contract.chain_status`, `contract.chain_trace`, `mcf.l_node_semantic_verdict`, `mcf.metric_publication_eligibility_result`, plus the C-FX / L-V1 result surfaces.
2. **Plans** an ordered dependency graph of governed write calls: entity admission → characteristic admission → BC admission + C5 confirm → registry publication → CC `field_selection` update → observation contract mapping → MC + MCV creation → MVB + MFC insertion → M13 eval → M14 activate.
3. **Emits a harness v1.1-compatible packet** describing the steps as `sc.create_draft` / `oc.create_draft` / `cc.field_selection.update` / governed-write packets.
4. **Halts at human-judgment boundaries** (e.g., "what is the `canonical_value_set` for this status?", "which of two candidate BCs should bind here?") and returns a structured decision request to the operator.
5. **Does NOT apply.** The operator runs the existing harness v1.1 governed-apply to enact the packet.

CEE is a service at `bc-core/src/registry/mcf/chain-enrichment/` with:

- One controller: `ChainEnrichmentController`
- One service: `ChainEnrichmentService`
- One planner: `ChainEnrichmentPlanner` (pure function: state → ordered plan)
- One new table: `mcf.chain_enrichment_plan` (plan journal)

V0 has **zero new write surface for chain substrate.** It is a pure planner over the harness v1.1 apply contract. The harness still owns governance, audit, dry-run, classifier, idempotent replay — all that infrastructure is leveraged, not duplicated.

## Locked V0 scope (operator-authorized 2026-06-16)

- **Single target kind:** `source_contract_gap_plan` (NOT `seed_metric`)
- **CEE v0 emits harness-compatible packet, NO auto-apply**
- **Pairs with B-track FSCM** via harness v1.1 `sc.create_draft` governed apply path
- Builds on existing governance rails instead of new write surface
- The planner-executor split is between CEE (planner) and harness (executor) for v0; CEE owns both planner and executor only in v1+

## Locked decisions

| D | Topic | Choice |
|---|---|---|
| D1 | Trigger model | **A** — operator-invoked only in v0/v1. Background sweep deferred to v2. |
| D2 | M12-from-CEE | **C** — deterministic prerequisite gate. CEE auto-invokes M12 only when all deterministic prerequisites are met. Otherwise halts. **Note for v0:** since v0 is plan-only (no auto-apply), M12 auto-invocation is not in scope for v0 — deferred to v1. |
| D3 | Rollback | **A** — per-step idempotency. Each governed-endpoint call commits independently. Plan journal records step-level state. Failed step does not unwind prior steps. |

## Surface — V0 (plan-only)

```
POST /api/chain-enrichment/plans
  body: { target: { kind: 'source_contract_gap_plan',
                    source_specimen_ref: { source_system_code, source_table_code, ... } } }
  auth: @PlatformOnly() @Roles('platform_admin')
  response: { plan_uid, target_summary, gap_count, harness_packet_json, halted_at_step?, decision_request? }

GET /api/chain-enrichment/plans/:planUid
  auth: @PlatformOnly() @Roles('platform_admin' | 'mcf_author' | 'mcf_publisher')
  response: full plan + harness_packet_json
```

The `harness_packet_json` is consumed by the existing harness v1.1 governed-apply CLI — the operator runs it manually. CEE never calls the harness; it just hands it the packet.

## Surface — V1 (with execution, deferred)

```
POST /api/mcf/chain-enrichment/runs
  body: { target: { kind: 'seed_metric' | 'mcv' | 'intake', uid: string },
          plan_only: boolean (default true), ... }
  auth: @PlatformOnly() @Roles('mcf_publisher')
  response: { run_uid, status, plan_summary, halted_at_step?, decision_request? }
```

V1 surface designed but not implemented until CEE v0 + harness integration proves stable.

## Audit-first loop

The intended production path becomes:

```
CEE (plans) → harness v1.1 (governed apply) → CAS (verifies) → release verdict
```

CEE proposes; harness applies; CAS verifies. If CAS's `pre_m13_audit` returns `FAIL` or `OPERATOR_REVIEW`, the M13 gate refuses. CEE then re-plans against the failure findings. The auditor (CAS) is authoritative on "correct enough to proceed."

## Rules

1. **CEE NEVER bypasses governed services.** Every write goes through an existing HTTP endpoint via the harness. CEE has no special DB privileges and no INSERT/UPDATE/DELETE on business substrate.
2. **CEE NEVER auto-resolves human-judgment boundaries.** Specifically: `canonical_value_set` authoring, BC ambiguity resolution, `semantic_role` assignment, formula intent rewriting all halt for operator input.
3. **CEE NEVER mutates tenant bindings.** Tenant onboarding is a separate flow.
4. **CEE plan-journal rows are immutable.** Append-only.
5. **CEE re-checks resumes.** On resume, each previously-completed step is re-verified against current substrate; if it is no longer valid (e.g., a BC was superseded between runs), the plan is re-computed from that point.
6. **CEE is bounded.** Max 30 steps per plan. Beyond that, halt for operator review — runaway plans indicate a deeper substrate problem.
7. **CEE reports drift.** When re-check fails on resume, it emits a `chain_enrichment_drift_detected` audit log and halts.

## Substrate — V0 (plan-only)

```sql
CREATE TABLE mcf.chain_enrichment_plan (
  plan_uid                   uuid PRIMARY KEY,
  target_kind_code           text NOT NULL CHECK (target_kind_code = 'source_contract_gap_plan'),
  target_ref_json            jsonb NOT NULL,
  computed_at                timestamptz NOT NULL DEFAULT now(),
  computed_by_sub            text NOT NULL,
  gap_summary_json           jsonb NOT NULL,
  harness_packet_json        jsonb NOT NULL,
  halted_at_step_index       integer,
  decision_request_json      jsonb,
  cee_engine_version         text NOT NULL DEFAULT 'cee-v0'
);
CREATE INDEX idx_cep_target ON mcf.chain_enrichment_plan(target_kind_code, computed_at DESC);
```

V0 emits plans only — no execution journal needed since harness owns execution.

## Substrate — V1 (with execution, deferred)

```sql
CREATE TABLE mcf.chain_enrichment_run (
  run_uid                    uuid PRIMARY KEY,
  target_kind_code           text NOT NULL CHECK (target_kind_code IN ('seed_metric','mcv','intake')),
  target_uid                 uuid NOT NULL,
  initiated_by_sub           text NOT NULL,
  initiated_at               timestamptz NOT NULL DEFAULT now(),
  plan_json                  jsonb NOT NULL,
  step_results_json          jsonb NOT NULL DEFAULT '[]',
  current_status_code        text NOT NULL CHECK (current_status_code IN
    ('planned','executing','halted_for_decision','completed','aborted','failed')),
  halted_at_step_index       integer,
  decision_request_json      jsonb,
  completed_at               timestamptz,
  cee_engine_version         text NOT NULL DEFAULT 'cee-v1'
);
CREATE INDEX idx_cee_run_target ON mcf.chain_enrichment_run(target_kind_code, target_uid);
CREATE INDEX idx_cee_run_status ON mcf.chain_enrichment_run(current_status_code) WHERE completed_at IS NULL;
```

No new `certification_record.action_code` needed — each governed endpoint CEE calls already mints its own certs. The CEE run journal links to those cert records by storing `certification_record_uid` per step result.

## Out of scope

- Substrate hand-edits, raw DML
- Bypassing C5 confirm
- Auto-resolving human-judgment decisions
- Mutating tenant bindings
- Real-time runtime evaluation
- Cross-tenant chain operations
- M12/M12.5/M13/M14 invocation from CEE v0 (deferred to v1 per D2)

## Foundation gate

**Repair location (V0):** **D** (evaluation boundary implementation) bordering **F** (read model). CEE v0 produces a plan (read-derived from current substrate) that is consumed by a separate execution surface (the harness). The new substrate `mcf.chain_enrichment_plan` is operational state, not contract state.

**Why not B:** no new grammar — the plan describes calls to existing contract surfaces via existing harness packet schemas.

**Why not E:** the plan table is operational, not contract substrate. It does not participate in contract chain semantics. The plan's `harness_packet_json` references existing artifacts but does not add new ones.

**Three pre-action answers (V0):**

1. **Why here?** The gap-detection-to-packet derivation has no current home; chain status detects, but no service produces an actionable packet.
2. **Why not upper layers?** No new grammar needed — packets reference existing artifact families (SC, AC, OC, CC) through harness v1.1.
3. **Why not lower layers?** No evaluation change. CEE v0 is pure read-derive over existing substrate.

## Risks

1. **Plan explosion** — poorly-bound seed metrics could generate plans that touch dozens of substrate elements. Mitigation: 30-step ceiling + operator review.
2. **M12 LLM cost** (deferred until v1) — if CEE invokes M12 in many chains, vendor spend grows. Mitigation: D2 requires deterministic prerequisites; LLM cost only incurred when chain is genuinely ready.
3. **Substrate drift mid-run** — another operator could mutate substrate while a CEE run is in flight. Mitigation: per-step re-validation, halt on drift.
4. **Step-level idempotency reliance** — CEE depends on each governed endpoint being idempotent. Audit (separate effort) to verify all governed endpoints are. If any are not, fix at source before CEE v1 owns execution.
