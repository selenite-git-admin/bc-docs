# Source-System Docket structure — folder-per-system co-located onboarding bundle (extends D385)

## Context

Today's flat source page is a reference sheet, not a docket: onboarding artifacts (contracts in the Contract Registry DB, catalog in Mongo, evidence in workstream-dated trees, reader flavors in process chapters) are scattered and unlinked — the frontmatter admission_contract_versions[] hook is empty on every one of the 61 systems. A co-located folder keeps everything for one system together and navigable, and the linking hooks make it discoverable. The repo already proves the pattern with metric-work-records/<metric_slug>/ (folder-per-metric). Folder-per-system (over the lighter manifest+evidence-subtree option) was the founder's explicit choice for maximal co-location. Rolling to all systems standardises onboarding-material capture ahead of the D525 per-source-realization audit program, which needs each system's contracts/catalog/evidence indexed in one place. Prove-on-one first because the bulk step rewrites intra-repo links across 61 files and must not be done blind.

## Decision

Founder-decided 2026-07-13. Per-source-system documentation moves from the flat one-markdown-per-system layout (D385 / DEC-6cb4f3) to a co-located FOLDER-PER-SYSTEM "docket" that keeps all onboarding reference material for one source system together and registered/logged. The convention is rolled across all source systems (currently 61 SRC-* pages), via prove-on-one then staged migration (D268 one-then-many).

STRUCTURE. Each system becomes a folder `docs/reference/source-systems/<slug>/` containing:
- `index.md` — the registered docket COVER/MANIFEST. Carries the SRC-* frontmatter registry (uid, slug, proof_status, reader_flavors[], admission_contract_versions[], governing_adrs[]) plus new linking hooks (catalog_ref, evidence_records[], docket_files[]). Holds the reference narrative (proof status, reader-flavor binding, what's admitted, legal/licensing, technical, source-system semantics/pitfalls, known gaps, references, changelog) — i.e. the current D385 reference page — and a Docket Contents nav to the sibling files.
- `contracts.md` — SC/AC/OC/CC authored against this system + Contract Registry links; the hook that fills the perpetually-empty admission_contract_versions[].
- `catalog.md` — seed catalog + source-registration entries (provider/system/version/module/table/field) for this system.
- `onboarding-log.md` — dated per-onboarding runbook-execution log (customer-side + BareCount-side), session-referenced.
- `evidence.md` — proof entries: simulator runs, first-hand verifications, metric snapshots realized; the ground for proof_status promotion.

REGISTRY STABILITY. The SRC-* uid is unchanged and moves to index.md, so the DevHub doc scanner (uid/frontmatter-keyed) is unaffected. Only markdown relative links and `supersedes:` paths need rewriting (`<slug>.md` → `<slug>/`), which is a mechanical, reviewable pass.

WHAT D385 KEEPS. The proof-status taxonomy (designed/shape_tested/first_hand_proven), the zero-claims-run-ahead-of-proof rule, the required reference sections, and the governance are all retained. This ADR changes only the physical layout (flat file → folder bundle) and adds the linking hooks that close the real gap: a source page today neither contains nor links the concrete contracts, catalog loads, and evidence realized against that system.

ROLLOUT. (1) `_template/` docket skeleton + README convention. (2) Prove on ONE existing system (SAP ECC — richest page) end-to-end incl. link rewrite + validation. (3) External-auditor + founder review of the exemplar. (4) Staged migration of the remaining ~60, most of which are `designed`-only (thin). (5) New systems are born directly in the docket shape. `index.md` "Authoring a new source page" guidance updated to the folder convention.

## Amendment 1 — Authority semantics (2026-07-13, external-auditor review)

The external auditor (Codex) accepted the structure in principle but required the docket's **authority semantics** be made explicit before catalog-wide rollout, to prevent the docket becoming an accidental parallel registry or evidence store. This amendment is normative and governs the template and all migrations.

**Authority hierarchy (normative).**

> Source-System Dockets are **documentation projections**. They do not create source registration, contract identity, catalog authority, evidence, or audit decisions. All such objects are referenced by exact governed UID and digest from their owning substrate. Proof maturity is scope-bound and distinct from source-realization audit verdict. CI fails when a docket reference cannot be resolved or disagrees with its authority.

Concretely:
1. **Registries own identities** — the source catalog (`source.*`), Contract Registry (`contract.*`), and seed store own source/catalog/contract identity.
2. **The audit substrate owns evidence and decisions** — evidence, PASS/REJECT/OPERATOR_REVIEW, NCs, CRs, signatures, and re-audit history are append-only/signed there.
3. **The docket is a Git-versioned human projection and index** of those authorities. Git history is change evidence, not the append-only signed authority.

**Required changes to the structure defined above:**
- Every docket file declares `authority_role: projection`. Values copied from governed systems must be generated or CI-reconciled, never hand-asserted as truth.
- `proof_status` is retained for D385 compatibility **as evidence-maturity only** (`designed`/`shape_tested`/`first_hand_proven` = *what evidence exists, at what scope*). It must never imply an audit verdict, and must be accompanied by machine-readable scope references: `proof_scope_refs[]`, `source_realization_refs[]`, `audit_decision_refs[]`. The referenced governed objects — not the YAML values — are authoritative.
- Proof is scoped to an **exact source-realization package** (source provider/system/release, reader/runtime version+digest, catalog/mapping roots, SC/AC/OC/CC set or contract-set digest, affected MCV/closure roots, evidence-set digest, pseudonymous instance ref for first-hand). `index.md` links the package UID/digest and displays a projection; it does not restate the package identity. The bare `SRC-*` uid is a documentation identity, **not** the audit-usable coordinate.
- Proof-maturity vocabulary and audit-verdict vocabulary are **orthogonal**, rendered in separate fields/sections with separate identities; neither is inferred from the other. Any rendered audit decision carries a "derived projection" label + the exact governed decision UID/digest.
- `evidence.md` and `onboarding-log.md` outcome/proof fields must be **emitted or referenced** (governed evidence UID/digest; onboarding-session receipt), not author-asserted. Narrative prose may remain hand-authored.
- **First-hand proof is entity/scope-specific**, never a global promotion of a whole source system from one customer run.
- **Customer identity stays out of Git** — only a pseudonymous instance UID, evidence digest, allowed scope, and disclosure-safe summary. Tenant/customer identity and raw evidence live in the restricted audit/evidence store.
- Community-catalogue seed material is **provisional seed provenance**, not verified source truth, until exact release/DDIC (or vendor-equivalent) verification.
- "Standard field semantics are universal" claims are **qualified**: release, support-package, industry-extension, and customizing differences remain in scope for a source realization.

**CI enforcement (to build):** a docket reference that cannot be resolved to its authority object, or that disagrees with it, fails CI. Full auditor response: `barecount-devhub/artifacts/metric-audit/RESPONSE-Codex-source-system-docket-d526-2026-07-13.md` (sha256 ed2b0403…c6ce57).

## Amendment 2 — four-quadrant docket content (2026-07-13, founder)

`index.md`'s **Legal & Licensing** and **Technical** sections each carry **two planes**, recording only **official, general research** (section numbers vary per page — e.g. a system with a Commercial section shifts Technical from §5 to §6; the planes attach to the named sections, not fixed numbers) (no tenant names, contracts, credentials, endpoints, approvals, or legal conclusions — those are governed-substrate objects linked by pseudonymous UID/digest):

| | **Legal & Licensing (§4)** | **Technical (§5)** |
|---|---|---|
| **Platform Plane** (vendor, general) | Vendor API/licensing policy; permitted vs prohibited access methods; read- vs write-back; user/licence types; maintenance/support horizon; official refs + applicability + retrieval date + digest | Supported releases/components; protocols & auth methods; metadata/schema discovery; API limits/pagination/delta; network requirements; known source semantics & restrictions |
| **Tenant Connection Plane** (what any customer must have/do, general) | Required vendor entitlement; MSA/DPA/Order-Form/security approvals that must exist; module/data-scope authorization; residency/retention/subprocessor; customer-side authorizer roles | Customer-side setup steps; service account & minimum permissions; endpoint/network allow-listing; modules/entities to enable; connection-test & first-chain proof requirements; release/customization verification |

The Tenant Connection Plane states **general requirements**, never a specific tenant's values — consistent with Amendment 1 (docket = projection; actual tenant authorization + execution evidence live in the restricted governed substrate).

## Amendment 3 — governed-evidence-backed maturity, exact-coordinate fields, digest-bound research (2026-07-13, external-auditor sign-off review)

From the Codex sign-off review (`barecount-devhub/artifacts/metric-audit/REVIEW-Codex-source-system-docket-d526-signoff-2026-07-13.md`, sha256 718723d8…66d1d9). Structure accepted in principle; three normative tightenings before catalog-wide rollout:

1. **`proof_status` requires governed evidence.** Evidence maturity is not an author assertion — `shape_tested`/`first_hand_proven` may be projected only when a governed proof-scope/evidence object exists and `proof_scope_refs[]` / `source_realization_refs[]` are populated. Absent that, project `designed` and keep any prior narrative as explicitly **ungoverned historical background**. (SAP ECC exemplar downgraded `shape_tested`→`designed` accordingly.)

2. **Exact governed reference coordinates are mandatory frontmatter, not prose.** Every manifest carries machine-readable `source_registry_ref` (provider/system/release UID+version), `reader_flavor_versions[]` (version UID+digest), `catalog_root` (snapshot/mapping-root digest), `contract_set_ref` (version-set/digest), `admission_contract_versions[]`, and `official_research_refs[]`. The slug fields (`system_type_code`, `reader_flavors[]`, `catalog_ref`) are **labels**; authority is the referenced object. Human Markdown links may accompany but never substitute for the coordinate.

3. **Legal/technical projections are research, not clearance.** No blanket "sanctioned/banned/zero-cost/licence-category" conclusions. Each condition is rendered as a researched candidate/condition, supported by a **version-exact official source** recorded in `official_research_refs[]` (source + exact version + retrieval date + content digest), and — at runtime — by the exact-release **Platform Source Access Policy** + **Tenant Source Authorization** (deferred authority objects, AuditHub/bc-core-owned). Credential handling points only to governed secret-ingress (`credential_ref` + receipt); no raw secret in Git/ticket/chat/email/docket/audit.

**Capability gates for rollout (all three confirmed by the auditor):**
- (G1) governed evidence UIDs/digests + onboarding-session receipts must exist before evidence/outcome rows are audit-grade;
- (G2) CI fails when a docket reference is missing, unresolved, stale, or disagrees with its authority;
- (G3) **official research references must be versioned + digest-bound before legal/technical projections are presented as verified** — CI validates the structured reference; human/legal review determines applicability.

Aligns with the Codex two-plane onboarding-assurance note: the docket is the documentation-projection layer; authority objects and the activation predicate remain AuditHub/bc-core-owned and post-Track-C.
