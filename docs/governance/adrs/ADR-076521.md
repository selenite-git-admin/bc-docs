# Apex Tenant Binds to SAP S/4HANA Reader Chain; bc-sdg as Profile-Parameterised SDG-SAP Service

## Context

The error in DEC-049cb1 was not a detail ‚Äî it was the framing. Three converging insights forced the supersession:

**(1) The Trust Chain claim demands canonical source identity.** The demo's positioning is *"this is what your S/4HANA data looks like; only the connection URL is simulated."* For that claim to be visible to a prospect inspecting the Trust Chain, `source.source_system.system_name` must read `s4hana`, not `apex-emitter`. DEC-049cb1's custom source system would have produced a Trust Chain that announces "this is synthetic" in the most operator-visible field ‚Äî undoing the entire pitch.

**(2) The architecture was already documented.** `bc-docs-v3/docs/source-systems/sap-s4hana.md` was authored 2026-04-28 with `proof_status: shape_tested` against the bc-sdg simulator, reader flavors `sap-s4-cloud-odata` and `sap-s4-onprem-cds` backed by `SapOdataV4Executor`. DEC-049cb1 was authored 2026-05-08 without reading that page. The pattern wasn't novel architecture ‚Äî it was undiscovered prior art. The ADR's omission was a research failure, not a design failure.

**(3) Multi-profile reuse was not modeled.** Apex Motors is a tenant brand for the auto-OEM-industry storyboard. Future tenants on pharma / FMCG / real-estate would each be a brand on a parallel profile. SDG-SAP is the unified emitter; profiles parameterise it; tenants are deployments. Naming everything "apex-*" coupled a brand to platform infrastructure. The 4200 server's existing 6-profile system is the right model; DEC-049cb1 bypassed it by creating `src/apex/` as a parallel module.

The corrected architecture lands four wins simultaneously:
- **Less code.** No custom executor (~150 LOC), no snapshot server (~80 LOC). Reuse `SapOdataV4Executor` and bc-sdg's 4200 server.
- **Correct Trust Chain identity.** `source_system=s4hana` visible to prospects.
- **Multi-tenant-ready.** Future apex-pharma binds the same SAP reader chain with a different profile selector in the connection URL.
- **Documentation alignment.** sap-s4hana.md, source-systems/index.md, and DEC-d2cdb9/DEC-6cb4f3 already describe this. The ADR catches up to the docs rather than diverging from them.

Cost of the correction: revert ~300 LOC across two commits, delete 5 platform DB rows (additive ‚Äî apex tenant + tbc_apex_dev DB stay), update demo-operations.md and demo-plan-cfo-pack.md framings, queue schema rename + module restructure as separate sessions. Components A, B, and C semantic content (~85% of work-by-LOC) preserved.

The rationale for accepting the cost rather than retrofitting around DEC-049cb1: every additional week with `apex-emitter` rooted in the catalog means more downstream work (binding, MC chain, demo health audits) hardcoding the wrong identity. The longer the correction is deferred, the more invasive it becomes. Catching it now ‚Äî before Phase 2 catalog activation builds on top ‚Äî is the cheapest moment.

**Supersedes DEC-049cb1.** That earlier ADR locked a custom `apex-emitter` source_system + `ApexSnapshotProtocolReader` executor + `/snapshot` push endpoint. It diverged from the architecture already documented in `bc-docs-v3/docs/source-systems/sap-s4hana.md` (authored 2026-04-28, ten days before DEC-049cb1 was written). The Trust Chain it would have produced reported `source_system=apex-emitter` ‚Äî visibly synthetic ‚Äî undermining the demo's positioning that *"the only difference between this and your real S/4HANA deployment is the connection URL."*

This ADR locks the corrected architecture grounded in the existing source-systems framework (DEC-6cb4f3 / D385) and the SAP API admission stance (DEC-d2cdb9 / D384).

## 0. Source system identity

The apex tenant's source system is **SAP S/4HANA Cloud** ‚Äî the existing `source.source_system` row with `system_name='s4hana'` (UUID `019d56d1-b36b-7fe4-8f4c-a95c134da2c3`). NO new source_system row. The Trust Chain reads exactly as a real S/4HANA customer's would.

Canonical reference: `bc-docs-v3/docs/source-systems/sap-s4hana.md` ¬ß2 ‚Äî reader flavors `sap-s4-cloud-odata` and `sap-s4-onprem-cds`, both backed by the existing `SapOdataV4Executor`.

## 1. Reader chain (all existing ‚Äî no new code)

| Layer | Apex tenant uses | Existing? |
|---|---|---|
| Source system | `s4hana` | ‚úÖ existing |
| Source version | S/4HANA Cloud (already in catalog) | ‚úÖ existing |
| Reader | sap-s4-cloud-odata or equivalent FI/AR/AP/MM reader registered against s4hana | ‚úÖ register if missing per documented onboarding flow |

The single architectural difference between apex and a real S/4HANA customer is `runtime.connection.endpoint_uri` pointing at bc-sdg, not the customer's SAP server. Everything else ‚Äî source_system, reader, flavor, executor, connector, contracts, observation/admission boundaries ‚Äî is shared.

## 2. SDG-SAP as profile-parameterised service

bc-sdg's existing 4200 OData server is the SDG-SAP service. It already supports six stochastic profiles (`auto-parts`, `chem-co`, `fmcg`, `mfg-co`, `pharma-co`, `real-estate-co`) with `/s4cloud`, `/s4onprem`, `/ecc` landscape routes.

For apex tenant: extend the 4200 server with a 7th profile. Per `bc-sdg/CLAUDE.md` the existing 4200 server is the right transport; the 6100 in-memory simulator stays for sandbox1 reader-chain testing.

**Profile abstraction (locked):**
- A profile = world-state shape + master data + (optional) story patterns + industry semantics
- Profiles live as data + code under `bc-sdg/src/profiles/<profile-slug>/` (the existing pattern)
- Tenants bind to ONE profile via `runtime.connection`'s endpoint URL (profile selector via path or query parameter ‚Äî implementation detail in subsequent session)
- The first storyboard-driven profile is `apex-motors` (auto-OEM industry, the demo storyboard's narrative)
- Future tenants on pharma / FMCG / etc. instantiate parallel profiles with different patterns

**Story patterns are a NEW capability layered onto the profile interface.** Existing profiles are stochastic; apex-motors profile adds deterministic story-event patterns (the 7 patterns from the storyboard). Subsequent ADR may further formalise the storyboard-vs-stochastic distinction; for now, both modes coexist under the unified `Profile` interface.

## Errata

### 2026-05-10 ‚Äî Apex demo path pivots to coherent snapshots; daemon framing extended/superseded by DEC-b0839a (D399)

DEC-b0839a (D399) ‚Äî "SDG Coherent Snapshots and Multi-Projection Architecture" ‚Äî pivots the Apex demo path from a continuously-living source system to a one-shot coherence-gated snapshot model. The trigger: 30-minute CFO Pack demos never observe wall-clock time-pass within a session, so the daemon-shaped state machinery + cadence + Lambda decomposition was paying for an outcome the demo never displays. DEC-b0839a also locks a multi-profile + multi-projection architecture so Apex (the profile) is decoupled from S/4 (one of many possible projections of that profile).

**What stays authoritative in this ADR (DEC-076521):**

- ¬ß0 ‚Äî Source system identity. `source_system='s4hana'` for Apex on the demo path. Trust Chain reads exactly as a real S/4HANA customer's would.
- ¬ß1 ‚Äî Reader chain. Canonical `sap-s4-cloud-odata` flavor backed by `SapOdataV4Executor`; bc-sdg's V4 OData server is the data origin under a profile-parameterised SDG-SAP framing. No push-to-bc-core. No 6100 Apex shortcut.
- ¬ß2 first paragraph ‚Äî Profile abstraction (profile = world-state shape + master data + industry semantics; profiles live as code under `src/profiles/<profile-slug>/`; the first storyboard-driven profile is `apex-motors`).
- ¬ß3 row A ‚Äî World-state persistence in `sdg_world` schema with `profile_code` partition (column-level scoping; specific table shapes are decided per-slice under the snapshot model).
- ¬ß3 row C ‚Äî Bookkeeping-correct posting generator (semantic content preserved; `core/document-balancer.ts`'s `assertBalanced` and `buildCustomerInvoice` are still the per-document validators, called by the snapshot generator instead of the arc-start emitter).
- ¬ß3 row D ‚Äî Bind-not-push (`runtime.connection` row + `runtime.reader_binding` for Apex tenant).

**What is extended/superseded for the Apex demo path:**

- ¬ß2 paragraph beginning "Story patterns are a NEW capability layered onto the profile interface" ‚Äî the runtime arc/state-machine framing is replaced. Storyboard patterns become **narrative events** placed at deterministic dates inside the snapshot's historical window; arc cadence machinery and per-tick state advancement are not used on the demo path. The 7 storyboard patterns themselves remain authoritative as the events the snapshot must contain ‚Äî what changes is when they are computed (snapshot generation time, not daily ticks).
- ¬ß3 row B ‚Äî Pattern engine (registry kept as code per Demo Operations hard contract, but tick/decide/advance-arc semantics replaced by snapshot-time placement rules).
- ¬ß3 row E ‚Äî Lambda + EventBridge orchestration (daily emit + monthly close + quarterly pattern-refresh + hourly coherence + monthly rollover). Replaced by a single `generateProfileSnapshot()` invocation, run on demand or quarterly (operator-triggered). Lambda decomposition is not on the Apex demo critical path; deployment shape is parked.

**What is parked, not deleted as a concept:**

The daemon framing remains a valid model for a future long-lived lab/sandbox profile that explicitly wants live cadence semantics. A future ADR on its own track may revive it for that profile without contradicting DEC-b0839a or this ADR. The Apex demo profile does not.

**What this means for in-flight implementation work:**

- The apex-motors manifest's `patternsProvided[]` field reframes to `narrativeEvents[]` with placement-rule semantics.
- The `delinquent_fleet` emitter's document-construction logic is preserved as a callable invoked by the snapshot generator.
- The deterministic batch-id idempotency contract (Phase 1) transfers as snapshot-id determinism on `(profile_code, as_of_date, seed, generator_version)`.
- The existing `sdg_world.vbrk` table from Phase 2.1 is treated as a first-adapter staging table; whether it is promoted to a neutral core fact (`customer_invoice`) or kept as S/4-projection staging with a new neutral core table beside it is decided per the next slice's DBCP request ‚Äî see DEC-b0839a "Caution on existing `vbrk` table".

See DEC-b0839a ¬ß"Schema and code implications for the next session" for the full per-table picture.
