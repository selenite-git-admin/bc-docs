# D226: bc-docs Restructuring — Essentials/Domains/Tier Documentation Model

## Context

bc-docs has grown to 706 documents (466 non-ADR + 240 ADRs) with no reading order or priority tiering:

| Area | Count | Issue |
|------|-------|-------|
| architecture/ (flat) | ~35 files | Mixed concerns |
| architecture/modules/ | ~200 files | Aspirational specs, never validated |
| architecture/decisions/ | 240 ADRs | Well-structured (D221 working) |
| architecture/component-cards/ | 14 files | Good |
| architecture/database/ | 10 files | Good |
| foundation/specification/ | 74 files | Locked spec |
| component-references/ | 13 files | Key docs but flat |
| playbooks/ | 22 files | Mixed quality |
| sources/ + user-stories/ | 108 files | Reference material |
| security/ | 14 files | Risk taxonomy |
| provisional-patent/ | 35 files | Locked |

**Problems:**
1. No reading order — new sessions cannot distinguish the 10 critical docs from 200+ aspirational specs
2. Flat architecture/ dump with no domain grouping
3. Module docs (governance, subscription) are ~200 files for things not yet built
4. No "start here" per domain
5. DevHub doc registry has 1,440 entries with no priority tiering

## Decision

Restructure bc-docs into a 3-tier documentation model wired to the DevHub session protocol.

### Tier 1: Essentials (~10 docs) — session-critical

Located in `essentials/`. Every session should have context on these.

| Document | Source | Purpose |
|----------|--------|---------|
| execution-model.md | component-references/ | Feed-forward execution model |
| contract-chains.md | architecture/ | Source → Metric → AI chain topology |
| data-object-model.md | component-references/ | SO, CO, Metric Snapshot, AO |
| binding-chain.md | architecture/database/ | Field resolution path |
| evaluation-boundaries.md | NEW (consolidated) | 4 boundaries + state produced |
| database-schema-map.md | architecture/database/ | 78 tables, 11 schemas |
| port-table.md | extract from CLAUDE.md | Service ports and dev setup |
| forbidden-vocabulary.md | extract from CLAUDE.md | Pipeline-free terminology |
| naming-conventions.md | D148 ADR | ISO 11179 naming standard |

### Tier 2: Domains (~30 docs) — per-domain entry points

Located in `domains/`. Each domain gets a README entry point.

### Tier 3: Deep Specifications — on-demand

Existing locations remain:
- `foundation/specification/` — Locked (no change)
- `architecture/decisions/` — 240 ADRs (D221 working)
- `architecture/component-cards/` — Object-level cards
- `architecture/database/` — DBML, schema-map
- `playbooks/` — Operational guides
- `security/` — Risk taxonomy
- `provisional-patent/` — Locked

### What Gets Archived

`architecture/modules/governance/` and `architecture/modules/subscription/` (~200 files) move to `archived/module-specs/`. Rebuilt as needed from actual implementation.

### DevHub Integration

1. `essentials/` docs → `authority: authoritative` in doc registry
2. `domains/*/README.md` → `authority: authoritative`
3. Session boot → DevHub returns recommended_reading based on project + domain
4. CLAUDE.md → "Reading Order" section pointing to essentials/ first

## Options Considered

### Option A: Flat restructure within architecture/
Simple folder moves. **Rejected:** Does not solve the reading order problem.

### Option B: 3-tier model with essentials/domains (CHOSEN)
Clear hierarchy with session protocol integration.

### Option C: Wiki-style with cross-linking
Every doc links to related docs. **Rejected:** Link maintenance overhead, no clear priority.

## Consequences

### Positive
- New sessions have a clear 10-doc fast-path
- Domain work gets a focused entry point
- ~200 aspirational docs stop polluting active corpus
- DevHub can prioritize docs by tier

### Negative
- Migration effort: moving files requires updating DevHub doc registry
- Some docs span multiple domains
- Cross-references may break during restructure

### Implementation Order
1. Create essentials/ and domains/ skeleton
2. Copy (not move initially) highest-value docs to new locations
3. Write new docs: observation-contracts.md, business-objects.md, business-fields.md, metric-dag.md
4. Update DevHub doc registry authority levels
5. Update CLAUDE.md reading order
6. Archive aspirational module specs
7. After validation period, remove duplicate copies from old locations
