# Bump self-verifier algorithm to mcf-verifier-v2: apply declared metric filter clauses during fixture playback

## Context

D-M10-4 forever-locks mcf-verifier-v1; semantic change requires ADR-governed bump. v1 hashed filter clauses into filter_set_hash (package signature) but never executed them at fixture playback, producing unfiltered actual values. Confirmed via Paid Customer Invoice Count v2 (MCV db3e1bd0…): expected=2, actual=3 — filter `WHERE status='paid'` ignored. v2 applies the filters; v1 rows remain immutable historical evidence.


## Context

The M10 self-verifier (`mcf-verifier-v1`) executes a metric's formula AST against a Section A fixture and compares the actual output to the Section B expected output. The verifier was forever-locked under DEC-c3e57f / D-M10-4 governance — any semantic change requires an ADR-governed bump.

In June 2026, while diagnosing the stuck MCV for Paid Customer Invoice Count v2 (`db3e1bd0-051b-401f-8278-e3cd84e622a7`), the operator confirmed:

- The metric declares one filter clause: `WHERE status (BC 0a860227…) = 'paid'`.
- The fixture's 3 rows carry per-row `filter_inputs[0a860227]` valued `paid, paid, open`.
- Expected output: 2 (count distinct invoices with status='paid').
- Verifier actual output: **3** (counted all 3 rows — filter ignored).

Root-cause investigation showed:
1. `FormulaExecutionEngine.execute(ast, sectionA, sectionC)` accepts no filter-clause parameter.
2. The 9 AST node kinds (`variable_ref`, `literal`, `aggregate`, `arithmetic`, `comparison`, `case`, `window`, `time_anchor_resolution`, `bucket_assign`) include no `filter` node.
3. `mcf.metric_filter_clause` rows participate in `filter_set_hash` via `FormulaCanonicalizationService.computeFilterSetHash` (so the filters are *bound* into the package signature) but are never *executed*.
4. `FixtureContext.filterClauses` is already loaded by `loadFixtureContext` in `fixture-structural-check.service.ts`; the M9 structural check verifies that fixture rows carry the required `filter_inputs` per BC UID. The data plumbing exists; only the application step is missing.

This is a verifier defect, not a contract defect. Renaming or amending `mcf-verifier-v1` in place would silently retroactively change the verdict semantics of every preserved v1 evidence row — a Foundation Invariant III violation. The disciplined path is an ADR-governed algorithm bump.

## Decision

Bump the self-verifier algorithm identifier from `mcf-verifier-v1` to `mcf-verifier-v2`. v2 extends v1 by applying declared `mcf.metric_filter_clause` rows during fixture playback before the formula AST evaluates.

### v2 semantics

1. **`where` clauses applied as a pre-filter step** on each Section A variable's rowset, before any AST node sees the rowset. A row is kept only if it satisfies every declared `where` clause (conjunctive semantics, matching SQL).

2. **Clause matching is by `bound_business_concept_id`**, not by display label or column name. For each `where` clause with `bound_business_concept_id = bcId` and `operator_code = op` and `literal_value_json = lit`:
   - Read `filter_input = row.filter_inputs[bcId]` from the fixture row.
   - Apply the operator (e.g., `eq`: `filter_input === lit`).
   - Drop the row if the predicate is false.

3. **Missing filter input behavior:** if `row.filter_inputs[bcId]` is undefined for a non-null-shaped operator, the row is DROPPED (conservative; M9 structural check enforces presence at fixture write time, so this is defense-in-depth).

4. **Null literal behavior:**
   - `is_null` operator: keep row iff input is null/undefined.
   - `is_not_null` operator: keep row iff input is not null/undefined.
   - Other operators against a null literal value: rejected by the DDL CHECK constraint `mfc_operator_literal_chk`; engine throws if encountered.

5. **Supported operators in v2 first pass: all 11** admitted by the DDL CHECK (`eq, ne, lt, lte, gt, gte, in, not_in, is_null, is_not_null, between`). Implementation cost is uniform across the operator set — supporting one operator is structurally identical to supporting all, so the v2 first pass covers all declared operators. Rejecting unsupported operators at engine time risks future-author surprise; the DDL CHECK already constrains the operator vocabulary.

6. **`having` clause: deferred.** No active or stuck metric uses `having` (substrate count = 0 as of 2026-06-23). If a `having` clause is present in `filterClauses`, the v2 engine throws `NotImplementedError` (loud failure rather than silent skip). A future v3 — or v2.1 ADR amendment — adds post-aggregate filtering when an active metric requires it.

7. **`pre_filter` clause: treated identically to `where` in v2.** Per the operator's decision and the DDL semantics (`pre_filter` applies before any aggregate, same as `where` at the rowset-input layer in the current AST grammar), the engine applies both `pre_filter` and `where` in a single pre-filter step. If a future grammar distinguishes them (e.g., adds a phase that runs *between* aggregates), the distinction can be reintroduced.

## Compatibility

1. **Old `mcf-verifier-v1` rows in `mcf.metric_self_verification_result` remain immutable historical evidence.** Foundation Invariant III preserves them as written. v1 rows continue to be valid records of "what the verifier did at the time it ran, under v1 semantics that did not apply filters."

2. **New verifier runs under v2 write rows with `verifier_algorithm_version = 'mcf-verifier-v2'` alongside any pre-existing v1 rows for the same fixture.** The unique constraint `uq_msvr_fixture_version_pkg_hash` keys on `(fixture_uid, verifier_algorithm_version, bound_package_signature_hash_at_run)`, so v1 + v2 rows coexist for the same `(fixture, package_signature)` triple.

3. **Publication Review (PE-MC-10) reads the latest self-verification result for the MCV.** When a v2 row exists for an MCV's fixture, PE-MC-10 picks the latest by `executed_at` regardless of version — meaning a fresh v2 row supersedes the v1 row for verdict purposes. The v1 row is not deleted; it remains as audit evidence of the prior verifier behavior.

4. **Activation evidence (M14, `mcf-m14-v2`) is unaffected by the verifier bump.** Activation evidence is stamped on `metric_publication_eligibility_result` rows under its own version; the verifier version applies to the upstream self-verification path.

5. **No DDL changes are required.** No schema migration. No new tables. No new columns. The package_signature_hash composition is unchanged (it already includes `filter_set_hash`). The only persisted change is new rows with a new `verifier_algorithm_version` value.

### Cleared Invoice Amount handling

CIA's MCV `57ea07d0…` is the only active metric with filter clauses. Two scenarios under v2:

- **Scenario A (likely):** CIA's fixture rows happen to satisfy both filter clauses (e.g., `clearing_date is not null` for all rows AND `status eq <expected>` for all rows). v2 produces an identical actual value to v1; verdict stays PASS; no operator-facing change.
- **Scenario B:** at least one CIA fixture row fails a filter; v2 evaluates the formula against a smaller filtered rowset; actual value differs from v1; verdict may flip to FAIL.

Foundation IV preserves CIA's active status regardless of scenario (activation is irrevocable). If Scenario B occurs:
- The v2 FAIL row joins the v1 PASS row in `mcf.metric_self_verification_result`.
- A subsequent Publication Review re-evaluation under `mcf-m13-v6` would flip PE-MC-10 to REJECT (PE-MC-10 reads the latest verifier_result).
- CIA's activation status does NOT roll back, but a documented inconsistency surfaces.

**Required mitigation:** before merging v2 production code, run a read-only v2 dry-run against CIA's fixture (test-mode invocation that does not write to substrate, or a separate scratch row with explicit cleanup) to forecast which scenario applies. If Scenario B, prepare a CIA fixture-amendment plan (re-author the fixture under correct filter semantics, append the corrected fixture via the M12.5 Append Fixture flow). Defer Production Review re-evaluation of CIA until the dry-run result is known.

## Consequences

### Positive
- Stuck MCV `Paid Customer Invoice Count v2` becomes recoverable via routine R1 Re-evaluate.
- Filter-clause-bearing metrics can now be authored with confidence that their declared filters will be enforced at verification time.
- Closes the silent semantic gap between `filter_set_hash` participation and runtime application.

### Negative / risk
- Cleared Invoice Amount may surface a v2 vs v1 verdict difference (Scenario B), requiring fixture amendment and an operator-documented inconsistency window until the corrected fixture verifies.
- `having` clause support remains deferred; if a future active metric requires `having`, an additional v3 bump (or v2.1 amendment) is needed.

### Operator-gated open items
1. **`having` clause:** defer to v3 — DECISION recorded.
2. **`pre_filter` clause:** treat as `where` — DECISION recorded (zero active metrics currently use it).
3. **CIA dry-run before merge:** REQUIRED — operator confirms before code-write begins.
4. **All-active-MC v2 backfill dry-run** (run v2 against ARPI, Billing Volume, IPCT fixtures to confirm no surprise change): recommended but not blocking, since all three have zero filter clauses and v2 is functionally identical to v1 for filter-free metrics. Operator may opt for focused regression instead.

## §4 Amendment — 2026-06-23: Publication Review evidence-state fingerprint

R3 Step 3 surfaced a follow-up gap. The v1 → v2 verifier-engine bump correctly writes v2 evidence rows alongside immutable v1 rows (proven end-to-end via the live fixture-append against PCIC v2's MCV `db3e1bd0-051b-401f-8278-e3cd84e622a7`), but the M13 Publication Review evaluator's idempotency cache (`findExistingEvaluation`) keys on a 5-tuple that omits the latest fixture/verifier_result state. After fixture-append, the cache returns prior PE rows that still reference the old v1 verifier_result, blocking PE-MC-10 from reflecting the new v2 PASS evidence.

**Extension.** Add an `evidence_state_fingerprint` axis to the cache tuple, stamped on every PE row's `evidence_json`, computed from `(latest_fixture_uid, latest_verifier_result_uid, verifier_algorithm_version, verdict_code, bound_package_signature_hash_at_run, stale_fixture_flag)` under the version tag `mcf-m13-evidence-state-fingerprint-v1`. Mirrors the existing `chain_state_fingerprint` pattern that PR-2 established. Widen the substrate partial unique index (`idx_mcf_mper_mcv_check_eval_pkg_chain` → `idx_mcf_mper_mcv_check_eval_pkg_chain_evidence`) to include the new axis as a 6th slot. Update the `readStalePredecessorPeRowUids` OR-chain to surface predecessors that drifted on the evidence axis.

**Independence.** The two fingerprints are kept separate (not folded). `chain_state_fingerprint` covers registry-side declarations (CC field_selection + OC field_mappings + OC source_references + BCF binding lifecycle states); `evidence_state_fingerprint` covers substrate-side evidence in `mcf.metric_self_verification_*`. They have independent change cadences and independent observability requirements.

**Compatibility.** Old PE rows (pre-amendment) carry NULL `evidence_state_fingerprint` in their evidence_json; `COALESCE(..., '')` puts them in a distinct uniqueness slot from post-amendment rows. Foundation Invariant V preserved: no row deletions, no row updates.

**Implementation surface.** ~50 production LoC in `bc-core/src/registry/mcf/metric-publication-eligibility-evaluator.service.ts` (one new `computeEvidenceStateFingerprint` method + threading through `evaluateInTx`, `findExistingEvaluation`, `runFirstEvaluation`, `composePeRowInputs`, `readStalePredecessorPeRowUids`) + one DDL migration pair (`docker/redesign/15-mcf-m13-pe-mc-evidence-fingerprint-index.sql` + rollback) + spec test additions.

**Implementation status.** Drafted in PR (status: draft) — migration NOT applied to live DB yet. ADR status remains `decided`. Flip to `implemented` only after the PR merges and live recovery for PCIC v2 (re-running Publication Review under the widened cache → PE-MC-10 PASS) completes.

**Design document.** `bc-docs-v3/docs/implementation/mms-publication-review-evidence-fingerprint-design-2026-06-23.md` carries the full rationale, alternatives considered, test plan, and rollback semantics.
