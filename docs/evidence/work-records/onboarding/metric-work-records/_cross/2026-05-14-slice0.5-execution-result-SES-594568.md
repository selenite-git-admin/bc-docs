## 2. What changed

### 2.1 The bug being fixed

`McOnboardingService.preview()` was reading `resolved_schema` and
`cc_field_mapping` literally, ignoring the D365 (DEC-a8e8fc) rule that
`fiscal_period` and `fiscal_year` are stamped on the CO payload at
canonical resolution time when the CC declares `posting_date_field`.

Two other consumers of the same rule already honor it:

| Consumer | Honors `derivedFactColumnsForCc`? |
|---|---|
| Schema-provisioner (DDL generator) | yes |
| Chain-status (grain-completeness check) | yes |
| **MC onboarding preview (validateGrainMappings + validateGrainAgainstCcSchema)** | **NO — fixed by this slice** |

The shared helper `derivedFactColumnsForCc(body)` already exists in
`bc-core/src/registry/../schema-provisioner/derived-cc-columns.ts`. Its
docstring already names the two consumers and the consequence of missing
the rule. Slice 0.5 adds the third consumer.

### 2.2 Code changes (mc-onboarding.service.ts)

- **Import** `derivedFactColumnsForCc` from `../schema-provisioner/derived-cc-columns`.
- **`validateGrainMappings`** — before the `cfRepo.hasCcMapping` lookup,
  consult `collectDerivedColumnsForCcs(ccNames)`. If the grain key is
  in that set, pass with detail `"Grain '<key>' derived from CC
  posting_date_field (D365 / DEC-a8e8fc)"`. Otherwise fall through to
  the existing mapping check (unchanged).
- **`collectCcSchemaFields`** — after collecting
  `resolved_schema.properties` keys, union
  `derivedFactColumnsForCc(body)` into the returned set. One inline
  call to the shared helper; rule not duplicated.
- **`collectDerivedColumnsForCcs`** (new private helper, ~12 lines) —
  mirrors the shape of `collectCcSchemaFields` but returns only the
  D365-derived columns.

### 2.3 Tests (mc-onboarding.service.spec.ts)

New `describe('slice (0.5) grain D365 gate consistency')` block, three
cases (C1/C2/C3 from the slice 0.5 plan):

| # | Case | Assertion |

| C1 | fiscal_period grain passes when bound CC declares `posting_date_field` | `valid: true`; `grain_cf_mapped` passes with detail matching `/D365/`; `grain_in_cc_schema` passes |
| C2 | fiscal_period grain fails when bound CC does NOT declare `posting_date_field` | `valid: false`; both gates fail; details name `'fiscal_period'` |
| C3 | explicitly-mapped grain (company_code) still passes via cc_field_mapping (no regression) | `grain_cf_mapped` passes via the existing path (detail matches `/mapped in CC/`, not `/D365/`) |
