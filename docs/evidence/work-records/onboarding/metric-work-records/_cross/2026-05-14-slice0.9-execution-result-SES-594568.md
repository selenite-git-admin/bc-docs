## 2. The bug surfaced

Slice (1) attempt-3 (post-slice-0.8) hit HTTP 409 from `McOnboardingService.create`. Cause: `findContractByName` correctly detected the existing draft MC from prior failed attempts and refused to create a duplicate parent — clean conflict behavior, but no recovery path. The wrapper had no resume-aware branch.

## 3. What changed (seed-promotion.service.ts)

1. **Inject `ContractService`** alongside existing `McOnboardingService` and `MetricDefinitionRepository` for direct `activateVersion`.
2. **`PromoteSeedResult.resumedFromDraft: boolean`** added (response/log only).
3. **`promoteSeed` Phase 4** runs `mcOnboarding.preview(dto)` to build the canonical "expected envelope," then calls the new `createOrResumeMc`.
4. **`createOrResumeMc`** branches:
   - No existing non-archived MC → fresh create path (unchanged).
   - Existing draft v1.0.0 with body matching preview → activate via `contractService.activateVersion`.
   - Existing draft with mismatched body → `ConflictException`.
   - Existing v1.0.0 in any other state (active/review/approved/superseded) → `ConflictException` (idempotent active-retry deferred per scope).
   - Existing MC without v1.0.0 → `ConflictException`.
5. **`lookupExistingMcForSeed`** uses raw SQL (`db.execute(sql\`...\`)`) keyed on `(metric_definition_id, metric_contract_name, archived_at IS NULL)` — explicitly avoids `ContractRepository.findContractByName` because that method does NOT filter archived rows.
6. **`payloadsMatch` + `canonicalize`** helpers: deep canonical compare on `contract_json.body` plus `header.{description, tags, kind, name}`. Ignores dynamic `header.contract_id` and `header.created_at`.
