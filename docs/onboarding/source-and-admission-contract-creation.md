# Source and Admission Contract Creation

## Scope

This chapter records the governed sequence by which a Source Contract (SC) and its paired Admission Contract (AC) are created against a registered source catalog object. It names the prerequisites the procedure assumes, the artifacts the procedure produces (one SC plus one AC, both versioned), the body keys an SC declares (per the SC body-purity discipline, DEC-242d60), the body keys an AC declares (per the AC master-shape discipline, DEC-ca4c1e), the quality gates the procedure enforces, the canonical SC and AC naming convention, and the bulk SC-plus-AC generation surface for system-scale onboarding. It records the boundary between SC-and-AC creation and the contract families that consume them (Observation Contract Creation, Reader Creation). It records the as-built drift between the procedure and the SC and AC state captured by the readiness baseline.

This chapter does not redefine the contract grammar that introduces Source Contract and Admission Contract as the entry-boundary contract families (The Contract Grammar), the admission and observation runtime acts (Admission and Observation), the source catalog the contracts reference (Sources and the Catalog), or the chain integrity rules that govern contract composition (Quality Gates and Chain Integrity).

**Governing source.** outline.md §4.6; The Contract Grammar.

## What the Procedure Produces

A complete SC-and-AC creation produces:

| Artifact | Persistent store | Created by |
|---|---|---|
| Source Contract identity | `contract.source_contract` | Step 1 |
| Source Contract version 1.0.0 (active) | `contract.contract_version` (with `category: source`) | Step 2-3 |
| Admission Contract identity | `contract.admission_contract` | Step 4 |
| Admission Contract version 1.0.0 (active) | `contract.contract_version` (with `category: admission`) | Step 5-6 |
| Activation transitions | `contract.contract_version_governance` | Step 3 and Step 6 |

The two artifacts are paired: every AC depends on an active SC. A standalone AC creation (without a paired SC creation) is admissible only when an active SC for the same source object already exists.

**Governing source.** The Contract Grammar; Admission and Observation.

## Naming Convention

The chapter records the canonical naming pattern that the registration surface enforces (D289 in the v2 SOP shorthand).

| Artifact | Name pattern | Display pattern |
|---|---|---|
| Source Contract | `sc__{system}__{table}` | `SC {SYSTEM} {TABLE}` |
| Admission Contract | `ac__{system}__{table}` | `AC {SYSTEM} {TABLE}` |

Both patterns use double underscores between the prefix and the system slug, and between the system slug and the table slug. The display name uses spaces and uppercase. The patterns are auto-generated by the wizard and are editable; an actor may override the auto-generated name only when an existing SC or AC name collides.

**Governing source.** The Contract Grammar.

## Prerequisites

| Precondition | Why it is required |
|---|---|
| Cognito authenticated session for a platform actor | Contract mutations are `@PlatformOnly()` JWT-guarded |
| Source catalog populated (at least one Source System with Source Tables and Source Fields) | The SC binds a `source_object_id`; the AC references the SC and inherits its source object |
| Meta-schemas seeded (`barecount/source/v1`, `barecount/admission/v1`) | The chapter validates the contract envelope against the registered meta-schema; absence of the meta-schema rejects the create |
| The bc-core contract surface is reachable | The procedure writes via `POST /api/contracts` and `POST /api/contracts/{id}/versions` |

A precondition that fails is not bypassed.

**Governing source.** Sources and the Catalog; The Contract Grammar.

## Step 1: Create the Source Contract Identity

The actor selects a Source Catalog object (drilling Provider to System to Version to Module to Table) and submits the SC identity:

```
POST /api/contracts
{
  "name": "sc__ecc__bkpf",
  "displayName": "SC ECC BKPF",
  "category": "source",
  "sourceObjectId": "<object-uuid>",
  "description": "Source contract for BKPF",
  "tags": ["BKPF", "ecc"]
}
```

The service validates that the source object exists, is not archived, and carries `catalogStatus: approved`. A draft or archived source object is not admissible at SC creation; the chapter does not admit an SC against a source object that has not passed Source Registration.

**Governing source.** Sources and the Catalog.

## Step 2: Create the Source Contract Version

The SC version 1.0.0 carries the contract envelope (`barecount/source/v1`) with a header (16 required keys per the structural-completeness discipline; D253 in the v2 SOP shorthand) and a body. The body declares the source-side structure only:

| Body key | Form |
|---|---|
| `table_code` | The source table code (e.g., `BKPF`) |
| `label` | The display label for the contract (e.g., `SC ECC BKPF`) |
| `source_system_version_id` | The Source Version identifier from the catalog |
| `module_code` | The Source Module code (e.g., `fi`) |
| `fields[]` | Array of field declarations with `field_code`, `label`, `type`, `nullable`, `z_extension` |
| `primary_key[]` | Array of field codes that compose the source table's primary key |

The SC body is pure declaration per DEC-242d60. It declares structure only; it does not declare runtime behavior, validation rules, or drift detection logic. Those concerns belong to the Admission Contract.

```
POST /api/contracts/{contractId}/versions
{
  "$contract": "barecount/source/v1",
  "version": "1.0.0",
  "header": { ... },
  "body": {
    "table_code": "BKPF",
    "label": "SC ECC BKPF",
    "source_system_version_id": "<version-uuid>",
    "module_code": "fi",
    "fields": [
      { "field_code": "BUKRS", "label": "BUKRS", "type": "string", "nullable": true, "z_extension": false }
    ],
    "primary_key": ["MANDT", "BUKRS", "BELNR", "GJAHR"]
  }
}
```

**Governing source.** The Contract Grammar.

## Step 3: Activate the Source Contract Version

```
POST /api/contracts/{contractId}/versions/1.0.0/activate
```

Activation transitions `governance_state` from `draft` to `active` and writes a row to `contract.contract_version_governance`. The SC is then admissible as a reference for an Admission Contract or for a downstream Observation Contract.

**Governing source.** Quality Gates and Chain Integrity.

## Step 4: Create the Admission Contract Identity

The actor selects the active SC the AC will govern and submits the AC identity:

```
POST /api/contracts
{
  "name": "ac__ecc__bkpf",
  "displayName": "AC ECC BKPF",
  "category": "admission",
  "sourceObjectId": "<same-object-uuid-as-SC>",
  "description": "Admission contract for BKPF",
  "tags": ["BKPF", "ecc"]
}
```

The service validates that an active SC exists for the same `sourceObjectId`. An AC identity created against a draft SC, an archived SC, or a source object with no SC is rejected.

**Governing source.** The Contract Grammar.

## Step 5: Create the Admission Contract Version

The AC version 1.0.0 carries the `barecount/admission/v1` envelope with a body that declares admission rules:

| Body key | Form |
|---|---|
| `source_contract_version_id` | The active SC version this AC governs |
| `rulebook_version` | The DQC rulebook version this AC honors (baseline `v1`) |
| `structural_validation` | Object with `expected_field_count`, `unknown_field_policy` (`warn` or `quarantine`), `missing_field_policy` (`warn` or `quarantine`) |
| `field_rules[]` | Array of field-level DQC rules, each with `rule_id`, `category`, `type`, `severity`, `action`, `description` |
| `record_rules[]` | Array of record-level DQC rules with the same structure |
| `outcome_thresholds` | Object with `warn_pct_limit`, `quarantine_pct_limit`, `action_on_exceed` (e.g., `hold_batch`) |
| `validation_policy` | One of `warn`, `quarantine`, `block` |
| `max_observation_age` | ISO 8601 duration (e.g., `P30D`) |

The AC body is the bronze-layer DQC declaration; the master shape is locked at DEC-ca4c1e. Rules are scoped with three prefixes (D247 in the v2 SOP shorthand for the scope-prefix rule):

| Prefix | Scope | Authored by |
|---|---|---|
| `GL-*` | Global; applies to all entities | Platform actor |
| `PL-{table}-*` | Platform; applies per entity | Platform actor |
| `TN-{table}-*` | Tenant; applies per entity per tenant | Tenant via `contract_binding` |

```
POST /api/contracts/{contractId}/versions
{
  "$contract": "barecount/admission/v1",
  "version": "1.0.0",
  "header": { ... },
  "body": {
    "source_contract_version_id": "<sc-uuid>",
    "rulebook_version": "v1",
    "structural_validation": {
      "expected_field_count": 166,
      "unknown_field_policy": "warn",
      "missing_field_policy": "quarantine"
    },
    "field_rules": [
      { "rule_id": "GL-001", "category": "completeness", "type": "not_null", "severity": "warning", "action": "warn", "description": "All fields not null" }
    ],
    "record_rules": [],
    "outcome_thresholds": { "warn_pct_limit": 10, "quarantine_pct_limit": 5, "action_on_exceed": "hold_batch" },
    "validation_policy": "warn",
    "max_observation_age": "P30D"
  }
}
```

**Governing source.** The Contract Grammar; Admission and Observation.

## Step 6: Activate the Admission Contract Version

```
POST /api/contracts/{contractId}/versions/1.0.0/activate
```

The AC version transitions to `active`. The SC-and-AC pair becomes operative.

**Governing source.** Quality Gates and Chain Integrity.

## Bulk Generation of Coupled SC-and-AC Pairs

For system-scale onboarding (admitting many source tables in one operation), the platform exposes a bulk generation surface that creates SC-and-AC pairs in coupled batches. The surface lives at `POST /api/contracts/generate-source-chain`:

```
POST /api/contracts/generate-source-chain
{
  "systemName": "sap_ecc",
  "limit": 50,
  "dryRun": true
}
```

A dry run returns the list of source objects eligible for SC-and-AC creation along with the planned names and field counts; no rows are written. A live run creates the SC-and-AC pairs in sequence, skips an object whose SC already exists (catching the conflict silently), continues past per-pair AC errors without rolling back the SC, and reports created counts, skipped counts, and elapsed time.

The bulk surface produces the same artifacts as the single-pair procedure. The naming follows the SC and AC naming convention (D289 in the v2 SOP shorthand) recorded earlier in this chapter. The SC envelope is D253-compliant `barecount/source/v1` with `body.fields[].field_code`. The AC envelope is D253-compliant `barecount/admission/v1` with global DQC rules. Both are activated immediately. The bulk surface honors the same gates the single-pair procedure honors; an SC-and-AC pair that fails any gate is rejected without admitting an invalid pair.

**Governing source.** The Contract Grammar.

## Quality Gates

The procedure enforces six gates at the API surface.

| Gate | Where enforced | What it checks |
|---|---|---|
| Source binding | `POST /api/contracts` (SC and AC) | `source_object_id` references an approved, non-archived catalog object |
| Meta-schema validation | `POST /api/contracts/{id}/versions` | The `contract_json` envelope conforms to the registered `barecount/{family}/v1` meta-schema |
| Structural completeness | `POST /api/contracts/{id}/versions` | All required header keys (16) and body keys (7 for SC, 8 for AC) are present with valid values; partial envelopes are rejected (D253 in the v2 SOP shorthand) |
| SC dependency | `POST /api/contracts` (AC only) | The AC's identity creation requires an active SC for the same source object (D288 in the v2 SOP shorthand for the contract-creation framework) |
| DQC rule validity | `POST /api/contracts/{id}/versions` (AC only) | `field_rules[]` and `record_rules[]` conform to the eight DQC categories; rule identifiers honor the `GL-*`, `PL-*`, `TN-*` scope prefixes (DEC-ca4c1e for AC master shape; D247 for the scope-prefix rule) |
| Governance state machine | `POST /api/contracts/{id}/versions/{version}/activate` | Transitions are `draft` to `review` to `approved` to `active`; `active` to `superseded` on a new active version; backwards transitions are rejected |

A row that lands in any of the contract tables without passing the corresponding gates is in the table but is not the platform's authoritative content. The audit substrate flags it for review.

**Governing source.** The Contract Grammar; Quality Gates and Chain Integrity.

## Drift Inventory

| Drift item | Form |
|---|---|
| Drift detection lives in SC `header`, not body | DEC-242d60 moved drift detection out of the SC body to keep the body pure. The drift configuration carries on the SC header; older SCs created before the move may carry drift fields in the body and are flagged for migration |
| Tenant DQC rules under `TN-*` prefix not yet routed via `contract_binding` | The DTO admits `TN-*` rule identifiers in the AC body, but the runtime path that lifts tenant rules out of `contract_binding.extensions_json` and merges them into AC enforcement is partial; tenant DQC enrichment is a queued surface in Tenancy and Binding |
| Bulk surface skips silently on conflict | The bulk generation surface catches `ConflictException` (existing SC with the same name) and continues. The skipped SC does not appear in the live-run summary; it is logged at the service layer. An actor reviewing a bulk run reads the log to recover the skipped list |
| AC errors do not roll back the paired SC | A bulk pair where the SC creates but the AC fails leaves an orphan SC in the catalog. The orphan SC is admissible to a downstream OC reference, but the missing AC means admission rules are not enforced. The orphan is flagged for repair |
| Activate-immediately is the default | The wizard and the bulk surface activate the new version immediately by default. The chapter records this as a default; the actor may opt to leave the version in `draft` and run a separate activation step for manual review |

**Governing source.** The Contract Grammar; Tenancy and Binding; Audit and Activity Logging.

## Governing Decisions

| Decision | Scope in this chapter |
|---|---|
| DEC-242d60 | Establishes SC body purity; the SC declares structure only and drift detection moves to the SC header |
| DEC-ca4c1e | Establishes the AC master shape with DQC integrated, structural validation, and the GL/PL/TN scope-prefix rule |

Several adjacent governance points (D018 contract families, D246 AC body as DQC bronze layer, D252 AC structural validation, D253 structural completeness, D288 contract creation framework, D289 SC and AC naming convention) are referenced in the v2 SOP shorthand but do not have standalone ADR files in the reviewed source set; their record lives in `legacy-v2/docs/sops/contract-creation-sop.md` and in this chapter. New ADR files may be filed if the policies are restated outside the v2 SOP.

**Governing source.** Decisions: ADR Registry.
